# 12. Make Sum Divisible by P (LC 1590) — Medium
🔗 [Open on LeetCode](https://leetcode.com/problems/make-sum-divisible-by-p/)

**Pattern:** Prefix Sum & Hashing — **prefix sum mod p + `remainder → latest index` map** (the *shortest-subarray* fork of the mod-K family: same machinery as [[5-subarray-sums-divisible-by-k]] / [[6-continuous-subarray-sum]], but we keep the **most recent** index per remainder instead of a frequency or the earliest index)
**Trigger:** "**remove the smallest subarray** so the **remaining** sum is divisible by `p`" → the removed chunk must itself have `sum % p == target` where `target = totalSum % p`. Sweep prefixes; for each `j` look up the **complement remainder** and minimize `j − latestIndex[need]`.

## Problem (1-liner)
Given positive `nums` and integer `p`, remove the **smallest** contiguous subarray (possibly empty, but **not** the whole array) so the **remaining** sum is divisible by `p`. Return that subarray's length, or `-1` if impossible.
**Example:** `nums=[3,1,4,2], p=6 → 1` (remove `[4]`, rest = 6) · `nums=[6,3,5,2], p=9 → 2` (remove `[5,2]`) · `nums=[1,2,3], p=3 → 0` (already divisible)
**Constraints:** `1 ≤ n ≤ 10⁵`, `1 ≤ nums[i] ≤ 10⁹`, `1 ≤ p ≤ 10⁹` — values are large ⇒ **`long long` running sum**; `p` can be huge ⇒ `unordered_map`, not a `vector`.

## Idea (1-liner)
Let `target = totalSum % p`. If `target == 0` the array is already divisible → answer `0`. Otherwise we must delete the **shortest** subarray whose own sum `≡ target (mod p)`, since peeling off that excess leaves the rest at remainder `0`. Sweep the prefix remainder `curr`; the subarray `(i, j]` works iff `prefix[i] % p == (curr − target + p) % p`. Keep `remainder → latest index` so `j − latestIndex[need]` is the **shortest** gap. **O(n) time, O(min(n, p)) space.**

## Approach
1. Compute `target = totalSum % p` (fold with `% p` as you go to avoid overflow). If `target == 0`, return `0` — nothing to remove.
2. Sweep `j` keeping the running prefix remainder `curr`. We want a removed subarray `(i, j]` with `(prefix[j] − prefix[i]) % p == target`, which rearranges to:
   `prefix[i] % p == (curr − target + p) % p`  — call this `need`.
   The `+ p` before the final `% p` keeps `need` non-negative.
3. Maintain `lastIndex: remainder → most recent index` and **overwrite on every step**. Because we keep the *latest* matching index, `j − lastIndex[need]` is automatically the *smallest* possible removal for that remainder — this is the whole "how do I get the **minimum** length?" answer.
4. Seed `lastIndex[0] = -1` (the empty prefix), so a removal starting at index 0 is reachable.
5. Track `best`; start it at `n` as the "impossible-so-far" sentinel. If it never improves below `n`, the only fix would be deleting the whole array, which is banned → return `-1`.

## Code (clean canonical)
```cpp
class Solution {
public:
    int minSubarray(vector<int>& nums, int p) {
        long long total = 0;
        for (int x : nums) total = (total + x) % p;     // fold to avoid overflow
        int target = (int)total;                         // == totalSum % p
        if (target == 0) return 0;                        // already divisible

        int n = nums.size();
        unordered_map<int,int> lastIndex{{0, -1}};        // remainder -> latest index; empty-prefix sentinel
        long long curr = 0;
        int best = n;                                     // impossible-so-far sentinel
        for (int j = 0; j < n; ++j) {
            curr = (curr + nums[j]) % p;
            int need = (int)((curr - target + p) % p);    // complement remainder we must have seen
            auto it = lastIndex.find(need);
            if (it != lastIndex.end())
                best = min(best, j - it->second);         // latest index -> shortest removal
            lastIndex[(int)curr] = j;                     // keep MOST RECENT index for this remainder
        }
        return best == n ? -1 : best;
    }
};
```
**TC:** O(n).  **SC:** O(min(n, p)) — at most `n` distinct remainders stored.

## The fork that defines this problem — index *recency* matters
All four mod-family problems sweep prefix remainders; what they store per remainder is the only difference:

| Problem | What we store per remainder | Why | Goal |
|---|---|---|---|
| [[5-subarray-sums-divisible-by-k]] (974) | **frequency** (count) | every earlier match forms a valid pair | **count** subarrays |
| [[6-continuous-subarray-sum]] (523) | **earliest** index, insert only on first sight | maximizes the gap ⇒ length `≥ 2` check | **exists** (longest-leaning) |
| [[7-contiguous-array]] (525) | **earliest** index | maximizes `i − firstIdx` | **longest** subarray |
| **1590 (this one)** | **latest** index, **overwrite every step** | minimizes `j − lastIndex` | **shortest** subarray |

Mnemonic: **earliest index → longest; latest index → shortest; frequency → count.** Overwriting (1590) vs. insert-only-on-first-sight (523/525) is the single line that flips longest into shortest — get it backwards and you'll report a longer removal than needed.

## Why your "wait for a repeated remainder" instinct stalled
That instinct solves a *different* problem — finding a subarray that is *itself* divisible by `p` (target remainder `0`, the [[5-subarray-sums-divisible-by-k]] mechanic). Here the **removed** part must equal `target`, not `0`, so you don't wait for `curr` to repeat — you look for the **complement** `need = (curr − target + p) % p`. And "minimum length for that remainder" isn't a search at all: it falls out for free from storing the *most recent* index per remainder.

## Other Approaches
- **Brute force O(n²)** — try every subarray to remove, check the remaining sum. `n = 10⁵` ⇒ 10¹⁰ ops, TLE.
- **Two-pointer / sliding window** — tempting because we want a "shortest" window, but the predicate is modular (`sum % p == target`), not monotonic in window length, so shrinking-while-too-big has no meaning. The remainder map is the right tool, same gate as in [[5-subarray-sums-divisible-by-k]].

## Pitfalls
- **Overflow.** `n · max(nums[i]) = 10⁵ · 10⁹ = 10¹⁴` blows past `int`. Use `long long` for the total (or fold with `% p` while summing, as above).
- **Storing earliest instead of latest.** Earliest index maximizes the gap → you'd return a needlessly *long* removal. For **shortest**, overwrite with the newest index every step.
- **Forgetting `lastIndex[0] = -1`.** Without the empty-prefix sentinel, removals that start at index 0 are unreachable and you can miss the true answer (or wrongly return `-1`).
- **Negative `need`.** `curr - target` can be negative; the `+ p` before `% p` normalizes it into `[0, p−1]`. Same C++ negative-modulo trap flagged across the family in [[5-subarray-sums-divisible-by-k]].
- **Returning `0` length as "impossible".** `best == n` means no proper subarray worked → `-1` (can't remove the whole array). Don't conflate it with the `target == 0` early return, which legitimately answers `0`.
- **Off-by-one in the gap.** Removing `(i, j]` has length `j − i`; with `lastIndex[0] = -1`, a prefix `nums[0..j]` removal correctly yields `j − (−1) = j + 1`.
