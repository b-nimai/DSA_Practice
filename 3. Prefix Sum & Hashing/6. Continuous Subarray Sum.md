# 6. Continuous Subarray Sum (LC 523) — Medium
🔗 [Open on LeetCode](https://leetcode.com/problems/continuous-subarray-sum/)

**Pattern:** Prefix Sum & Hashing — **prefix sum mod K + map of the *earliest index* per remainder** (the "exists / longest" cousin of the counting twin [[5-subarray-sums-divisible-by-k]]; same "two equal remainders bracket a sum divisible by K" mechanic, but here we store **first index**, not a **frequency**, because the question is existence-with-a-length-floor, not a count)
**Trigger:** "does there **exist** a subarray of length **≥ 2** whose sum is a **multiple of k**" → sweep the prefix sum, and at each index ask **"have I seen this `sum mod k` before, at least 2 indices back?"** Two prefixes sharing a remainder carve out a subarray whose sum is a multiple of `k`; the length floor becomes `i − firstIdx ≥ 2`. Keep **only the first** index for each remainder so the gap can grow.

## Problem (1-liner)
Given `nums` and integer `k`, return `true` if there is a contiguous subarray of **length ≥ 2** whose sum is a multiple of `k` (0 counts as a multiple of any `k`).
**Example:** `nums=[23,2,4,6,7], k=6 → true` (`[2,4]`) · `nums=[23,2,6,4,7], k=6 → true` · `nums=[23,2,6,4,7], k=13 → false` · `nums=[5,0,0,0], k=3 → true` (`[0,0]`)
**Constraints:** `1 ≤ n ≤ 10⁵`, `0 ≤ nums[i] ≤ 10⁹`, `0 ≤ sum ≤ 2³¹−1`, `1 ≤ k ≤ 2³¹−1` — sums can hit `2³¹−1` (fits `int`, but stay alert), and the **length-≥-2** rule is the whole twist.

## Idea (1-liner)
A subarray `nums[i..j]` is a multiple of `k` iff `ps[j+1] ≡ ps[i] (mod k)`. So track each remainder's **earliest** index; when the current remainder reappears with `i − firstIdx ≥ 2`, a valid (length ≥ 2) subarray exists. One pass, hash map keyed by remainder: **O(n) time, O(min(n, k)) space.**

## Approach
1. Maintain a running `sum` and a map `rem → earliest index that produced it`.
2. **Seed `{0: -1}`** — the empty prefix has remainder 0 sitting "before" index 0. This makes a subarray `nums[0..i]` that is itself a multiple of `k` work out as `i − (−1) = i + 1 ≥ 2`, and removes any need for a special `rem == 0` branch.
3. For each index `i`:
   - `sum += nums[i]`, then `rem = sum % k` (inputs are non-negative here, so no `((%+k)%k)` normalization needed — but it's harmless to keep).
   - If `rem` is already in the map: it's a repeat. Check the **length floor** — `i − firstIdx ≥ 2` ⇒ return `true`.
   - Else record `map[rem] = i`. **Insert only on first sight** — never overwrite.
4. Return `false` if the sweep ends with no qualifying repeat.

## The bug this note exists for — overwriting the index
The natural-but-wrong version writes `mpp[rem] = i` **every** iteration. That keeps the *latest* index, so the gap `i − storedIdx` is never more than 1 and the length-≥-2 check can never fire across a long run.

Trace on `nums=[5,0,0,0], k=3` (every prefix sum is 5, so `rem = 2` throughout):

| i | rem | overwrite version stores | gap | result | correct (keep first) |
|---|-----|--------------------------|-----|--------|----------------------|
| 0 | 2 | mpp[2]=0 | — | — | mpp[2]=0 |
| 1 | 2 | →1 | 1−0=1 | skip | gap 1−0=1, skip |
| 2 | 2 | →2 | 2−1=1 | skip | **2−0=2 ✓ true** |
| 3 | 2 | →3 | 3−2=1 | skip | 3−0=3 ✓ |

Overwriting returns `false`; keeping the **first** index returns `true` (the subarray `[0,0]`). **Rule of thumb:** *exists / longest* problems store the **earliest** index; *count* problems store a **frequency**. Mixing them up is this whole family's classic trap.

## Code (clean canonical)
```cpp
class Solution {
public:
    bool checkSubarraySum(vector<int>& nums, int k) {
        unordered_map<int, int> first{{0, -1}};   // remainder -> earliest index; empty prefix at -1
        int sum = 0;
        for (int i = 0; i < (int)nums.size(); ++i) {
            sum += nums[i];
            int rem = sum % k;
            auto it = first.find(rem);
            if (it != first.end()) {
                if (i - it->second >= 2) return true;   // length >= 2 satisfied
            } else {
                first[rem] = i;                          // store ONLY the first occurrence
            }
        }
        return false;
    }
};
```
**TC:** O(n).  **SC:** O(min(n, k)) — at most `k` distinct remainders, at most `n` insertions.

## My Solution (original — failed `[5,0,0,0], k=3`)
```cpp
class Solution {
public:
    bool checkSubarraySum(vector<int>& nums, int k) {
      unordered_map<int, int>mpp;
      int sum = 0;
      for (int right = 0; right < nums.size(); right++) {
        sum += nums[right];
        int rem = sum % k;
        if (rem == 0 && right >= 1) return true;
        auto it = mpp.find(rem);
        if (it != mpp.end() && right - it->second >= 2) return true;
        mpp[rem] = right;          // ❌ overwrites — keeps latest index, gap never grows
      }
      return false;
    }
};
```
Right skeleton, one fatal line. **Refinements rolled into the canonical version:**
- **Insert only on first sight.** `mpp[rem] = right` ran unconditionally and clobbered the earliest index, so on a constant-remainder run the gap stayed at 1 and the `≥ 2` check never fired (the `[5,0,0,0]` failure). Guard it: only record when the remainder is new.
- **Seed `{0, -1}`** to delete the `if (rem == 0 && right >= 1)` special case — the sentinel makes prefix-from-0 subarrays fall out of the same `i − firstIdx ≥ 2` test.

## Family map — where this sits
| Problem | "Bucket key" per prefix | Sentinel | Decision rule |
|---|---|---|---|
| [[4-subarray-sum-equals-k]] (LC 560) | `sum` value | `cnt[0] = 1` (freq) | `count += cnt[sum − k]` |
| [[5-subarray-sums-divisible-by-k]] (LC 974) | `sum mod k` | `freq[0] = 1` (freq) | `count += freq[rem]` |
| **523 (this one)** | `sum mod k` | `first[0] = -1` (index) | exists iff `i − first[rem] ≥ 2` |
| [[contiguous-array-525]] (LC 525) | running `+1/−1` balance | `seen[0] = -1` (index) | `len = i − seen[balance]` (longest) |
| [[1248-nice-subarrays]] (LC 1248) | running odd-count | `cnt[0] = 1` (freq) | `count += cnt[odds − k]` |

Shared template: **pick a per-prefix bucket key, seed the empty-prefix sentinel, lookup-then-record.** The fork is *what you store*: **frequency** for "count", **earliest index** for "exists / longest".

## Other Approaches
- **Brute force O(n²)** — every `(i, j)` with `j ≥ i+1`, check `(sum) % k == 0`. At `n = 10⁵` that's 10¹⁰ ops — TLE.
- **Prefix-sum array + O(n²) pair scan** — same wall; the remainder map is what collapses the inner loop to O(1).
- **No two-pointer.** There's no monotone "shrink while > target" predicate — the condition is modular, not a threshold — so windows don't apply (same gate as [[5-subarray-sums-divisible-by-k]]).

## Pitfalls
- **Overwriting the stored index.** The headline bug: for *exists / longest*, store the **first** index per remainder so the gap can grow to `≥ 2`. Overwriting keeps the latest and silently fails on constant-remainder runs like `[5,0,0,0], k=3`.
- **Off-by-one on the length floor.** The rule is `i − firstIdx ≥ 2` (gap ≥ 2 ⇒ subarray length ≥ 2). With the `{0,-1}` sentinel a prefix-from-0 needs `i ≥ 1`, which `i − (−1) ≥ 2` enforces automatically.
- **Forgetting the `{0, -1}` sentinel.** Without it you must special-case `rem == 0`; with it the single `i − firstIdx ≥ 2` test covers every case.
- **Confusing this with the counting twin.** 974 stores a **frequency** and sums it; 523 stores an **index** and compares it. Same remainder key, opposite bookkeeping.
- **`k` is large, inputs non-negative.** `k` can be up to `2³¹−1`, so a flat `vector<int>(k)` would blow memory — **use `unordered_map`** here (unlike 974 where `k ≤ 10⁴`). Inputs are `≥ 0`, so `sum % k` is already non-negative; the `((%+k)%k)` fold is unnecessary but harmless.
