# 8. Count Number of Nice Subarrays (LC 1248) — Medium
🔗 [Open on LeetCode](https://leetcode.com/problems/count-number-of-nice-subarrays/)

**Pattern:** Prefix Sum & Hashing — **prefix count-of-odds + frequency map of past prefixes** (the "count exactly K" twin of [[4-subarray-sum-equals-k]] / [[5-subarray-sums-divisible-by-k]]; here the per-prefix "value" is *how many odd numbers seen so far*, and the recode `odd → +1, even → +0` turns "k odd numbers" into "prefix-odd-count difference of k")
**Trigger:** "**count** the number of subarrays containing **exactly k odd numbers**" → maintain a running `odds` count; at each index ask **"how many earlier prefixes had `odds − k` odds?"** Two prefixes whose odd-counts differ by exactly `k` carve out a subarray with exactly `k` odds. Seed the map with `{0: 1}` so a prefix that *itself* has `k` odds (subarray starting at index 0) is counted.

## Problem (1-liner)
Given `nums` and integer `k`, return the **total number** of contiguous subarrays that contain **exactly `k` odd numbers** (a "nice" subarray).
**Example:** `nums=[1,1,2,1,1], k=3 → 2` · `nums=[2,4,6], k=1 → 0` · `nums=[2,2,2,1,2,2,1,2,2,2], k=2 → 16`
**Constraints:** `1 ≤ n ≤ 5·10⁴`, `1 ≤ nums[i] ≤ 10⁵`, `1 ≤ k ≤ n`. Only the **parity** of each element matters — the actual values are irrelevant.

## Idea (1-liner)
Recode each element to its parity (`odd → 1, even → 0`); then "subarray with exactly `k` odds" becomes "subarray whose recoded sum is exactly `k`" — which is **literally [[4-subarray-sum-equals-k]]** on the parity array. Sweep a running odd-count and, at each prefix, add the number of earlier prefixes with `odds − k` odds. One pass, frequency map keyed by odd-count: **O(n) time, O(n) space** (or O(k) — odd-counts only span `0..n`).

## Approach
1. Maintain a running `odds` (number of odd values in the prefix up to current index) and `cnt`, a map **odd-count → how many earlier prefixes produced it**.
2. **Seed `cnt[0] = 1`** — the empty prefix has 0 odds. Without it, a subarray that starts at index 0 and already has exactly `k` odds is missed.
3. For each element `x`:
   - `odds += x & 1` — bump only on odd values.
   - `res += cnt[odds - k]` — every earlier prefix with `odds − k` odds starts a window ending here that holds exactly `k` odds (0 if the key is absent).
   - `cnt[odds]++` — record the current prefix **after** the lookup.
4. The lookup-before-insert ordering is the same "check before insert" lesson from [[1-two-sum]] / [[4-subarray-sum-equals-k]].

## Code (clean canonical)
```cpp
class Solution {
public:
    int numberOfSubarrays(vector<int>& nums, int k) {
        unordered_map<int, int> cnt{{0, 1}};           // odd-count -> #prefixes (empty-prefix sentinel)
        int odds = 0, res = 0;
        for (int x : nums) {
            odds += x & 1;                             // recode: odd -> +1, even -> +0
            res += cnt[odds - k];                      // earlier prefixes that open a k-odd window
            cnt[odds]++;                               // record AFTER lookup
        }
        return res;
    }
};
```
**TC:** O(n).  **SC:** O(n) — or O(k), since reachable odd-counts live in `0..n`.

## My Solution (also correct)
```cpp
class Solution {
public:
    int numberOfSubarrays(vector<int>& nums, int k) {
      unordered_map<int, int>oddCount;
      int odd = 0, res = 0;
      for (int i = 0; i < nums.size(); i++) {
        if (nums[i] % 2) odd++;
        if (odd == k) res++;
        if (oddCount.count(odd - k)) res += oddCount[odd - k];
        oddCount[odd]++;
      }
      return res;
    }
};
```
Correct logic, identical algorithm, passes every LeetCode test. The `if (odd == k) res++;` line exists **only because the map was never seeded with the empty prefix** (`oddCount[0]`) — it manually credits the subarray that starts at index 0 and has exactly `k` odds. **Refinements rolled into the canonical version:**
- **Seed `cnt[0] = 1` instead of special-casing.** Once the empty-prefix sentinel is in the map, `cnt[odds - k]` already counts the index-0 case (`odds == k ⇒ odds - k == 0 ⇒ cnt[0]`), so the standalone `if (odd == k) res++;` disappears. This is the canonical form of the family and reads as "same template as 560" at a glance.
- **`cnt[odds - k]` over `count()` + `[]`.** Your `if (oddCount.count(odd - k)) res += oddCount[odd - k];` does **two** hash lookups; `res += cnt[odds - k]` does one (a missing key default-constructs to 0). The only spurious side effect is inserting a `odds - k` key that's negative, which never matches a real prefix and costs nothing.
- **`x & 1` over `nums[i] % 2`**, and **range-for** over an indexed loop when the index isn't used. Marginal, but clearer.

## Why two-pointer *does* also work here (unlike 974)
Because all elements are positive (parity recode is `0/1`, never negative), the odd-count is **monotonic non-decreasing** along the array — so the sliding-window `atMost` trick applies:
`exactly(k) = atMost(k) − atMost(k − 1)`, where `atMost(K)` counts subarrays with **≤ K** odds via a window that shrinks while it holds more than `K` odds. This is the same "exactly K" decomposition used in [[992-subarrays-with-k-different-integers]] and gives **O(1) extra space**. The hashmap version is still the more direct read of the family; know both.

1248 is 560 with the input recoded to parities. The shared template: **define a per-prefix bucket key, seed the empty-prefix sentinel, lookup-then-record.**

## Other Approaches
- **Brute force O(n²)** — for every start, extend and count odds until it hits `k` (and slightly beyond). At `n = 5·10⁴` that's ~2.5·10⁹ ops — TLE.
- **Sliding window `atMost(k) − atMost(k−1)`** — O(n) time, **O(1) space**, strictly lighter than the map. Best when interview asks to cut space. (See section above.)
- **Gap-counting / "count evens between odd positions"** — record the indices of odd numbers; for the window bounded by the `i`-th and `(i+k)`-th odd, the number of nice subarrays is `(gap of evens on the left + 1) × (gap on the right + 1)`. Same O(n), most elegant but easiest to off-by-one.

## Pitfalls
- **Forgetting the empty-prefix sentinel.** Either seed `cnt[0] = 1` **or** special-case `if (odds == k) res++` — but you need exactly one. Your original took the second route; the canonical version takes the first. Doing neither drops every subarray that starts at index 0.
- **`count()` + `[]` is a double lookup.** `cnt[odds - k]` (or `find`) is one hash probe; the `count` guard is redundant since a missing key reads as 0.
- **Recording before lookup.** `cnt[odds]++` must come **after** `res += cnt[odds - k]`, otherwise when `k == 0` a prefix would pair with itself. (Here `k ≥ 1`, so it's latent, but keep the order — it's the family invariant.)
- **Don't overthink the values.** Only `nums[i] & 1` matters; never sum the actual numbers. No overflow concern — `odds ≤ n ≤ 5·10⁴` fits `int` easily.
- **`% 2` on negatives** would be a trap in problems with negative inputs (C++ `-3 % 2 == -1`), but constraints here guarantee `nums[i] ≥ 1`, so `nums[i] % 2` and `x & 1` agree.
