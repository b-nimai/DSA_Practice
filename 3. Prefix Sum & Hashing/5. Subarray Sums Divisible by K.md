# 5. Subarray Sums Divisible by K (LC 974) — Medium
🔗 [Open on LeetCode](https://leetcode.com/problems/subarray-sums-divisible-by-k/)

**Pattern:** Prefix Sum & Hashing — **prefix sum mod K + frequency map of past remainders** (the modular twin of [[4-subarray-sum-equals-k]]; same "count earlier prefixes matching a target value" mechanic, except the "value" is the prefix's remainder mod K)
**Trigger:** "**count** the number of subarrays whose sum is **divisible by K**" → maintain a running `sum`; at each index ask **"how many earlier prefixes share my current `sum mod K`?"** Two prefixes with the same remainder carve out a subarray whose sum is a multiple of K. Seed the map with `{0: 1}` so a prefix that's *itself* divisible by K is counted.

## Problem (1-liner)
Given `nums` and integer `k`, return the **total number** of contiguous non-empty subarrays whose sum is divisible by `k`.
**Example:** `nums=[4,5,0,-2,-3,1], k=5 → 7` · `nums=[5], k=9 → 0`
**Constraints:** `1 ≤ n ≤ 3·10⁴`, `−10⁴ ≤ nums[i] ≤ 10⁴`, `2 ≤ k ≤ 10⁴` — **negatives are allowed** (rules out two-pointer) and **K is small and bounded** (enables a flat `vector` instead of a hash map).

## Idea (1-liner)
A subarray `nums[i..j]` has sum divisible by `k` iff `ps[j+1] ≡ ps[i] (mod k)`. So while sweeping the prefix sum, count **how many earlier prefixes share the current remainder** — that's exactly the number of valid subarrays ending here. One pass, frequency array of size `k`: **O(n) time, O(k) space.**

## Approach
1. Maintain a running `sum` (prefix sum up to current index) and `freq`, a `vector<int>(k, 0)` mapping **remainder → how many earlier prefixes produced it**.
2. **Seed `freq[0] = 1`** — the empty prefix has remainder 0. Without it, a prefix that is itself divisible by `k` (subarray starting at index 0) is missed.
3. For each element `x`:
   - `sum += x`
   - `rem = ((sum % k) + k) % k` — **normalize** the remainder into `[0, k-1]`. C++'s `%` returns a sign that follows the dividend, so `-7 % 5 == -2`; the `+k) % k` fold gives the canonical non-negative remainder. Without this, `-2` and `3` would land in different buckets even though they represent the same class mod 5.
   - `count += freq[rem]++` — post-increment on `[]` returns the **old** value (0 if first time seen), then bumps the counter. One lookup, no separate `find` / sentinel check needed because the array is dense.
4. The lookup-before-insert ordering is the same "check before insert" lesson from [[1-two-sum]] / [[4-subarray-sum-equals-k]].

## Code (clean canonical)
```cpp
class Solution {
public:
    int subarraysDivByK(vector<int>& nums, int k) {
        vector<int> freq(k, 0);                        // remainder -> count of earlier prefixes
        freq[0] = 1;                                   // empty-prefix sentinel
        int sum = 0, count = 0;
        for (int x : nums) {
            sum += x;
            int rem = ((sum % k) + k) % k;             // normalize negatives into [0, k-1]
            count += freq[rem]++;                      // read old value, then bump
        }
        return count;
    }
};
```
**TC:** O(n).  **SC:** O(k) — bounded and tiny here (`k ≤ 10⁴`), strictly better than the hash-map version.

## My Solution (also correct)
```cpp
class Solution {
public:
    int subarraysDivByK(vector<int>& nums, int k) {
      unordered_map<int, int>remainder;
      remainder[0] = 1;
      int sum = 0, result = 0;
      for (int i = 0; i < nums.size(); i++) {
        sum += nums[i];
        int rem = ((sum % k) + k) % k;
        auto it = remainder.find(rem);
        if (it != remainder.end()) result += it->second;
        remainder[rem]++;
      }
      return result;
    }
};
```
Correct logic, identical algorithm. **Refinements rolled into the canonical version:**
- **`vector<int>(k, 0)` over `unordered_map<int,int>`.** Remainders live in `[0, k-1]` with `k ≤ 10⁴`, so a flat array avoids hashing, collisions, and bucket rehashes. Reach for `vector` over `unordered_map` whenever keys are dense bounded ints — same lesson the 303 note made on `ps`.
- **Two map operations collapse into one.** `freq[rem]++` returns the current value (0 if absent) **and** bumps in a single expression, so `count += freq[rem]++` replaces the `find` + iterator-check + `remainder[rem]++` block. With a `vector` this also drops to one memory write.
- **Range-for** (`for (int x : nums)`) over indexed `for` when the index isn't used.

## Why two-pointer doesn't work here
Same gate as in [[4-subarray-sum-equals-k]]: negatives break monotonicity, so "shrink while sum > target" is meaningless — and there's no "target" anyway, the predicate is "sum mod k == 0". The map/array version is the only general tool for **counting subarrays whose sum has a modular property**, and the constraint `−10⁴ ≤ nums[i]` is exactly what tells you so.

## Other Approaches
- **Brute force O(n²)** — enumerate every `(i, j)` and check `sum % k == 0`. At `n = 3·10⁴` that's 9·10⁸ ops — TLE.
- **Prefix sum + O(n²) pair scan** — same wall; the freq array is what collapses the inner loop to O(1).
- **Hash map version** — works (your original), but strictly worse than `vector<int>(k)` when `k` is small and bounded.

## Pitfalls
- **Forgetting to normalize negatives.** `((sum % k) + k) % k` is mandatory in C++ because `%` returns a remainder with the **sign of the dividend** — `-7 % 5 == -2`, not `3`. Without normalization, equivalent prefixes land in different buckets and the count silently underreports. (Python `%` already normalizes; this trap is C++/Java/Go specific.)
- **Forgetting `freq[0] = 1`.** Without the empty-prefix sentinel, a prefix that is itself divisible by `k` (subarray `nums[0..j]` with sum a multiple of `k`) is dropped. Same lesson as [[4-subarray-sum-equals-k]].
- **Recording before lookup.** `freq[rem]++` *before* using the old value lets a single prefix pair with itself. Always **read old count → then bump**; `count += freq[rem]++` does both atomically and in the right order.
- **Using `unordered_map` reflexively.** It's the right tool when keys are unbounded or sparse (as in LC 560 where `sum` can be anything). Here `rem ∈ [0, k-1]` is dense and bounded ⇒ `vector` wins on both time (no hash) and clarity.
- **Don't try two-pointer.** Negatives break the monotonic-window assumption — same trap flagged in [[4-subarray-sum-equals-k]].
- **Overflow.** Worst case `|sum| = n · max|nums[i]| = 3·10⁴ · 10⁴ = 3·10⁸`, well within `int`. No need for `long long` on this problem's constraints, but stay alert when `n · max|nums|` approaches `2·10⁹`.
