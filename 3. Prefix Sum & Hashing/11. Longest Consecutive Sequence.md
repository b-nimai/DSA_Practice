# 11. Longest Consecutive Sequence (LC 128) — Medium
🔗 [Open on LeetCode](https://leetcode.com/problems/longest-consecutive-sequence/)

**Pattern:** Prefix Sum & Hashing — **hash set as an O(1) membership oracle** (the odd one out in this folder: no prefix sum at all, just the family's core "have I seen value X?" lookup from [[1-two-sum]], used here to walk a number line instead of matching a complement)
**Trigger:** "**longest run of consecutive integers**, unsorted, **must be O(n)**" → sorting is the obvious O(n log n) trap the constraint forbids. Dump everything into a `unordered_set`, then **only start counting from a value that has no left neighbour** (`x-1` absent ⇒ `x` is a sequence start). Walk right `x+1, x+2, …` while present. The "start only at heads" guard is what keeps the nested walk **O(n) total**, not O(n²).

## Problem (1-liner)
Given an unsorted `nums`, return the length of the **longest run of consecutive integers** (values differing by 1), in **O(n)** time.
**Example:** `[100,4,200,1,3,2] → 4` (the run `1,2,3,4`) · `[0,3,7,2,5,8,4,6,0,1] → 9` · `[1,0,1,2] → 3`
**Constraints:** `0 ≤ n ≤ 10⁵`, `−10⁹ ≤ nums[i] ≤ 10⁹` — **n can be 0** (answer 0), values span the full `int` range (no array-indexing by value — must hash), and **duplicates exist** (the set collapses them for free).

## Idea (1-liner)
A run is identified by its **smallest** element. Put all values in a hash set; for each value that is a **start** (its predecessor `x-1` isn't in the set), walk forward counting `x+1, x+2, …`. Each value is visited at most twice — once in the membership scan, once in exactly one forward walk — so the whole thing is **O(n) time, O(n) space**.

## Approach
1. Insert every value into an `unordered_set` — this **dedups** automatically and gives O(1) average membership.
2. Iterate the **set** (not `nums` — avoids re-walking duplicates).
3. For each `x`, ask **"is `x` a sequence start?"** ⇒ `!st.count(x - 1)`. Only starts are allowed to launch a walk; interior values are skipped in O(1).
4. From a start, walk `while (st.count(x + 1)) x++, len++;` and update `longest = max(longest, len)`.
5. **Why it's O(n), not O(n²):** the inner `while` only ever runs from a head, and a head walks each member of its own run exactly once. Across all heads, every value is touched by at most one walk ⇒ total walk work is O(n).

## Code (clean canonical)
```cpp
class Solution {
public:
    int longestConsecutive(vector<int>& nums) {
        unordered_set<int> st(nums.begin(), nums.end());   // dedup + O(1) membership
        st.reserve(nums.size() * 2);                       // fewer rehashes on big inputs

        int longest = 0;
        for (int x : st) {
            if (st.count(x - 1)) continue;                 // not a sequence start, skip
            int cur = x, len = 1;
            while (st.count(cur + 1)) { cur++; len++; }     // walk the run rightward
            longest = max(longest, len);
        }
        return longest;
    }
};
```
**TC:** O(n) — each value is visited at most twice.  **SC:** O(n) for the set.

## My Solution (also correct)
```cpp
class Solution {
public:
    int longestConsecutive(vector<int>& nums) {
      unordered_set<int>st;
      for (int num : nums) st.insert(num);

      int longest = 0;
      for (int num : st) {
        int count = 1;
        if (!st.count(num - 1)) {
          while (st.count(num + 1)) num++, count++;
          longest = max(longest, count);
        }
      }
      return longest;
    }
};
```
Correct and already optimal — same O(n) algorithm, the head-only guard is exactly right. **Refinements rolled into the canonical version (all cosmetic / constant-factor — no complexity change):**
- **`st.reserve(nums.size()*2)`** before inserting — the only change with measurable effect. Pre-sizing avoids repeated rehashing as the set grows toward 10⁵ elements.
- **Range-construct the set** (`unordered_set<int> st(nums.begin(), nums.end())`) instead of the manual insert loop — one line, same work.
- **`if (...) continue;` over wrapping the body in `if (!st.count(num-1))`** — flattens nesting; the "skip non-heads early" reads as the guard it is.
- **Don't mutate the loop variable.** Walking with a separate `cur` instead of `num++` keeps the iterator value untouched (harmless here since it's a by-value copy, but clearer). The comma-operator `num++, count++` also tends to draw reviewer eyes — a braced `{ cur++; len++; }` says the same thing plainly.
- `st.contains(x-1)` (C++20) reads better than `st.count(x-1)` if your compiler allows it.

## Why the head-only guard is the whole trick
Without `!st.count(num - 1)`, every value would launch a forward walk, and a length-`L` run would be re-walked from each of its `L` members ⇒ **O(n·L) = O(n²)** worst case (e.g. `1,2,3,…,n`). Gating the walk to start **only at heads** means each run is walked exactly once. This is the same "do O(1) work to decide whether to do the expensive work" discipline as **check-before-insert** in [[1-two-sum]].

## Why not sort?
Sorting then scanning for consecutive runs is the natural first idea and is **correct** — but it's **O(n log n)**, and the problem explicitly demands **O(n)**. Sorting is the trap the constraint is written to rule out. The hash set buys you the missing log factor by turning "is `x±1` present?" into an O(1) question.

## Other Approaches
- **Sort + linear scan — O(n log n).** Sort, then sweep tracking the current run length, resetting on a gap and skipping duplicates. Correct but violates the O(n) requirement; the right answer to "why not this?" in an interview.
- **Union-Find.** Union each `x` with `x+1` when both present, then take the largest component. Same O(n·α) asymptotics but strictly more code and bookkeeping than the set walk — no reason to reach for it here.
- **Sorted set / `map` for "smooth streaming".** Overkill for a one-shot batch query.

## Pitfalls
- **Forgetting the head-only guard.** Walking from every value (not just sequence starts) silently turns the solution O(n²) — it still *passes small cases*, so this is a correctness-of-complexity trap, not a wrong-answer trap. The `!st.count(x-1)` check is mandatory.
- **Iterating `nums` instead of the set.** Works, but duplicates make you re-evaluate the same start multiple times. Iterate the **set** so each distinct value is considered once.
- **`n == 0`.** Empty input must return `0`; the `longest = 0` initialization handles it — don't assume at least one element.
- **Reaching for sorting reflexively.** The O(n) constraint is the entire point of the problem; sorting answers a different (easier) question.
- **Using a `set` (tree) instead of `unordered_set`.** A `std::set` makes membership O(log n) ⇒ overall O(n log n), quietly defeating the goal. The hash set's **O(1)** lookup is load-bearing.
