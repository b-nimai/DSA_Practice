# 3. Range Sum Query 2D - Immutable (LC 304) — Medium
🔗 [Open on LeetCode](https://leetcode.com/problems/range-sum-query-2d-immutable/)

**Pattern:** Prefix Sum & Hashing — **2D prefix sum + inclusion–exclusion** (the natural lift of [[2-range-sum-query]] into a second dimension)
**Trigger:** "many `sumRegion(r1, c1, r2, c2)` queries on an immutable matrix" → precompute `ps[i][j]` = sum of the top-left rectangle `(0,0)..(i-1,j-1)`. Every query is **4 lookups with `+ − − +`** signs. The `(m+1) × (n+1)` zero border kills the `row1 == 0` / `col1 == 0` edge cases — same trick as the `n+1` shift in 303.

## Problem (1-liner)
Build a `NumMatrix(matrix)` that answers `sumRegion(row1, col1, row2, col2)` — the sum of all cells inside the rectangle with corners `(row1,col1)` (top-left) and `(row2,col2)` (bottom-right, inclusive) — in O(1) per query, with up to `10^4` queries on a `200 × 200` matrix.
**Example:** `sumRegion(2,1,4,3) = 8` · `sumRegion(1,1,2,2) = 11` · `sumRegion(1,2,2,4) = 12`

## Idea (1-liner)
Brute force per query is O(m·n) ⇒ 10⁴ × 4·10⁴ = 4·10⁸ ops, too slow at the upper bound. A **2D prefix sum** `ps[i][j] = sum of the top-left submatrix ending just before (i,j)` reduces every query to **inclusion–exclusion on four corners** — O(m·n) build, O(1) query, O(m·n) extra space.

## Approach
1. **Constructor (`O(m·n)`):** allocate `ps` of size `(m+1) × (n+1)`, all zeros. The full row 0 and column 0 of `ps` stay zero — that's the empty-prefix sentinel border. For `i = 1..m`, `j = 1..n`:
   ```
   ps[i][j] = matrix[i-1][j-1] + ps[i-1][j] + ps[i][j-1] - ps[i-1][j-1]
   ```
   Read: "this cell + the column-strip above + the row-strip to the left − the corner that got added twice." Same `+ − − +` shape as the query.
2. **Query (`O(1)`):** with `(r1, c1, r2, c2)`,
   ```
   sumRegion = ps[r2+1][c2+1] - ps[r1][c2+1] - ps[r2+1][c1] + ps[r1][c1]
   ```
   Picture four top-left rectangles ending at `(r2,c2)`, `(r1-1,c2)`, `(r2,c1-1)`, `(r1-1,c1-1)`. Combining them with `+ − − +` carves out exactly the query rectangle.
3. **Why size `(m+1) × (n+1)` and not `m × n`?** With a plain `prefix[i][j] = sum of (0,0)..(i,j)` the query becomes `prefix[r2][c2] - prefix[r1-1][c2] - prefix[r2][c1-1] + prefix[r1-1][c1-1]`, which blows up when `r1 == 0` or `c1 == 0`. The **+1 shift** makes the entire row 0 and column 0 the empty-prefix sentinel — no special case.
4. The build and the query are **the same inclusion–exclusion formula** in opposite directions. That symmetry is the whole pattern — internalize it once and 2D problems stop being scary.

## Code (clean canonical)
```cpp
class NumMatrix {
    vector<vector<int>> ps;                            // ps[i][j] = sum of (0,0)..(i-1,j-1)
public:
    NumMatrix(vector<vector<int>>& matrix) {
        int m = matrix.size(), n = matrix[0].size();
        ps.assign(m + 1, vector<int>(n + 1, 0));       // zero border = empty-prefix sentinel
        for (int i = 1; i <= m; i++)
            for (int j = 1; j <= n; j++)
                ps[i][j] = matrix[i-1][j-1]
                         + ps[i-1][j] + ps[i][j-1]     // strips above + to the left
                         - ps[i-1][j-1];               // subtract double-counted corner
    }

    int sumRegion(int r1, int c1, int r2, int c2) {
        return ps[r2+1][c2+1]                          // big rectangle
             - ps[r1][c2+1] - ps[r2+1][c1]             // minus top strip + left strip
             + ps[r1][c1];                             // add back double-removed corner
    }
};
```
**TC:** O(m·n) build, O(1) per query.  **SC:** O(m·n) for `ps`.

## Other Approaches
- **Recompute per query** — O(m·n) per call. Trivially correct, blows up at 10⁴ queries × 4·10⁴ cells ≈ 4·10⁸ ops. The whole reason this problem exists is to force you off this.
- **Row-only prefix sums** — store `rowPS[i][j] = sum of row i up to column j`. Query loops `r1..r2` and does one subtraction per row → O(r2 − r1 + 1) per query, build O(m·n). A nice halfway step that shows the 2D version is just "do the 1D trick on both axes at once," but strictly dominated here.
- **Segment tree / 2D BIT** — O(log m · log n) per query, O(m·n log m log n) preprocessing. Overkill because the matrix is **immutable**; reach for these only when point updates appear (the 2D analogue of [[range-sum-mutable]] / LC 308).

## Pitfalls
- **Off-by-one on the shift.** `matrix[i][j]` lives at `ps[i+1][j+1]`. Mixing the two indexing schemes mid-formula is the #1 bug in this problem. Stay disciplined: the matrix is 0-indexed, `ps` is 1-indexed; translate **at the boundary** of the formula, not inside it.
- **Wrong sign in inclusion–exclusion.** Both build and query are `+ − − +` — same shape, mirrored direction. If you flip a sign you'll pass the example but fail edge rectangles touching row 0 / col 0.
- **Forgetting the zero border.** If you size `ps` as `m × n` you'll need explicit `if (r1 == 0)` / `if (c1 == 0)` guards in every query — bug-prone under interview pressure. Pick `(m+1)×(n+1)` and the bug class disappears (same lesson as 303's `n+1` trick).
- **`int` overflow.** Constraints: `|matrix[i][j]| ≤ 10⁴`, `m, n ≤ 200` ⇒ |prefix sum| ≤ 4·10⁸, fits in `int`. For looser bounds widen to `long long`.
- **Don't recompute inside `sumRegion`.** The whole point is moving work to the constructor; if your query touches `matrix[][]`, you've defeated it.
- **Family link:** this is the 2D lift of [[2-range-sum-query]]. The inclusion–exclusion idea also shows up in counting-with-constraints problems (e.g. "at most K distinct = atMostK − atMostK-1" in [[sliding-window-992]]) — same mental move in a different costume. Lock in the **`+ − − +` template** here; it transfers.
