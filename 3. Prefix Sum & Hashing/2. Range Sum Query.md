# 2. Range Sum Query - Immutable (LC 303) — Easy
🔗 [Open on LeetCode](https://leetcode.com/problems/range-sum-query-immutable/)

**Pattern:** Prefix Sum & Hashing — **prefix sum for O(1) range queries** (the seed problem of the prefix-sum half of the family)
**Trigger:** "many `sumRange(l, r)` queries on an immutable array" → precompute `ps[i] = nums[0] + … + nums[i-1]` once; every query is `ps[r+1] - ps[l]` in O(1). The `+1` shift kills the `l == 0` edge case.

## Problem (1-liner)
Build a `NumArray(nums)` that answers `sumRange(left, right)` (inclusive) in O(1) per query, with up to `10^4` queries on an array of size `10^4`.
**Example:** `nums=[-2,0,3,-5,2,-1]` · `sumRange(0,2)=1` · `sumRange(2,5)=-1` · `sumRange(0,5)=-3`

## Idea (1-liner)
Brute force per query is O(n) ⇒ O(n·q) = 10^8 ops, too slow at the upper bound. A **prefix-sum array** `ps[i] = sum of first i elements` reduces every query to a single subtraction `ps[r+1] - ps[l]` — O(n) build, O(1) query, O(n) extra space.

## Approach
1. **Constructor (`O(n)`):** allocate `ps` of size `n+1` with `ps[0] = 0`. For `i = 0..n-1`, set `ps[i+1] = ps[i] + nums[i]`. After this loop, `ps[i]` = sum of `nums[0..i-1]`.
2. **Query (`O(1)`):** `sumRange(l, r) = ps[r+1] - ps[l]`.
3. **Why size `n+1` and not `n`?** With a plain `prefix[i] = sum of nums[0..i]` you'd need `prefix[r] - prefix[l-1]`, which breaks when `l == 0`. The **`n+1` shift** makes `ps[0] = 0` the empty-prefix sentinel — no special case.
4. The class stores `ps` as a member so the O(n) work happens **once** in the constructor and gets amortized across the 10^4 queries.

## Code (clean canonical)
```cpp
class NumArray {
    vector<int> ps;                                    // ps[i] = sum of nums[0..i-1]
public:
    NumArray(vector<int>& nums) : ps(nums.size() + 1, 0) {
        for (int i = 0; i < (int)nums.size(); i++)
            ps[i + 1] = ps[i] + nums[i];               // build once, O(n)
    }

    int sumRange(int left, int right) {
        return ps[right + 1] - ps[left];               // O(1) per query
    }
};
```
**TC:** O(n) build, O(1) per query.  **SC:** O(n) for `ps`.

## My Solution (also correct)
```cpp
class NumArray {
  unordered_map<int, int> mpp;
public:
    NumArray(vector<int>& nums) {
      mpp[-1] = 0;                                     // sentinel for left==0
      int sum = 0;
      for (int i = 0; i < nums.size(); i++) {
        sum += nums[i];
        mpp[i] = sum;
      }
    }

    int sumRange(int left, int right) {
      return mpp[right] - mpp[left - 1];
    }
};
```
Correct logic — same prefix-sum idea with an explicit `-1` sentinel. **Why prefer the canonical `vector` version:**
- **`vector<int>` over `unordered_map<int,int>`:** keys are dense contiguous indices `0..n-1`, so the hash map is pure overhead (hashing, buckets, pointer chasing). `vector` is contiguous → better cache locality, tighter memory.
- **`mpp[left - 1]` silently inserts** when `left == 0` and the sentinel `-1` happens to be missing — `operator[]` on a map mutates it during what should be a read. The `n+1`-sized vector approach has no equivalent pitfall.
- **Index-shift over sentinel-key** is the idiom you'll reuse in 304 (2D prefix sums add a row/col of zeros for the same reason) and 560 (`cnt[0] = 1` is the empty-prefix sentinel in *count* form).

## Other Approaches
- **Recompute per query** — O(n) per call. Trivially correct, blows up at 10^4 queries × 10^4 elements ≈ 10^8 ops. The whole reason this problem exists is to force you off this.
- **Caching answers by `(l, r)` pair** — up to `n²` distinct pairs ⇒ O(n²) memory, and the first call to each pair is still O(n). Prefix sum dominates on both axes.
- **Segment tree / Fenwick (BIT)** — O(log n) per query, O(n log n) preprocessing. Overkill here because the array is **immutable**; reach for these in [[range-sum-mutable]] (LC 307) where point updates appear.

## Pitfalls
- **Off-by-one on the shift.** `ps[r+1] - ps[l]` (size `n+1`, `ps[0]=0`) ✅. `ps[r] - ps[l-1]` (size `n`) needs an `l == 0` guard — easy to forget under interview pressure. Pick the `n+1` version and the bug class disappears.
- **`int` overflow.** Constraints: `|nums[i]| ≤ 10^5`, `n ≤ 10^4` ⇒ |prefix sum| ≤ 10^9, fits in `int`. For wider ranges (e.g. [[437-path-sum-iii]] on long paths) widen to `long long`.
- **Don't recompute inside `sumRange`.** The whole point is moving work to the constructor; if your query loop touches `nums[]`, you've defeated it.
- **Hash map vs vector for dense integer keys.** Whenever the keys are `0..n-1`, default to `vector`. Reserve `unordered_map` for sparse / non-integer / unknown-range keys — that's the lesson this problem teaches before you hit [[4-subarray-sum-equals-k]] where a map is genuinely needed.
- **Family seed:** this is the prefix-sum prototype for [[3-range-sum-2d]] (2D inclusion-exclusion), [[4-subarray-sum-equals-k]] (prefix + frequency map for *count*), [[5-subarray-sums-divisible-by-k]] (prefix mod K), and any "many range aggregates on an immutable array" problem. Lock in the **`n+1`-sized, `ps[0]=0`** template — it transfers directly.
