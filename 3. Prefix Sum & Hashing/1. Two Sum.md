# 1. Two Sum (LC 1) — Easy
🔗 [Open on LeetCode](https://leetcode.com/problems/two-sum/)

**Pattern:** Prefix Sum & Hashing — **complement lookup via hash map** (the seed problem of the whole family)
**Trigger:** "find two indices whose values sum to `target`, can't use the same element twice" → for every `nums[i]` ask **"have I already seen `target - nums[i]`?"** in O(1) via `unordered_map<value, index>`; **check before insert** so an element never matches itself

## Problem (1-liner)
Given `nums` and `target`, return the indices of the two numbers that add up to `target`. Exactly one solution exists, you may not reuse the same element, indices in any order.
**Examples:** `[2,7,11,15], 9` → `[0,1]` · `[3,2,4], 6` → `[1,2]` · `[3,3], 6` → `[0,1]`

## Idea (1-liner)
Brute force tries every pair in O(n²); instead, sweep once with a hash map of `value → index` and, for each new `nums[i]`, look up its **complement** `target - nums[i]` — if it's in the map, you've already seen the other half of the pair; otherwise record `nums[i]` for future queries. The "check then insert" order is what makes duplicates like `[3,3]` work without an element pairing with itself.

## Approach
1. **Hash map** `mpp: value → first-seen index` (empty to start).
2. **Single pass** `i = 0..n-1`:
   - `req = target - nums[i]` is the value we'd need to have already seen.
   - If `mpp.find(req) != mpp.end()` ⇒ return `{mpp[req], i}` (earlier index first, matches "any order" + natural ordering).
   - Otherwise insert `mpp[nums[i]] = i` and continue.
3. **Order matters:** look up the complement **before** writing `nums[i]` into the map, otherwise `[3,3]` would match index 0 against itself.
4. Problem guarantees a solution, so the loop always returns; the trailing `return {}` is just to silence the compiler.

**Why sorting + two-pointer is wrong here:** that's [LC 167 Two Sum II](https://leetcode.com/problems/two-sum-ii-input-array-is-sorted/) on a *sorted* array — sorting destroys the original indices the problem asks for. Use it as a contrast: same shape, different input contract ⇒ different pattern.

## Code (clean canonical)
```cpp
class Solution {
public:
    vector<int> twoSum(vector<int>& nums, int target) {
        unordered_map<int, int> mpp;                       // value -> index seen so far
        for (int i = 0; i < (int)nums.size(); i++) {
            int req = target - nums[i];
            auto it = mpp.find(req);
            if (it != mpp.end()) return {it->second, i};   // complement seen earlier
            mpp[nums[i]] = i;                              // record AFTER the lookup
        }
        return {};                                         // unreachable per constraints
    }
};
```
**TC:** O(n) — single pass, O(1) avg hash ops.  **SC:** O(n) — up to `n-1` entries in the map.

## Other Approaches
- **Brute force two loops** — O(n²) / O(1). Useful as a baseline / sanity check; immediately rejected by the follow-up "less than O(n²)".
- **Sort + two pointers on indexed copies** — pair each value with its original index, sort by value, then two-pointer for the sum. O(n log n) / O(n), correct but strictly worse than the hash map and more code. Only worth it if hashing is banned or values are tightly bounded.
- **Two-pass hash map** — fill the map first, then scan looking up complements. Same complexity but needs a `req == nums[i] && mpp[req] != i` guard to avoid self-pairing; the **one-pass "check before insert"** version sidesteps that entirely.

## Pitfalls
- **Insert-then-check** flips the order and breaks duplicates: with `[3,3], target=6`, inserting `mpp[3]=0` first, then looking up `3`, would return `[0,0]` — invalid.
- **`mpp.count(req)` + `mpp[req]`** is two hash lookups; `mpp.find(req)` returns the iterator in one. Negligible on LC but it's the idiomatic C++ pattern and matters at scale.
- **Don't sort** — sorting renumbers the array, and the problem asks for **original** indices. Reach for sort + two-pointer on [[2-two-sum-ii]] (sorted-input variant) instead.
- **Integer overflow** in `target - nums[i]`: with the given constraints (`|nums[i]|, |target| ≤ 10^9`) the subtraction fits in `int` — no `long long` needed here. For larger ranges (e.g. [[12-4sum]]) you do need to widen.
- **Family seed:** this is the prototype for [[4-subarray-sum-equals-k]] (complement = `sum - k` on a running prefix sum), [[6-3sum]] / [[7-4sum]] (fix outer index, two-sum on the rest), and any "have I seen X before?" hashing problem. Lock in the *"check before insert"* invariant — it transfers directly.
