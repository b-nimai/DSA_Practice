# 7. Contiguous Array (LC 525) — Medium
🔗 [Open on LeetCode](https://leetcode.com/problems/contiguous-array/)

**Pattern:** Prefix Sum & Hashing — **running +1/−1 balance + map of the *earliest index* per balance** (the "longest" sibling of [[6-continuous-subarray-sum-523]]; same earliest-index bookkeeping as 523, but the per-prefix key is a **±1 balance** instead of a remainder, and we maximize a length instead of testing existence)
**Trigger:** "**longest** subarray with an **equal number of 0s and 1s**" → recode `0 → −1`, `1 → +1`; now "equal counts" becomes "**subarray sums to 0**", i.e. two prefixes with the **same running balance**. Track each balance's **earliest** index; when a balance repeats, the span between is a valid subarray of length `i − firstIdx`.

## Problem (1-liner)
Given a binary array `nums`, return the **maximum length** of a contiguous subarray with an equal count of `0`s and `1`s.
**Example:** `[0,1] → 2` · `[0,1,0] → 2` · `[0,1,1,1,1,1,0,0,0] → 6` (`[1,1,1,0,0,0]`)
**Constraints:** `1 ≤ n ≤ 10⁵`, `nums[i] ∈ {0,1}` — the binary alphabet is what lets the `0→−1` recoding turn "equal counts" into "prefix balance repeats".

## Idea (1-liner)
Map `0 → −1`, `1 → +1` and keep a running `balance`. A subarray `nums[i..j]` has equal 0s and 1s iff `balance` is the same at both ends (the segment sums to 0). So record each balance's **earliest** index; when the current balance was seen before, `i − firstIdx` is a candidate answer. One pass: **O(n) time, O(n) space.**

## Approach
1. Keep a running `balance` (`+1` for a `1`, `−1` for a `0`) and a map `balance → earliest index that produced it`.
2. **Seed `{0: -1}`** — the empty prefix has balance 0 sitting "before" index 0. This makes a subarray that's balanced from index 0 compute as `i − (−1) = i + 1`, and removes any need for a `balance == 0` special case.
3. For each index `i`:
   - `balance += nums[i] ? 1 : -1`.
   - If `balance` is already in the map: the span since its first occurrence is balanced ⇒ `res = max(res, i − firstIdx)`.
   - Else record `map[balance] = i`. **Insert only on first sight** — keeping the earliest index maximizes future spans.
4. Return `res`.

## Code (clean canonical)
```cpp
class Solution {
public:
    int findMaxLength(vector<int>& nums) {
        unordered_map<int, int> first{{0, -1}};   // balance(#1 − #0) -> earliest index
        int balance = 0, res = 0;
        for (int i = 0; i < (int)nums.size(); ++i) {
            balance += nums[i] ? 1 : -1;
            auto it = first.find(balance);
            if (it != first.end()) res = max(res, i - it->second);  // same balance ⇒ equal 0s/1s between
            else first[balance] = i;                                // store only the FIRST index
        }
        return res;
    }
};
```
**TC:** O(n).  **SC:** O(n) — up to `n+1` distinct balances in `[−n, n]`.

## My Solution (also correct)
```cpp
class Solution {
public:
    int findMaxLength(vector<int>& nums) {
      // diff(one - zero) -> index
      unordered_map<int, int>diff_mpp;
      int one = 0, zero = 0, res = 0;
      for (int i = 0; i < nums.size(); i++) {
        if(nums[i] == 0) zero++;
        else one++;
        if(zero == one) res = zero + one;
        auto it = diff_mpp.find(one - zero);
        if (it != diff_mpp.end()) {
          int index = diff_mpp[one - zero];
          res = max(res, i - index);
        } else diff_mpp[one - zero] = i;
      }
      return res;
    }
};
```
Correct logic, identical algorithm (`one − zero` *is* the balance). **Refinements rolled into the canonical version:**
- **Seed `{0, -1}` to delete the `zero == one` branch.** That special case only existed to catch subarrays balanced from index 0; the sentinel folds it into the single `i − firstIdx` formula. Same trick as the `{0,-1}` seed in [[6-continuous-subarray-sum-523]].
- **One `balance` counter instead of `one` + `zero`.** Only their difference matters: `balance += nums[i] ? 1 : -1`.
- **Reuse the iterator.** `int index = diff_mpp[one - zero]` is a *second* hash lookup after the `find`; use `it->second` for one lookup per step.

## Family map — where this sits
| Problem | "Bucket key" per prefix | Sentinel | Decision rule |
|---|---|---|---|
| [[4-subarray-sum-equals-k]] (LC 560) | `sum` value | `cnt[0] = 1` (freq) | `count += cnt[sum − k]` |
| [[5-subarray-sums-divisible-by-k]] (LC 974) | `sum mod k` | `freq[0] = 1` (freq) | `count += freq[rem]` |
| [[6-continuous-subarray-sum-523]] (LC 523) | `sum mod k` | `first[0] = -1` (index) | exists iff `i − first[rem] ≥ 2` |
| **525 (this one)** | running `+1/−1` balance | `first[0] = -1` (index) | `res = max(res, i − first[balance])` (longest) |
| [[1248-nice-subarrays]] (LC 1248) | running odd-count | `cnt[0] = 1` (freq) | `count += cnt[odds − k]` |

Shared template: **pick a per-prefix bucket key, seed the empty-prefix sentinel, lookup-then-record.** The fork is *what you store*: **frequency** for "count", **earliest index** for "exists / longest" (523, 525). 525's twist is purely the `0 → −1` recoding that turns a counting balance into a summable prefix.

## Other Approaches
- **Brute force O(n²)** — for every start `i`, extend `j` tracking the running balance, record length when it hits 0. `n = 10⁵` ⇒ 10¹⁰ ops, TLE.
- **Prefix-sum array + O(n²) pair scan** — same wall; the balance→index map collapses the inner loop to O(1).
- **No sliding window.** "Equal counts" isn't a monotone window predicate (adding an element can move balance either way), so grow/shrink doesn't apply — the index map is the right tool.

## Pitfalls
- **Overwriting the stored index.** For *longest*, keep the **first** index per balance so spans can grow; overwriting with the latest index shrinks every candidate. Same rule as [[6-continuous-subarray-sum-523]].
- **Forgetting the `{0, -1}` sentinel.** Without it, a subarray balanced from index 0 needs a special `balance == 0` branch; the sentinel makes the single `i − firstIdx` formula cover it.
- **Recoding direction is arbitrary but must be consistent.** `0 → −1, 1 → +1` or the reverse both work; just don't mix mid-loop.
- **Confusing with the counting twins.** 560/974/1248 store a **frequency** and *sum* it; 523/525 store an **index** and *compare/max* it. Same map shape, opposite bookkeeping.
- **Double hash lookup.** `find` then `map[key]` hashes twice; reuse the iterator's `->second`. Minor, but it's the idiomatic fix.
