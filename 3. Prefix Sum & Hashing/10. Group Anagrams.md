# 10. Group Anagrams (LC 49) — Medium
🔗 [Open on LeetCode](https://leetcode.com/problems/group-anagrams/)

**Pattern:** Prefix Sum & Hashing — **canonical key + bucket map** (a membership-family cousin of [[1-two-sum]] and [[11-longest-consecutive-sequence]]: instead of asking "have I seen value X?", you collapse each item to a **canonical signature** so all equivalents hash to the same bucket. The art is choosing a key that is *identical for anagrams and distinct for everything else*)
**Trigger:** "**group the items that are equivalent under some transform**" (here: rearrangement). Build a `unordered_map<key, vector<string>>`; map each word to a key that is **invariant across anagrams** — its **letter-count signature** — and `push_back` into that bucket. Anagrams share a signature ⇒ land in the same vector. Return the map's values.

## Problem (1-liner)
Given `strs`, group strings that are anagrams of one another (same multiset of letters); return the groups in any order.
**Example:** `["eat","tea","tan","ate","nat","bat"] → [["bat"],["nat","tan"],["ate","eat","tea"]]` · `[""] → [[""]]` · `["a"] → [["a"]]`
**Constraints:** `1 ≤ n ≤ 10⁴`, `0 ≤ |strs[i]| ≤ 100`, **lowercase only** (⇒ a fixed 26-slot count vector is the natural signature). Empty strings are valid and group together.

## Idea (1-liner)
Two words are anagrams iff they have the **same letter counts**. So reduce each word to a canonical key — its 26-letter count signature — and bucket words by that key in a hash map. **O(N·K)** time (N words, K = max length), **O(N·K)** space.

## Approach
1. `unordered_map<string, vector<string>> mpp` — key = canonical signature, value = the group.
2. For each word, build its **count signature**: a length-26 tally of `ch - 'a'`. Anagrams produce identical signatures; non-anagrams differ in at least one slot.
3. `mpp[key].push_back(word)` — `operator[]` default-constructs an empty vector on first sight, so no "exists?" check is needed.
4. Collect `mpp`'s values into the result (order irrelevant).

## Code (clean canonical)
```cpp
class Solution {
public:
    vector<vector<string>> groupAnagrams(vector<string>& strs) {
        unordered_map<string, vector<string>> mpp;
        for (const string& str : strs) {
            string key(26, 0);                       // 26-byte count signature
            for (char ch : str) key[ch - 'a']++;     // counts ≤ 100, fit in a char
            mpp[key].push_back(str);
        }
        vector<vector<string>> result;
        result.reserve(mpp.size());
        for (auto& it : mpp) result.push_back(std::move(it.second));
        return result;
    }
};
```
**TC:** O(N·K) — each word scanned once to build its key.  **SC:** O(N·K) for the map + output.

## My Solution (also correct)
```cpp
class Solution {
public:
    vector<vector<string>> groupAnagrams(vector<string>& strs) {
      unordered_map<string, vector<string>>mpp;

      for (string &str : strs) {
        vector<int>freq(26, 0);
        for (char ch : str) {
          freq[ch - 'a']++;
        }

        string key = "";
        for (int num : freq) key += to_string(num) + "#";

        mpp[key].push_back(str);
      }

      vector<vector<string>>result;
      for (auto &it : mpp) result.push_back(it.second);

      return result;
    }
};
```
Correct and — importantly — already the **better key choice** (count signature, not sorted string), and you're already iterating `strs` by reference (`string &str`) so no per-word copy. **Refinements rolled into the canonical version (all constant-factor / style — no complexity change):**
- **Raw-byte key over `to_string(num)+"#"`.** Counts are ≤ 100 < 256, so each fits in one `char`; a fixed 26-byte string *is* the signature directly. This drops 26 `to_string` allocations + concatenations per word. The `#` separator existed only to stop `"1,11"` colliding with `"11,1"` — a fixed-width 26-byte layout removes that ambiguity, so the delimiter isn't needed.
- **`const string& str`** (vs `string& str`) — you avoid the copy with the plain reference already; `const` just documents that the words aren't mutated.
- **`std::move(it.second)`** into the result — the map is discarded after, so move the group vectors instead of copying them.
- **`result.reserve(mpp.size())`** before the final loop avoids vector reallocations.

## The two key choices — why count beats sort
| Key strategy | How | Per-word cost | Total |
|---|---|---|---|
| **Sorted string** | `sort(word)`; anagrams share sorted form | O(K log K) | **O(N·K log K)** |
| **Count signature (this)** | 26-slot letter tally | O(K) | **O(N·K)** |

Both are correct; the count signature is a `log K` factor faster because tallying is linear while sorting isn't. With a fixed 26-letter alphabet the signature is also tiny and fixed-width. Sorting is the more *obvious* key — recognizing the count signature is the upgrade.

## Family map — where this sits
| Problem | Hash structure | The key / question | Family role |
|---|---|---|---|
| [[1-two-sum]] (LC 1) | `unordered_map<val,idx>` | "is complement `target − x` seen?" | complement lookup |
| [[11-longest-consecutive-sequence]] (LC 128) | `unordered_set<val>` | "is neighbour `x ± 1` present?" | membership / number-line walk |
| **49 (this one)** | `unordered_map<sig, group>` | "what canonical signature collapses equivalents?" | canonical-key bucketing |

49 is the family's **canonical-key** instance: the hash isn't answering "seen before?" but "**which bucket does this collapse to?**" — the same trick behind grouping/dedup-by-property problems.

## Other Approaches
- **Sorted-string key — O(N·K log K).** Same map, key = `sort(word)`. Correct, slightly slower, one line shorter. The common textbook answer.
- **Prime-product hash.** Map each letter to a prime, multiply; anagrams share the product. Cute but **overflows** for length-100 words — needs big integers, so not worth it.

## Pitfalls
- **Count key without a separator while using digits.** If you encode counts as `to_string` *without* a delimiter, `[1,11]` and `[11,1]` both render `"111"` and collide. Either keep the `#` delimiter (your original, safe) **or** use the fixed-width raw-byte signature (no delimiter needed). Don't mix: digits without separators is the bug.
- **Copying strings in the loop.** `for (string str : strs)` copies each word; use `const string&` (or `string&` as you did).
- **Forgetting empty strings group together.** `""` has an all-zero signature and buckets correctly — the count loop just runs zero times, which is fine.
- **Reaching for sorting reflexively.** It works, but the count signature is strictly better given the bounded lowercase alphabet — recognizing that is the point of the problem.
