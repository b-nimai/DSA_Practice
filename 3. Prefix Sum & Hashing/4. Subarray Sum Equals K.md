# 4. Subarray Sum Equals K (LC 560) — Medium
🔗 [Open on LeetCode](https://leetcode.com/problems/subarray-sum-equals-k/)

**Pattern:** Prefix Sum & Hashing — **prefix sum + frequency map of past prefix values** (the seed problem for *counting* subarrays with a sum property; the natural step up from [[2-range-sum-query]] which only *answers* range sums)
**Trigger:** "**count** the number of subarrays with sum = K" → maintain running prefix `sum`; at each index ask **"how many earlier prefixes equal `sum − k`?"** because each such prefix carves out a valid subarray ending here. Seed the map with `{0: 1}` (the empty prefix) so subarrays starting at index 0 are counted.

## Problem (1-liner)
Given `nums` and integer `k`, return the **total number** of contiguous non-empty subarrays whose sum equals `k`.
**Example:** `nums=[1,1,1], k=2 → 2` · `nums=[1,2,3], k=3 → 2`
**Constraints:** `1 ≤ n ≤ 2·10⁴`, `−1000 ≤ nums[i] ≤ 1000`, `−10⁷ ≤ k ≤ 10⁷` — **negatives are allowed**, which is the whole point.

## Idea (1-liner)
A subarray `nums[i..j]` sums to `k` iff `ps[j+1] − ps[i] = k`, i.e. `ps[i] = ps[j+1] − k`. So while sweeping the prefix sum, count **how many times the value `sum − k` has appeared as a prefix before** — that's exactly the number of valid subarrays ending at this index. One pass, hash-map for the frequency: **O(n) time, O(n) space.**

## Approach
1. Maintain a running `sum` (= prefix sum up to current index) and `cnt`, an `unordered_map<int,int>` mapping **prefix-sum value → how many indices have produced it**.
2. **Seed `cnt[0] = 1`** — the empty prefix. Without it, a subarray starting at index 0 with sum exactly `k` is missed (because there's no "earlier prefix of value 0" recorded yet).
3. For each element `x`:
   - `sum += x`
   - `count += cnt[sum − k]` — every earlier index whose prefix sum equals `sum − k` is the **left edge** of a valid subarray ending here.
   - `cnt[sum]++` — record **after** the lookup so an element can't pair with itself when `k == 0`.
4. The lookup-before-insert ordering is the same "check before insert" lesson from [[1-two-sum]].

## Code (clean canonical)
```cpp
class Solution {
public:
    int subarraySum(vector<int>& nums, int k) {
        unordered_map<int,int> cnt{{0, 1}};            // empty-prefix sentinel
        cnt.reserve(nums.size() + 1);
        int sum = 0, count = 0;
        for (int x : nums) {
            sum += x;
            auto it = cnt.find(sum - k);               // one hash lookup, not two
            if (it != cnt.end()) count += it->second;
            cnt[sum]++;                                // record AFTER the lookup
        }
        return count;
    }
};
```
**TC:** O(n) average (hash ops O(1)).  **SC:** O(n) for `cnt`.

## My Solution (also correct)
```cpp
class Solution {
public:
    int subarraySum(vector<int>& nums, int k) {
      unordered_map<int, int>preSum;
      preSum[0] = 1;
      int count = 0, sum = 0;
      for(int i = 0; i < nums.size(); i++) {
        sum += nums[i];
        int rem = sum - k;
        if(preSum.find(rem) != preSum.end()) count += preSum[rem];
        preSum[sum]++;
      }
      return count;
    }
};
```
Correct logic, identical algorithm. **Refinements rolled into the canonical version:**
- **`find` + `[]` = two hash lookups for one key.** Keep the iterator returned by `find` and read `it->second` — one lookup. The 303 note already endorsed this idiom; reinforce it here.
- **Indexed `for` with unused `i`.** Range-for (`for (int x : nums)`) drops the noise — pattern of choice whenever you don't need the index.
- **`reserve(n+1)`** prevents bucket rehashing mid-loop. Micro, but free.

## What if `nums` has **no negatives**? — Two-pointer variant
The map approach works regardless of sign. **But** with non-negative inputs the running prefix becomes **monotonic**, which unlocks **sliding window** — O(n) time, **O(1) space**. This is the better answer when the constraint allows it. Two flavors:

### Case A — strictly positive (`nums[i] ≥ 1`, `k ≥ 1`)
Window sum strictly grows with `right`, strictly shrinks with `left`. At most one valid `left` per `right`, so count direct hits.
```cpp
int subarraySumPositive(vector<int>& nums, int k) {
    int l = 0, sum = 0, count = 0;
    for (int r = 0; r < (int)nums.size(); r++) {
        sum += nums[r];
        while (sum > k) sum -= nums[l++];              // shrink while window is too big
        if (sum == k) count++;                         // ≤ 1 valid window ends at r
    }
    return count;
}
```

### Case B — non-negative with zeros allowed
A `0` extends a valid window without changing its sum, so **multiple** windows can end at the same `r`. The "count direct hits" version undercounts (e.g. `[1,0,0,1], k=1` → truth = 6, naive count = 4). Use the **`atMost(k) − atMost(k−1)`** trick (same idiom as [[992-k-distinct]]):
```cpp
long atMost(vector<int>& nums, int t) {
    if (t < 0) return 0;
    long l = 0, sum = 0, cnt = 0;
    for (int r = 0; r < (int)nums.size(); r++) {
        sum += nums[r];
        while (sum > t) sum -= nums[l++];
        cnt += r - l + 1;                              // # subarrays ending at r with sum ≤ t
    }
    return cnt;
}
int subarraySumNonNeg(vector<int>& nums, int k) {
    return (int)(atMost(nums, k) - atMost(nums, k - 1));
}
```

### Why two-pointer **fails** on LC 560's actual constraints
Negatives break the monotonicity: adding to `right` can *decrease* the sum, so "shrink while sum > k" doesn't terminate correctly. The map version is the only general solution — that's exactly why the problem permits `nums[i] = −1000`.

| Constraint on `nums` | Right tool | TC / SC |
|---|---|---|
| All `> 0`, `k > 0` | Sliding window — count direct hits | O(n) / O(1) |
| All `≥ 0`, `k ≥ 0` | Sliding window — `atMost(k) − atMost(k−1)` | O(n) / O(1) |
| Any negatives (LC 560) | Prefix sum + hash map (no shortcut) | O(n) / O(n) |

## Other Approaches
- **Brute force two loops** — enumerate every `(i, j)` and sum. O(n²) at best (O(n³) if you re-sum each time). At `n = 2·10⁴`, O(n²) = 4·10⁸ — TLE.
- **Prefix sum array + double loop** — O(n²) checking `ps[j+1] − ps[i] == k`. Same wall as above. The hash map is what cuts the inner loop to O(1).
- **Divide & conquer** — O(n log n) at best, much harder to write than the map version. No real reason to reach for it here.

## Pitfalls
- **Forgetting `cnt[0] = 1`.** Without the empty-prefix sentinel, any subarray starting at index 0 with sum exactly `k` is silently dropped. This is the prefix-sum analog of "check before insert" from [[1-two-sum]] — the sentinel makes the boundary case fall into the general case.
- **`cnt[sum]++` *before* the lookup.** If `k == 0` and `nums[i] == 0`, recording first lets a single element pair with itself ⇒ overcount. Always **lookup → then record.**
- **`operator[]` silently inserts.** `count += cnt[sum - k]` mutates the map when the key is absent. It's "correct" (default-constructs to 0) but bloats the map and slows iteration. Prefer `find`-then-iterator.
- **Two hash lookups for one key.** `find(...) != end()` followed by `mpp[key]` is two hash probes. Save the iterator from `find` and read `it->second`.
- **Overflow.** `|sum|` can reach `n · max|nums[i]| = 2·10⁴ · 10³ = 2·10⁷`; `sum − k` stays under `3·10⁷`. `int` is fine here. For looser constraints (e.g. [[437-path-sum-iii]] on deep paths) widen to `long long`.
- **Don't try two-pointer when negatives are allowed.** It compiles, passes the example, and silently miscounts on `nums = [1, -1, 1]`. The monotonicity gate must be checked **before** reaching for sliding window — same recognition step you'll need in [[974-subarray-divisible-by-k]].
