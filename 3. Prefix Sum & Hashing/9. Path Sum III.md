# 9. Path Sum III (LC 437) — Medium
🔗 [Open on LeetCode](https://leetcode.com/problems/path-sum-iii/)

**Pattern:** Prefix Sum & Hashing — **prefix-sum frequency map carried down a DFS, with backtracking** (it's [[4-subarray-sum-equals-k]] lifted from an array onto root-to-node *paths* of a tree; the "array" is whatever chain of nodes you're currently standing on, and DFS + undo keeps the map scoped to exactly that chain)
**Trigger:** "**count** the downward root-to-leaf-ish paths whose node sum = `targetSum`, path may start/end anywhere" → a downward path is a contiguous slice of some root→node chain, so run 560's running-prefix-sum trick **along each DFS path**: at each node `count += freq[runningSum − target]`, record `freq[runningSum]++`, then **decrement on the way back up** so siblings don't see each other's prefixes.

## Problem (1-liner)
Given a binary tree and `targetSum`, return the **number of downward paths** (parent→child only, need not touch root or a leaf) whose node values sum to `targetSum`.
**Example:** `root=[10,5,-3,3,2,null,11,3,-2,null,1], target=8 → 3` · `root=[5,4,8,11,null,13,4,7,2,null,null,5,1], target=22 → 3`
**Constraints:** `0 ≤ n ≤ 1000`, **`−10⁹ ≤ Node.val ≤ 10⁹`**, `−1000 ≤ targetSum ≤ 1000`. The wide value range is the whole trap: a running path sum can hit `~10¹²`, so it **must be accumulated in `long long`**.

## Idea (1-liner)
Any valid path is a contiguous segment of the root→current-node chain. So it's exactly "count subarrays summing to `k`" ([[4-subarray-sum-equals-k]]) — only the "array" is the current DFS path. Carry a `runningSum` and a frequency map of prefix sums **seen along this path**; at each node add `freq[runningSum − target]`, then push `runningSum`, recurse, and **pop it on backtrack** so the map always reflects just the current path. **O(n) time, O(h) live map.**

## Approach
1. DFS carrying `sum` (prefix sum from root to the current node) and a map `freq: prefixSum → #occurrences along this path`.
2. **Seed `freq[0] = 1`** — the empty prefix, so a path *starting at the root* that already equals `target` is counted (same sentinel as 560).
3. At each node:
   - `sum += node->val`.
   - `count += freq[sum − target]` — every earlier prefix on this path equal to `sum − target` marks the start of a downward sub-path ending here that sums to `target`. **Use `find`, not `freq[sum−target]`**, so a missing key isn't default-inserted mid-traversal.
   - `freq[sum]++` — record this prefix **after** the lookup.
4. Recurse into left and right, summing their counts.
5. **`freq[sum]--` on the way out** (and erase if it hits 0). This is what makes the map path-scoped: once you leave a subtree, its prefixes must not be visible to a sibling subtree. Forgetting this is the classic bug — it leaks prefixes across branches.

## Code (clean canonical)
```cpp
class Solution {
    unordered_map<long long, int> freq;
    int target;
    int dfs(TreeNode* node, long long sum) {
        if (!node) return 0;
        sum += node->val;                              // long long: path sum can reach ~1e12
        int count = 0;
        auto it = freq.find(sum - target);
        if (it != freq.end()) count += it->second;     // earlier prefixes that open a target path
        freq[sum]++;                                   // record AFTER the lookup
        count += dfs(node->left, sum) + dfs(node->right, sum);
        if (--freq[sum] == 0) freq.erase(sum);         // undo on backtrack → map stays path-scoped
        return count;
    }
public:
    int pathSum(TreeNode* root, int targetSum) {
        target = targetSum;
        freq[0] = 1;                                   // empty-prefix sentinel
        return dfs(root, 0);
    }
};
```
**TC:** O(n) — each node visited once, O(1) map work per node.  **SC:** O(h) live map (h = height) + O(h) recursion stack.

## My Solution (also correct)
```cpp
class Solution {
  int findNumberOfPath(TreeNode* root, unordered_map<long long, int>&mpp, int target, long long sum) {
    int count = 0;
    if (!root) return count;
    sum += root->val;
    long long req = sum - target;

    auto it = mpp.find(req);
    if (it != mpp.end()) count += it->second;
    mpp[sum]++;

    count += findNumberOfPath(root->left, mpp, target, sum);
    count += findNumberOfPath(root->right, mpp, target, sum);

    if(--mpp[sum] == 0) mpp.erase(sum);
    return count;
  }
public:
    int pathSum(TreeNode* root, int targetSum) {
      unordered_map<long long, int>mpp{{0, 1}};
      return findNumberOfPath(root, mpp, targetSum, 0);
    }
};
```
Correct and optimal — passes every test. This was reached after fixing **two** bugs along the way (see Pitfalls). The map-by-reference, `long long` sum, `find`-not-`[]`, and decrement-on-backtrack are all already right. **Refinements rolled into the canonical version (all minor / stylistic):**
- **Erase zero-count entries: `if (--mpp[sum] == 0) mpp.erase(sum);`** Plain `mpp[sum]--` leaves a stale `{sum: 0}` key behind after backtracking, so the map drifts toward O(n) size. Erasing on zero keeps it truly **O(h)** — bounded by the current path length. (Correctness is identical either way; `count += it->second` just adds 0 for a dead key.)
- **Hoist `target` + `freq` to members.** They're threaded unchanged through every call; making them members trims the signature. Pure taste — the explicit-parameter version is equally valid and some interviewers prefer it.

## Why `find` over `mpp[req]` (a trap you correctly avoided)
`count += mpp[req]` is shorter but `operator[]` **default-inserts** `{req: 0}` when absent — mutating the map mid-DFS and defeating the O(h) bound (those phantom keys never get decremented/erased). `find` + iterator reads without inserting. Keep `find`.

## Other Approaches
- **Brute force, two nested DFS — O(n²) / O(n log n).** Outer DFS picks every node as a path start; inner DFS walks all downward paths from it counting those that hit `target`. Simple, no map, no overflow concern if summed in `long long`. Fine at `n ≤ 1000` but the prefix-sum version is strictly better and the same code length.
- **Prefix-sum map (this one) — O(n).** The canonical optimal answer; the only one that generalizes cleanly to large `n`.

## Pitfalls
- **`unordered_set` instead of a frequency `map` (undercount bug).** A set only records *whether* a prefix sum appeared, not how many times. On `root=[0,1,1], target=1` the prefix sum `0` occurs twice (sentinel + root), so the two paths `1` and `0→1` collapse into one count → returns less than the correct **4**. You **must** count occurrences, not presence.
- **`int` running sum (overflow bug).** With `Node.val` up to `10⁹` and up to 1000 nodes, a path sum reaches `~10¹²`. On `root=[1000000000,1000000000,null,294967296,null,1000000000,null,1000000000,null,1000000000], target=0`, 32-bit wrap makes prefix sums alias the sentinel `0` and each other, fabricating **2** phantom paths where the answer is **0**. Accumulate `sum` (and `sum − target`) in **`long long`**.
- **Forgetting to backtrack (`freq[sum]--`).** Without the undo, a right subtree sees the left subtree's prefixes — paths that aren't on a single downward chain get counted. The decrement is what keeps the map scoped to the current path.
- **Recording before the lookup.** `freq[sum]++` must come **after** `count += freq[sum − target]`; otherwise when `target == 0` a node's own prefix would match itself.
- **`operator[]` in the lookup.** `count += freq[sum − target]` silently inserts a 0 key — use `find`. (See section above.)
- **Decrement-then-erase order.** `if (--freq[sum] == 0) freq.erase(sum);` — decrement first, then erase if zero. Erasing unconditionally would wipe a key that a still-active ancestor prefix needs.
