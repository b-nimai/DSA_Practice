# 6. Next Greater Element II (LC 503) — Medium
🔗 [Open on LeetCode](https://leetcode.com/problems/next-greater-element-ii/)

**Pattern:** Monotonic Stack — **next-greater on a circular array: same template, scanned twice**
**Trigger:** "next greater element, but the array is **circular** (you may wrap around)" → run the standard NGE monotonic stack over the array **twice** (logically `2n` indices via `i % n`, or an explicit seed pass); the first sweep plants the wrap-around candidates, the second reads the answers. Duplicates exist → answer by **index**, no value map.

## Problem (1-liner)
For each element of a **circular** array, return the first **strictly greater** element found scanning forward (wrapping past the end back to the start); `-1` if the element is the maximum.
**Example:** `[1,2,1]` → `[2,-1,2]` (the last `1` wraps around to find `2`); `[1,2,3,4,3]` → `[2,3,4,-1,4]`.

## Idea (1-liner)
A circular scan from position `i` sees at most the next `n-1` elements — exactly what a linear scan over the array **concatenated with itself** sees. So don't actually duplicate the array: run the usual right-to-left monotonic stack for `2n` steps with `i % n`. The first (right) half of the walk pre-loads the stack with every element's wrap-around candidates; the second half computes each `res[i]` against a stack that already "knows" the elements circularly to the right. Still O(n): each element is pushed/popped at most twice.

## Approach (canonical — one loop, 2n iterations)
1. Stack of **values**, strictly increasing top-to-bottom (candidates circularly to the right).
2. Loop `i` from `2n−1` down to `0`, acting on `num = nums[i % n]`.
3. Pop while `top <= num` — anything ≤ `num` is dominated by `num` for everyone further left.
4. Only when `i < n` (second sweep) record `res[i] = top` or `-1`. Push `num` every iteration.
5. The maximum of the array always empties its way to `-1` — at least one `-1` is guaranteed.

## Code (clean canonical)
```cpp
class Solution {
public:
    vector<int> nextGreaterElements(vector<int>& nums) {
        int n = nums.size();
        vector<int> res(n, -1);
        stack<int> st;                          // values, increasing top-to-bottom
        for (int i = 2 * n - 1; i >= 0; i--) {
            int num = nums[i % n];
            while (!st.empty() && st.top() <= num) st.pop();   // equal is not "greater"
            if (i < n) res[i] = st.empty() ? -1 : st.top();    // only the second sweep writes
            st.push(num);
        }
        return res;
    }
};
```
**TC:** O(n) — 2n iterations, each element pushed/popped ≤ 2 times.  **SC:** O(n) — stack (+ output).

## My Solution
**Two explicit right-to-left passes — correct, O(n), and equivalent to the canonical `2n` loop.** ✓ Pass 1 runs the pop-and-push over the whole array purely to *seed* the stack with the circular ("wrap-around") candidates; pass 2 repeats the scan, now reading `res[i]` from a stack that already contains everything circularly to the right. The `>=` pop is essential here (unlike 496, duplicates exist — `[1,2,2]`: the first `2` must not claim the second `2` as its "greater"). Two readable refinements available, neither changing complexity:
- Fold the two passes into the single `2n` loop with `i % n` and the `i < n` write-guard — the interview-standard form, half the code.
- Pass 1 doesn't need to track answers, only monotonicity — which your version already exploits. Fine as is; just be able to *say* it's the unrolled form of the modulo loop.

## Other Approaches
- **Brute force** — for each `i`, scan up to `n−1` steps forward with `(i+k) % n`. O(n²); at n = 10⁴ that's ~10⁸ checks — borderline TLE and misses the point.
- **Left-to-right `2n` loop** — indices on the stack; for `i` in `0…2n−1`, while `nums[i % n] > nums[st.top()]` pop `j` and set `res[j] = nums[i % n]`; only push when `i < n`. Same O(n); "resolve at pop time", the shape that generalizes to 739/84. Note the mirrored guard: push only in the **first** half (L→R) vs write only in the **second** half (R→L).
- **Actually doubling the array** — build `nums+nums`, run plain NGE, keep the first `n` answers. Works, O(n) extra memory and a copy; the `i % n` form is the same idea without paying for it.

## Pitfalls
- **Pop on `<=`, not `<`** — duplicates are real here (no uniqueness constraint, unlike 496). `[1,2,2]`: equal must not count as greater.
- **No value→answer map** — duplicates also kill the 496 hashmap trick; write answers **by index** directly.
- **Guard which half writes/pushes** — R→L `2n` loop: write `res` only when `i < n` (else the seeding half records garbage positions); L→R variant: *push* only when `i < n` (else elements enter twice and can self-answer). Getting the guard on the wrong half is the classic bug.
- **An element is never its own next-greater** — strictness (`<=` pop) plus the scan order already ensures this; "the max returns −1" is a quick sanity check on any output.
- **Don't physically wrap forever** — the search space is at most `n−1` steps; doubling the index range (`2n`) fully covers it. There is no need for a third pass.
- **`int` loop with `i % n`** — keep `i` as `int` and bounds as `2 * n - 1`; mixing `size_t` here invites the `i >= 0` infinite-loop bug on unsigned types.
