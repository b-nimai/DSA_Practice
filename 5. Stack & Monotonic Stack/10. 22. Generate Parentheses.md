# 10. Generate Parentheses (LC 22) — Medium
🔗 [Open on LeetCode](https://leetcode.com/problems/generate-parentheses/)

**Pattern:** Backtracking (the stack made implicit) — **build the string one paren at a time, guarding each choice so every prefix stays valid; the recursion's call stack *is* the stack.**
**Trigger:** "generate **all** well-formed combinations of `n` pairs" → enumerate, but prune the moment a prefix can't extend to a valid string → DFS with two counters (`open`/`close` remaining) + two guards = **constrained backtracking**.

## Problem (1-liner)
Given `n` pairs of parentheses, return **every** distinct well-formed string of length `2n` (each `(` matched, never a `)` before its opener).

**Examples:** `n=1` → `["()"]` · `n=3` → `["((()))","(()())","(())()","()(())","()()()"]` (Catalan(3) = 5 strings).

## Idea (1-liner)
Grow the string left→right. Track `open` = `(` still available and `close` = `)` still available. **Add `(`** whenever `open > 0`. **Add `)`** only when `close > open` — i.e. an unmatched `(` is still on the (implicit) stack. When both hit 0 the string is complete and valid → record it. The two guards make every leaf a valid answer, so there's **zero validity check at the base case**.

## Approach (canonical — DFS with two counters + backtrack)
1. State carried down: `open`, `close` (parens **remaining** to place), shared `string& str`, result `vector`.
2. **Base case:** `open == 0 && close == 0` → push `str`, return.
3. **Open branch (`open > 0`):** append `(`, recurse `(open-1, close)`, `pop_back()` to undo.
4. **Close branch (`close > open`):** append `)`, recurse `(open, close-1)`, `pop_back()` to undo.
   - Why `close > open`? Unmatched `(` currently in `str` = `close − open` (placed-opens minus placed-closes). A `)` is legal only when that's `> 0`.
5. Start from `str = ""`, `buildParenthesis(n, n, …)` — let the recursion place even the first `(`.

## Code (clean canonical)
```cpp
class Solution {
    void buildParenthesis(int open, int close, vector<string>& res, string& str) {
        if (open == 0 && close == 0) { res.push_back(str); return; }

        if (open > 0) {                 // can open while opens remain
            str += "(";
            buildParenthesis(open - 1, close, res, str);
            str.pop_back();             // backtrack
        }
        if (close > open) {             // close only with an unmatched '(' open
            str += ")";
            buildParenthesis(open, close - 1, res, str);
            str.pop_back();             // backtrack
        }
    }
public:
    vector<string> generateParenthesis(int n) {
        vector<string> res;
        string str = "";
        buildParenthesis(n, n, res, str);
        return res;
    }
};
```
**TC:** O(4ⁿ / √n) — the number of valid strings is the nth **Catalan** number ≈ 4ⁿ/(n^1.5), each built in O(n); the `close > open` prune kills invalid prefixes so it's **not** the naive 2^(2n).  **SC:** O(n) recursion depth + O(n) for `str` (output excluded).

## My Solution
**Same DFS, written `open < close` for the close guard** — equivalent to `close > open`, correct. The path to it (interview hints) hit three classic bugs worth remembering:
- **Dangling `return`** — `if (cond) push(str); return;` puts `return` *outside* the `if` (no braces), so the function returns after the first push and the recursion never explores. Wrap the body: `{ push; return; }`.
- **Inverted close guard** — wrote `open > close`. Since `(` is always spent first, `open` is never `>` `close`, so the `)` branch **never fires** → `res` comes back **empty**. The correct test is `close > open` (a.k.a. `open < close`).
- **No backtrack** — `str += "("` mutates the **shared** `string&`; without `pop_back()` the sibling `)` branch sees the already-grown string. Append → recurse → `pop_back()` (or pass `str` by value).
```cpp
if (open < close) {           // == close > open ; both correct
    str += ")";
    buildParenthesis(open, close - 1, res, str);
    str.pop_back();
}
```

## Family map
- **Validation → generation** → [[1-20-valid-parentheses]] *checks* one string with a stack; this *generates* every valid string. Both live on the invariant "a `)` needs a prior unmatched `(`" (`close > open` here = "stack non-empty" there).
- **Backtracking template** → append a choice → recurse → undo (`pop_back`). Same skeleton as subsets/permutations/combinations; here the "pruning" guards (`open>0`, `close>open`) are what keep it linear-per-leaf instead of generating-then-filtering.
- **Implicit vs explicit stack** → unlike [[9-394-decode-string]] (explicit two-stack parser), here the **call stack** holds the pending `(`; the recursion *is* the stack.
- **Not monotonic** → no ordering invariant / no pop-while like [[4-739-daily-temperatures]] / [[7-901-online-stock-span]]; popping here is just backtracking.

## Other Approaches
- **Generate-all-then-filter** — emit all 2^(2n) `(`/`)` strings, keep the valid ones via a balance check. Correct but exponentially wasteful; the two guards above prune at the source. Don't.
- **Pass `str` by value** — drop the `pop_back()` and take `string str` by copy; each branch gets its own copy. Simpler to reason about, but O(n) copy per node vs O(1) append+undo. The `&` + `pop_back` version is the one to memorize.
- **Closure-number / DP** — build size-`n` answers from `"(" + a + ")" + b` over all splits `a`-pairs inside, `b`-pairs after. Elegant Catalan recurrence; overkill for an interview unless asked.

## Pitfalls
- **Braces on the base case** — `if (open==0 && close==0) { push; return; }`; a naked `if (cond) push; return;` returns unconditionally and the recursion dies after one push.
- **Close guard is `close > open`, not `open > close`** — invert it and the `)` branch never runs → empty result. Equivalent correct forms: `close > open` or `open < close`.
- **Backtrack the shared string** — append → recurse → `pop_back()`; forgetting the undo corrupts the sibling branch (or pass by value instead).
- **Counters are *remaining*, not *used*** — start both at `n`, recurse with `open-1` / `close-1`; mixing "remaining" and "used" semantics flips every comparison.
- **No validity check needed at the leaf** — the guards guarantee every completed string is well-formed; if you feel the urge to re-validate at the base case, a guard is wrong.
- **`open < 0 || close < 0` guard is dead code** — with `open>0` and `close>open` the counters never go negative; harmless but unnecessary.
