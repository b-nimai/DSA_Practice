# 3. Evaluate Reverse Polish Notation (LC 150) — Medium
🔗 [Open on LeetCode](https://leetcode.com/problems/evaluate-reverse-polish-notation/)

**Pattern:** Stack — **evaluate a postfix expression: operands wait on a stack, an operator consumes the top two**
**Trigger:** "evaluate an expression with **no parentheses**, given in **postfix / Reverse Polish**" → operands accumulate, and an operator always acts on the **two most recent** operands → that "most recent first" is exactly LIFO → push numbers, on an operator pop two, compute, push the result back.

## Problem (1-liner)
Given tokens of a valid **postfix** arithmetic expression (operands + `+ - * /`), evaluate it and return the integer result. Division **truncates toward zero**; all intermediate values fit in a 32-bit int.
**Example:** `["2","1","+","3","*"]` → `((2+1)*3) = 9`. `["4","13","5","/","+"]` → `(4 + 13/5) = 6`.

## Idea (1-liner)
In postfix, an operator's two operands are always the **last two computed values**. A stack gives "last two" in O(1): push every number; when you hit an operator, **pop the second operand, then the first** (order matters for `-` and `/`), compute, and push the result so it can serve as an operand later. After one left-to-right pass the lone value on the stack is the answer.

## Approach
1. Stack of ints.
2. For each token: if it's an **operator**, pop `b` (top = right/second operand), then `a` (left/first operand); push `a op b`.
3. Else it's a number → `push(stoi(token))`.
4. **Order is critical:** the top of the stack is the *right* operand. `a - b` and `a / b`, not `b - a`. Getting it backwards passes `+`/`*` but fails `-`/`/`.
5. Final answer is the single remaining `st.top()`.

## Code (clean canonical)
```cpp
class Solution {
public:
    int evalRPN(vector<string>& tokens) {
        stack<int> st;
        for (string &s : tokens) {
            if (s == "+" || s == "-" || s == "*" || s == "/") {
                int b = st.top(); st.pop();      // top = right operand
                int a = st.top(); st.pop();      // next = left operand
                if      (s == "+") st.push(a + b);
                else if (s == "-") st.push(a - b);
                else if (s == "*") st.push(a * b);
                else               st.push(a / b);   // C++ int / truncates toward 0 ✓
            } else {
                st.push(stoi(s));                // handles negatives like "-11"
            }
        }
        return st.top();
    }
};
```
**TC:** O(n) — one pass, each token pushed/popped O(1).  **SC:** O(n) — stack holds the pending operands.

class Solution {

                else               st.push(num1 / num2);
            } else {

            }

    }

## Family map
- **Matching / parse-on-a-stack** (same pattern's other face) → [[1-20-valid-parentheses]] pushes *openers* and pops on a closer; RPN pushes *operands* and pops on an operator. Same "push pending state, resolve when the trigger token arrives" shape.
- **Augmented stack** → [[2-155-min-stack]] piggybacks an aggregate on each element; here the stack just holds raw operands.
- **Infix → postfix (Shunting-yard)** → RPN is the *output* of converting normal `a+b*c` infix; "Basic Calculator" (224/227) is the harder cousin that must handle precedence/parentheses *while* evaluating.
- **Monotonic stack** → [[4-daily-temperatures]] — different use of the same structure (next-greater spans), not expression eval.

## Other Approaches
- **`switch` on `s[0]` instead of the `||` chain** — dispatch on the single operator char. Cleaner, but you must still detect numbers by `s.size() > 1 || isdigit(s[0])`, **not** by `s[0] == '-'` (that misreads `"-11"` as the minus operator). Same O(n)/O(n).
- **`unordered_set<string>` of operators** — readable but slower than the 4-way `||` for just four fixed strings; no real gain.
- **Two-pointer / in-place on the vector** — overwrite `tokens` as a manual stack to save the explicit container. Micro-optimization only; the explicit stack is clearer.
- **Recursive eval from the right** — pop tokens from the back, recursing for each operand. Works but uses O(n) call-stack and is harder to read than the linear scan.

## Pitfalls
- **Operand order** — the **top** of the stack is the **right** operand. Use `a - b` / `a / b` where `b` was popped first. Swapping them silently passes `+`/`*` but breaks `-`/`/` — the classic RPN bug.
- **Don't classify operators by `s[0]`** — `"-11"`, `"-200"` are *negative numbers*. A `s[0]=='-'` (or `!isdigit(s[0])`) test misfires on them; compare the **whole string** or guard with `s.size() > 1`.
- **Division truncates toward zero, not floor** — the problem demands toward-zero. C++ `int/int` already does this (`-7/2 == -3`), so it's free in C++; in Python you'd need `int(a/b)`, since `//` floors (`-7//2 == -4`).
- **`stoi` only** — don't hand-roll parsing that forgets the leading `-`. `stoi` handles the sign.
- **No empty/edge guard needed** — the problem guarantees a *valid* expression, so the stack always has ≥2 operands at an operator and exactly one value at the end. (Defensive code would still guard `!empty()`.)
- **32-bit safety** — guaranteed all intermediates fit in `int`, so plain `int` is fine; no `long long` required here (unlike Min Stack's sentinel trick).
