# 1. Valid Parentheses (LC 20) — Easy
🔗 [Open on LeetCode](https://leetcode.com/problems/valid-parentheses/)

**Pattern:** Stack — **matching pairs / nesting** (the seed problem of the whole stack family)
**Trigger:** "are the brackets `()[]{}` balanced **and** correctly nested?" → a closer must cancel the **most recent** unmatched opener → **LIFO** → push openers, on a closer pop iff `top` is its matching opener, valid iff stack ends empty

## Problem (1-liner)
Given a string `s` of just `'(' ')' '{' '}' '[' ']'`, decide if it's valid: every opener is closed by the **same type**, in the **correct order**, and every closer has a matching opener.
**Examples:** `"()"` → true · `"()[]{}"` → true · `"(]"` → false · `"([])"` → true · `"([)]"` → false

## Idea (1-liner)
"Correct order" means the **last opened** bracket must be the **first closed** — that's a stack. Push every opener; when a closer arrives, the only opener that can legally match it is the one on top. If the top matches, pop it (the pair is resolved); if it doesn't match (or the stack is empty), the string is invalid. After the scan, a non-empty stack means unclosed openers remain.

## Approach
1. **Stack of chars** holding the still-unmatched openers.
2. **Single pass** over `s`:
   - **Opener** `(`, `{`, `[` ⇒ push it.
   - **Closer** ⇒ valid only if the stack is **non-empty** *and* `top` is the **matching opener**; then pop. Otherwise `return false` (wrong type, or a closer with nothing open).
3. **End:** return `st.empty()` — leftover openers (`"((("`) ⇒ false.
4. **Why `!st.empty()` guards matter:** a leading closer like `")"` would call `top()` on an empty stack (UB) without the check.

**Two ways to write the closer check** — both O(n):
- **Compare-on-pop** (your version): three `else if` lines, one per closer, that only pop when `top` is the right opener. No map, just char compares — slightly faster.
- **Map-the-closer**: an `unordered_map<char,char>` from closer → opener collapses the three branches into one: `if (st.empty() || st.top() != match[c]) return false; st.pop();`. Scales if bracket types grow.

## Code (clean canonical)
```cpp
class Solution {
public:
    bool isValid(string s) {
        if (s.size() & 1) return false;                       // odd length can't balance
        stack<char> st;
        unordered_map<char,char> match = {{')','('}, {'}','{'}, {']','['}};
        for (char c : s) {
            if (c == '(' || c == '{' || c == '[') {
                st.push(c);                                   // opener: remember it
            } else {                                          // closer
                if (st.empty() || st.top() != match[c]) return false;
                st.pop();                                     // matched pair resolved
            }
        }
        return st.empty();                                    // no openers left over
    }
};
```
**TC:** O(n) — each char pushed/popped at most once.  **SC:** O(n) — worst case all openers (`"((((("`).

## My Solution
```cpp
class Solution {
public:
    bool isValid(string s) {
        stack<char>st;
        for (int i = 0; i < s.size(); i++) {
            if (s[i] == '(' || s[i] == '{' || s[i] == '[') {
                st.push(s[i]);
            }
            else if (s[i] == ')' && !st.empty() && st.top() == '(') st.pop();
            else if (s[i] == '}' && !st.empty() && st.top() == '{') st.pop();
            else if (s[i] == ']' && !st.empty() && st.top() == '[') st.pop();
            else return false;
        }
        return st.empty();
    }
};
```
**Verdict:** correct and clean — the compare-on-pop style is arguably *faster* than the map version (plain char compares, no hashing). Possible polish: add the `s.size() & 1` early-exit, and a range-based `for (char c : s)` since the index `i` isn't used for anything else.

## Other Approaches
- **Map-the-closer** (shown in canonical) — one branch instead of three; same O(n)/O(n).
- **String-as-stack / repeated replace** — keep deleting `"()"`, `"[]"`, `"{}"` until none remain; valid iff the string empties. Elegant but O(n²) (each pass rescans) — rejected for the linear stack.
- **Counter only** (track counts of each bracket) — **wrong**: counts can't detect *order*, so `"([)]"` passes a pure-count check. A stack is mandatory because nesting order matters.

## Pitfalls
- **Call `top()` on an empty stack** = undefined behaviour. Every closer branch must guard `!st.empty()` first — a leading/excess closer (`")"`, `"(]"` after the `(` pops) hits this.
- **Forgetting the final `empty()` check** — `"((("` never returns false mid-loop; the only signal is the leftover stack. Returning `true` unconditionally at the end is the classic bug.
- **Matching type, not just "a closer closed something"** — `"(]"` must fail: the top is `(` but the closer is `]`. The `top() == matching-opener` test (not merely "stack non-empty") is what enforces *same type*.
- **Odd length is an instant false** — a balanced string has even length; `if (s.size() & 1) return false;` skips all the work for guaranteed-invalid inputs (minor, but free).
- **Pattern seed:** this is the prototype for every "push state, pop on the closing token" parser — [[3-evaluate-reverse-polish-notation]], [[8-simplify-path]], [[9-decode-string]]. Lock in the LIFO + *match-the-top* invariant; it transfers directly.
