# 4. Daily Temperatures (LC 739) — Medium
🔗 [Open on LeetCode](https://leetcode.com/problems/daily-temperatures/)

**Pattern:** Monotonic Stack — **next-greater-element: stack of *indices*, kept temperature-decreasing**
**Trigger:** "for each day, how many days until a **strictly warmer** one?" = **next greater element**, but you need the **distance** not the value → keep a stack of **indices** whose temperatures are decreasing; the gap `i - j` is the answer.

## Problem (1-liner)
For each day `i`, return how many days you wait for the **next strictly warmer** temperature; `0` if none ever comes.
**Example:** `[73,74,75,71,69,72,76,73]` → `[1,1,4,2,1,1,0,0]`.

## Idea (1-liner)
A brute force compares every pair → O(n²). The insight: as you scan, a day stays "unanswered" only until the **first** later day that's warmer — so keep the unanswered days on a stack with **decreasing** temperatures. When a warmer day arrives, it resolves *all* the shorter days on top in one sweep, and each index is pushed/popped once → O(n). Store **indices** (not temps) because the answer is a **distance** `i - j`.

## Approach (canonical — left-to-right)
1. `res` all zeros (default = "no warmer day"). Stack of **indices**, temperatures strictly decreasing top-to-bottom.
2. For each `i`: while the stack's top day is **colder than** `T[i]` (`T[i] > T[st.top()]`), it just found its warmer day → pop `j`, set `res[j] = i - j`.
3. Push `i` (it's now the newest unanswered day).
4. Days never resolved stay `0`. Each index enters/leaves the stack once ⇒ O(n).

## Code (clean canonical)
```cpp
class Solution {
public:
    vector<int> dailyTemperatures(vector<int>& T) {
        int n = T.size();
        vector<int> res(n, 0);
        stack<int> st;                                   // indices waiting for a warmer day; temps decreasing
        for (int i = 0; i < n; i++) {
            while (!st.empty() && T[i] > T[st.top()]) {  // i is the warmer day j was waiting for
                int j = st.top(); st.pop();
                res[j] = i - j;                          // distance, not value
            }
            st.push(i);
        }
        return res;
    }
};
```
**TC:** O(n) — each index pushed and popped at most once.  **SC:** O(n) — stack + result.

## My Solution
**Right-to-left variant — correct, refined to store only indices.** ✓ Scan from the end; pop every day `<= T[j]` (keep only strictly-warmer future days), so the top is the next warmer day and `top - j` is the wait. Equivalent O(n)/O(n); originally used `pair<int,int>` and recomputed `n-1-i` repeatedly — refined to index-only and a downward loop.
```cpp
class Solution {
public:
    vector<int> dailyTemperatures(vector<int>& T) {
        int n = T.size();
        vector<int> res(n, 0);
        stack<int> st;                                   // indices of strictly-warmer future days
        for (int j = n - 1; j >= 0; j--) {
            while (!st.empty() && T[st.top()] <= T[j]) st.pop();   // <= : equal temp isn't "warmer"
            if (!st.empty()) res[j] = st.top() - j;      // distance to next warmer day
            st.push(j);
        }
        return res;
    }
};
```
**Left-to-right vs right-to-left:** both O(n). Left-to-right resolves each day *when its answer becomes known* and is the textbook "next greater element" template (generalizes best — memorize it as the default). Right-to-left discards days that can never be an answer (already-seen and not-warmer) before reading the top.

## Other Approaches
- **Brute force** — for each `i`, scan forward to the first warmer day. O(n²), TLE-risk at n = 10⁵.
- **Reverse DP / jump table** — scan right-to-left; for a colder neighbor, *jump* using already-computed answers instead of a stack: `j += res[j]` until you find a warmer day or fall off. O(n) amortized, **O(1) extra space** (besides output). Slick but trickier to derive than the stack.
- **Bucketed "next warmer index" array** — exploit the tiny value range (30–100): keep `nextWarmer[temp]`; for each `i` from the right, the answer is the min index over hotter buckets. O(100·n) — only worth it for the bounded-range trivia.

## Pitfalls
- **Store indices, not temperatures** — the answer is a *distance* (`i - j`); from a temperature alone you can't recover the gap. (Original solution carried a `pair` to do this; index-only is leaner.)
- **Strict vs non-strict** — "warmer" is **strict**. Left-to-right pop on `T[i] > T[st.top()]`; right-to-left pop on `T[st.top()] <= T[j]`. Equal temperatures must **not** resolve a day — getting the `<`/`<=` boundary wrong is the classic bug.
- **Default to 0, don't overwrite** — initialize `res` to 0 and only set it when a warmer day is found; days left on the stack correctly stay 0.
- **Decreasing vs increasing stack** — for *next-greater* the stack stays **decreasing**; flipping the comparison silently solves the wrong problem (next-smaller).
- **`top()`/`pop()` on empty is UB** — always guard `!st.empty()` first.
- **Index bookkeeping in the reverse scan** — looping the index directly (`for j = n-1 … 0`) avoids recomputing `n-1-i` and the off-by-one bugs that come with it.
