# 5. Next Greater Element I (LC 496) — Easy
🔗 [Open on LeetCode](https://leetcode.com/problems/next-greater-element-i/)

**Pattern:** Monotonic Stack — **the textbook next-greater-element template, plus a hashmap to answer queries by value**
**Trigger:** "for each element, what's the **first greater element to its right**?" — and here queries come from a *second* array → precompute next-greater for **all** of `nums2` with a monotonic stack, store in a `value → answer` map (values are unique!), then answer each `nums1[i]` by lookup.

## Problem (1-liner)
`nums1` is a subset of `nums2` (all values unique). For each `x` in `nums1`, find `x` in `nums2` and return the **first element greater than it to its right**; `-1` if none.
**Example:** `nums1 = [4,1,2]`, `nums2 = [1,3,4,2]` → `[-1,3,-1]` (nothing right of 4 is bigger; right of 1 the first bigger is 3; nothing right of 2).

## Idea (1-liner)
Don't answer queries one-by-one (that's O(n1·n2)) — **invert it**: compute the next-greater answer for *every* element of `nums2` in one O(n2) monotonic-stack pass, drop the results in a hashmap keyed by value (legal only because **all values are unique**), then each `nums1` query is an O(1) lookup. This "precompute on the superset, look up by key" decomposition is the whole trick; the stack part is identical to [[4-739-daily-temperatures]].

## Approach (canonical — right-to-left)
1. Scan `nums2` **from the right**, keeping a stack whose values are **strictly increasing top-to-bottom** (every element on it is a *candidate* next-greater for things further left).
2. For the current `num`: pop while `top <= num` — anything ≤ `num` can never be the answer for an element left of `num`, because `num` itself stands in front of it and is bigger-or-equal.
3. After popping, the top (if any) **is** the next greater element of `num` → record `nextGreater[num] = top` (else `-1`). Push `num`.
4. Map each `nums1[i]` through the map. Each element of `nums2` is pushed/popped once ⇒ **O(n1 + n2)** — exactly the follow-up's ask.

## Code (clean canonical)
```cpp
class Solution {
public:
    vector<int> nextGreaterElement(vector<int>& nums1, vector<int>& nums2) {
        unordered_map<int, int> nextGreater;   // value -> its NGE (values unique, so keying by value is safe)
        stack<int> st;                          // candidates to the right, increasing top-to-bottom
        for (int i = nums2.size() - 1; i >= 0; i--) {
            int num = nums2[i];
            while (!st.empty() && st.top() <= num) st.pop();   // can't be NGE for anyone left of num
            nextGreater[num] = st.empty() ? -1 : st.top();
            st.push(num);
        }
        vector<int> res(nums1.size());
        for (int i = 0; i < (int)nums1.size(); i++) res[i] = nextGreater[nums1[i]];
        return res;
    }
};
```
**TC:** O(n1 + n2) — one stack pass + one lookup pass.  **SC:** O(n2) — map + stack.

## My Solution
**Right-to-left monotonic stack + hashmap — correct and already the optimal O(n1+n2) answer to the follow-up.** ✓ Same as the canonical above. Two micro-refinements applied: dropped the redundant `-1` fill of `result` (every slot is overwritten by the map lookup anyway), and noted that since values are unique, `>=` vs `>` in the pop condition never differs here — but `top <= num` is the right *invariant* to memorize ("an equal element can never be anyone's *greater*"), and it's what keeps the template correct when duplicates exist (e.g. Daily Temperatures, Stock Span).

## Other Approaches
- **Brute force** — for each `nums1[i]`, find it in `nums2`, scan right for something bigger. O(n1·n2); passes at n ≤ 1000 but fails the follow-up.
- **Left-to-right stack** — push values; while `num > top`, the popped value's NGE is `num` (`nextGreater[popped] = num`). Same O(n), but elements left on the stack get **no map entry**, so lookups need a `find()`-with-default. The right-to-left form writes an explicit entry for every element — cleaner here. Know both: left-to-right ("resolve at pop time") is the shape Daily Temperatures and Histogram need.
- **No-map variant** — answer each query with its own rightward scan from `x`'s position. Simpler code, but O(n1·n2) again; the map is what decouples "compute" from "query".

## Pitfalls
- **Pop on `<=`, not `<`** — an element *equal* to `num` is not "greater" for anything to the left, and `num` (closer) dominates it anyway. Unique values hide this bug here; duplicates elsewhere expose it.
- **Keying the map by value only works because values are unique** — with duplicates you must key by index (as in 739). Check that constraint before reusing this exact shape.
- **Don't search `nums2` per query** — the inner `find` is what makes the naive solution O(n1·n2); precompute once, look up many.
- **Values vs indices on the stack** — answer is a *value* → values suffice; the moment the answer is a distance/span, switch to indices.
- **`top()` on empty is UB** — record the answer with the `st.empty() ? -1 : st.top()` guard *before* pushing `num`.
- **Initialize-then-overwrite** — filling `res` with `-1` and then assigning every slot is harmless but redundant; the `-1`s already live in the map.
