# 11. Largest Rectangle in Histogram (LC 84) — Hard
🔗 [Open on LeetCode](https://leetcode.com/problems/largest-rectangle-in-histogram/)

**Pattern:** Monotonic Stack — **for each bar, the widest span it can rule = previous-smaller boundary → next-smaller boundary**
**Trigger:** "fixed-width bars; biggest axis-aligned rectangle?" Each bar `h[i]` can be the **shortest** bar of some rectangle; that rectangle stretches left until a **strictly smaller** bar and right until a **strictly smaller** bar → `area = h[i] * (right − left − 1)`. Find both boundaries for every bar in one pass with an **increasing** monotonic stack.

## Problem (1-liner)
Given `heights[]` (each bar width 1), return the **maximum area** of a rectangle that fits entirely under the histogram outline.

**Example:** `[2,1,5,6,2,3]` → `10`. The bars `5,6` give `2×5=10`: bar `5` (index 2) extends right through `6` until the `2` at index 4, and left until the `1` at index 1 → width `4−1−1 = 2`, height `5`.

## Idea (1-liner)
Anchor on **height**: every maximal rectangle is exactly as tall as its **shortest** bar, so try each bar as that shortest one and ask *how wide can it go?* It widens left and right until it hits a bar **shorter** than itself. So for each `i` we need its **previous smaller** index `L` and **next smaller** index `R`; width = `R − L − 1`. A single **increasing** monotonic stack computes both: when bar `i` pops bar `j`, `i` is `j`'s next-smaller (right wall) and the new stack top is `j`'s previous-smaller (left wall).

## Approach (canonical — index stack + sentinel)
1. Keep a stack of **indices** whose heights are **strictly increasing** bottom→top.
2. Iterate `i` from `0` to `n` (one past the end). Treat `h = (i==n) ? 0 : heights[i]` — the trailing **0 sentinel** is shorter than everything, so it flushes the whole stack at the end (no separate drain loop).
3. While the stack is non-empty **and** `heights[top] >= h`: pop `j`. Bar `j`'s rectangle is now fully bounded:
   - **right wall** = `i` (first bar `≤ h[j]` to the right),
   - **left wall** = new `top` after popping (or `−1` if stack empties),
   - `width = stack.empty() ? i : i − stack.top() − 1`, `area = heights[j] * width`.
4. Push `i`. Each index is pushed once and popped once ⇒ **O(n)**.

## Code (clean canonical)
```cpp
int largestRectangleArea(vector<int>& heights) {
    int n = heights.size(), maxArea = 0;
    stack<int> st;                                  // indices, heights strictly increasing
    for (int i = 0; i <= n; i++) {
        int h = (i == n) ? 0 : heights[i];          // sentinel 0 drains the stack
        while (!st.empty() && heights[st.top()] >= h) {
            int height = heights[st.top()]; st.pop();
            int width = st.empty() ? i : i - st.top() - 1;
            maxArea = max(maxArea, height * width);
        }
        st.push(i);
    }
    return maxArea;
}
```
**TC:** O(n) — each index pushed/popped once.  **SC:** O(n) — stack (worst case strictly increasing heights).

## My Solution
**The "carry the left index" merge variant — correct, and now refined.** Original passed all tests but had a redundant `if` guarding a `while`, a duplicated final drain loop, and stored the value alongside the index. The merge idea is good though: instead of reading the left wall off the new stack top, you **carry the leftmost index** the new shorter bar can extend back to, and store `{startIndex, height}`.

**Refinements applied:**
- **Sentinel `0` iteration** (`i <= size`) deletes the duplicated post-loop drain — the area formula is written **once**.
- Initialize `start = i` and let the `while` overwrite it ⇒ the outer `if/else` and its duplicated `push` collapse into one path.

```cpp
int largestRectangleArea(vector<int>& heights) {
    int maxArea = 0;
    stack<pair<int,int>> st;                        // {startIndex, height}
    for (int i = 0; i <= (int)heights.size(); i++) {
        int h = (i == (int)heights.size()) ? 0 : heights[i];
        int start = i;                              // how far left this height can extend
        while (!st.empty() && st.top().second >= h) {
            auto [idx, ht] = st.top(); st.pop();
            maxArea = max(maxArea, (i - idx) * ht); // i = next-smaller (right wall)
            start = idx;                            // absorb the popped bar's left reach
        }
        st.push({start, h});
    }
    return maxArea;
}
```
**Carry-index vs read-the-new-top:** identical O(n)/O(n). The canonical `i - st.top() - 1` keeps the stack to bare indices and matches the sibling problems' "prev-smaller / next-smaller width" formula — memorize that one. Your merge variant is intuitive (you *see* the rectangle growing leftward) and is the natural way most people first derive it.

## Other Approaches
- **Brute force O(n²)** — for each `i`, expand left/right while bars `≥ heights[i]`, area = `heights[i] × span`. Conceptually the "each bar as the shortest" idea without the stack; TLEs at n = 10⁵.
- **Divide & conquer O(n log n) avg** — best rectangle either uses the global-min bar across the whole width, or lies entirely in the left/right half; recurse. Worst case O(n²) on sorted input; needs a sparse table to stay log-linear.
- **Two precomputed boundary arrays** — fill `prevSmaller[]` and `nextSmaller[]` in two separate stack passes, then `area = h[i] * (next[i] − prev[i] − 1)`. Same O(n), more explicit/teachable, but two passes + two arrays vs the one-pass stack.

## Pitfalls
- **`heights[top] >= h` uses `≥`, not `>`** — on **equal** heights you must still pop. Popping on `>` only can miscompute width for runs of equal bars; `≥` keeps the *leftmost* equal bar as the surviving boundary, and the equal bar that "should" have measured wider gets its full width when the shorter sentinel/next bar pops it. (`[2,2]` → must give 4.)
- **Forgetting the sentinel / final drain** — bars still on the stack at the end never met a right wall; without the trailing `0` (or an explicit drain loop) you lose the tallest monotone-increasing suffix (e.g. `[1,2,3,4]`).
- **Width when the stack empties** — if popping clears the stack, the popped bar extends all the way left from index 0 ⇒ `width = i`, **not** `i - top - 1` (there's no left wall). Guard with `st.empty() ? i : ...`.
- **`int` overflow** — fine here (`10⁵ × 10⁴ = 10⁹` < INT_MAX ≈ 2.1×10⁹), but it's *close*; bump to `long long` if constraints grow.
- **`top()`/`pop()` on empty is UB** — guard `!st.empty()` first, always.
- **Off-by-one in the merge variant** — the carried `start` must be the **popped** bar's start, set *inside* the loop; setting it once before the loop or to `i-1` silently truncates wide rectangles.
