# Tree BFS тАФ Index

60-second whole-pattern recall. Read the **Trigger** column and try to recall the **Idea** before peeking.

**Core skeleton:** `q.push(root); while(!q.empty()){ int sz = q.size(); for(sz times){ pop; collect; push non-null children; } record level; }` тАФ the load-bearing line is `int sz = q.size()` **snapshotted before the inner loop**: at that instant the queue *is* exactly one level. Every problem below is this skeleton with a different "what do I collect per level."

## When to reach for Tree BFS
1. The question is about **levels / depth-grouping** тАФ "level order", "zigzag", "right side view", "level averages", "bottom-up".
2. You need the **minimum depth** or shortest path in an unweighted tree тАФ BFS short-circuits at the **first leaf** (DFS would explore everything).
3. You need **per-level** aggregation (sum/max/count). Reach for **Tree DFS** instead when the answer combines info from *both subtrees* post-order rather than per level.

## Common Pitfalls Across the Pattern
- **Snapshot the level size into a local before the inner loop** тАФ never read `q.size()` in the loop condition; it grows as children are pushed and levels merge. This is *the* Tree BFS bug.
- **Empty tree guard** тАФ return the empty result for `!root`; don't push a null root.
- **Push only non-null children** (unless the problem needs null markers, e.g. serialize) so the size snapshot equals the real level width.
- **Left-before-right** child order тАФ reversing silently mirrors every level.
