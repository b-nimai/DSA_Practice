# 11. 3Sum Closest (LC 16) — Medium
🔗 [Open on LeetCode](https://leetcode.com/problems/3sum-closest/)

**Pattern:** Two Pointers — sort + outer loop + inner two-pointer (track best, no exact target)
**Trigger:** triplet sum *closest* to a target; minimise `|target - sum|`

## Problem (1-liner)
Pick 3 numbers at distinct indices whose sum is **closest** to `target`; return that sum. Exactly one optimal answer.
**Example:** `nums = [-1,2,1,-4]`, `target = 1` → `2` (`-1 + 2 + 1`)

## Idea (1-liner)
Same skeleton as 3Sum, but instead of comparing `sum` to `0` to record/skip, keep the running best `|target - sum|`; move pointers by `sum < target` (grow) / `sum > target` (shrink); `sum == target` is the global best — return immediately.

## Approach
1. Sort `nums`. Init `closest` to any valid triple (e.g. `nums[0]+nums[1]+nums[2]`), or a large sentinel.
2. For `i = 0..n-3`:
   - `l = i+1`, `r = n-1`.
   - While `l < r`:
     - `sum = nums[i] + nums[l] + nums[r]`.
     - If `abs(target - sum) < abs(target - closest)` → `closest = sum`.
     - `sum < target` → `l++`  ·  `sum > target` → `r--`  ·  else `return sum`.
3. Return `closest`.

## Code (clean canonical)
```cpp
int threeSumClosest(vector<int>& nums, int target) {
    sort(nums.begin(), nums.end());
    int n = nums.size();
    int closest = nums[0] + nums[1] + nums[2];
    for (int i = 0; i < n - 2; i++) {
        int l = i + 1, r = n - 1;
        while (l < r) {
            int sum = nums[i] + nums[l] + nums[r];
            if (abs(target - sum) < abs(target - closest)) closest = sum;
            if (sum < target) l++;
            else if (sum > target) r--;
            else return sum;          // exact hit — can't do better
        }
    }
    return closest;
}
```
**TC:** O(n²)  **SC:** O(1) extra (sort uses O(log n) stack)

## My Solution
```cpp
class Solution {
public:
    int threeSumClosest(vector<int>& nums, int target) {
        sort(nums.begin(), nums.end());
        int closest = INT_MAX / 2;
        for(int i = 0; i < nums.size() - 2; i++) {
          int start = i + 1, last = nums.size() - 1;
          while(start < last) {
            int sum = nums[i] + nums[start] + nums[last];
            if(abs(target - sum) < abs(target - closest)) {
              closest = sum;
            }

            if(sum < target) start++;
            else if(sum > target) last--;
            else return sum;
          }
        }
        return closest;
    }
};
```

> **Bug I had first:** moved the pointers by `closest < target` / `closest > target` instead of `sum`. `closest` is the *best so far across all iterations* — it says nothing about whether the *current* triple is too small or too big, so the pointers wandered the wrong way and skipped the real answer. The move decision must be driven by the triple you just looked at.

## Other Approaches
- **Brute force:** all triplets, track min `|target - sum|` → O(n³) / O(1)
- Same two-pointer with a small pruning: if `nums[i]+nums[i+1]+nums[i+2] > target` you can break; if `nums[i]+nums[n-2]+nums[n-1] < target` you can `continue`.

## Pitfalls
- Move pointers by **`sum` vs `target`**, never by `closest` vs `target`.
- Initialise `closest` to a *real* triple sum (or `INT_MAX/2`) — not `INT_MAX` (overflow in `abs(target - closest)`).
- `nums.size()` is unsigned; `size - 2` underflows if `n < 2`. Constraints give `n ≥ 3`, but `int n = nums.size();` then `i < n - 2` is the safe habit.
- No duplicate-skipping needed — we return a sum, not a list of triples.
- `sum == target` → return now; nothing beats distance 0.
