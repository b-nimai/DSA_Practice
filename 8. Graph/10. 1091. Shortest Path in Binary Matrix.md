# 10. 1091. Shortest Path in Binary Matrix (LC 1091) — Medium
🔗 [Open on LeetCode](https://leetcode.com/problems/shortest-path-in-binary-matrix/)

**Pattern:** Graph BFS — **single-source shortest path in an unweighted grid, 8-directional** (the "fewest cells/steps from corner to corner" seed)
**Trigger:** "shortest clear path top-left → bottom-right, move in **8 directions**, count cells visited" → unweighted shortest path ⇒ **plain BFS**; the first time you *pop* the target, that distance is the answer

## Problem (1-liner)
`n×n` binary grid. A clear path goes `(0,0) → (n-1,n-1)` over only `0`-cells, stepping **8-directionally** (edges *and* corners). Return the **number of cells** on the shortest such path, or `-1` if none exists.
**Examples:** `[[0,1],[1,0]]` → `2` (diagonal) · `[[0,0,0],[1,1,0],[1,1,0]]` → `4` · `[[1,0,0],...]` → `-1` (start blocked).
**Constraints:** `1 ≤ n ≤ 100`; cells are `0`/`1`. **Edge:** if `grid[0][0]` or `grid[n-1][n-1]` is `1`, answer is `-1` immediately.

## Idea (1-liner)
Fewest cells on a path in an **unweighted** graph = **BFS** by distance. Seed the queue with `(0,0)` at distance `1`, expand all **8** neighbours ring by ring; the first time the target is dequeued its stored distance is the shortest. No path ⇒ queue drains ⇒ `-1`.

> **Load-bearing insight:** length counts **cells, not moves**, so start at `1` (not `0`). Because the grid is unweighted, BFS is optimal — DFS/backtracking would enumerate *every* path and blow up exponentially.

## Approach
1. **Guard the corners:** `grid[0][0]` or `grid[n-1][n-1]` blocked → return `-1`.
2. **Seed BFS:** push `{(0,0), 1}`; mark `visited[0][0]`.
3. **Expand:** pop `{(r,c), d}`; if it's the target `(n-1,n-1)` return `d`. Else for each of **8** directions, if the neighbour is in-bounds, `0`, and unvisited → **mark visited on push** and enqueue `{nbr, d+1}`.
4. **Drained without hitting target** → return `-1`.

> **Mark visited at push, not pop** — the standard BFS discipline; otherwise a cell gets enqueued by several neighbours and the queue bloats (redundant work / TLE).

## Code (clean canonical)
```cpp
class Solution {
public:
    int shortestPathBinaryMatrix(vector<vector<int>>& grid) {
        int n = grid.size();
        if (grid[0][0] || grid[n-1][n-1]) return -1;

        vector<vector<int>> visited(n, vector<int>(n, 0));
        queue<pair<pair<int,int>,int>> q;          // {{r,c}, dist(cells so far)}
        q.push({{0, 0}, 1});
        visited[0][0] = 1;

        int dr[] = {-1,-1,-1, 0,0, 1,1,1};
        int dc[] = {-1, 0, 1,-1,1,-1,0,1};         // all 8 directions

        while (!q.empty()) {
            auto [cell, d] = q.front(); q.pop();
            auto [r, c] = cell;
            if (r == n-1 && c == n-1) return d;    // first pop of target = shortest

            for (int i = 0; i < 8; ++i) {
                int nr = r + dr[i], nc = c + dc[i];
                if (nr < 0 || nr >= n || nc < 0 || nc >= n) continue;
                if (visited[nr][nc] || grid[nr][nc]) continue;
                visited[nr][nc] = 1;               // mark on push
                q.push({{nr, nc}, d + 1});
            }
        }
        return -1;
    }
};
```
**TC:** O(n²) — each cell enqueued once, 8 neighbours each.  **SC:** O(n²) — `visited` + queue.

## My Solution
Your **BFS (first block)** is correct and optimal — same shape as above, with a tidy `valid()` lambda. Nothing to fix; only cosmetics (`array<int,8>` over C-arrays, or fold the target check where you dequeue — you already do). ✅

Your **DFS + backtracking (second block)** is **logically correct but TLEs**. It explores *every* path and takes the `min`, undoing `visited` on the way out (`visited[row][col] = false`). That's exponential — roughly **O(8^(n²))** — and with `n ≤ 100` any moderately open grid (even ~20×20 all-zero) hangs. There's **no memo rescue**: the shortest distance to a cell depends on the visited set along the path, so you can't cache it. Shortest path on an unweighted grid is inherently a **BFS** problem — keep the BFS, drop the DFS.

## Other Approaches
- **Plain BFS** *(optimal)* — the canonical block. O(n²) / O(n²). The default for "fewest steps/cells, unweighted."
- **BFS marking distance in-place** — overwrite `grid[r][c]` (or a `dist` grid) instead of carrying `d` in the queue; saves the pair-of-pair. Same O(n²), a touch cleaner, but mutates input.
- **A\*** (heuristic = Chebyshev distance `max(|dr|,|dc|)` to target) — same worst case but expands fewer cells in practice on large open grids; overkill for `n ≤ 100`, worth naming as the "can you do better than BFS?" answer.
- **DFS + backtracking** — correct but **exponential ⇒ TLE**. Never use backtracking for shortest path on an unweighted graph. (This is your second solution.)
- **Bidirectional BFS** — search from both corners, meet in the middle; smaller frontier on large grids. Nice to mention, rarely needed here.

## Pitfalls
- **DFS/backtracking is the wrong tool** — shortest path + unweighted ⇒ BFS. Backtracking = exponential TLE, and no memo fixes it.
- **Length counts cells, not edges** — start distance at `1`, so the single-cell grid `[[0]]` returns `1`.
- **8 directions, not 4** — include all diagonals (`dr/dc` of length 8), unlike the 4-dir flood-fill problems.
- **Guard the endpoints up front** — blocked start *or* end ⇒ `-1` before touching the queue.
- **Mark visited on push, not on pop** — else the same cell is enqueued many times (bloat / TLE).
- **Bounds before value** — check `nr/nc` in range before reading `grid[nr][nc]`.
- **Return distance only on *dequeue* of the target** — with per-cell BFS that first pop is guaranteed shortest.
