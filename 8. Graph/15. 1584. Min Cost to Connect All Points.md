# 15. 1584. Min Cost to Connect All Points (LC 1584) — Medium
🔗 [Open on LeetCode](https://leetcode.com/problems/min-cost-to-connect-all-points/)

**Pattern:** Weighted Graph — **connect *all* nodes at minimum total cost** ⇒ **MST (Prim / Kruskal)**. *Not* a shortest-path problem.
**Trigger:** "make **all** points connected for the **minimum total cost**", "exactly one simple path between any two" (= a **tree**). No `src`/`dst` ⇒ nothing to shortest-path *to* ⇒ **MST**, not Dijkstra. Edges aren't given — the graph is **complete and implicit**, weight `(i,j)` = Manhattan distance.

## Problem (1-liner)
Given `points[i] = [xi, yi]`, cost of joining two points = `|xi-xj| + |yi-yj|`. Return the **minimum total cost** to connect all points into one tree.
**Examples:** `[[0,0],[2,2],[3,10],[5,2],[7,0]]` → `20` · `[[3,12],[-2,5],[-4,1]]` → `18` (`|3-(-2)|+|12-5| = 12`, `|-2-(-4)|+|5-1| = 6`).
**Constraints:** `1 ≤ n ≤ 1000`, `-10⁶ ≤ xi, yi ≤ 10⁶`, all points distinct. **Edge:** `n = 1` ⇒ `0` (already connected, nothing to join).

## Idea (1-liner)
Every pair of points is an edge ⇒ build the **complete graph** (`n(n-1)/2` edges, Manhattan weights), then run **Prim**: repeatedly absorb the **cheapest edge leaving the tree** until all `n` nodes are in. An MST on `n` nodes has exactly `n-1` edges; their sum is the answer.

> **Load-bearing insight:** the *nodes* are **indices**, not coordinates. `points[i]` is just node `i`'s label; the coordinates only exist to produce the weight. Once you see that, this is the same Prim you'd run on any given edge list.
> **Why MST and not Dijkstra:** Dijkstra minimises the distance **from one source to each node**; MST minimises the **total edge cost of the whole tree**. "Connect everything cheaply" has no source — the shortest-path tree is generally *not* the MST.

## Approach
1. **Build the graph:** for `i < j`, `w = |xi-xj| + |yi-yj|`; push `{j,w}` into `adjList[i]` **and** `{i,w}` into `adjList[j]` (undirected). `j = i+1` avoids self-loops and duplicates.
2. `visited[n] = false`, min-heap of `{weight, node}`, `total = 0`. Push `{0, 0}` — start anywhere, cost 0 to add the first node.
3. Pop the cheapest `{dis, node}`. **If `visited[node]` → `continue`** (stale entry, already absorbed).
4. Mark visited, `total += dis`, and push every unvisited neighbour `{w, nbr}`.
5. Loop until the heap drains; return `total`.

> **Why the heap is keyed by edge weight, not path cost:** in Dijkstra you push `dist[u] + w` (cumulative). In Prim you push **`w` alone** — you only care what it costs to *attach* this node to the tree, never how you got there. That one-token difference is Prim vs Dijkstra.

## Code (clean canonical)
```cpp
class Solution {
public:
    int minCostConnectPoints(vector<vector<int>>& points) {
        int n = points.size();

        vector<vector<pair<int,int>>> adjList(n);           // node = index, not coordinate
        for (int i = 0; i < n; i++) {
            int x1 = points[i][0], y1 = points[i][1];
            for (int j = i + 1; j < n; j++) {               // j = i+1 → each pair once
                int x2 = points[j][0], y2 = points[j][1];
                int dis = abs(x1 - x2) + abs(y1 - y2);
                adjList[i].push_back({j, dis});
                adjList[j].push_back({i, dis});             // undirected → both ways
            }
        }

        // Prim's Algorithm
        vector<bool> visited(n, false);
        priority_queue<pair<int,int>, vector<pair<int,int>>, greater<>> pq;   // {weight, node}
        int total = 0;
        pq.push({0, 0});                                    // start anywhere, free

        while (!pq.empty()) {
            auto [dis, node] = pq.top(); pq.pop();
            if (visited[node]) continue;                    // stale duplicate
            visited[node] = true;
            total += dis;

            for (auto [nbr, w] : adjList[node])
                if (!visited[nbr]) pq.push({w, nbr});       // push w, NOT dis + w
        }
        return total;
    }
};
```
**TC:** O(n² log n) — `n²` edges, each pushed once. **SC:** O(n²) — the adjacency list dominates.

**Dry run** (`[[3,12],[-2,5],[-4,1]]`, answer `18`): weights `0-1 = 12`, `0-2 = 18`, `1-2 = 6`.

| pop | visited? | total | pushed |
|---|---|---|---|
| `{0,0}` | no → take | `0` | `{12,1}`, `{18,2}` |
| `{12,1}` | no → take | `12` | `{6,2}` |
| `{6,2}` | no → take | **`18`** | — |
| `{18,2}` | **yes → skip** | `18` | — (the stale entry the `continue` exists for) |

## My Solution
This *is* my solution — verified: `20`, `18`, `0` on the samples, and `n=1000` in **86 ms**. Correct as written, nothing to fix.

**One refinement worth knowing:** the adjacency list is `n(n-1)/2 = 499,500` edges (~8 MB) at `n = 1000` — and it buys nothing, since any weight is O(1) recomputable from the coordinates. **Dense Prim** drops it entirely:

```cpp
// O(n²) time, O(n) space — no graph built at all
int n = points.size();
vector<int> minDist(n, INT_MAX);        // cheapest edge from tree → v
vector<bool> inMST(n, false);
minDist[0] = 0;
int total = 0;
for (int it = 0; it < n; ++it) {
    int u = -1;                                              // closest node outside the tree
    for (int v = 0; v < n; ++v)
        if (!inMST[v] && (u == -1 || minDist[v] < minDist[u])) u = v;
    inMST[u] = true;
    total += minDist[u];
    for (int v = 0; v < n; ++v) {                            // weights on the fly
        if (inMST[v]) continue;
        int w = abs(points[u][0]-points[v][0]) + abs(points[u][1]-points[v][1]);
        minDist[v] = min(minDist[v], w);
    }
}
return total;
```
Measured: **86 ms → 5 ms** at `n = 1000`. `minDist[v]` *is* the adjacency info, kept in O(n) instead of O(n²). Rule of thumb: **build an adjacency list when edges are sparse and given; skip it when the graph is complete and the weight is a formula.**

## Other Approaches
- **Brute force — try every spanning tree.** Cayley: `n^(n-2)` trees on `n` nodes. **O(n^(n-2))** — mention it only to name why greedy MST is remarkable (the cut property makes local-cheapest globally optimal).
- **Better — Prim with a heap over an explicit adjacency list.** **O(n² log n)** / O(n²). *My solution.* Natural if you already think in Dijkstra-shaped code; the `log` and the memory are both avoidable here.
- **Alternative — Kruskal + DSU.** Build all `n²/2` edges as `{w,i,j}`, sort, union greedily, stop after `n-1` unions. **O(n² log n)** / O(n²). Same ballpark; better when edges come *given* and sparse, and it's the natural fit if the problem already smells like Union-Find.
- **Optimal — dense Prim, no graph.** **O(n²)** / **O(n)**. Beats the heap because the graph is complete: `E = n²` makes `E log V` strictly worse than `V²`. **This is the expected answer at `n ≤ 1000`.**

## Pitfalls
- **Pushing `dis + w` instead of `w`** — that's Dijkstra, and it silently returns a shortest-path-tree cost, not the MST. Prim pushes the **edge weight alone**.
- **`j = 0` in the inner loop** — creates self-loops (`w = 0`) and duplicates every edge. Start at `j = i + 1`.
- **Forgetting `if (visited[node]) continue;`** — a node sits in the heap once per incident edge; without the skip you'd add the same node's cost several times and overcount.
- **Marking visited at push instead of pop** — locks in the *first* edge seen rather than the cheapest. In Prim, visited is set **on pop**. (Opposite of plain BFS, where you mark on discovery.)
- **`n = 1`** — heap pops `{0,0}`, `total = 0`, done. Works with no special case; say so out loud.
- **Overflow** — max cost is ~`1000 × 4·10⁶` ≈ `4·10⁹`… but an MST has only `n-1` edges and LeetCode guarantees the answer fits in `int`. Mention `long long` as the safe habit.
- **Reaching for Dijkstra** — no `src`/`dst` in the statement is the tell. "Connect all" ⇒ MST.
