# 11. 127. Word Ladder (LC 127) — Hard
🔗 [Open on LeetCode](https://leetcode.com/problems/word-ladder/)

**Pattern:** Graph BFS — **shortest path on an *implicit* unweighted graph** (words = nodes, one-letter edits = edges)
**Trigger:** "transform `beginWord` → `endWord`, one letter at a time, each step must be a dictionary word; return the **length of the shortest** such chain" → unweighted shortest path ⇒ **BFS**; generate neighbours by **mutating each position over 'a'–'z'**, not by pairwise comparison

## Problem (1-liner)
Given `beginWord`, `endWord`, and a `wordList`, each adjacent pair in the chain differs by exactly **one letter** and every intermediate word is in the list (`beginWord` needn't be). Return the **number of words** in the shortest chain, or `0` if none.
**Examples:** `hit → cog`, list `[hot,dot,dog,lot,log,cog]` → `5` (`hit→hot→dot→dog→cog`) · same without `cog` → `0` (target unreachable).
**Constraints:** `len ≤ 10`, `wordList ≤ 5000`, lowercase; all words same length; words unique. **Edge:** `endWord` not in list ⇒ `0` immediately.

## Idea (1-liner)
Words are nodes; an edge joins two words differing by one letter. Shortest chain = shortest path on an **unweighted** graph ⇒ **BFS**. Don't build the graph — generate a word's neighbours on the fly by replacing each position `i` with every letter `'a'..'z'` and keeping those still in the set. First time `endWord` is popped, its level is the answer.

> **Load-bearing insight:** never compare every pair of words (`O(N²·L)`). Instead, for each of `L` positions try 26 letters and hash-lookup the result — `O(26·L)` neighbours per word. The **hash set is both the dictionary and the visited set**: erase a word when you enqueue it.

## Approach
1. **Set + early out:** `st = set(wordList)`; if `endWord ∉ st` return `0`.
2. **Seed BFS:** push `{beginWord, 1}` (length counts words, so start at `1`).
3. **Expand:** pop `{word, level}`; if `word == endWord` return `level`. For each position `i`, save the char, try `'a'..'z'`: if the mutated word is in `st`, push `{mutated, level+1}` and **erase it from `st`** (mark visited on discovery); restore the char.
4. **Queue drains** without reaching target ⇒ return `0`.

> **Erase on enqueue, not on pop** — otherwise the same word is reached by many neighbours and re-enqueued, bloating the queue (redundant work / TLE). Restoring `word[i]` after the inner loop keeps the string intact for the next position.

## Code (clean canonical)
```cpp
class Solution {
public:
    int ladderLength(string beginWord, string endWord, vector<string>& wordList) {
        unordered_set<string> st(wordList.begin(), wordList.end());
        if (!st.count(endWord)) return 0;                 // unreachable target

        queue<pair<string,int>> q;                        // {word, chain length}
        q.push({beginWord, 1});                           // length counts words → start at 1

        while (!q.empty()) {
            auto [word, level] = q.front(); q.pop();
            if (word == endWord) return level;            // first pop = shortest

            for (int i = 0; i < (int)word.size(); ++i) {
                char original = word[i];
                for (char ch = 'a'; ch <= 'z'; ++ch) {
                    word[i] = ch;
                    if (st.count(word)) {
                        q.push({word, level + 1});
                        st.erase(word);                   // mark visited on discovery
                    }
                }
                word[i] = original;                       // restore for next position
            }
        }
        return 0;                                         // drained, never reached endWord
    }
};
```
**TC:** O(N · L² · 26) — `N` words, each tries `L` positions × 26 letters, each set lookup hashes an `L`-char string.  **SC:** O(N · L) — set + queue.

## My Solution
Your BFS is **correct and optimal** — carries the chain length in the queue pair, mutates-and-restores each position, and erases on discovery (the right visited discipline). Nothing to fix. ✅ Minor stylistic options only: freeze `sz = q.size()` and count **levels** instead of carrying `count` in the pair, and cache `word.size()`.

## Other Approaches
- **Pairwise-comparison graph (brute force)** — for every pair of words check if they differ by one letter, build an adjacency list, then BFS. Building it is **O(N²·L)** — with `N = 5000` that's ~25M·L, borderline TLE. The mutation trick avoids it.
- **Mutation-based BFS** *(optimal, canonical)* — the block above. O(N·L²·26) / O(N·L). Default answer.
- **Preprocessed wildcard buckets** — map each pattern like `h*t` → list of words; neighbours = same-bucket words. Also O(N·L²) build but sometimes cleaner; extra memory for the buckets.
- **Bidirectional BFS** *(the "can you do better?" answer)* — grow frontiers from **both** `beginWord` and `endWord`, always expand the **smaller** set, stop when they intersect. Same worst-case O but explores far fewer nodes (two radius-`k/2` balls vs one radius-`k` ball). More bookkeeping (two visited sets, swap-smaller-frontier) — mention it in interviews.

## Pitfalls
- **Length counts words, not edges** — start the level at `1`; `hit→hot→dot→dog→cog` is **5**, not 4.
- **`endWord` must be in the list** — check up front; not in list ⇒ `0` (Example 2).
- **Erase on enqueue, not on pop** — the set doubles as visited; erasing late re-enqueues words → TLE.
- **Restore `word[i]`** after the 26-letter sweep, else later positions mutate a corrupted string.
- **Don't build the O(N²) pairwise graph** — mutate positions instead; N up to 5000 makes pairwise costly.
- **`beginWord` needn't be in the list** — still valid as the start; don't gate on it.
