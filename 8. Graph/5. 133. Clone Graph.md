# 5. 133. Clone Graph (LC 133) — Medium
🔗 [Open on LeetCode](https://leetcode.com/problems/clone-graph/)

**Pattern:** Graph traversal — **deep copy via `old → new` hash map** (the "clone a structure with cycles" seed)
**Trigger:** "return a **deep copy** of a graph that has **cycles / shared neighbours**" → keep a map from *original node* to *its clone*; register the clone **before** recursing so a cycle finds it instead of looping forever

## Problem (1-liner)
Given a reference to a node in a **connected undirected** graph (each node = `int val` + `vector<Node*> neighbors`), return a **deep copy**: brand-new nodes mirroring the exact structure, no pointer shared with the original.
**Examples:** `adjList = [[2,4],[1,3],[2,4],[1,3]]` → identical 4-node cycle, freshly allocated · `[[]]` → one node, no neighbours · `[]` → `nullptr` (empty graph).
**Constraints:** `0 ≤ nodes ≤ 100`, `val` unique, no repeated edges, no self-loops, graph connected.

## Idea (1-liner)
The graph has **cycles** (undirected ⇒ every edge is a 2-cycle), so a naive recursive copy would clone the same node endlessly. Keep a map `oldToNew : Node* → Node*`. When you first meet a node, **create its clone and store it in the map *immediately*, before touching neighbours** — then recursing into a neighbour that points back finds the clone already there and just links to it.

> **Load-bearing insight:** register the clone in the map **before** recursing on neighbours. That map entry is *both* your memo (don't re-clone) *and* your cycle-breaker (don't recurse forever). Register after the loop and A↔B recurses infinitely.

## Approach
1. **Map `oldToNew`** from original node → its clone.
2. **`clone(cur)`:** if `cur` already in map → return the stored clone (memo + cycle guard).
3. Else make `copy = new Node(cur->val)`, **store `oldToNew[cur] = copy` right away**.
4. For each neighbour, `copy->neighbors.push_back(clone(nbr))` — recurse; cycles resolve via the map.
5. Return `copy`. Top level: `node ? clone(node) : nullptr` (handle the empty graph).

> **Why "before" matters:** on `[[2],[1]]` (A↔B) — clone A, store A→A'; recurse B; clone B, store B→B'; B recurses A, finds A' in the map, links it; done. Store *after* the loop → B's recursion into A never finds A' → infinite recursion.

## Code (clean canonical)
```cpp
/*
class Node { public: int val; vector<Node*> neighbors; ... };
*/
class Solution {
public:
    Node* cloneGraph(Node* node) {
        unordered_map<Node*, Node*> oldToNew;

        function<Node*(Node*)> clone = [&](Node* cur) -> Node* {
            if (oldToNew.count(cur)) return oldToNew[cur];   // memo + cycle guard

            Node* copy = new Node(cur->val);
            oldToNew[cur] = copy;                            // register BEFORE recursing
            for (Node* nbr : cur->neighbors)
                copy->neighbors.push_back(clone(nbr));
            return copy;
        };

        return node ? clone(node) : nullptr;                 // empty graph → nullptr
    }
};
```
**TC:** O(V + E) — each node cloned once, each edge walked once.  **SC:** O(V) — map + recursion stack.

*(Recursive lambda uses `std::function<Node*(Node*)>` so it can call itself and capture the map by `[&]` — see [[../cpp-lambdas|C++ Lambdas]].)*

## Other Approaches
- **DFS + hash map** *(optimal, canonical — the block above)* — O(V+E) / O(V). The default answer; concise and interview-friendly.
- **BFS + hash map** *(equally optimal)* — clone the start node, push original to a queue; pop a node, and for each neighbour: if unseen, create its clone + enqueue; always link `clone[cur]->neighbors.push_back(clone[nbr])`. Same O(V+E) / O(V). Reach for it when recursion depth is a worry (not here, `≤100` nodes) or the interviewer asks for iterative.
- **Two-pass** *(more verbose, sometimes asked)* — pass 1: create every clone (fill the map, no edges). Pass 2: wire neighbours from the map. Clear separation of "make nodes" vs "make edges", but two traversals — the one-pass DFS/BFS above subsumes it.

## Pitfalls
- **Store the clone before recursing** — the single biggest trap; storing after the neighbour loop → infinite recursion on cycles.
- **Undirected = cycles everywhere** — every edge is bidirectional, so you *will* revisit; the map is mandatory, not optional.
- **Empty graph** — `node == nullptr` must return `nullptr` (Example 3), not a new node.
- **Key by the original `Node*`, not by `val`** — `val` is unique here so it'd work, but pointer-keying is the general-safe habit (values can repeat in variants).
- **Recursion depth** — fine at `≤100`, but on a huge graph deep DFS can overflow the stack → switch to BFS.
- **Don't reuse original neighbour pointers** — a "deep" copy means every node *and* every neighbour link points into the cloned graph only.

**Family map:** sits in grid/graph traversal alongside flood-fill [[1-200-number-of-islands|Number of Islands]]; unlike the grid problems this is a **pointer graph** and the twist is *reconstruction with cycles* via a memo map — the same `visited`/memo discipline as [[3-994-rotting-oranges|Rotting Oranges]] and [[4-130-surrounded-regions|Surrounded Regions]], but the map stores *the clone*, not just "seen".
