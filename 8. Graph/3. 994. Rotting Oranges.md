# 3. 994. Rotting Oranges (LC 994) — Medium
🔗 [Open on LeetCode](https://leetcode.com/problems/rotting-oranges/)

**Pattern:** Graph BFS — **multi-source BFS, level = time** (the "minimum minutes/steps in a grid" seed)
**Trigger:** "every minute all rotten cells infect neighbours **simultaneously**; minimum minutes until none fresh" → seed the queue with **all** sources at once and expand **ring by ring**; each ring drained = one minute

## Problem (1-liner)
Grid cells are `0` empty, `1` fresh, `2` rotten. Each minute, every fresh orange 4-adjacent to a rotten one becomes rotten. Return the **minimum minutes** until no fresh orange remains, or `-1` if some fresh orange can never be reached.
**Examples:** `[[2,1,1],[1,1,0],[0,1,1]]` → `4` · `[[2,1,1],[0,1,1],[1,0,1]]` → `-1` (bottom-left marooned) · `[[0,2]]` → `0` (already none fresh).
**Constraints:** `1 ≤ m, n ≤ 10`.

## Idea (1-liner)
The minute a cell rots = its **shortest distance to the *nearest* initially-rotten orange**. Shortest-distance-from-many-sources in an unweighted grid = **multi-source BFS**. Put *all* rotten oranges in the queue as the "minute-0 frontier" so they advance **in parallel**, one ring per minute. Count the rings; if any fresh orange is left unreached, return `-1`.

> **Load-bearing insight:** this is **not** DFS. DFS reports the length of the arm it happened to walk, and its `visited` gives *first-writer-wins* (a farther source can claim a cell before a closer one). Parallel spread + "minimum time" ⇒ BFS by levels, seeded with every source.

## Approach
1. **Scan once:** push every rotten cell `{i,j}` into the queue; count `fresh`.
2. **Early out:** `fresh == 0` → return `0` (Example 3).
3. **BFS by minutes:** while queue non-empty, **freeze `sz = q.size()`** and drain exactly that ring. For each cell, rot every fresh 4-neighbour: mark it `2` **on discovery**, `--fresh`, push it.
4. If anything rotted this ring, `++minutes`.
5. **Return** `fresh == 0 ? minutes : -1` — leftover fresh = unreachable.

> **The `sz = q.size()` snapshot is what separates minutes** — you drain the current ring; the children you push are next minute's ring. Same trick as Tree BFS level-order ([[../7. Tree BFS/_|LC 102]]).

## Code (clean canonical)
```cpp
class Solution {
public:
    int orangesRotting(vector<vector<int>>& grid) {
        int rows = grid.size(), cols = grid[0].size(), fresh = 0;
        queue<pair<int,int>> q;
        for (int i = 0; i < rows; ++i)
            for (int j = 0; j < cols; ++j) {
                if (grid[i][j] == 2) q.push({i, j});
                else if (grid[i][j] == 1) ++fresh;
            }
        if (fresh == 0) return 0;                       // nothing to rot

        int minutes = 0, dr[] = {0, 0, 1, -1}, dc[] = {1, -1, 0, 0};
        while (!q.empty() && fresh > 0) {               // stop once all rotted
            int sz = q.size();                          // freeze this minute's ring
            for (int k = 0; k < sz; ++k) {
                auto [r, c] = q.front(); q.pop();
                for (int d = 0; d < 4; ++d) {
                    int nr = r + dr[d], nc = c + dc[d];
                    if (nr >= 0 && nr < rows && nc >= 0 && nc < cols && grid[nr][nc] == 1) {
                        grid[nr][nc] = 2;               // rot on discovery = mark visited
                        --fresh;
                        q.push({nr, nc});
                    }
                }
            }
            ++minutes;                                  // this ring rotted something
        }
        return fresh == 0 ? minutes : -1;
    }
};
```
**TC:** O(m·n) — each cell enqueued once.  **SC:** O(m·n) — the queue. *(`grid` doubles as the visited set — mutates input.)*

## My Solution (the bug I hit)
First attempt used **DFS** — passes small cases then fails, because DFS spreads **sequentially** down one arm instead of all sources advancing together. On `[[2,1,1,1,2]]` the true answer is `2` (both ends meet in the middle) but DFS walks one source across and reports `3`. **"Parallel spread + minimum time" is unfixable in DFS — switch to multi-source BFS.**

The BFS rewrite was the right shape but had a **one-character bug**:
```cpp
if (grid[nr][nc] == 1) {
    grid[nr][nc] == 2;   // ❌ == compares (discarded); cell stays 1
    ...
    q.push({nr, nc});
}
```
`==` instead of `=` means the cell is **never marked rotten**, so it's re-discovered and re-pushed forever → the queue never drains, `fresh` goes negative → **infinite loop / TLE**. The grid write *is* the visited marker; skip it and the "rot on discovery" invariant breaks. Fix: `grid[nr][nc] = 2;`. *(Also: don't reuse the loop variable `i` for both the ring drain and the direction index — rename the inner one to `d`; not a bug but a readability trap.)*

## Other Approaches
- **Multi-source BFS** *(optimal)* — the canonical block above. O(m·n) / O(m·n). The go-to for "minimum minutes/steps, simultaneous spread."
- **Single-source BFS per rotten orange, take min distance to each fresh** — correct but O((#rotten)·m·n); pointless when one multi-source pass gets it. Multi-source is just "all starts at distance 0 in one queue."
- **DFS with a `time` grid** — feasible only if you record the *earliest* minute each cell rots and overwrite when a **shorter** path arrives (`if (t < best[nr][nc])` recurse). Works but re-explores cells → worse than BFS and easy to get wrong. Not worth it here.
- **Simulation (repeat full-grid sweeps until stable)** — each minute scan the grid, rot neighbours into a *next* set, repeat until nothing changes. O(minutes·m·n). Simple, fine for `≤10×10`, but BFS is cleaner in one pass.

## Pitfalls
- **DFS is the wrong tool** — parallel spread + shortest-time ⇒ BFS. The single biggest trap on this problem.
- **`=` vs `==`** — the mark-rotten write must be an assignment; `==` gives an infinite loop.
- **Freeze `q.size()` before draining** — reading it live mixes this minute's and next minute's cells, corrupting the count.
- **Mark on discovery (push), not on pop** — else a cell is enqueued by several neighbours and counted multiple times.
- **`fresh == 0` up front** — Example 3 (`[[0,2]]`) must return `0`, and it prevents a spurious `+1` minute on an all-rotten/empty grid.
- **Off-by-one minute** — with `while (!q.empty())` alone, the last (childless) ring still bumps `minutes`; guard with `&& fresh > 0` (as above) or a `rottedThisMinute` flag.
- **`-1` only for *unreachable* fresh** — decide by `fresh > 0` at the end, not by any time value.
