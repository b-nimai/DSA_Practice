# 6. 207. Course Schedule (LC 207) — Medium
🔗 [Open on LeetCode](https://leetcode.com/problems/course-schedule/)

**Pattern:** Graph — **topological sort as cycle detection** (Kahn's BFS indegree-peel)
**Trigger:** "given **dependencies / prerequisites**, can you finish everything / is there a valid order?" → build a DAG and try to topologically sort it; **if you can't order every node, there's a cycle** → impossible

## Problem (1-liner)
`numCourses` labelled `0…n-1`; `prerequisites[i] = [a, b]` means **take `b` before `a`**. Return `true` iff you can finish all courses — i.e. the dependency graph has **no cycle**.
**Examples:** `n=2, [[1,0]]` → `true` (do 0 then 1) · `n=2, [[1,0],[0,1]]` → `false` (0↔1 deadlock).
**Constraints:** `1 ≤ numCourses ≤ 2000`, `0 ≤ prerequisites.length ≤ 5000`, all pairs unique.

## Idea (1-liner)
Model each `[a, b]` as a directed edge **`b → a`** ("b unlocks a"). A valid schedule exists iff this graph is a **DAG**. Run **Kahn's algorithm**: repeatedly take any course with **indegree 0** (no unmet prereqs), remove it, and decrement its dependents' indegrees. If you manage to process **all `n` nodes**, it's acyclic → `true`; if the queue empties early, the leftover nodes sit in a cycle → `false`.

> **Load-bearing insight:** you never need the actual ordering — just the **count** of nodes you managed to peel. `count == numCourses` *is* the acyclic test. A cycle can never reach indegree 0 (each node in it is held by another), so it's never enqueued.

## Approach
1. **Build graph + indegree:** for each `[a, b]`, add edge `adj[b].push_back(a)` and `indegree[a]++`. Edge direction = prereq → course.
2. **Seed queue** with every node whose `indegree == 0` (no prerequisites — can start immediately).
3. **Peel (BFS):** pop `node`, `count++`; for each neighbour, `--indegree[nbr]`, and when it hits `0` push it (all its prereqs now satisfied).
4. **Answer:** `count == numCourses` — every node got scheduled ⇒ no cycle.

> **Why the direction is `b → a`:** you want to start from courses with no prerequisites and flow *toward* the ones they unlock. Indegree of `a` = how many prereqs `a` still waits on; it reaches 0 exactly when all are done.

## Code (clean canonical)
```cpp
class Solution {
public:
    bool canFinish(int numCourses, vector<vector<int>>& prerequisites) {
        vector<vector<int>> adj(numCourses);
        vector<int> indegree(numCourses, 0);
        for (auto& p : prerequisites) {          // p = [a, b] → edge b -> a
            adj[p[1]].push_back(p[0]);
            indegree[p[0]]++;
        }

        queue<int> q;
        for (int i = 0; i < numCourses; ++i)
            if (indegree[i] == 0) q.push(i);      // no prereqs → ready

        int count = 0;
        while (!q.empty()) {
            int node = q.front(); q.pop();
            ++count;
            for (int nbr : adj[node])
                if (--indegree[nbr] == 0) q.push(nbr);
        }
        return count == numCourses;               // all peeled ⇒ acyclic
    }
};
```
**TC:** O(V + E) — build + each node/edge touched once.  **SC:** O(V + E) — adjacency list + indegree + queue.

Two fixes — the BFS/peel logic itself was already right:
- **Loop over edges, not courses** — `for (auto& p : prerequisites)`; each `p` is one `[a, b]` pair, so `p[0]`/`p[1]` are the two ints. Iterating `prerequisites[i]` makes the element an `int` and `it[1]` is a compile error.
- **Increment indegree of `p[0]` (`a`), not `p[1]` (`b`)** — `[a, b]` is edge `b → a`, so it's `a` that gains an unmet prerequisite.

*(Naming nit: `inorder` reads like an in-order traversal — `indegree` is the conventional name and states the intent.)*

## Code — DFS variant (three-color cycle detection)
Same graph, **same `b → a` build** — no direction flip. Swap the indegree array + queue for a `state` color array and recursion. A cycle shows up as a **back edge**: reaching a neighbour that is still **in-progress** (on the current recursion path).
```cpp
class Solution {
    vector<vector<int>> adj;
    vector<int> state;               // 0 = unvisited, 1 = in-progress (on stack), 2 = done
public:
    bool canFinish(int numCourses, vector<vector<int>>& prerequisites) {
        adj.assign(numCourses, {});
        state.assign(numCourses, 0);
        for (auto& p : prerequisites)          // p = [a, b] → edge b -> a  (identical build)
            adj[p[1]].push_back(p[0]);

        for (int i = 0; i < numCourses; ++i)   // graph may be a forest → start everywhere
            if (state[i] == 0 && hasCycle(i)) return false;
        return true;
    }
private:
    bool hasCycle(int node) {
        state[node] = 1;                        // mark on current path
        for (int nbr : adj[node]) {
            if (state[nbr] == 1) return true;               // back edge → cycle
            if (state[nbr] == 0 && hasCycle(nbr)) return true;
        }
        state[node] = 2;                        // fully explored, safe — never revisit
        return false;
    }
};
```
**TC:** O(V + E) — each node colored once, each edge walked once.  **SC:** O(V + E) — adjacency list + `state` + up to O(V) recursion stack.

> **BFS vs DFS — what actually changes:** *not* the edge direction (both run on `b → a`). BFS uses an `indegree` array + queue and answers via `count == numCourses`; DFS uses a 3-color `state` array + recursion and answers via "found a back edge?". Two things bite in DFS: **(1)** you must loop `for i in 0..n` and start from every unvisited node — the graph can be disconnected, so one call won't reach all (BFS gets this free from seeding *all* indegree-0 nodes); **(2)** the `state==2` "done" color is essential — without it you'd re-explore shared subtrees and blow up to exponential.

## Other Approaches
- **Kahn's BFS (indegree peel)** *(the canonical block above)* — iterative, no recursion depth worry, and naturally yields a topo order if you collect popped nodes. Default pick for interviews.
- **DFS three-color cycle detection** *(equally optimal)* — mark nodes `0` unvisited / `1` in-progress / `2` done. DFS a node; if you hit an **in-progress (`1`)** neighbour, that's a **back edge → cycle → return false**. `visited` state prevents re-work. O(V+E) / O(V) stack. Reach for it when the interviewer asks recursively or you also need the post-order (which reversed = topo order, as in [[7-210-course-schedule-ii|Course Schedule II]]).
- **DFS with a per-path `recStack`** — same idea as three-color, an explicit "on current path" set instead of the in-progress color.

## Pitfalls
- **Edge direction** — `[a, b]` means `b → a` (b before a). Flip it and you detect the mirror graph; for pure yes/no cycle it happens to still work, but the topo order comes out reversed and Course Schedule II breaks.
- **Indegree on the right node** — increment `indegree[a]` (the course that has the prereq), not `indegree[b]`.
- **`count == numCourses`, not `q` emptiness** — the queue always drains; the *count* is what distinguishes acyclic (all peeled) from a cycle (some never reach indegree 0).
- **No prerequisites** — `prerequisites` empty ⇒ every node indegree 0 ⇒ all pushed ⇒ `true`. Handled for free.
- **Self-loop `[a, a]`** — indegree of `a` is 1 with an edge to itself; it never reaches 0 → `false`, which is correct (needs itself first).
- **Isolated nodes count too** — a course with no edges is indegree 0, gets pushed and counted; don't forget to size `adj`/`indegree` to `numCourses`, not to the edge list.
