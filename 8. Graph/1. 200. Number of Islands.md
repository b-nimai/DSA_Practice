# 1. 200. Number of Islands (LC 200) — Medium
🔗 [Open on LeetCode](https://leetcode.com/problems/number-of-islands/)

**Pattern:** Graph DFS/BFS — **grid flood-fill / connected components** (the seed problem of the whole grid family)
**Trigger:** "count connected blobs / regions / islands of `1`s in a grid" → scan every cell; each **unvisited land** cell is a new component — bump the counter, then **flood-fill** to sink its entire island so it's never re-counted

## Problem (1-liner)
Given an `m × n` grid of `'1'` (land) / `'0'` (water), return the number of **islands** — maximal groups of land connected **4-directionally** (up/down/left/right, *not* diagonal). All outer edges are surrounded by water.
**Examples:** one big connected blob → `1` · three separate blobs → `3`.
**Constraints:** `1 ≤ m, n ≤ 300` (so up to 90,000 cells — a snake-shaped island can force **90k-deep recursion**).

## Idea (1-liner)
The grid *is* an implicit graph: each land cell is a node, edges join 4-adjacent land cells. **Number of islands = number of connected components.** Walk the grid; the first time you hit a land cell you haven't seen, you've found a brand-new component — count it once, then traverse (DFS/BFS) every cell reachable from it and **mark it visited** so the rest of that island is absorbed into the same count. Any later scan lands only on water or already-visited cells until the *next* fresh island.

## Approach
1. **Outer scan:** loop every cell `(i, j)`.
2. When `grid[i][j] == '1'` **and** not visited → `++islands`, then launch a flood-fill from it.
3. **Flood-fill (DFS):** mark the cell visited; for each of the 4 neighbours, recurse if in-bounds, land, and unvisited.
4. The fill drains the *entire* island before returning, so the outer loop never double-counts it.

> **Load-bearing insight:** mark visited **the moment you enter** a cell (before/at the push), not when you pop it. Marking on pop lets the same cell get queued/recursed from several neighbours → re-counting or blow-up. "Count on first discovery, sink immediately."

## Code (clean canonical)
```cpp
class Solution {
    void dfs(vector<vector<char>>& grid, int r, int c) {
        int rows = grid.size(), cols = grid[0].size();
        if (r < 0 || r >= rows || c < 0 || c >= cols || grid[r][c] != '1') return;
        grid[r][c] = '0';                                  // sink it => marks visited in-place
        dfs(grid, r + 1, c); dfs(grid, r - 1, c);
        dfs(grid, r, c + 1); dfs(grid, r, c - 1);
    }
public:
    int numIslands(vector<vector<char>>& grid) {
        int rows = grid.size(), cols = grid[0].size(), islands = 0;
        for (int i = 0; i < rows; ++i)
            for (int j = 0; j < cols; ++j)
                if (grid[i][j] == '1') { ++islands; dfs(grid, i, j); }
        return islands;
    }
};
```
**TC:** O(m·n) — each cell is visited once.  **SC:** O(m·n) worst-case recursion stack (one snake-shaped island); no separate `visited` grid since we sink to `'0'` in place. *Caveat: mutates the input.*

## My Solution
```cpp
class Solution {
    void dfs(vector<vector<char>>& grid, vector<vector<int>>& visited, int row, int col) {
        int rows = grid.size(), cols = grid[0].size();
        visited[row][col] = 1;
        int drow[] = {0, 0, -1, 1};
        int dcol[] = {1, -1, 0, 0};
        for (int i = 0; i < 4; i++) {
            int newRow = row + drow[i], newCol = col + dcol[i];
            if (newRow >= 0 && newRow < rows && newCol >= 0 && newCol < cols &&
                grid[newRow][newCol] == '1' && !visited[newRow][newCol])
                dfs(grid, visited, newRow, newCol);
        }
    }
public:
    int numIslands(vector<vector<char>>& grid) {
        int rows = grid.size(), cols = grid[0].size();
        vector<vector<int>> visited(rows, vector<int>(cols, 0));
        int islands = 0;
        for (int i = 0; i < rows; i++)
            for (int j = 0; j < cols; j++)
                if (grid[i][j] == '1' && !visited[i][j]) { islands++; dfs(grid, visited, i, j); }
        return islands;
    }
};
```
**Correct**, same O(m·n) time. Marks `visited[row][col]=1` **on entry** (right call), and the `drow/dcol` arrays are the clean, extensible way to encode the 4 directions (swap in the 8-diagonal set for problems that need it). Only trade-off vs canonical: keeps a separate `visited` grid → O(m·n) **extra** space and preserves the input. Keep this version whenever the grid **must not be mutated**; use the sink-in-place version when it can.

## Other Approaches
- **DFS, sink in place** *(optimal on space)* — the canonical block above. O(m·n) time, O(m·n) worst-case **stack**, no extra grid. Reach for it by default when mutating the grid is allowed.
- **BFS with a queue** — same O(m·n) time / O(min(m,n)) frontier space, but **iterative** so it can't stack-overflow. With `m,n ≤ 300` a 90k-deep DFS recursion is a genuine risk — prefer BFS (or an explicit stack) when depth could be huge:
  ```cpp
  queue<pair<int,int>> q; q.push({i, j}); grid[i][j] = '0';
  int dr[]={1,-1,0,0}, dc[]={0,0,1,-1};
  while (!q.empty()) {
      auto [r, c] = q.front(); q.pop();
      for (int d = 0; d < 4; ++d) {
          int nr = r + dr[d], nc = c + dc[d];
          if (nr>=0 && nr<rows && nc>=0 && nc<cols && grid[nr][nc]=='1') {
              grid[nr][nc] = '0';          // mark on push, not on pop
              q.push({nr, nc});
          }
      }
  }
  ```
- **Union-Find (DSU)** — union each land cell with its right/down land neighbour, answer = number of distinct roots among land cells. O(m·n·α(m·n)) ≈ linear. Overkill for the static grid, but the **go-to** when land is added incrementally and you must report the island count after each addition → [[2-305-number-of-islands-ii|Number of Islands II]] (LC 305, 🔴).

## Pitfalls
- **Marking visited too late** — sink/mark **on discovery** (entry or push), never on pop. Marking on pop lets multiple neighbours enqueue the same cell → double-count or TLE.
- **Recursion depth** — up to 90k-deep DFS on a 300×300 all-land grid can blow the native stack. Say this out loud in an interview and offer BFS / explicit stack.
- **Diagonals don't connect** — only 4 directions here. Using the 8-direction set silently merges islands that should stay separate (that's a *different* problem).
- **`'1'`/`'0'` are chars, not ints** — compare against `'1'`, not `1`.
- **Re-reading `grid.size()`/`cols` is fine, but don't index before the bounds check** — order the `if` so out-of-bounds short-circuits *before* `grid[nr][nc]`.
- **Sink-in-place mutates the caller's grid** — if the grid is needed afterwards (or the signature forbids mutation), keep the separate `visited` array.
