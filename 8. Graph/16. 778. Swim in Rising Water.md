# 16. 778. Swim in Rising Water (LC 778) — Hard
🔗 [Open on LeetCode](https://leetcode.com/problems/swim-in-rising-water/)

**Pattern:** Weighted Grid — **minimize the *maximum* cell on a path** ⇒ **bottleneck / minimax path** (Dijkstra with `max` instead of `+`, or MST/DSU, or binary search + BFS).
**Trigger:** "water level rises to `t`", "you can pass a cell only if its value **≤ t**", "**minimum `t`** to get from A to B". The answer is a **threshold**, not a sum — the cost of a path is its **worst cell**, and you want the path whose worst cell is smallest. Any phrasing of the form *"minimize the maximum X along the route"* (max load on a bridge, weakest link, min capacity) is this problem.

## Problem (1-liner)
`n × n` grid, `grid[i][j]` = elevation. At time `t` every cell with elevation `≤ t` is swimmable; you move 4-directionally between swimmable cells in **zero time**. Return the **minimum `t`** at which `(0,0)` can reach `(n-1,n-1)`.
**Examples:** `[[0,2],[1,3]]` → `3` (the destination itself is `3`, so you can't be there sooner) · `[[0,1,2,3,4],[24,23,22,21,5],[12,13,14,15,16],[11,17,18,19,20],[10,9,8,7,6]]` → `16`.
**Constraints:** `1 ≤ n ≤ 50`, `0 ≤ grid[i][j] < n²`, **all values distinct** (⇒ they're a permutation of `0..n²-1`). **Edge:** `n = 1` ⇒ answer `grid[0][0]` (`= 0`, since values are `< 1`) — you're already at the target.

## Idea (1-liner)
Cost of a path = **max cell on it**, not the sum. So run Dijkstra where "distance" means *highest water level needed so far*: relax with **`max(t, grid[nbr])`** instead of `t + w`. Pop the destination ⇒ that max **is** the answer.

> **Load-bearing insight:** swimming is free, so *time* is only ever spent **waiting**. You never pay for distance — you pay once, for the single tallest cell you're forced to cross. Replace `+` with `max` in Dijkstra and everything else about the algorithm (heap, greedy pop, visited) survives untouched, because `max` is monotone the same way `+` is.
> **Why the destination is a lower bound:** the answer is always `≥ max(grid[0][0], grid[n-1][n-1])` — you must stand on both. Example 1 is exactly that bound.

## Approach
1. Min-heap of `{time, row, col}`; push `{grid[0][0], 0, 0}`, mark `visited[0][0]`.
2. Pop the smallest `time`. If it's `(n-1, n-1)` → **return `time`**.
3. For each of the 4 neighbours in bounds and unvisited: mark visited, push `{max(time, grid[nr][nc]), nr, nc}`.
4. The heap can't drain before reaching the target (the grid is fully connected at `t = n²-1`), so step 2 always fires.

> **Why `visited` can be set on *push* here (unlike normal Dijkstra):** the key of a cell is bounded below by its own elevation, and the greedy pop order means the first time a cell is discovered it is already reached with the optimal minimax cost. There's no cheaper relaxation to come, so no stale entries — one push per cell, no `if (visited) continue;` needed.

## Code (clean canonical)
```cpp
class Solution {
public:
    int swimInWater(vector<vector<int>>& grid) {
        int n = grid.size();
        vector<vector<bool>> visited(n, vector<bool>(n, false));

        using T = tuple<int,int,int>;                       // {time, row, col}
        priority_queue<T, vector<T>, greater<T>> pq;
        pq.push({grid[0][0], 0, 0});
        visited[0][0] = true;

        int dr[] = {0, 0, 1, -1};
        int dc[] = {1, -1, 0, 0};

        while (!pq.empty()) {
            auto [time, r, c] = pq.top(); pq.pop();
            if (r == n - 1 && c == n - 1) return time;      // first pop = optimal

            for (int i = 0; i < 4; i++) {
                int nr = r + dr[i], nc = c + dc[i];
                if (nr < 0 || nr >= n || nc < 0 || nc >= n) continue;
                if (visited[nr][nc]) continue;
                visited[nr][nc] = true;
                pq.push({max(time, grid[nr][nc]), nr, nc});  // max, NOT +
            }
        }
        return -1;                                           // unreachable on a full grid
    }
};
```
**TC:** O(n² log n) — each cell enters the heap once. **SC:** O(n²) — heap + visited.

**Dry run** (`[[0,2],[1,3]]`, answer `3`):

| pop | time | target? | pushed |
|---|---|---|---|
| `(0,0)` | `0` | no | `{max(0,2),0,1}=2`, `{max(0,1),1,0}=1` |
| `(1,0)` | `1` | no | `{max(1,3),1,1} = 3` |
| `(0,1)` | `2` | no | — (both nbrs visited) |
| `(1,1)` | **`3`** | **yes** | → return `3` |

**Also correct** — this is the **Prim's-flavoured** variant. The heap key is the cell's own elevation (a *static* value, not path-dependent), so it's really "grow the region cheapest-cell-first"; the answer is the **max of everything popped** before the target appears, which is exactly the water level at which the two corners join. Same O(n² log n) / O(n²).

> **Don't delete the running max.** Pops here are **not** monotonically non-decreasing, because a newly discovered neighbour can be cheaper than the cell that revealed it. Single row `0 → 5 → 1`: pop `0` (push `5`), pop `5` (push `1`), pop `1`. Answer is `5`, and only `reqTime` captures it. In the canonical version the `max` is baked into the key, which is why *that* one can `return time` directly. Know which version you're writing.

## Other Approaches
- **Brute force — try every `t` from `0` upward, BFS each time.** For `t = 0,1,2,…` flood from `(0,0)` over cells `≤ t` and stop at the first `t` that reaches the corner. **O(n⁴)** / O(n²). Correct but wasteful — the feasibility test is **monotone in `t`**, which begs for binary search.
- **Better — binary search on `t` + BFS/DFS.** `lo = max(grid[0][0], grid[n-1][n-1])`, `hi = n²-1`; `check(t)` = flood using only cells `≤ t`. **O(n² log n)** / O(n²). The most *explainable* solution and the easiest to get right under pressure — say this one out loud first, then optimize.
- **Optimal (and the expected answer) — Dijkstra with `max`.** **O(n² log n)** / O(n²). One pass, no outer search. *My solution is the Prim's-shaped equivalent.*
- **Optimal (best complexity) — Union-Find / offline threshold sweep.** Values are a **permutation of `0..n²-1`**, so bucket them: `pos[v] = cell` in O(n²) with **no sort**. Then raise `t = 0,1,2,…`, activate cell `pos[t]`, union it with active neighbours, and return `t` the moment `find(0) == find(n²-1)`. **O(n² α(n²)) ≈ O(n²)** / O(n²). The distinct-values constraint exists precisely to enable this — flagging it scores points even if you code the Dijkstra.

## Pitfalls
- **Writing `time + grid[nr][nc]` instead of `max(...)`** — turns a bottleneck problem into a shortest-*sum* problem and silently returns a much larger number. `max` vs `+` is the whole problem.
- **Returning the popped `time` in the Prim's-style variant** — wrong, because pops aren't sorted (`0 → 5 → 1`). Keep the running max, or switch to the canonical key.
- **Checking the target at push instead of pop** — the first *discovery* of the corner may not be via the minimax path. Only a **pop** carries the optimality guarantee (same rule as every Dijkstra).
- **Bounds after indexing** — guard `nr/nc` before touching `grid[nr][nc]`.
- **Starting binary search at `lo = 0`** — harmless but wasteful; `lo = max(grid[0][0], grid[n-1][n-1])` is a provable floor.
- **`n = 1`** — the start *is* the target, popped immediately ⇒ `grid[0][0]`. Works with no special case; say so.
- **Forgetting the answer is a `t`, not a path length** — the return value is a water level, so `visited` bookkeeping never needs to track distance.
