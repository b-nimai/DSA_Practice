# 14. 787. Cheapest Flights Within K Stops (LC 787) — Medium
🔗 [Open on LeetCode](https://leetcode.com/problems/cheapest-flights-within-k-stops/)

**Pattern:** Weighted Graph — **shortest path with a hop limit** ⇒ **BFS on a `(node, stops)` state with a FIFO queue** (alt: Bellman-Ford, `k+1` rounds). *Plain Dijkstra breaks here.*
**Trigger:** "cheapest price from `src` to `dst` with **at most `k` stops**" → the cost function has a **second dimension (hops)**, so a state is `(node, hopsUsed)`, not just `node`. Expand in **hop order** (plain FIFO queue) and carry `stops` in the state — never finalize greedily by cost.

## Problem (1-liner)
`n` cities, directed weighted flights `[from, to, price]`. Return the **cheapest** price `src → dst` using **at most `k` stops** (= at most **`k+1` edges**), or `-1` if no such route.
**Examples:** `n=4, flights=[[0,1,100],[1,2,100],[2,0,100],[1,3,600],[2,3,200]], src=0,dst=3,k=1` → `700` (`0→1→3`; the cheaper `0→1→2→3`=400 uses 2 stops ✗) · `n=3, [[0,1,100],[1,2,100],[0,2,500]], src=0,dst=2,k=1` → `200` · same with `k=0` → `500`.
**Constraints:** `2 ≤ n ≤ 100`, `E ≤ n(n-1)/2`, `1 ≤ price ≤ 10⁴`, `0 ≤ k < n`, no duplicate edges, `src ≠ dst`. **Edge:** `k=0` ⇒ only a direct flight; no route ⇒ `-1`.

## Idea (1-liner)
Ordinary **BFS with a `stops` counter in the state**. Push `{cost, stops, node}` into a **plain FIFO queue**; refuse to expand any entry with `stops > k`. Because a FIFO pops in **non-decreasing hop order**, the cheap `distance[]` prune is safe, and the hop limit is enforced by one `continue`.

> **Load-bearing insight:** the greedy "cheapest cost wins" invariant is **false** here. A path can be *cheaper but too long*; once it claims `distance[node]`, a **pricier path with fewer hops** — the one that could still legally reach `dst` — gets pruned away. Cost alone is not a valid dominance test; the state must carry hops.
> **Why FIFO rescues it:** hops only ever grow by 1 per push, so pops are sorted by hops. Whatever already sits in `distance[nbr]` was written by a path with **≤ hops**, so if it's also **≤ cost** it truly dominates ⇒ discarding is sound. A **min-heap** destroys that ordering — the stored entry can have *more* hops, dominate nothing, and still prune. That single difference is the whole problem.

## Approach
1. Build `adjList` from `flights`: `adjList[from].push_back({to, price})`.
2. `distance` of size `n` = `INT_MAX`, `distance[src] = 0`. Push `{0, 0, src}` = `{cost, stops, node}`.
3. **Pop from the front** (FIFO, *not* a heap). If `stops > k` → `continue`: this route has already used `k+1` edges, it may not fly again.
4. For each `(nbr, w)`: if `cost + w < distance[nbr]` → update and push `{cost + w, stops + 1, nbr}`.
5. Return `distance[dst] == INT_MAX ? -1 : distance[dst]`.

> **Why the answer is legal:** `distance[]` is only ever written at push time, and we only push from entries with `stops ≤ k`, so every value stored used **≤ k+1 edges**. No post-filter needed.
> **Why the `stops > k` check goes at pop, not push:** an entry with `stops = k+1` is a *valid answer* (it used exactly `k+1` edges) — it just may not be extended. Dropping it at push time would lose real routes.

## Code (clean canonical)
```cpp
class Solution {
public:
    int findCheapestPrice(int n, vector<vector<int>>& flights, int src, int dst, int k) {
        vector<vector<pair<int,int>>> adjList(n);
        for (auto& f : flights) adjList[f[0]].push_back({f[1], f[2]});

        vector<int> distance(n, INT_MAX);
        distance[src] = 0;

        queue<tuple<int,int,int>> q;        // {cost, stops, node}  — FIFO, never a heap
        q.push({0, 0, src});

        while (!q.empty()) {
            auto [cost, stops, node] = q.front(); q.pop();
            if (stops > k) continue;        // already spent k+1 flights → can't fly again
            for (auto [nbr, w] : adjList[node]) {
                if (cost + w < distance[nbr]) {
                    distance[nbr] = cost + w;
                    q.push({cost + w, stops + 1, nbr});
                }
            }
        }
        return distance[dst] == INT_MAX ? -1 : distance[dst];
    }
};
```
**TC:** O(k·E) — each node can be re-enqueued at most once per hop level.  **SC:** O(n + E) — adjacency list + queue.

**Dry run** (`n=3, [[0,1,100],[1,2,100],[0,2,500]], src=0, dst=2, k=0`, answer `500`):

| pop | `stops > k`? | relaxations | queue after |
|---|---|---|---|
| `{0,0,0}` | `0 > 0` no | `distance[1]=100`, `distance[2]=500` | `{100,1,1}`, `{500,1,2}` |
| `{100,1,1}` | `1 > 0` **yes → skip** | — (this is exactly the 2-flight route being blocked) | `{500,1,2}` |
| `{500,1,2}` | `1 > 0` yes → skip | — | empty |

→ `distance[2] = 500` ✓ (with `k=1` the second row would expand and give `200`).

## My Solution
This *is* my solution — verified against brute force on 50 000 random graphs, 0 mismatches. The **only** variant that fails is swapping the FIFO for a `priority_queue`:

```cpp
// ❌ PQ version — min-heap by cost, pruned with a cost-only distance[]
priority_queue<tuple<int,int,int>, vector<tuple<int,int,int>>, greater<>> pq;  // {dis, stops, node}
...
while (!pq.empty()) {
    auto [dis, stops, node] = pq.top(); pq.pop();     // ← top() instead of front() is the whole bug
    if (stops > k) continue;
    for (auto [nbr, w] : adjList[node])
        if (dis + w < distance[nbr]) { distance[nbr] = dis + w; pq.push({dis + w, stops + 1, nbr}); }
}
```

**Failing case:** `n=6, k=2, src=0, dst=5, flights=[[0,1,1],[1,2,1],[2,4,1],[0,3,5],[3,4,1],[4,5,1]]` → true answer `7` (`0→3→4→5`, 3 edges = 2 stops ✓), PQ returns `-1`.

| pop | what happens |
|---|---|
| `(0,0,0)` | `dist[1]=1`, `dist[3]=5` |
| `(1,1,1)` | `dist[2]=2` |
| `(2,2,2)` | **`dist[4]=3`** — via **3 edges** |
| `(3,3,4)` | `stops 3 > k` → skipped, never expands to `5` |
| `(5,1,3)` | relax 4: `5+1=6` **not <** `dist[4]=3` → **never pushed**, the legal 2-edge route is lost |

The **queue** version survives because it pops in **non-decreasing hop order**: `(5,1,3)` is processed *before* any 3-edge entry can write `3` into `dist[4]`, so `dist[4]=6` gets enqueued and reaches `5`.

**Fix if you insist on the heap:** make the prune hop-aware — `best[node][stops]` (2-D `n × (k+2)`), or order the heap by **stops** instead of cost (which is just this FIFO plus a pointless `log`). The FIFO is the better answer.

## Other Approaches
- **Brute force — DFS all paths with a hop budget.** Explore every route, cut at `k+1` edges, keep the min. **O(n^k)** / O(k) stack. Correct, exponential, TLE — mention it, then move on.
- **Same idea, level-drained form.** Identical to the code above but with `int sz = q.size()` per level and `stops` as a loop counter instead of a tuple field. Same complexity; carrying `stops` in the state is less bookkeeping and reads better.
- **Dijkstra with hop-aware state.** Heap of `{cost, stops, node}` pruned on `best[node][stops]`. **O(E·k·log(E·k))** / O(n·k). Works, but strictly more memory + a `log` for nothing here.
- **Bellman-Ford, `k+1` rounds with a `temp` copy.** Loop `i = 0..k`; each round snapshots `temp = dist`, relaxes every edge reading `dist` / writing `temp`, then `dist = temp`. **O(k·E)** / O(n), no adjacency list. Shortest code, but the `temp` copy is easy to get wrong — **without it one round chains two flights** (in `[[0,1,100],[1,2,100],[0,2,500]], k=0` it returns `200` instead of `500`) and the answer starts depending on the *order edges appear in the input*. Mention it as the alternative; lead with the queue.

## Pitfalls
- **`k` stops = `k+1` edges** — off-by-one is the #1 bug. Guard is `stops > k` (an entry *at* `stops == k` may still fly once more).
- **Using a `priority_queue` instead of a `queue`** — the single-token change that breaks everything. Cheapest-first finalization ignores hops; a cheap long path evicts a pricier short one. See the trace above.
- **Pruning at push instead of at pop** — an entry with `stops = k+1` is still a legal *answer*, it just can't be extended. Drop it at push and you lose real routes.
- **Never marking nodes visited** — deliberate. A node **must** be re-enqueued when reached more cheaply; there is no `visited[]` in this problem, only the `distance[]` improvement test.
- **Overflow** — here `cost` comes from the state (already finite), so `cost + w` is safe. In the Bellman-Ford variant you *must* guard `if (dist[u] == INT_MAX) continue;` before `dist[u] + w`.
- **`-1` vs `0`** — `src != dst` is guaranteed, so an untouched `dst` really is unreachable ⇒ `-1`.
