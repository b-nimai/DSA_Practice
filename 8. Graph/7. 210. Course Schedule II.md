# 7. 210. Course Schedule II (LC 210) — Medium
🔗 [Open on LeetCode](https://leetcode.com/problems/course-schedule-ii/)

**Pattern:** Graph — **topological sort** (return the order, not just yes/no)
**Trigger:** "given **dependencies / prerequisites**, output **a valid order** to do everything (empty if impossible)" → build a DAG, topologically sort it; **can't order every node ⇒ cycle ⇒ return `[]`**

## Problem (1-liner)
`numCourses` labelled `0…n-1`; `prerequisites[i] = [a, b]` means **take `b` before `a`**. Return **any valid ordering** of all courses, or **`[]`** if impossible (a cycle).
**Examples:** `n=2, [[1,0]]` → `[0,1]` · `n=4, [[1,0],[2,0],[3,1],[3,2]]` → `[0,1,2,3]` or `[0,2,1,3]` · `n=1, []` → `[0]`.
**Constraints:** `1 ≤ numCourses ≤ 2000`, `0 ≤ prerequisites.length ≤ n·(n-1)`, all pairs distinct, `a ≠ b`.

## Idea (1-liner)
Same DAG as [[6. 207. Course Schedule]] — edge **`b → a`** ("b unlocks a") — but now **emit the ordering**, not just the acyclic yes/no. Two ways: **Kahn's BFS** collects nodes *in the order they're peeled* (that sequence **is** the topo order); **DFS** collects nodes in **post-order** (finish time) and **reverses** it. Either way, if you can't place all `n` nodes, there's a cycle → `[]`.

> **Load-bearing insight:** in 207 you only needed the **count** of peeled nodes; here you **keep the sequence**. In Kahn's the pop-order *is* the answer (append on pop). In DFS the answer is the **reverse of post-order** — a node is finished only after everything it depends on, so reversing puts prerequisites first.

## Approach (Kahn's BFS — default)
1. **Build graph + indegree:** for each `[a, b]`, `adj[b].push_back(a)` and `indegree[a]++`.
2. **Seed queue** with every indegree-0 node (no prereqs).
3. **Peel & record:** pop `node`, **`topo.push_back(node)`**; for each neighbour `--indegree[nbr]`, push when it hits 0.
4. **Answer:** `topo.size() == numCourses` ⇒ return `topo`; else a cycle blocked some nodes ⇒ return `{}`.

## Code (clean canonical — Kahn's BFS)
```cpp
class Solution {
public:
    vector<int> findOrder(int numCourses, vector<vector<int>>& prerequisites) {
        vector<vector<int>> adj(numCourses);
        vector<int> indegree(numCourses, 0);
        for (auto& p : prerequisites) {          // p = [a, b] → edge b -> a
            adj[p[1]].push_back(p[0]);
            indegree[p[0]]++;
        }

        queue<int> q;
        for (int i = 0; i < numCourses; ++i)
            if (indegree[i] == 0) q.push(i);      // no prereqs → ready

        vector<int> topo;
        while (!q.empty()) {
            int node = q.front(); q.pop();
            topo.push_back(node);                 // pop-order IS the topo order
            for (int nbr : adj[node])
                if (--indegree[nbr] == 0) q.push(nbr);
        }
        return topo.size() == numCourses ? topo : vector<int>{};   // short ⇒ cycle
    }
};
```
**TC:** O(V + E) — build + each node/edge touched once.  **SC:** O(V + E) — adjacency + indegree + queue + output.

## My Solution (both correct ✅)
Two clean solutions, both building the graph the canonical way `adj[b].push_back(a)`:

**DFS — 3-color post-order + cycle detection:**
```cpp
class Solution {
    bool hasCycle(vector<vector<int>>& adj, vector<int>& state, vector<int>& topo, int node) {
        state[node] = 1;                              // 1 = in-progress (on path)
        for (int nbr : adj[node]) {
            if (state[nbr] == 1) return true;         // back edge → cycle
            if (!state[nbr] && hasCycle(adj, state, topo, nbr)) return true;
        }
        state[node] = 2;                              // 2 = done
        topo.push_back(node);                         // post-order finish
        return false;
    }
public:
    vector<int> findOrder(int numCourses, vector<vector<int>>& prerequisites) {
        vector<vector<int>> adj(numCourses);
        for (auto& it : prerequisites) adj[it[1]].push_back(it[0]);   // b -> a

        vector<int> state(numCourses, 0), topo;
        for (int i = 0; i < numCourses; ++i)
            if (!state[i] && hasCycle(adj, state, topo, i)) return {};

        reverse(topo.begin(), topo.end());            // reverse post-order = topo order
        return topo;
    }
};
```
**Two things this DFS *must* have** (and yours does): **(1)** the `state==1` in-progress color so a back edge is detected → `[]` on a cycle (a plain binary visited can't detect cycles, and would return a garbage full order); **(2)** with edge direction `b → a`, post-order **must be reversed** — prerequisites finish first (bottom of the reversed list). Flip either one and the order breaks. *(Earlier buggy draft built `a → b` and reversed → double-flip = wrong order, and used binary visited → missed cycles. Both fixed here.)*

Your **BFS** solution is the canonical block above (indegree peel, `topo.size() != numCourses → {}`).

## Other Approaches
- **Kahn's BFS (indegree peel)** *(canonical block)* — iterative, no recursion-depth worry, **cycle detection is free** (short output ⇒ cycle). Default interview pick. O(V+E) / O(V+E).
- **DFS post-order + reverse** *(your solution, equally optimal)* — 3-color to detect the back edge; reverse the finish order. Reach for it when asked recursively or when you also want the post-order for other work. O(V+E) time, O(V) stack.
- **Brute force (no real contender)** — repeatedly scan for any indegree-0 node, emit it, remove its edges, repeat. That's just Kahn's done in O(V²) instead of O(V+E) because you re-scan every pass. Same idea, worse constant — mention it only to contrast against the queue.

## Pitfalls
- **Edge direction `b → a`** — `[a, b]` = b before a. Get it backwards and the order comes out reversed (207 tolerates it for yes/no; **210 does not**).
- **BFS: append on pop** — the topo order is the *pop sequence* (`topo.push_back(node)` when you dequeue), not something you reverse.
- **DFS: reverse the post-order** — post-order alone lists dependents before prerequisites; you must `reverse` it. (Opposite of BFS, which needs no reverse.)
- **DFS needs 3 colors, not a bool** — binary visited can't tell "on the current path" from "already finished", so it **can't detect cycles** → would wrongly return a full order instead of `[]`.
- **Cycle ⇒ `[]`** — BFS: `topo.size() != numCourses`; DFS: back edge found. Don't forget the empty-array contract.
- **Isolated / no-prereq nodes** — indegree 0, enter the queue immediately and appear in the order; size `adj`/`indegree` to `numCourses`.
