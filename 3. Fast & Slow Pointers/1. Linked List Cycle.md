# 1. Linked List Cycle (LC 141) — Easy
🔗 [Open on LeetCode](https://leetcode.com/problems/linked-list-cycle/)

**Pattern:** Fast & Slow Pointers — Floyd's Tortoise & Hare (cycle detection)
**Trigger:** "does a linked list contain a cycle?" → two pointers, `slow += 1`, `fast += 2`; meet ⇒ cycle

## Problem (1-liner)
Given the head of a singly linked list, return `true` if there exists a node reachable again by following `next` pointers (i.e. a cycle), else `false`.
**Example:** `head = [3,2,0,-4], pos = 1` → `true` (tail's `next` points at index 1)

## Idea (1-liner)
Walk two pointers at different speeds — if a cycle exists the fast one will lap the slow one and they will collide inside the loop; otherwise the fast pointer falls off the end (`nullptr`).

## Approach
1. `slow = fast = head`.
2. While `fast && fast->next`: advance `slow` by 1, `fast` by 2. If `slow == fast` → cycle, return `true`.
3. Loop exits ⇒ fast hit a null ⇒ no cycle, return `false`.

**Why they must meet (when a cycle exists):** once both pointers are inside the cycle, the gap from `slow` to `fast` (going forward) shrinks by exactly 1 every step (fast gains 1 per step). So they meet in ≤ `cycle_length` more steps — no skipping past each other.

## Code (clean canonical)
```cpp
bool hasCycle(ListNode *head) {
    ListNode *slow = head, *fast = head;
    while (fast && fast->next) {
        slow = slow->next;
        fast = fast->next->next;
        if (slow == fast) return true;
    }
    return false;
}
```
**TC:** O(n)  **SC:** O(1)

## My Solution
```cpp
class Solution {
public:
    bool hasCycle(ListNode *head) {
      ListNode *slow = head, *fast = head;
      while(fast && fast->next) {
        slow = slow->next;
        fast = fast->next->next;
        if(slow == fast) return true;
      }
      return false;
    }
};
```

> **First cut:** hash-set of visited node pointers — insert as you walk; if `find` hits, return `true`; if you walk off the end, `false`. Correct and easy to reason about, but O(n) extra space. Floyd's matches the time complexity with O(1) space, and the same two-pointer setup extends naturally to **finding the cycle start** ([2. Linked List Cycle II](2.%20Linked%20List%20Cycle%20II.md)), so it's the version worth memorizing.

## Other Approaches
- **Hash set of visited nodes** — O(n) / O(n). Mention as the "obvious" approach in an interview, then upgrade to Floyd's.
- **Destructive marking** (mutate `val` to a sentinel, or `next` to a known node) — O(n) / O(1) but corrupts the input. Don't.

## Pitfalls
- Loop guard must be `fast && fast->next` — checking only `fast` will dereference `nullptr` on `fast->next->next` when the list has even length and ends.
- Compare *after* moving, not before — otherwise the initial `slow == fast` (both `head`) returns `true` spuriously.
- Empty list (`head == nullptr`): the `while` exits immediately ⇒ returns `false`. No separate guard needed.
- Don't confuse with [2. Linked List Cycle II](2.%20Linked%20List%20Cycle%20II.md): that one needs a *second phase* (reset one pointer to head, walk both at speed 1) to find the cycle's entry node.
