# 9. Find the Duplicate Number (LC 287) — Medium
🔗 [Open on LeetCode](https://leetcode.com/problems/find-the-duplicate-number/)

**Pattern:** Fast & Slow Pointers — Floyd's cycle detection on the array-as-function `f(x) = nums[x]` (not a linked list!)
**Trigger:** "`n+1` integers in `[1, n]`, one repeats, find it **without modifying the array** in O(1) space" → treat `i → nums[i]` as a functional graph; the duplicate is the **cycle entrance**

## Problem (1-liner)
An array of `n+1` integers, each in `[1, n]`, contains exactly one value that repeats (possibly many times). Return that value — without modifying `nums` and using O(1) extra space.
**Examples:** `[1,3,4,2,2]` → `2` · `[3,1,3,4,2]` → `3` · `[3,3,3,3,3]` → `3`

## Idea (1-liner)
Read each slot as a pointer (`i → nums[i]`). With `n+1` slots holding values in `[1, n]`, pigeonhole forces two indices to point to the same value ⇒ the graph has a **cycle**, and the node where the cycle begins is the duplicate — so run Floyd's tortoise & hare and return the cycle entrance.

## Approach
1. **Phase 1 — detect the meeting point.** Start `slow = fast = nums[0]`. Advance `slow = nums[slow]` (1×) and `fast = nums[nums[fast]]` (2×) in a `do/while` until they meet inside the cycle. (`do/while` is essential — both start equal, so a top-tested `while` would exit immediately.)
2. **Phase 2 — find the cycle entrance.** Reset a fresh pointer to the start (`nums[0]`); walk it and `slow` at 1× each. They collide exactly at the cycle entrance — the duplicate.
3. **Return** that value.

**Why the entrance is the duplicate:** the only node with *two* incoming arrows is the repeated value (two different indices `i, j` with `nums[i] = nums[j] = dup`). Two incoming arrows = where the cycle closes ⇒ the duplicate is the cycle's entry node.

**Why `a ≡ L−b (mod L)`:** same Floyd's phase-2 math as [Linked List Cycle II](https://leetcode.com/problems/linked-list-cycle-ii/) (LC 142) — distance from head to entry ≡ distance from meeting point to entry, modulo cycle length, so two 1× walkers from those two starts meet at the entry.

## Code (clean canonical)
```cpp
class Solution {
public:
    int findDuplicate(vector<int>& nums) {
        int slow = nums[0], fast = nums[0];
        do {                                   // phase 1: find meeting point
            slow = nums[slow];
            fast = nums[nums[fast]];
        } while (slow != fast);

        int finder = nums[0];                  // phase 2: find cycle entrance
        while (finder != slow) {
            finder = nums[finder];
            slow   = nums[slow];
        }
        return finder;
    }
};
```
**TC:** O(n)  **SC:** O(1) — array untouched.

## Other Approaches
- **Binary search on the value range** — search `m` in `[1, n]`; count how many `nums[i] ≤ m`. If `count > m`, the duplicate lies in `[lo, m]`, else `[m+1, hi]`. O(n log n) / O(1), array untouched. Slower than Floyd's but a solid fallback if the cycle framing doesn't click in an interview.
- **Negative marking** — for each `x`, flip the sign of `nums[abs(x)]`; the first already-negative slot reveals the duplicate. O(n) / O(1) but **modifies the array** → violates the constraint (acceptable only if restoration is allowed).
- **Hash set / frequency count** — trivial O(n) time but O(n) space; violates the constant-space requirement.
- **Bit counting** — count set bits per position across `nums` vs. `[1, n]`; O(n·32) / O(1). Works but fiddly; rarely the expected answer.

## Pitfalls
- **It's NOT a linked-list problem** — the leap is reading the array as a function `f(x) = nums[x]`. Values in `[1, n]` are valid indices, which is what makes the pointer-chase well-defined.
- **`do/while`, not `while`** — `slow` and `fast` both start at `nums[0]`, so a pre-tested loop exits instantly. Step first, then compare.
- **Phase 2 must reset to the start used in phase 1** (`nums[0]`) and walk **both at 1×** — resetting to index `0` or walking at 2× breaks the entrance math.
- **Don't modify the array** (no sorting, no sign-flipping) and **no extra O(n) space** — these constraints are the whole point; hash-set/sort "solutions" miss it.
- **Cycle in a function iteration variant** — same backbone as [Happy Number](https://leetcode.com/problems/happy-number/) (LC 202, `f` = digit-square-sum) and [Linked List Cycle II](https://leetcode.com/problems/linked-list-cycle-ii/) (LC 142, entrance-finding). Here `f(x) = nums[x]`.
