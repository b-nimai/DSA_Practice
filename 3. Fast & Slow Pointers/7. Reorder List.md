# 7. Reorder List (LC 143) — Medium
🔗 [Open on LeetCode](https://leetcode.com/problems/reorder-list/)

**Pattern:** Fast & Slow Pointers — find-middle + reverse-half + interleave-merge
**Trigger:** "reorder to `L0→Ln→L1→Ln-1→…`, modify **nodes not values**, in O(1) space" → split at the middle, reverse the back half, then weave the two halves together

## Problem (1-liner)
Given the head of a singly linked list `L0→L1→…→Ln`, reorder it in place to `L0→Ln→L1→Ln-1→L2→Ln-2→…`. You may only relink nodes, not change their values.
**Examples:** `[1,2,3,4]` → `[1,4,2,3]` · `[1,2,3,4,5]` → `[1,5,2,4,3]`

## Idea (1-liner)
The target list is the **first half** interleaved with the **reversed second half** — so do exactly that: cut the list in two, reverse the back half, and zip the two halves together one node at a time.

## Approach
1. **Find the split point** with slow/fast. Start `fast = head->next` so that on even length the *first* half is the longer/equal one — `slow` then ends on the tail of the first half.
2. **Cut:** `newHead = slow->next; slow->next = nullptr`.
3. **Reverse** the second half (`newHead`) in place → standard prev/curr reversal.
4. **Interleave-merge** the two halves: repeatedly splice one node from the second list after the current node of the first. **Save both `next` pointers before rewiring** each pair, and loop while the (shorter/equal) second half still has nodes.

**Why `fast = head->next`:** it makes `slow` stop on the *last node of the first half* rather than the first node of the second. The first half is then equal-or-one-longer than the second, which is exactly what the `L0→Ln→L1…` interleave wants (the first half "leads"). With `fast = head` you'd have to split one node later.

## Code (clean canonical)
```cpp
class Solution {
    ListNode* reverse(ListNode* head) {
        ListNode* prev = nullptr;
        while (head) {
            ListNode* next = head->next;
            head->next = prev;
            prev = head;
            head = next;
        }
        return prev;
    }
public:
    void reorderList(ListNode* head) {
        if (!head || !head->next || !head->next->next) return;

        // 1. split: slow ends on tail of first half
        ListNode *slow = head, *fast = head->next;
        while (fast && fast->next) { slow = slow->next; fast = fast->next->next; }
        ListNode* second = slow->next;
        slow->next = nullptr;

        // 2. reverse second half
        second = reverse(second);

        // 3. interleave-merge (second is shorter or equal)
        ListNode* first = head;
        while (second) {
            ListNode *f = first->next, *s = second->next;
            first->next = second;
            second->next = f;
            first = f;
            second = s;
        }
    }
};
```
**TC:** O(n) (find middle + reverse + merge, each linear)  **SC:** O(1)

## My Solution
```cpp
class Solution {
    ListNode* reversedList(ListNode* head) {
        ListNode *prev = nullptr;
        while(head) {
            ListNode *next = head->next;
            head->next = prev;
            prev = head;
            head = next;
        }
        return prev;
    }
public:
    void reorderList(ListNode* head) {
       if(!head->next || !head->next->next) return;
       ListNode *slow = head, *fast = head->next;
       while(fast && fast->next) {
           fast = fast->next->next;
           slow = slow->next;
       }
       ListNode *newHead = slow->next;
       slow->next = nullptr;
       ListNode *start1 = head;
       ListNode *start2 = reversedList(newHead);

       while(start1 && start2) {
           ListNode *start1Next = start1->next, *start2Next = start2->next;
           start1->next = start2;
           start1 = start1Next;
           start2->next = start1;
           start2 = start2Next;
       }
    }
};
```

> **Verdict:** correct and optimal — O(n) time, O(1) space, all three steps right. Traced clean on both `[1,2,3,4]` → `[1,4,2,3]` and `[1,2,3,4,5]` → `[1,5,2,4,3]`. The merge correctly caches `start1Next`/`start2Next` *before* rewiring (the part most people break).
>
> **Refinements (style only):**
> - **Guard dereferences `head` first:** `if(!head->next || ...)` reads `head->next` without checking `head`. Constraints guarantee ≥1 node so it can't crash here, but `if(!head || !head->next || !head->next->next)` is the safe habit.
> - **Loop condition:** the second half is never longer than the first, so `while(second)` alone suffices; `start1 && start2` is equally safe, just redundant.
> - **Name:** `reversedList` reads like it returns a copy — `reverse` better signals in-place mutation.

## Other Approaches
- **Store nodes in a vector, two-pointer index from both ends** — `O(n)` time but `O(n)` space; relink `v[0]→v[n-1]→v[1]→v[n-2]…`. Simplest to reason about but loses the O(1)-space point of the exercise.
- **Deque** — push all nodes, alternately pop front/back. Same `O(n)` space tradeoff.

## Pitfalls
- **Forgetting to cut the first half** (`slow->next = nullptr`) → the merge loops forever or revisits nodes; the first half must terminate.
- **Rewiring before saving `next`:** you must cache both `first->next` and `second->next` *before* reassigning, or you lose the rest of a list.
- **Wrong split parity:** with `fast = head` the second half can come out longer, leaving a stray node mis-placed; `fast = head->next` keeps the first half leading. (Either works if your merge handles the leftover correctly, but the leading-first-half split is cleanest.)
- **Small lists:** length ≤ 2 is already reordered — early-return on `!head->next->next` avoids needless work and a degenerate split.
- **This is the "find middle + reverse half" variant** shared with [Palindrome Linked List](https://leetcode.com/problems/palindrome-linked-list/) (LC 234) — same backbone, different finish (compare vs. interleave).
