# 3. Middle of the Linked List (LC 876) — Easy
🔗 [Open on LeetCode](https://leetcode.com/problems/middle-of-the-linked-list/)

**Pattern:** Fast & Slow Pointers — speed-ratio trick (no cycle involved)
**Trigger:** "return the middle node" in **one pass**, O(1) space → two pointers, `fast` at 2×

## Problem (1-liner)
Return the middle node of a singly linked list; if the list has two middles (even length), return the **second** one.
**Examples:** `[1,2,3,4,5]` → `3` · `[1,2,3,4,5,6]` → `4`

## Idea (1-liner)
If `fast` moves twice as fast as `slow` and both start at `head`, then when `fast` reaches the end, `slow` is at the half-way mark — exactly the middle (second middle on even length).

## Approach
1. `slow = fast = head`.
2. While `fast && fast->next`: `slow = slow->next`; `fast = fast->next->next`.
3. Return `slow`.

**Why the loop guard:** `fast->next->next` requires *two* nodes ahead, so check both. The guard naturally produces the **second middle** on even length — when `fast` lands on the last node (odd), `fast->next == nullptr` and we stop; when `fast` falls off (even), we've already advanced `slow` one extra step past the first middle.

## Code (clean canonical)
```cpp
ListNode* middleNode(ListNode* head) {
    ListNode *slow = head, *fast = head;
    while (fast && fast->next) {
        slow = slow->next;
        fast = fast->next->next;
    }
    return slow;
}
```
**TC:** O(n)  **SC:** O(1)

## My Solution
```cpp
class Solution {
public:
    ListNode* middleNode(ListNode* head) {
      ListNode *fast = head, *slow = head;
      while(fast && fast->next) {
        fast = fast->next->next;
        slow = slow->next;
      }
      return slow;
    }
};
```

> **No refinements needed** — this is the canonical Floyd's-style middle-finder. Only nits are stylistic: declare `slow` before `fast` (convention, since `slow` is the returned pointer), prefer `nullptr` over `NULL`. Algorithm cannot be improved.

## Other Approaches
- **Two-pass: count then walk `n/2`** — O(n) / O(1), but two passes instead of one. The speed-ratio trick is strictly better.
- **Store nodes in a vector, return `arr[arr.size()/2]`** — O(n) / O(n). Wasteful.

## Pitfalls
- **First vs second middle:** this template returns the **second** middle on even length. If you need the **first** middle (e.g. for [LC 143 Reorder List](https://leetcode.com/problems/reorder-list/) or [LC 234 Palindrome Linked List](https://leetcode.com/problems/palindrome-linked-list/) — you want to split with the first half ≤ second), use `fast = head->next` instead of `fast = head`. Same loop body.

  | Setup | `[1,2,3,4]` returns |
  |-------|---------------------|
  | `slow = fast = head` (this problem) | `3` (second middle) |
  | `slow = head; fast = head->next` | `2` (first middle) |
- Loop guard must be `fast && fast->next` — checking only `fast` will dereference null on `fast->next->next` when the list has even length.
- Empty list `head == nullptr`: loop exits immediately, returns `nullptr`. Per constraints `n ≥ 1` here, but the code is safe regardless.
- Compare *moves*, not *equality*: this isn't cycle detection — there's no `if(slow == fast) ...`. Don't muscle-memory it in from [1. Linked List Cycle](1.%20Linked%20List%20Cycle.md).
