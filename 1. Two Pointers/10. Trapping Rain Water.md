# 10. Trapping Rain Water (LC 42) — Hard
🔗 [Open on LeetCode](https://leetcode.com/problems/trapping-rain-water/)

**Pattern:** Two Pointers — opposite ends + running `leftMax` / `rightMax`
**Trigger:** elevation map / water trapping / "min of two side maxes"

## Problem (1-liner)
Each bar has width 1; compute total water trapped above the bars.
**Example:** `[0,1,0,2,1,0,1,3,2,1,2,1]` → `6`

## Idea (1-liner)
Water at `i` = `min(maxLeft[i], maxRight[i]) - h[i]`. With two pointers, the **smaller side's** running max is the binding wall — commit water there safely.

## Approach
1. `l = 0`, `r = n - 1`, `lMax = rMax = 0`, `water = 0`.
2. While `l < r`:
   - If `h[l] < h[r]` (right side guarantees a taller wall):
     - `h[l] >= lMax` → `lMax = h[l]`
     - else `water += lMax - h[l]`
     - `l++`
   - Else (symmetric on right):
     - `h[r] >= rMax` → `rMax = h[r]`
     - else `water += rMax - h[r]`
     - `r--`

## Code (clean canonical)
```cpp
int trap(vector<int>& h) {
    int l = 0, r = h.size() - 1;
    int lMax = 0, rMax = 0, water = 0;
    while (l < r) {
        if (h[l] < h[r]) {
            (h[l] >= lMax) ? lMax = h[l] : water += lMax - h[l];
            l++;
        } else {
            (h[r] >= rMax) ? rMax = h[r] : water += rMax - h[r];
            r--;
        }
    }
    return water;
}
```
**TC:** O(n)  **SC:** O(1)

## My Solution
```cpp
class Solution {
public:
    int trap(vector<int>& height) {
      int left = 0, right = height.size() - 1;
      int leftMax = 0, rightMax = 0, water = 0;
      while(left < right) {
        if(height[left] < height[right]) {
          if(height[left] >= leftMax) leftMax = height[left];
          else water += leftMax - height[left];
          left++;
        } else {
          if(height[right] >= rightMax) rightMax = height[right];
          else water += rightMax - height[right];
          right--;
        }
      }
      return water;
    }
};
```

## Other Approaches
- **Brute force:** for each `i`, scan left & right for max → O(n²) / O(1)
- **Prefix max + suffix max arrays:** precompute `maxL[i]`, `maxR[i]`, then sum `min - h[i]` → O(n) / O(n)
- **Monotonic decreasing stack:** for each bar, pop and add water layer-by-layer → O(n) / O(n)

## Why is the smaller side's max binding?
If `h[l] < h[r]`, the right side has at least one bar `≥ h[r] > h[l]`. So the **true** rightMax (whatever it is) is `≥ h[r]`, hence `≥ lMax`. So `min(lMax, trueRightMax) = lMax` → safe to commit water at `l` using only `lMax`.

## Pitfalls
- Compare `h[l] < h[r]`, then update **that side's** max first
- Use `>=` (not `>`) when updating max so `water` doesn't go negative
- Stack approach computes water in horizontal layers, not vertical — different mental model
