# 11. 124. Binary Tree Maximum Path Sum (LC 124) — Hard
🔗 [Open on LeetCode](https://leetcode.com/problems/binary-tree-maximum-path-sum/)

**Pattern:** Tree DFS — **post-order "bend at a node"**; the answer is a global max updated at every node, while each call *returns* only a single straight arm to its parent (twin of [[4-543-diameter-of-binary-tree|543 Diameter]], summing values instead of counting edges)
**Trigger:** "**max path sum**, path may start/end **anywhere** and need not pass the root" → DFS returning `val + max(0, bestArm)` upward, and at each node update a global with `val + leftArm + rightArm`

## Problem (1-liner)
A path is any sequence of edge-connected nodes (each used once); return the **largest sum of node values** over all non-empty paths. The path need not touch the root.
**Examples:** `[1,2,3]` → `2→1→3 = 6` · `[-10,9,20,null,null,15,7]` → `15→20→7 = 42` (skips the root) · `[-3]` → `-3` (a single node is a valid path).
**Constraints:** `1 … 3·10⁴` nodes, values in `[-1000, 1000]` → **negatives exist**, which is the whole difficulty.

## Idea (1-liner)
Every path has a single **highest point** where it "bends" — it goes down into the left subtree, up to this node, and down into the right. At each node the best bent path is `node + leftArm + rightArm`; track that in a **global max**. But you can only hand your parent a **straight arm** (it will bend at *its* own apex), so you *return* `node + max(leftArm, rightArm)`. Crucially, **a negative arm is never worth taking** — clamp each child's arm to `0` before using it (skipping it gives a better path).

## Approach
1. **Base:** `null` → `0` (an absent child contributes nothing).
2. Recurse to get `left` and `right` arms, and **clamp each to `0`**: `left = max(0, dfs(node->left))`. A negative arm means "don't extend into that subtree."
3. **Update the global** with the path that bends here: `best = max(best, node->val + left + right)`.
4. **Return the straight arm** the parent can extend: `node->val + max(left, right)`.
5. Answer is the global, **not** the root's return value.

> **The bug everyone hits (and the one in my first attempt): forgetting to clamp the arm at `0`.** If you return `node->val + max(left, right)` with *unclamped* child arms, a negative arm propagates up and `max` between two negatives stays negative, silently corrupting every ancestor. Clamp the arms, not `node->val` — see Pitfalls.

## Code (clean canonical)
```cpp
class Solution {
    int dfs(TreeNode* root, int& best) {
        if (!root) return 0;

        int left  = max(0, dfs(root->left,  best));   // drop negative arms
        int right = max(0, dfs(root->right, best));

        best = max(best, root->val + left + right);    // best path bending here
        return root->val + max(left, right);           // straight arm for the parent
    }
public:
    int maxPathSum(TreeNode* root) {
        int best = INT_MIN;                            // single negative node can be the answer
        dfs(root, best);
        return best;
    }
};
```
**TC:** O(n) — each node visited once.  **SC:** O(h) recursion stack — O(log n) balanced, O(n) skewed.

## My Solution (buggy first cut → fix)
```cpp
// ❌ WRONG — returned arm is not clamped at 0
class Solution {
    int helper(TreeNode *root, int &maxi) {
        if (!root) return 0;
        int leftSum  = helper(root->left, maxi);
        int rightSum = helper(root->right, maxi);

        maxi = max(maxi, root->val + leftSum + rightSum);
        maxi = max(maxi, root->val);
        maxi = max(maxi, root->val + leftSum);
        maxi = max(maxi, root->val + rightSum);

        return max(leftSum, rightSum) + root->val;   // ← bug: arm not clamped to 0
    }
public:
    int maxPathSum(TreeNode* root) {
        int maxi = -1001;
        helper(root, maxi);
        return maxi;
    }
};
```
**Right shape, one missing clamp.** The four `maxi` updates correctly cover the *local* node, so the bend value at each node is fine **in isolation**. The defect is purely the **return**: `max(leftSum, rightSum) + root->val` ships a *negative* arm up to the parent. A parent that does `root->val + leftSum` then eats that negative instead of treating the branch as "not taken," and the error compounds up the tree.

**Failing case `[9,6,-3,null,null,-6,2,null,null,2,null,-6,-6,-6]` → got `15`, expected `16`:**
```
   -3
  /  \
-6    2     (A)
     /
    2       (B)   ← two -6 children
   / \
 -6   -6
```
- Node **2(B)** should return arm `2 + max(0,-6,-6) = 2`; buggy code returns `max(-6,-6)+2 = -4`.
- Node **2(A)** then computes `max(-4,0)+2 = 2` instead of the true `2+2 = 4`.
- Node **-3** gets arm `-1` instead of `1`; at root **9**, best path `6→9→-3→2→2 = 16` is missed → only `9+6 = 15`.

**Fix:** clamp every child arm to `0` *before* using it (`int left = max(0, helper(...))`). That single change makes 2(B)'s arm `2`, 2(A)'s `4`, -3's `1`, and the root path `16` surfaces. Bonus: with clamped arms ≥ 0, the four `maxi` lines collapse into one `root->val + left + right`, since "node alone" and "node + one arm" are just the cases where an arm is `0`.

## Other Approaches
- **Return a struct `{maxArm, bestPath}`** instead of a global — same O(n), purely stylistic; some prefer it to avoid the by-reference accumulator. The global/by-ref version is shorter and the interview default.
- **Two passes (compute all arms, then combine)** — needless; the single post-order pass already has both children's arms in hand at the parent, exactly when it needs them.

## Family map
- **Twin (the same "bend at a node, return one arm, update a global" skeleton):** [[4-543-diameter-of-binary-tree|543 Diameter]] — identical structure; diameter sums **edge counts** (`L + R`, arms = `1 + max(L,R)`, base `0`), this sums **node values** with the extra **clamp-at-0** because values can be negative. If you know one, you know both.
- **Seed:** [[1-104-maximum-depth-of-binary-tree|104 Maximum Depth]] — the `height = 1 + max(L, R)` recursion both 543 and 124 are built on.
- **Cousin (post-order combine of two children):** [[9-236-lowest-common-ancestor-of-a-binary-tree|236 LCA]] — recurse both sides, merge at the node; there the merged thing is "which targets were seen" rather than a sum.

## Pitfalls
- **Clamp the *arms* at `0`, never `node->val`** — a negative subtree should be dropped (`max(0, arm)`), but the node itself is mandatory in any path through it. Clamping `node->val` would wrongly let you "skip" a node mid-path, which paths can't do.
- **Initialize the global to `INT_MIN`, not `0`** — all-negative trees (and the single-node `[-3]`) have a negative answer. Starting at `0` returns `0`, which isn't a valid non-empty path sum. (`-1001` also works here since `val ≥ -1000`, but `INT_MIN` is the safe habit.)
- **Return value ≠ answer** — `dfs` returns a *straight arm* for the parent; the actual answer is the **global** updated with `val + left + right`. Returning `left + right + val` upward would let a parent reuse a forbidden bent path.
- **Bent path is `left + right + val`; arm is `max(left,right) + val`** — don't confuse the two. The bend can't be passed up (a parent can't pass *through* a node twice), so what you return and what you record are different quantities.
- **Don't forget either subtree can be the apex** — that's why the global is updated at *every* node, not just the root; the optimal path may sit entirely in one subtree and never reach the root.
