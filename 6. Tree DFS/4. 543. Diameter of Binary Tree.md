# 4. 543. Diameter of Binary Tree (LC 543) — Easy
🔗 [Open on LeetCode](https://leetcode.com/problems/diameter-of-binary-tree/)

**Pattern:** Tree DFS — **post-order "return info up" + side-channel global** (height recursion of [[1-104-maximum-depth-of-binary-tree|104]], but you steal `L + R` on the way up)
**Trigger:** "longest path between **any** two nodes (may not pass through root), counted in **edges**" → reuse the height recursion, and at every node update a global with `L + R`

## Problem (1-liner)
Return the **diameter** of a binary tree = length (in **edges**) of the longest path between *any* two nodes. The path **need not pass through the root**.
**Examples:** `[1,2,3,4,5]` → `3` (path `4→2→1→3` or `5→2→1→3`) · `[1,2]` → `1`
**Constraints:** `1 … 10^4` nodes, `-100 ≤ node.val ≤ 100`.

## Idea (1-liner)
The longest path **bends at exactly one node** — its highest point. At that node the path goes down the left subtree as far as possible and down the right subtree as far as possible, so its edge-count is `height(left) + height(right)`. We don't know *which* node is the bend, so run the ordinary height recursion ([[1-104-maximum-depth-of-binary-tree|104]]) and, **at every node**, update a global `maxi = max(maxi, L + R)`. One post-order pass finds the best bend.

## Approach
1. **Reuse height-in-edges:** `dfs(node)` returns the number of **edges** from `node` down to its deepest leaf. Empty child ⇒ `0`.
2. **Recurse** to get `L = dfs(left)`, `R = dfs(right)`.
3. **Side-channel update:** `maxi = max(maxi, L + R)` — the longest path that *bends here* (left-down + right-down edges).
4. **Return upward:** `1 + max(L, R)` — the parent only cares about the single deepest arm, not the bend.
5. Answer is `maxi`, **not** the function's return value. Pass `maxi` by reference (or keep it as a member).

> **Why "bend at one node" is enough:** any simple path in a tree has a unique top-most node. Checking `L + R` at every node therefore considers every possible path exactly once — no path is missed, none double-counted.

## Code (clean canonical)
```cpp
class Solution {
    int dfs(TreeNode* root, int& maxi) {
        if (!root) return 0;                  // empty child contributes 0 edges
        int L = dfs(root->left,  maxi);
        int R = dfs(root->right, maxi);
        maxi = max(maxi, L + R);              // longest path *bending* at this node (edges)
        return 1 + max(L, R);                 // height in edges: deepest single arm + this edge
    }
public:
    int diameterOfBinaryTree(TreeNode* root) {
        int maxi = 0;
        dfs(root, maxi);
        return maxi;
    }
};
```
**TC:** O(n) — each node visited once.  **SC:** O(h) recursion stack — O(log n) balanced, O(n) skewed.

## My Solution
```cpp
class Solution {
    int helper(TreeNode *root, int &maxi) {
        if (!root) return 0;
        if(!root->left && !root->right) return 1;   // redundant — see below

        int l = helper(root->left, maxi);
        int r = helper(root->right, maxi);

        maxi = max(maxi, l + r);
        return 1 + max(l, r);
    }
public:
    int diameterOfBinaryTree(TreeNode* root) {
        int maxi = 0;
        helper(root, maxi);
        return maxi;
    }
};
```
**Correct**, same O(n)/O(h) — passing `maxi` by reference is exactly the side-channel trick. The explicit leaf base case (`return 1`) is **redundant**: without it a leaf computes `l = r = 0` ⇒ `maxi` unchanged and `return 1 + max(0,0) = 1`, identical result. Dropping it removes a line and a second exit point for zero behavioral change. Minor style: the repo's other notes attach `*`/`&` to the type (`TreeNode* root`, `int& maxi`).

## Other Approaches
- **Return a pair `{height, diameter}`** instead of a by-reference global — purely functional, no shared state: `dfs` returns `{1 + max(hL,hR), max({dL, dR, hL + hR})}`. Same O(n)/O(h); preferred when you dislike side effects, but the global is shorter and just as clear here.
- **Two-pass / brute force** — for every node compute `height(left) + height(right)` by calling a separate `height()`. That recomputes heights repeatedly ⇒ **O(n²)** (O(n log n)-ish balanced). Don't; the single-pass side-channel is strictly better.

## Pitfalls
- **Returning `maxi` from the helper** — the helper must return **height** so the parent can extend the path; the diameter lives in the global. Returning `L + R` upward breaks the recursion.
- **`L + R` vs `1 + max(L, R)`** — *diameter at this node* sums both arms (`L + R`); *height for the parent* takes the deeper arm plus one (`1 + max(L, R)`). Two different combine steps in the same function — don't swap them.
- **Edges vs nodes** — this problem counts **edges**, so a single node is `0` and `[1,2]` is `1`. Because the null base returns `0`, `L + R` already yields edges directly; if you counted nodes (null → `1`) you'd be off by one and need `L + R - 1`.
- **Update at *every* node, not just the root** — the longest path may bend deep in a subtree ("may or may not pass through the root"). Updating `maxi` only at the root misses those.
