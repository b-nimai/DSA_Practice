## When to reach for Tree DFS
1. You need a **path / sum / depth / diameter** in a binary tree, or to **validate** a structural property.
2. The answer at a node depends on **information combined from both subtrees** â€” recurse, then merge `L` and `R`.
3. Reach for **BFS** instead when you need per-level info or **minimum** depth (BFS short-circuits at the first leaf).

## Common Pitfalls Across the Pattern
- **Base case value** â€” an empty child usually returns `0` (depth) or a neutral sentinel; returning the wrong base shifts every count by one.
- **`max(L, R)` vs `L + R`** â€” depth/height takes the deeper subtree (`max`); diameter-style "path through this node" sums both (`L + R`). Know which the question wants.
- **Nodes vs edges** â€” depth counted in nodes makes a single node `1`; in edges it's `0`. Re-read the definition each problem.
