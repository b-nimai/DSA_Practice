# 2. 100. Same Tree (LC 100) — Easy
🔗 [Open on LeetCode](https://leetcode.com/problems/same-tree/)

**Pattern:** Tree DFS — **parallel pre-order traversal of two trees** (compare structure + values lock-step)
**Trigger:** "are these two trees identical (same shape **and** same values)" → walk both at once; at every step check the **three base cases** (both null / one null / values differ), then recurse left-with-left and right-with-right

## Problem (1-liner)
Given roots `p` and `q`, return `true` iff the two trees are **structurally identical** and every corresponding node has the **same value**.
**Examples:** `[1,2,3] , [1,2,3]` → `true` · `[1,2] , [1,null,2]` → `false` (shape differs) · `[1,2,1] , [1,1,2]` → `false` (values differ)
**Constraints:** `0 … 100` nodes each, `-10^4 ≤ node.val ≤ 10^4`.

## Idea (1-liner)
Two trees are the same iff their roots match **and** their left subtrees are the same **and** their right subtrees are the same — a recursive definition that maps straight to code. Advance **both** pointers in lockstep; the recursion fails fast the moment shapes diverge (exactly one pointer null) or a pair of values disagrees. This is the two-pointer cousin of [[1-104-maximum-depth-of-binary-tree|max depth]]: same DFS skeleton, but you carry **two** nodes and return a `bool`.

## Approach
1. **Both null** ⇒ `true` — two empty (sub)trees are trivially equal; this is the success base case at the bottom of every matching path.
2. **Exactly one null** ⇒ `false` — shapes diverge (handled by the order: the both-null check already passed, so `||` catches the "one is null" case).
3. **Values differ** (`p->val != q->val`) ⇒ `false` — structure matches so far but content doesn't.
4. **Recurse in parallel:** `isSameTree(p->left, q->left) && isSameTree(p->right, q->right)` — `&&` short-circuits, so the first mismatch stops the work.

## Code (clean canonical)
```cpp
class Solution {
public:
    bool isSameTree(TreeNode* p, TreeNode* q) {
        if (p == nullptr && q == nullptr) return true;     // both empty -> equal
        if (p == nullptr || q == nullptr) return false;    // exactly one empty -> shape differs
        if (p->val != q->val)            return false;     // values disagree
        return isSameTree(p->left,  q->left)               // recurse both subtrees in lockstep
            && isSameTree(p->right, q->right);
    }
};
```
**TC:** O(min(n, m)) — stops at the smaller tree the moment shapes diverge; O(n) when equal.  **SC:** O(min(h_p, h_q)) recursion stack.

## My Solution
Identical to the canonical (split the final `&&` into two named `bool`s) — clean and correct. The only nit: naming `leftResult`/`rightResult` evaluates **both** before returning, which forfeits a tiny bit of short-circuiting. `return isSameTree(left…) && isSameTree(right…);` skips the right recursion the instant the left subtree mismatches. Negligible at n ≤ 100, but the inline form is the idiom.

## Other Approaches
- **Iterative with a stack/queue of pairs** — push `{p, q}`, pop, run the same three checks, push `{p->left,q->left}` and `{p->right,q->right}`. O(n)/O(n). Use when recursion depth is a concern (not here).
- **Serialize-and-compare** — preorder serialize both **with explicit null markers** (`#`) and compare the strings. The null markers are essential — without them `[1,2]` and `[1,null,2]` serialize the same. Slower and O(n) extra space; only worth it if you'd cache/compare many trees.

## Pitfalls
- **Base-case order matters:** check `both null` **before** `one null`. If you wrote `one null ⇒ false` first you'd reject two empty subtrees. The canonical order (`&&` null, then `||` null, then value) is what makes it correct.
- **Don't dereference before the null guards** — touching `p->val` while `p` is null is UB. The two null checks must come first.
- **Structure vs values are *both* required** — a value-only check passes `[1,2]` vs `[1,null,2]`; a structure-only check passes `[1,2,1]` vs `[1,1,2]`. You need all three base cases.
- **Cousin problems:** flip the recursion to compare a tree against its **mirror** and you get [[3-101-symmetric-tree|Symmetric Tree (101)]] (`left↔right`, `right↔left`); ask "is `q` a subtree of `p`" and you call `isSameTree` at every node of `p` → [[4-572-subtree-of-another-tree|Subtree of Another Tree (572)]]. Same lockstep skeleton, different pairing.
