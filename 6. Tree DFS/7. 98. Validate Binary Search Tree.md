# 7. 98. Validate Binary Search Tree (LC 98) — Medium
🔗 [Open on LeetCode](https://leetcode.com/problems/validate-binary-search-tree/)

**Pattern:** Tree DFS — **carry an allowed `(min, max)` range down**; every node must fit *its ancestors'* bounds, not just its parent's
**Trigger:** "is this a **valid BST**" (left subtree *strictly* <, right subtree *strictly* >, recursively) → DFS passing an open interval `(low, high)`; tighten one bound on each side as you descend

## Problem (1-liner)
Return `true` iff the tree is a valid BST: for **every** node, the *entire* left subtree is strictly less and the *entire* right subtree is strictly greater (not just the immediate children).
**Examples:** `[2,1,3]` → `true` · `[5,1,4,null,null,3,6]` → `false` (right child `4 < 5`) · `[5,4,6,null,null,3,7]` → `false` (`3` sits in 5's right subtree but `3 < 5`)
**Constraints:** `1 … 10^4` nodes, `-2^31 ≤ node.val ≤ 2^31 - 1` (full `int` range — bounds can hit `INT_MIN`/`INT_MAX`).

## Idea (1-liner)
A node is legal only if it lies inside the open interval `(low, high)` it inherited from its ancestors. Going **left**, the current node becomes the new **upper** bound (everything left must be `< node`); going **right**, it becomes the new **lower** bound. A leaf is fine as long as it fits, and `null` is trivially valid. This is why a *local* parent-vs-child check is not enough: a grandchild can satisfy its parent yet violate an ancestor.

## Approach
1. **Null ⇒ `true`** — an empty subtree can't break the order.
2. **Range test:** the node is valid only if `low < node->val < high` (strict on both sides — equal values are *not* allowed in this problem).
3. **Recurse, tightening one bound:**
   - Left child inherits `(low, node->val)` — node is the new ceiling.
   - Right child inherits `(node->val, high)` — node is the new floor.
4. **AND the two sides** — every node in *both* subtrees must hold, so it's `left && right`.
5. **Seed the root** with the widest possible interval: `low = -∞`, `high = +∞`.

> **Why range-passing beats local checks.** The naive "compare each node only to its direct children" passes `[5,4,6,null,null,3,7]` because `3 < 6` looks fine locally — but `3` is in 5's right subtree, so it must be `> 5`. The inherited interval carries that ancestor constraint down, so the violation is caught at `3` (its `low` is `5`).

## Code (clean canonical)
```cpp
class Solution {
    bool valid(TreeNode* node, long low, long high) {
        if (!node) return true;                       // empty subtree is fine
        if (node->val <= low || node->val >= high)    // must fit ancestors' range (strict)
            return false;
        return valid(node->left,  low, node->val)     // node is the new upper bound
            && valid(node->right, node->val, high);   // node is the new lower bound
    }
public:
    bool isValidBST(TreeNode* root) {
        return valid(root, LONG_MIN, LONG_MAX);       // widen past int so INT_MIN/MAX fit
    }
};
```
**TC:** O(n) — each node visited once.  **SC:** O(h) recursion stack — O(log n) balanced, O(n) skewed.

## My Solution
```cpp
class Solution {
    bool isValidBSTHelper(TreeNode *root, long long maxi, long long mini) {
        if (!root) return true;

        // check if the root value within range or not
        if (mini < root->val && root->val < maxi) {
            return isValidBSTHelper(root->left, root->val, mini)    // shrink max → root->val
                && isValidBSTHelper(root->right, maxi, root->val);  // raise min → root->val
        }

        return false;
    }
public:
    bool isValidBST(TreeNode* root) {
        long long maxi = LONG_MAX, mini = LONG_MIN;
        return isValidBSTHelper(root, maxi, mini);
    }
};
```
**Correct.** Same range-passing idea, just with the bounds in `(max, min)` order — going left you drop `maxi` to `root->val`, going right you lift `mini` to `root->val`. The strict `mini < val < maxi` handles the no-duplicates rule. Using `long long` (seeded `LONG_MAX`/`LONG_MIN`) is what saves you when `node->val == INT_MIN` or `INT_MAX` — with `int` bounds the comparison `val < INT_MAX` would wrongly reject a legal `INT_MAX` leaf. Same O(n)/O(h) as canonical.

**Your first attempt (why it failed):**
```cpp
// WRONG — only compares each node to its direct children
if ((root->left  && root->val <= root->left->val) ||
    (root->right && root->val >= root->right->val)) return false;
return isValidBST(root->left) && isValidBST(root->right);
```
This is a *local* check. It accepts `[5,4,6,null,null,3,7]` because at node `6` the child `3` satisfies `6 > 3`, but `3` is in **5's** right subtree and must be `> 5`. A node can be fine versus its parent yet violate a grandparent — you need the inherited interval, not a one-level comparison.

## Other Approaches
- **In-order traversal must be strictly increasing** — a BST's in-order walk yields sorted values. Traverse in-order keeping `prev` (a pointer or `long long`); if `cur->val <= prev` at any step, return `false`. Often the cleanest: O(n) time, O(h) space, and no bounds bookkeeping. Can be done iteratively with an explicit stack to dodge recursion depth.
- **Iterative range DFS** — push `(node, low, high)` triples on a stack; same logic without recursion, useful against a 10⁴-deep skewed tree.

## Pitfalls
- **Local parent-child check is insufficient** — the bug above. You must carry **ancestor** bounds down via an inherited `(low, high)` interval, not just compare a node to its immediate children.
- **Strict, not `<=` / `>=`** — duplicates are *invalid* here. The range test is `low < val < high`; using `<=`/`>=` on the bounds would wrongly allow equal values.
- **Integer overflow on the sentinels** — `node->val` spans the full `int` range incl. `INT_MIN`/`INT_MAX`. Seed bounds with `long long` (`LONG_MIN`/`LONG_MAX`), or use `optional`/pass the parent node instead of a numeric sentinel; plain `int` bounds break on `INT_MIN`/`INT_MAX` leaves.
- **Which bound tightens on which side** — left child lowers the **upper** bound to `node->val`; right child raises the **lower** bound. Swapping them silently inverts the BST rule.
- **In-order alternative needs strict `prev`** — initialize `prev` to a sentinel below `INT_MIN` (use `long long`) and compare `cur <= prev` for the strict-increase test.
