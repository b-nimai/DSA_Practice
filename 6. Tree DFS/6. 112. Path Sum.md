# 6. 112. Path Sum (LC 112) — Easy
🔗 [Open on LeetCode](https://leetcode.com/problems/path-sum/)

**Pattern:** Tree DFS — **root→leaf path with a running budget**; subtract as you descend, check at the leaf, OR the two branches
**Trigger:** "is there a **root-to-leaf** path that sums to `targetSum`" → DFS carrying the remaining target; at a **leaf** test `remaining == node->val`, and bubble up `left || right`

## Problem (1-liner)
Return `true` iff some **root-to-leaf** path's node values add up exactly to `targetSum` (a leaf has **no** children).
**Examples:** `[5,4,8,11,null,13,4,7,2,null,null,null,1], 22` → `true` (5→4→11→2) · `[1,2,3], 5` → `false` · `[], 0` → `false`
**Constraints:** `0 … 5000` nodes, `-1000 ≤ node.val ≤ 1000`, `-1000 ≤ targetSum ≤ 1000`.

## Idea (1-liner)
Walk down carrying how much target is **left** after spending the current node's value. The decision "does a qualifying path exist below me?" is `OR` over the two children — one good path anywhere is enough — and the test only fires at a **leaf**, because the path must *end* at a leaf. Empty tree is `false` (no path at all), which is why you can't test "remaining == 0" at a null child.

## Approach
1. **Null tree ⇒ `false`** — there is no root-to-leaf path, so even `targetSum == 0` is false.
2. **Spend** the current node: `remaining = targetSum - node->val`.
3. **Leaf check:** if `node` has no children, the path ends here ⇒ return `remaining == 0`.
4. **Recurse & OR:** otherwise return `hasPathSum(left, remaining) || hasPathSum(right, remaining)`. A null child contributes `false` and the `||` short-circuits the moment one side succeeds.

> **Why subtract instead of accumulate?** Carrying the *remaining* target lets each call answer independently with no shared mutable state — there is **nothing to undo**. The running-sum + `&sum` reference version works too, but forces correct backtracking (subtract `node->val` only after *both* children are explored). See *My Solution*.

## Code (clean canonical)
```cpp
class Solution {
public:
    bool hasPathSum(TreeNode* root, int targetSum) {
        if (!root) return false;                              // no root→leaf path

        int remaining = targetSum - root->val;
        if (!root->left && !root->right)                      // leaf: path ends here
            return remaining == 0;

        return hasPathSum(root->left,  remaining)
            || hasPathSum(root->right, remaining);            // one good side is enough
    }
};
```
**TC:** O(n) — each node visited once.  **SC:** O(h) recursion stack — O(log n) balanced, O(n) skewed.

## My Solution
```cpp
class Solution {
    bool helper(TreeNode *root, int target, int &sum) {
        sum += root->val;
        if (!root->left && !root->right) {
            if (sum == target) return true;
            sum -= root->val;
            return false;
        }

        bool leftRes = false;
        if (root->left) leftRes = helper(root->left, target, sum);
        if (leftRes) return true;

        bool rightRes = false;
        if (root->right) rightRes = helper(root->right, target, sum);
        if (rightRes) return true;

        sum -= root->val;        // undo only after BOTH children explored

        return false;
    }
public:
    bool hasPathSum(TreeNode* root, int targetSum) {
        if (!root) return false;
        int sum = 0;
        return helper(root, targetSum, sum);
    }
};
```
**Correct after the fix** — carries a running `sum` by reference and backtracks. The bug you fixed: the `sum -= root->val` was sitting **between** the left and right recursions, so the right subtree was explored *without* this node's value in `sum` (every right path off by `root->val`). Moving the subtract to after *both* children — the single "undo right before returning false" point — restores it. Same O(n)/O(h) as canonical. The subtract-the-budget version above sidesteps backtracking entirely (nothing shared, nothing to undo), so it's the safer default — but a `&sum` reference is the right muscle when the problem also needs the *actual path* (next problem up, 113/437).

## Other Approaches
- **Iterative DFS with an explicit stack** — push `(node, remainingAfterThisNode)` pairs; on popping a leaf whose remaining is `0`, return `true`. Same O(n)/O(h), avoids recursion-depth limits on a 5000-deep skewed tree.
- **Running-sum (accumulate) variant** — carry `currentSum` down and at the leaf test `currentSum == targetSum`. Equivalent to subtracting; the *My Solution* `&sum` version is this with shared state, which is why it needs backtracking.

## Pitfalls
- **Don't check `remaining == 0` at a null node** — that returns `true` for an empty tree with `targetSum == 0` (wrong, no path exists) and **double-counts**: a one-child node would let the missing side "succeed". The end-of-path test must be the **leaf** condition `!left && !right`, not `!root`.
- **Negative values mean you can't prune early** — values down to `-1000`, so a path can dip below and climb back to the target; never stop because `remaining < 0`.
- **Backtracking placement (the bug you hit)** — with a shared running `sum`, undo `node->val` **after both** children, at the single return-false exit. Subtracting between left and right corrupts the right subtree's sum. Or sidestep it: pass the budget by value.
- **It's `||`, not `&&`** — you need *one* qualifying path, not both subtrees to qualify. And it must reach a **leaf** — a partial path that hits the target mid-tree doesn't count.
