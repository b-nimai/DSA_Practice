# 10. 105. Construct Binary Tree from Preorder and Inorder Traversal (LC 105) — Medium
🔗 [Open on LeetCode](https://leetcode.com/problems/construct-binary-tree-from-preorder-and-inorder-traversal/)

**Pattern:** Tree DFS — **build, not traverse**; `preorder` hands you roots top-down, `inorder` tells you where each root splits its subtree into left/right
**Trigger:** "**reconstruct the tree** from preorder + inorder" → walk preorder left-to-right as a root stream (shared pointer), and for each root use its inorder position to carve the left/right index ranges

## Problem (1-liner)
Given the **preorder** and **inorder** traversals of a tree with **unique** values, rebuild and return the tree.
**Example:** `preorder=[3,9,20,15,7]`, `inorder=[9,3,15,20,7]` → root `3`, left subtree `{9}`, right subtree `{20,15,7}`.
**Constraints:** `1 … 3000` nodes, **all values unique** (this is what makes the inorder-index lookup possible).

## Idea (1-liner)
**Preorder = root, then left, then right** → the next unused preorder element is always the root of the subtree you're currently building. **Inorder = left, root, right** → once you know the root, its index in inorder splits the inorder range into *everything left of it* (left subtree) and *everything right of it* (right subtree). Consume preorder front-to-back with a **single shared index**, and recurse on inorder index ranges. A hashmap `value → inorder index` turns the "find the root in inorder" step from O(n) into O(1).

## Approach
1. **Precompute** `mpp[value] = its index in inorder` so the split point is O(1) per node.
2. Keep **one shared `preIdx`** (by reference) marching through `preorder`. Each call grabs `preorder[preIdx++]` as its root — this guarantees roots are consumed in exactly preorder order.
3. Carry the current **inorder window** `[inStart, inEnd]` (the slice of inorder belonging to this subtree).
4. **Base case:** `inStart > inEnd` → empty range → `nullptr`.
5. Look up `mid = mpp[rootVal]`. Recurse **left first** on `[inStart, mid-1]`, then **right** on `[mid+1, inEnd]`. Left must go first so `preIdx` advances through the entire left subtree before the right subtree claims its roots.

> **Why left-before-right is non-negotiable.** Preorder lays out *root → (whole left subtree) → (whole right subtree)*. Since `preIdx` is shared and advances on every node, you must finish consuming the left subtree's preorder slice before touching the right. Swap the two recursions and the indices desync → wrong tree.

## Code (clean canonical)
```cpp
class Solution {
    unordered_map<int,int> pos;          // inorder value -> index
    int preIdx = 0;

    TreeNode* build(vector<int>& preorder, int inStart, int inEnd) {
        if (inStart > inEnd) return nullptr;          // empty range

        int rootVal = preorder[preIdx++];             // next preorder elem is this subtree's root
        TreeNode* root = new TreeNode(rootVal);
        int mid = pos[rootVal];                       // split point in inorder

        root->left  = build(preorder, inStart, mid - 1);
        root->right = build(preorder, mid + 1, inEnd);
        return root;
    }
public:
    TreeNode* buildTree(vector<int>& preorder, vector<int>& inorder) {
        for (int i = 0; i < inorder.size(); i++) pos[inorder[i]] = i;
        return build(preorder, 0, inorder.size() - 1);
    }
};
```
**TC:** O(n) — each node built once, split is O(1) via the map.  **SC:** O(n) — hashmap + O(h) recursion stack.

## My Solution
```cpp
class Solution {
    TreeNode* builder(vector<int>& preorder, vector<int>& inorder, unordered_map<int, int>&mpp, int &preInd, int inIndS, int inIndE) {
        if (preInd >= inorder.size() || inIndS > inIndE) return nullptr;

        int num = preorder[preInd++];
        TreeNode *root = new TreeNode(num);
        int index = mpp[num];

        root->left = builder(preorder, inorder, mpp, preInd, inIndS, index - 1);
        root->right = builder(preorder, inorder, mpp, preInd, index + 1, inIndE);
        return root;
    }
public:
    TreeNode* buildTree(vector<int>& preorder, vector<int>& inorder) {
        unordered_map<int, int>mpp;
        for (int i = 0; i < inorder.size(); i++) mpp[inorder[i]] = i;
        int preInd = 0, inIndS = 0, inIndE = inorder.size();
        return builder(preorder, inorder, mpp, preInd, inIndS, inIndE);
    }
};
```
**Correct and optimal** — right algorithm (hashmap + shared `preInd` by reference), right complexity O(n)/O(n). Two small refinements:
- **Off-by-one in `inIndE`** — you pass `inorder.size()` but the recursion treats `inIndE` as the *last valid index*, so it should be `inorder.size() - 1`. It still gives the correct answer **only because** the extra `preInd >= inorder.size()` guard catches the overflow the too-large bound creates. Fix the bound and that guard becomes redundant — `inIndS > inIndE` alone terminates every branch.
- **Drop the `preInd >= inorder.size()` check** once `inIndE = size()-1`; with correct windows every consumed preorder element maps to exactly one valid inorder slice, so the range check is sufficient.

(Optional polish: `mpp.at(num)` if you want to pass `mpp` as `const&`, since `operator[]` can't be used on a const map.)

## Other Approaches
- **`inorder` index instead of a hashmap** — linear-scan inorder for the root each call: same logic, but O(n) per lookup → **O(n²)** total. Fine for tiny inputs, but the map is free to add and strictly better.
- **Pass subarray slices (copying)** instead of indices — far more intuitive to read, but each call copies vectors → O(n²) time / O(n²) space. Index-window + shared `preIdx` is the price-of-admission optimization.
- **Iterative with a stack** — process preorder left-to-right, using a stack of pending nodes and an inorder pointer to decide when to switch from "go left" to "pop and go right". O(n) and no recursion, but fiddly; recursion is the interview-default.

## Pitfalls
- **`inEnd` is the last index, not the size** — initial call must be `build(0, n-1)`. Passing `n` is an off-by-one that only "works" if a second guard masks it.
- **Recurse left before right** — `preIdx` is shared and advances on every node; the left subtree must fully consume its preorder slice first. Swapping the two lines silently builds the wrong tree.
- **Share `preIdx` by reference (or make it a member)** — a by-value `preIdx` resets per call and every subtree re-reads `preorder[0]`. Same trap as `230`'s `k`.
- **Uniqueness is load-bearing** — the `value → index` map assumes no duplicates. With repeated values the inorder split point is ambiguous and this approach breaks.
- **Don't linear-scan for the root** — it's the easy way to turn an O(n) solution into O(n²). Build the map up front.
