# 1. 104. Maximum Depth of Binary Tree (LC 104) — Easy
🔗 [Open on LeetCode](https://leetcode.com/problems/maximum-depth-of-binary-tree/)

**Pattern:** Tree DFS — **post-order "return info up"** (the seed problem of the whole family)
**Trigger:** "how deep / how tall is the tree" → recurse to the leaves, and let each node return **`1 + max(left, right)`** to its parent; the empty child returns `0`

## Problem (1-liner)
Given the root of a binary tree, return its **maximum depth** = number of nodes on the longest root→leaf path.
**Examples:** `[3,9,20,null,null,15,7]` → `3` · `[1,null,2]` → `2` · `[]` → `0`
**Constraints:** `0 … 10^4` nodes, `-100 ≤ node.val ≤ 100`.

## Idea (1-liner)
A tree's depth is defined recursively: an empty tree has depth `0`, and a non-empty tree is `1` (for the current node) plus the depth of its **deeper** subtree. That definition *is* the algorithm — recurse left and right, take the `max`, add one. This is the canonical **post-order** shape (work happens **after** the children return) and the template you reuse for height, [[4-543-diameter-of-binary-tree|diameter]], and [[5-110-balanced-binary-tree|balanced-check]].

## Approach
1. **Base case:** `root == nullptr` ⇒ return `0` (an empty tree contributes no depth).
2. **Recurse** into both children to get `L = depth(left)` and `R = depth(right)`.
3. **Combine upward:** return `1 + max(L, R)` — the `1` counts the current node, `max` picks the longer path.
4. The recursion naturally bottoms out at the null children of every leaf, so a leaf returns `1 + max(0, 0) = 1`.

## Code (clean canonical)
```cpp
class Solution {
public:
    int maxDepth(TreeNode* root) {
        if (root == nullptr) return 0;                       // empty tree => depth 0
        return 1 + max(maxDepth(root->left), maxDepth(root->right));
    }
};
```
**TC:** O(n) — each node visited once.  **SC:** O(h) recursion stack — O(log n) balanced, O(n) worst case (skewed tree; with n up to 10^4 that's ~10^4 deep, still fine).

## My Solution
```cpp
class Solution {
    int maxDepthHelper(TreeNode *root, int count) {
        if (root == nullptr) return count;
        int leftTreeDepth  = maxDepthHelper(root->left,  count + 1);
        int rightTreeDepth = maxDepthHelper(root->right, count + 1);
        return max(leftTreeDepth, rightTreeDepth);
    }
public:
    int maxDepth(TreeNode* root) { return maxDepthHelper(root, 0); }
};
```
**Correct**, same O(n)/O(h). This is the **top-down / accumulator** style: instead of building the answer up from the leaves, it carries a running `count` *down* and reports it at the null boundary. Off-by-one handled right — it increments **before** recursing, so the null base returns `count` (not `count + 1`).

**Top-down vs bottom-up — which to keep:** the canonical bottom-up version is preferred because (a) no helper, no extra parameter, fewer places to slip an off-by-one, and (b) it's the reusable `combine(L, R)` template that generalizes to diameter / balanced / max-path-sum, where you genuinely need each subtree's returned value. The accumulator style doesn't extend as cleanly to those.

## Other Approaches
- **Iterative DFS (stack of `{node, depth}`)** — push pairs, pop, `ans = max(ans, depth)`, push children with `depth+1`. O(n)/O(h). Use when recursion depth is a real risk.
- **BFS level-count** — pop level by level with a queue, `++depth` per level drained. O(n)/O(w) where `w` = max width. This is literally [[../7. Tree BFS/_|level-order]] with a counter; reach for it if the problem also asks for per-level info or **minimum** depth ([[../7. Tree BFS/_|LC 111]] — BFS short-circuits at the first leaf, recursion can't).

## Pitfalls
- **Forgetting the null base returns 0, not 1** — if you return `1` for `nullptr`, a single node reports depth `2`. The node is counted by the parent's `+1`, never by itself.
- **`max(L, R)` not `L + R`** — adding the subtrees computes something path-like, not depth. (Summing the two *and* tracking it separately is the trick for [[4-543-diameter-of-binary-tree|diameter]] — don't confuse the two.)
- **Depth (nodes) vs height (edges)** — this problem counts **nodes** on the path, so the answer for a single node is `1`. "Height in edges" would be `max(L,R)+1` measured differently; read the definition each time.
- **Family seed:** lock in the post-order shape `L = dfs(left); R = dfs(right); return f(L, R, node)`. [[4-543-diameter-of-binary-tree]] reuses this exact height recursion and updates a global with `L + R`; [[5-110-balanced-binary-tree]] reuses it and returns `-1` as a "not balanced" sentinel. Same skeleton, different combine step.
