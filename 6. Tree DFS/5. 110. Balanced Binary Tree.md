# 5. 110. Balanced Binary Tree (LC 110) — Easy
🔗 [Open on LeetCode](https://leetcode.com/problems/balanced-binary-tree/)

**Pattern:** Tree DFS — **post-order "return info up" + sentinel** (height recursion of [[1-104-maximum-depth-of-binary-tree|104]], but return `-1` to mean "already unbalanced")
**Trigger:** "is the tree height-balanced — every node's two subtree heights differ by ≤ 1" → compute height bottom-up and short-circuit with a `-1` sentinel the instant any subtree fails

## Problem (1-liner)
Return whether a binary tree is **height-balanced**: for **every** node, `|height(left) − height(right)| ≤ 1`.
**Examples:** `[3,9,20,null,null,15,7]` → `true` · `[1,2,2,3,3,null,null,4,4]` → `false` · `[]` → `true`
**Constraints:** `0 … 5000` nodes, `-10^4 ≤ node.val ≤ 10^4`.

## Idea (1-liner)
Balance is a property of **every** node, and checking it needs each node's height — so compute heights bottom-up ([[1-104-maximum-depth-of-binary-tree|104]]) and piggyback the balance check onto the same pass. The trick is a **sentinel**: `dfs` returns the real height when the subtree is balanced, or `-1` the moment it isn't. Once `-1` appears it propagates straight to the root, so one post-order pass answers the question in O(n).

## Approach
1. **Base case:** empty subtree ⇒ height `0`, and a `0`-height empty tree is trivially balanced.
2. **Recurse** to get `L = dfs(left)`, `R = dfs(right)`.
3. **Propagate failure:** if `L == -1` or `R == -1`, a descendant already failed ⇒ return `-1` (stop checking).
4. **Check here:** if `abs(L - R) > 1`, *this* node is the violation ⇒ return `-1`.
5. **Otherwise** the subtree is balanced ⇒ return its real height `max(L, R) + 1`.
6. The tree is balanced iff the root's call is **not** `-1`.

> **Why the sentinel beats the brute force:** the naive version calls a separate `height()` at every node — recomputing the same heights over and over → **O(n²)** (O(n log n) balanced). Folding the check into the height recursion visits each node once → **O(n)**.

## Code (clean canonical)
```cpp
class Solution {
    int dfs(TreeNode* root) {
        if (!root) return 0;                              // empty => height 0, balanced

        int L = dfs(root->left);
        int R = dfs(root->right);

        if (L == -1 || R == -1) return -1;               // a child already failed
        if (abs(L - R) > 1)     return -1;               // this node is the violation

        return max(L, R) + 1;                            // balanced => real height
    }
public:
    bool isBalanced(TreeNode* root) {
        return dfs(root) != -1;
    }
};
```
**TC:** O(n) — each node visited once.  **SC:** O(h) recursion stack — O(log n) balanced, O(n) skewed.

## My Solution
```cpp
class Solution {
    int helper(TreeNode *root) {
        if (!root) return 0;

        int l = helper(root->left);
        int r = helper(root->right);

        if (l == -1 || r == -1) return -1;
        if (abs(l - r) > 1) return -1;

        return max(l, r) + 1;
    }
public:
    bool isBalanced(TreeNode* root) {
        return helper(root) == -1 ? false : true;
    }
};
```
**Correct and textbook-optimal** — same O(n)/O(h), the proper single-pass sentinel approach (no recomputed heights). Only nit: `helper(root) == -1 ? false : true` is just `helper(root) != -1`. The two separate guard lines (child-failed vs this-node-failed) are a fine readability choice — you *could* merge them into `if (l == -1 || r == -1 || abs(l - r) > 1) return -1;`, but keeping them apart reads as two distinct ideas.

## Family map
- **Seed:** [[1-104-maximum-depth-of-binary-tree|104 Maximum Depth]] — the plain height recursion this is built on.
- **Sibling:** [[4-543-diameter-of-binary-tree|543 Diameter]] — same height recursion, but updates a **global** with `L + R` instead of returning a `-1` sentinel. Two ways to "steal extra info" from the height pass: side-channel global (diameter) vs sentinel return (balanced).
- **Boss:** [[124-binary-tree-maximum-path-sum|124 Max Path Sum]] — same post-order "return one arm, track the bend" shape on node values.

## Other Approaches
- **Brute force two-pass** — `isBalanced(node)` checks `abs(height(L) - height(R)) <= 1 && isBalanced(L) && isBalanced(R)`, with `height()` as a separate recursion. Recomputes heights ⇒ **O(n²)**. The sentinel is the optimization that collapses it to one pass.
- **Return a pair `{height, balanced}`** — functional, no sentinel overloading: each call returns both. Equivalent O(n)/O(h); the `-1` sentinel is just a compact encoding of the same two facts in one int.

## Pitfalls
- **`-1` must propagate before the abs-check** — if a child already returned `-1`, you must bail *before* computing `abs(L - R)`, otherwise `abs(-1 - R)` produces a garbage "height" and a false `true`. (Guard order: child-failed check first.)
- **Balance is a per-node property, not just the root** — `[1,2,2,3,3,null,null,4,4]` has a balanced-looking top but a deep subtree fails; you must check **every** node, which the post-order pass does for free.
- **Empty tree is balanced** — `[]` → `true`. The `!root → 0` base handles this since the root call returns a valid height, not `-1`.
- **Don't reach for the O(n²) two-pass** — it's the intuitive first idea and it TLEs/wastes work on skewed trees; the sentinel is the whole point of the problem.
