# 12. 297. Serialize and Deserialize Binary Tree (LC 297) — Hard
🔗 [Open on LeetCode](https://leetcode.com/problems/serialize-and-deserialize-binary-tree/)

**Pattern:** Tree traversal **round-trip** — encode the tree (incl. **null markers**) to a string, then rebuild it by replaying that string in the **same traversal order**. The null markers are what make the shape unambiguous, so a *single* traversal suffices (unlike [[10-105-construct-binary-tree-from-preorder-inorder|105]], which needs two because it has *no* nulls).
**Trigger:** "**serialize / deserialize** a binary tree", "convert tree ↔ string", "store/transmit a tree and reconstruct it" → pick one traversal (preorder DFS *or* level-order BFS), emit `#` for every null, and decode with a cursor over the tokens.

## Problem (1-liner)
Design `serialize(root) → string` and `deserialize(string) → root` so that `deserialize(serialize(t))` reproduces the **exact** tree (shape + values). The format is yours to invent.
**Examples:** `[1,2,3,null,null,4,5]` → some string → back to `[1,2,3,null,null,4,5]` · `[]` → `""` → `nullptr`.
**Constraints:** `0 … 10⁴` nodes, values in `[-1000, 1000]` → **negatives exist** (so a `-` is part of a token, don't split on it) and the **empty tree** is legal.

## Idea (1-liner)
A traversal alone is ambiguous (`[1,2]` could be 2-as-left or 2-as-right), so **also record every null** as a sentinel. Once nulls are explicit, the token stream encodes the shape completely, and you rebuild by **consuming tokens in the identical order** you wrote them — the structure falls out for free.

## Approach (BFS / level-order — my version)
1. **serialize:** push `root`; while the queue is non-empty, pop a node. If real → append `val,` and push **both** children (even null ones); if null → append `#,`. This is plain level-order with nulls kept.
2. **deserialize:** read the first token as the root, push it. Then for **each** dequeued node read **two** tokens (its left, then its right): a `#` means that child is null; anything else becomes a new node, is attached, and is **pushed** so its own children get filled later.
3. The queue in `deserialize` holds exactly the nodes still waiting for their two children — a mirror of how serialize enqueued them.

> The two directions use slightly different bookkeeping but produce/consume the **same comma string**: serialize enqueues nulls and emits `#` when it pops one; deserialize never enqueues nulls, it reads children in pairs. Compatible, just two lenses on the same level-order stream.

## Code (clean canonical — BFS, with `stringstream`)
```cpp
class Codec {
public:
    string serialize(TreeNode* root) {
        if (!root) return "";
        ostringstream out;
        queue<TreeNode*> q;
        q.push(root);
        while (!q.empty()) {
            TreeNode* node = q.front(); q.pop();
            if (node) {
                out << node->val << ',';
                q.push(node->left);
                q.push(node->right);
            } else {
                out << "#,";
            }
        }
        return out.str();
    }

    TreeNode* deserialize(string data) {
        if (data.empty()) return nullptr;
        istringstream in(data);
        string tok;
        queue<TreeNode*> q;

        getline(in, tok, ',');
        TreeNode* root = new TreeNode(stoi(tok));
        q.push(root);

        auto buildChild = [&]() -> TreeNode* {          // read one token, make a child
            getline(in, tok, ',');
            if (tok == "#") return nullptr;
            TreeNode* child = new TreeNode(stoi(tok));
            q.push(child);
            return child;
        };

        while (!q.empty()) {
            TreeNode* node = q.front(); q.pop();
            node->left  = buildChild();
            node->right = buildChild();
        }
        return root;
    }
};
```
**TC:** O(n) each way — every node and every null marker is written/read once.  **SC:** O(n) — the string plus an O(width) queue (O(n) worst case).

## My Solution (passed all tests — refinement notes)
My original was already correct and O(n)/O(n); the version above only **polishes** it. What changed and why:
- **`string +=` → `ostringstream`.** Repeated `str += ...` can reallocate the buffer as it grows (amortized fine in practice, but the streaming version avoids it and reads cleaner). Pure performance/style.
- **Deduplicated the left/right blocks** into one `buildChild` lambda — the two halves of my loop were identical except for which pointer they set. Behaviour is unchanged.
- **Kept** every correct decision I already had: emitting `#` for nulls, the trailing comma (harmless — `getline` just stops at EOF), parsing with `stoi` (handles the negative values), and the empty-string guards in both directions.

Nothing in the original was *wrong* — these are the only further refinements worth making.

## Other Approaches
- **Preorder DFS (the more common interview answer)** — usually shorter to write and reason about:
  ```cpp
  void enc(TreeNode* n, ostringstream& o){
      if(!n){ o << "#,"; return; }
      o << n->val << ',';
      enc(n->left, o); enc(n->right, o);
  }
  TreeNode* dec(istringstream& in){
      string t; getline(in, t, ',');
      if(t == "#") return nullptr;
      TreeNode* n = new TreeNode(stoi(t));
      n->left  = dec(in);          // order matters: left consumes its tokens first
      n->right = dec(in);
      return n;
  }
  ```
  Same O(n)/O(n); the recursion's call order *is* the cursor. The one subtlety: build **left before right** (the preorder stream lists the whole left subtree first), same discipline as 105's `preIdx`.
- **Postorder / inorder** — postorder works (decode right-before-left, reading the stream backwards); **inorder does not** uniquely determine a tree even with nulls, so don't reach for it.
- **LeetCode's own bracket format** — same level-order idea; people often trim trailing `null`s, but that's a cosmetic optimization, not required.

## Family map
- **Mirror problem:** [[10-105-construct-binary-tree-from-preorder-inorder|105 Build from Preorder + Inorder]] — also "reconstruct a tree," but there nulls are **absent** so one traversal is ambiguous and you need **two** (preorder + inorder) plus an index map. Here, **explicit null markers** make a *single* traversal enough. Contrast: 105 has no `#`, 297 lives or dies by it.
- **Both lean on a shared advancing cursor** — 105's `preIdx`, 297's `getline`/recursion order. "Build, don't traverse; consume tokens in traversal order."
- **Traversal seeds:** the preorder DFS variant is the same skeleton as every other Tree DFS note; the BFS variant is plain level-order (the BFS contrast called out in the index's "when to reach for BFS").

## Pitfalls
- **You must record nulls.** A bare traversal can't be inverted — `#` markers are the entire trick. Drop them and the shape is lost.
- **Pick one traversal and use the *same* one both ways.** serialize-as-preorder + deserialize-as-level-order won't round-trip. (My two functions differ in queue bookkeeping but encode/decode the *same* level-order stream — that's fine; a true traversal-order mismatch is not.)
- **Don't split on `-`; values can be negative.** Split only on the delimiter (`,`) and let `stoi` handle the sign. A naive char-by-char digit parse breaks on `-1000`.
- **Handle the empty tree explicitly** — `root == nullptr` → `""`, and `data.empty()` → `nullptr`. Constraints allow `[0, 10⁴]` nodes, so `0` is a real case.
- **Build left before right** in any DFS decode — the stream lists the left subtree's tokens first; swapping the two `dec(in)` calls silently mis-assigns whole subtrees because each call greedily eats tokens.
- **`+=`-building a giant string** isn't a *bug* but prefer `ostringstream`/`istringstream` — fewer reallocations and no re-creating a `stringstream` per call.
