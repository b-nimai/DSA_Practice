# 9. 236. Lowest Common Ancestor of a Binary Tree (LC 236) â€” Medium

**Pattern:** Tree DFS â€” **post-order "report what you found up"**; a node where `p` surfaces from one side and `q` from the other is the LCA
**Trigger:** "**lowest common ancestor** of two nodes in a *plain* binary tree (no BST order)" â†’ DFS returning the found node; the **first node that gets a non-null answer from *both* children** is the LCA

## Problem (1-liner)
Return the lowest node that has **both** `p` and `q` as descendants (a node counts as a descendant of itself).
**Examples:** `p=5, q=1` â†’ `3` Â· `p=5, q=4` (4 is under 5) â†’ `5` (a node can be its own ancestor) Â· `[1,2], p=1, q=2` â†’ `1`
**Constraints:** `2 â€¦ 10^5` nodes, unique values, `p â‰  q`, **both guaranteed to exist** in the tree.

## Idea (1-liner)
Search post-order. Each call answers "did I find `p` or `q` (or the LCA) in my subtree?" by returning a node or `null`. If **one** subtree reports a hit, pass that hit up. If **both** subtrees report a hit, *this* node is where `p` and `q` first meet â†’ it's the LCA. The "node is its own ancestor" rule falls out for free: when you land on `p` (or `q`) you return it immediately, and an ancestor that also sees the other node on its far side becomes the answer.

## Approach
1. **Base / hit:** if `root` is `null`, or `root == p`, or `root == q` â†’ return `root`. (Null = nothing here; matching `p`/`q` = "found one, stop digging.")
2. **Recurse both sides:** `left = LCA(root->left)`, `right = LCA(root->right)`.
3. **Combine:**
   - Both non-null â‡’ `p` and `q` split across the two subtrees â‡’ **`root` is the LCA**.
   - Exactly one non-null â‡’ both targets live on that side (or only one is found *so far*) â‡’ **bubble that node up**.
   - Both null â‡’ neither target here â‡’ return `null`.

> **Why returning the node on match is enough (the self-ancestor case).** When `p` is an ancestor of `q`, the call at `p` returns `p` *before* recursing deeper, so `p`'s parent sees `p` from one side and `null` from the other â†’ it forwards `p` up, and `p` ends up being the answer. You never need an explicit "is `q` below `p`?" check â€” guaranteed existence does the rest.

## Code (clean canonical)
```cpp
class Solution {
public:
    TreeNode* lowestCommonAncestor(TreeNode* root, TreeNode* p, TreeNode* q) {
        if (!root || root == p || root == q) return root;   // null, or found a target â†’ report up

        TreeNode* left  = lowestCommonAncestor(root->left,  p, q);
        TreeNode* right = lowestCommonAncestor(root->right, p, q);

        if (left && right) return root;   // p and q split across sides â†’ this is the LCA
        return left ? left : right;       // else forward whichever side found something
    }
};
```
**TC:** O(n) â€” each node visited once.  **SC:** O(h) recursion stack â€” O(log n) balanced, O(n) skewed.

## My Solution
```cpp
class Solution {
public:
    TreeNode* lowestCommonAncestor(TreeNode* root, TreeNode* p, TreeNode* q) {
        if (!root || root == p || root == q) return root;

        TreeNode *leftRes  = lowestCommonAncestor(root->left, p, q);
        TreeNode *rightRes = lowestCommonAncestor(root->right, p, q);

        if (!leftRes) return rightRes;
        else if (!rightRes) return leftRes;
        else return root;
    }
};
```
**Correct â€” textbook.** Same logic as canonical, just written as explicit null-checks: `!leftRes â†’ rightRes`, `!rightRes â†’ leftRes`, else (both non-null) `root`. Clean and idiomatic. The `return left ? left : right` form above just folds your two single-null branches into one line. Same O(n)/O(h).

## Other Approaches
- **Parent-pointer + ancestor set** â€” DFS once recording each node's parent in a map; walk `p` up to the root collecting its ancestors in a set, then walk `q` up until you hit the first node already in the set. O(n) time, O(n) space. Useful when you must answer **many** LCA queries or only have childâ†’parent links.
- **Path-to-target then diverge** â€” find rootâ†’`p` and rootâ†’`q` paths as lists, then the last common prefix node is the LCA. Simple to reason about but two passes + O(h) path storage.
- **Euler tour + sparse table (RMQ)** â€” heavy preprocessing (O(n)) for O(1) per query; only worth it under a high volume of LCA queries on a static tree.

## Pitfalls
- **Don't assume BST order** â€” this is a *plain* binary tree; there is no `<`/`>` shortcut. You must search both subtrees (that's 235 vs 236).
- **The match-returns-node base case handles self-ancestor** â€” returning `root` when `root == p || root == q` is what makes the "node is a descendant of itself" case (`p=5, q=4`) work. Don't add an extra "keep searching below p" step; it breaks it.
- **Both-non-null is the *only* LCA condition** â€” the LCA is the *single* node that first sees a hit from each side. Once that bubbles up, a higher node sees it from one side and `null` from the other and just forwards it â€” don't mistake those higher nodes for new LCAs.
- **Relies on "both `p` and `q` exist"** â€” guaranteed here. In **1644/1245** where they might be absent, this exact code returns a non-null node even when one is missing; you must add a found-count check before trusting the result.
- **Compare by identity, not value** â€” `root == p` (pointer), not `root->val == p->val`. Values are unique here so either works, but identity is the correct, general habit.
