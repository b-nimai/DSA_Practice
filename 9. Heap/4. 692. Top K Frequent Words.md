# 4. 692. Top K Frequent Words (LC 692) — Medium
🔗 [Open on LeetCode](https://leetcode.com/problems/top-k-frequent-words/)

**Pattern:** Heap — **Top-K by frequency + custom tie-break comparator**
**Trigger:** "k most frequent **words**, ties broken **lexicographically**" → [[3-347-top-k-frequent-elements|347]] with strings, but the default pair ordering breaks on ties — you must write a **custom comparator**, and in the size-K min-heap version the tie-break **inverts**.

## Problem (1-liner)
Return the **k most frequent** strings, sorted by frequency **descending**; equal-frequency words sorted **lexicographically ascending**. (Unlike 347, order *matters*.)
**Examples:** `["i","love","leetcode","i","love","coding"], k=2 → ["i","love"]` (`i` and `love` both ×2 — `"i" < "love"`) · `["the","day","is","sunny","the","the","the","sunny","is","is"], k=4 → ["the","is","sunny","day"]` (4,3,2,1).
**Constraints:** `n ≤ 500`, word length ≤ 10, lowercase only, `1 ≤ k ≤ #unique`. **Follow-up: O(n log k) time, O(n) space.**

## Idea (1-liner)
Same as [[3-347-top-k-frequent-elements|347]] — hash-map count, then a size-K min-heap over `{freq, word}` — except the heap needs a **custom comparator**: the min-heap's top must be the **worst** candidate (lowest freq; among ties, lexicographically **largest** word), so evictions always drop the right one. Pop into the answer **back-to-front**.

## Approach
1. Count: `unordered_map<string,int> freq` — O(n).
2. Define the comparator for a size-K min-heap: "worst on top" = `a.first > b.first` (lower freq on top); tie → `a.second < b.second` (**larger** word on top).
3. Push each `{f, word}`; if `size() > k`, pop → evicts the current worst.
4. The heap now pops in **worst→best** order, so fill the answer **from the back**: `res[i] = pq.top().second` for `i = k-1 … 0`.

> **Load-bearing insight:** in a size-K min-heap the comparator selects who gets **evicted**, not who comes out first — so *both* keys invert relative to the desired output order: freq flips (lower on top) **and** the lexicographic tie-break flips (larger word on top). Forgetting to flip the tie-break is *the* 692 bug.

## Code (clean canonical)
```cpp
class Solution {
public:
    vector<string> topKFrequent(vector<string>& words, int k) {
        unordered_map<string, int> freq;
        for (string& w : words) freq[w]++;

        // size-K min-heap: WORST candidate on top (low freq; tie → larger word)
        auto cmp = [](const pair<int,string>& a, const pair<int,string>& b) {
            if (a.first != b.first) return a.first > b.first; // lower freq on top
            return a.second < b.second;                       // larger word on top
        };
        priority_queue<pair<int,string>, vector<pair<int,string>>, decltype(cmp)> pq(cmp);
        for (auto& [w, f] : freq) {
            pq.push({f, w});
            if ((int)pq.size() > k) pq.pop();                 // evict worst
        }

        vector<string> res(k);
        for (int i = k - 1; i >= 0; i--) {                    // pops worst→best: fill backwards
            res[i] = pq.top().second;
            pq.pop();
        }
        return res;
    }
};
```
**TC:** O(n·L + m·L·log k), L = word length (string compares cost O(L)), m = unique words — meets the O(n log k) follow-up.  **SC:** O(n).

## My Solution (bug journey — worth remembering)
1. **Attempt 1:** default `priority_queue<pair<int,string>>` (max-heap, no comparator) → **WA on Example 1**: output `["love","i"]`. Default pair ordering puts the lexicographically **larger** string on top among frequency ties.
2. **Attempt 2:** wrote the comparator but as `bool cmp = [](...)` (lambda isn't a `bool` → CE; needs `auto`), passed `cmp` as the template arg (needs the **type**: `decltype(cmp)`, plus `pq(cmp)` in the constructor), and counted `i` down while still `push_back`-ing → result reversed (worst-first).
3. **Fixed:** `auto cmp` + `decltype(cmp)` + `pq(cmp)` + fill `res[i]` by index.

## Other Approaches
- **Sort all pairs** *(brute)* — collect `{word, freq}`, `sort` with comparator (freq desc, word asc), take first k. **O(n + m·L·log m)**, O(m). Simplest correct answer; fails only the follow-up bar.
- **Max-heap of all m, pop k** *(better)* — comparator in the *natural* direction (high freq on top, smaller word on top: `a.first < b.first`, tie `a.second > b.second`), pop k times. **O(m + k log m)**. Easier comparator to reason about — good fallback if the inverted one confuses you under pressure.
- **Size-K min-heap** *(optimal for the follow-up, above)* — **O(n log k)** / O(n). The comparator inversion is the price of the better bound.
- **Bucket sort by freq + sort each bucket** — `bucket[f]` = words with freq f; walk high→low, sort each bucket lexicographically, collect k. **O(n + m·L·log m)** worst case (one giant bucket) — the sort inside buckets means it's *not* O(n) like in [[3-347-top-k-frequent-elements|347]]; ties killed the linear bound.

## Pitfalls
- **Default pair ordering fails on ties** — `pair` compares `.second` ascending, so a default max-heap surfaces the lexicographically **largest** tied word. Example 1 is the canary: `["love","i"]` instead of `["i","love"]`.
- **The min-heap tie-break inverts** — "worst on top" means larger word on top (`a.second < b.second`). Copying the max-heap comparator into the min-heap version is the classic WA.
- **Lambda comparator mechanics** — `auto cmp = ...` (not `bool`); template arg is `decltype(cmp)`; pass the object to the constructor: `pq(cmp)`.
- **Output order** — the min-heap pops worst→best; fill `res` **backwards** (or reverse at the end). A countdown loop with `push_back` still appends forward — index does nothing.
