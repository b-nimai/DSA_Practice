# 5. 973. K Closest Points to Origin (LC 973) — Medium
🔗 [Open on LeetCode](https://leetcode.com/problems/k-closest-points-to-origin/)

**Pattern:** Heap — **Top-K with a size-K max-heap** (the "k *closest / smallest*" flip of [[1-215-kth-largest-element-in-an-array|215]])
**Trigger:** "k **closest** points / k **smallest** by some metric, any order" → keep a **max-heap of size k** keyed by distance; the top is the *farthest of the k best*, so pop it when a closer point arrives. Static array? **Quickselect / `nth_element`** does it in average O(n).

## Problem (1-liner)
Return the **k closest points to the origin** by Euclidean distance, in any order; the answer is unique up to ordering.
**Examples:** `[[1,3],[-2,2]], k=1 → [[-2,2]]` (√8 < √10) · `[[3,3],[5,-1],[-2,4]], k=2 → [[3,3],[-2,4]]`.
**Constraints:** `1 ≤ k ≤ n ≤ 10^4`, `-10^4 ≤ xi,yi ≤ 10^4`. So `k` is always valid; coords fit in `int` but `x²+y²` (≤ 2·10^8) is safest in `long long`.

## Idea (1-liner)
You only need the **k smallest distances**, and among those the *largest* is the one to evict when something closer shows up — so a **max-heap capped at size k** holds exactly the k closest seen so far. Whenever it overflows, popping the max drops the current farthest, so the survivors are always the top-k closest. **Compare squared distance** — you never need the actual √, it's a monotonic transform. O(n log k), works on a stream.

## Approach
1. Build a **max-heap** of `{dist², x, y}` (default `priority_queue` is a max-heap; `dist²` first so it orders by distance).
2. For each point: `push`; if `size() > k`, `pop()` the max (the current farthest).
3. After the pass the heap holds the **k closest**; drain it into the result.

> **Load-bearing insight:** k **closest / smallest** → **max**-heap of size k (top = the one to kick out). This is the exact mirror of kth **largest** → **min**-heap. And **never call `sqrt`** — compare `x²+y²`; the ordering is identical and you dodge all floating-point noise.

## Code (clean canonical)
```cpp
class Solution {
public:
    vector<vector<int>> kClosest(vector<vector<int>>& points, int k) {
        priority_queue<tuple<long long,int,int>> pq;              // max-heap by dist²
        for (auto& p : points) {
            long long d = 1LL*p[0]*p[0] + 1LL*p[1]*p[1];
            pq.push({d, p[0], p[1]});
            if ((int)pq.size() > k) pq.pop();                     // drop the farthest
        }
        vector<vector<int>> res;
        while (!pq.empty()) {
            auto [d, x, y] = pq.top(); pq.pop();
            res.push_back({x, y});
        }
        return res;
    }
};
```
**TC:** O(n log k) — n points, each heap op O(log k).  **SC:** O(k) heap. Best answer for the **streaming / n huge, k small** framing.

## My Solution
```cpp
class Solution {
public:
    vector<vector<int>> kClosest(vector<vector<int>>& points, int k) {
        priority_queue<tuple<double, int, int>> pq;
        for (int i = 0; i < points.size(); i++) {
            double dis = sqrt((points[i][0]*points[i][0]) + (points[i][1]*points[i][1]));
            if (i >= k) {
                auto [d, x, y] = pq.top();
                if (d > dis) { pq.push({dis, points[i][0], points[i][1]}); pq.pop(); }
            } else {
                pq.push({dis, points[i][0], points[i][1]});
            }
        }
        vector<vector<int>> result;
        while (!pq.empty()) { auto [dis, x, y] = pq.top(); pq.pop(); result.push_back({x, y}); }
        return result;
    }
};
```
**Correct**, same O(n log k) / O(k). Seeds the heap from the first `k` (`i < k`), then for later points peeks the max and replaces it only if closer. Push-then-pop order is safe: since `dis < d`, the old max stays on top and gets popped. Refinements: (1) **drop the `sqrt`** — compare `x²+y²` in `long long`, cheaper and float-free; (2) the canonical *push-then-pop-if-oversized* loop removes the `i >= k` / `i < k` bookkeeping and reads cleaner — same complexity. Watch `size()` vs `k` (unsigned compare → cast `k`).

## Other Approaches
- **Sort by distance, take first k** *(brute)* — `sort` on `x²+y²`, slice `[0,k)`. **O(n log n)** time, O(1)/O(n) extra. Simplest; fine when n is small.
- **Size-K max-heap** *(the canonical block)* — **O(n log k)** time, **O(k)** space. Reach for it on a **stream** or when `k ≪ n` (you never hold all points at once). Same skeleton as [[1-215-kth-largest-element-in-an-array|215]], polarity flipped.
- **Quickselect / `nth_element` on dist²** *(optimal on a static array)* — partition around the kth-smallest distance; the first k slots are then the answer (in any order — exactly what's asked). **Average O(n)**, worst O(n²) (randomize pivot), O(1) extra:
  ```cpp
  auto d2 = [](vector<int>& p){ return 1LL*p[0]*p[0] + 1LL*p[1]*p[1]; };
  nth_element(points.begin(), points.begin()+k, points.end(),
              [&](vector<int>& a, vector<int>& b){ return d2(a) < d2(b); });
  return vector<vector<int>>(points.begin(), points.begin()+k);
  ```
  The one-liner to state after describing the heap — fastest for a one-shot in-memory answer.

## Pitfalls
- **Max-heap for closest, not min-heap** — the classic polarity flip. k *closest/smallest* → keep the k best in a **max**-heap (top = the farthest, i.e. the one to evict).
- **Don't `sqrt`** — it's an unnecessary float op; compare `x²+y²`. If you must square, use `long long` (`10^4 · 10^4 = 10^8`, sum up to `2·10^8` — fits `int` but only just; `long long` is the safe habit).
- **Cap the heap at k, not n** — a heap of all n points is O(n log n) / O(n); the size-k version is O(n log k) / O(k).
- **`size()` is unsigned** — `pq.size() > k` mixes `size_t` and `int`; cast `k`.
- **Order doesn't matter** — the problem allows any order, so don't waste effort sorting the final k (and quickselect's unordered output is already valid).
