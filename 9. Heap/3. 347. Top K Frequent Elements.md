# 3. 347. Top K Frequent Elements (LC 347) — Medium
🔗 [Open on LeetCode](https://leetcode.com/problems/top-k-frequent-elements/)

**Pattern:** Heap — **Top-K keyed by frequency** (hash-map count → size-K min-heap)
**Trigger:** "k **most frequent** elements, any order, must beat O(n log n)" → count with a hash map, then run the [[1-215-kth-largest-element-in-an-array|215]] size-K min-heap **on the frequencies** (pair `{freq, num}`). O(n) exists via **bucket sort by frequency**.

## Problem (1-liner)
Return the **k most frequent** elements of `nums`, in any order. The answer is guaranteed unique (no frequency ties at the cut-off).
**Examples:** `[1,1,1,2,2,3], k=2 → [1,2]` · `[1], k=1 → [1]` · `[1,2,1,2,1,2,3,1,3,2], k=2 → [1,2]`.
**Constraints:** `1 ≤ n ≤ 10^5`, `-10^4 ≤ nums[i] ≤ 10^4`, `1 ≤ k ≤ #unique` — so k is always valid; a single-element array with `k=1` is the smallest case. **Follow-up: must beat O(n log n).**

## Idea (1-liner)
It's [[1-215-kth-largest-element-in-an-array|Kth Largest]] with one extra move: first collapse `nums` into `{num → freq}` with a hash map, then keep a **min-heap of size k ordered by frequency** — whatever survives the evictions is the k most frequent. O(n log k) beats the follow-up bar since k ≤ unique ≤ n.

## Approach
1. Count: `unordered_map<int,int> freq` — one O(n) pass.
2. For each `{num, f}` in the map: push `{f, num}` into a **min-heap** (frequency **first** — `priority_queue` compares `.first`).
3. If `size() > k`, pop — evicts the least frequent of the current candidates.
4. Drain the heap into the answer (order doesn't matter).

> **Load-bearing insight:** the heap operates on the **m unique elements, not the n raw values** — the hash map is the compression step. Frequency goes first in the pair so the heap orders by it; the element rides along as `.second`.

## Code (clean canonical)
```cpp
class Solution {
public:
    vector<int> topKFrequent(vector<int>& nums, int k) {
        unordered_map<int, int> freq;
        for (int x : nums) freq[x]++;                        // O(n) count

        // min-heap of {frequency, number}, capped at k
        priority_queue<pair<int,int>, vector<pair<int,int>>, greater<>> pq;
        for (auto& [num, f] : freq) {
            pq.push({f, num});
            if ((int)pq.size() > k) pq.pop();                // evict least frequent
        }

        vector<int> ans;
        while (!pq.empty()) { ans.push_back(pq.top().second); pq.pop(); }
        return ans;
    }
};
```
**TC:** O(n + m log k), m = unique elements (≤ n) — beats O(n log n) as required.  **SC:** O(n) for the map (heap only O(k)).

## Other Approaches
- **Sort by frequency** *(brute)* — count, copy `{freq, num}` pairs to a vector, sort descending, take first k. **O(n + m log m)** time, O(m) space. Simple, but fails the follow-up when m ≈ n.
- **Max-heap of all m pairs, pop k** *(better)* — push every `{freq, num}` into a max-heap, pop k times. **O(m + k log m)** (heapify is linear), O(m) space. Fine, but the size-K min-heap wins when k ≪ m.
- **Size-K min-heap** *(the canonical block, above)* — **O(n + m log k)** / O(n). The default interview answer.
- **Bucket sort by frequency** *(optimal, O(n))* — frequency is bounded by n, so make `buckets[f] = elements appearing f times` and walk buckets from `n` down to `1` collecting until you have k:
  ```cpp
  vector<int> topKFrequent(vector<int>& nums, int k) {
      unordered_map<int, int> freq;
      for (int x : nums) freq[x]++;
      vector<vector<int>> bucket(nums.size() + 1);          // index = frequency
      for (auto& [num, f] : freq) bucket[f].push_back(num);
      vector<int> ans;
      for (int f = nums.size(); f >= 1 && (int)ans.size() < k; f--)
          for (int num : bucket[f]) {
              ans.push_back(num);
              if ((int)ans.size() == k) break;
          }
      return ans;
  }
  ```
  **O(n)** time, O(n) space. State this as the optimal follow-up after coding the heap.
- **Quickselect on the freq pairs** — partition the m pairs around frequency rank; **average O(m)**, worst O(m²). Valid but bucket sort is simpler and deterministic here.

## Pitfalls
- **Pair order:** push `{freq, num}`, not `{num, freq}` — `priority_queue` orders by `.first`; swapping them silently ranks by value, and the examples may still pass.
- **Heap over unique elements, not raw values** — pushing all n values (skipping the count step) answers a different question entirely.
- **Min-heap polarity** (inherited from [[1-215-kth-largest-element-in-an-array|215]]): *most* frequent → **min**-heap of size k; unconditional pop before size check shrinks below k.
- **Bucket size is n+1** — max possible frequency is n; `bucket[n]` must exist. Walk **high→low** and stop at exactly k.
- **Unsigned `size()`** — cast when comparing `pq.size() > k` / `ans.size() < k`.
