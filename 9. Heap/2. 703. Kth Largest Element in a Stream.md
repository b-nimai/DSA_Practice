# 2. 703. Kth Largest Element in a Stream (LC 703) — Easy
🔗 [Open on LeetCode](https://leetcode.com/problems/kth-largest-element-in-a-stream/)

**Pattern:** Heap — **Top-K online**: a **size-K min-heap** maintained across many `add` calls
**Trigger:** "class that keeps returning the **kth largest so far** after each new value / real-time cut-off" → hold the **k largest** in a min-heap; every `add` pushes then pops **only if oversized**; `top()` is the answer.

## Problem (1-liner)
Design `KthLargest(k, nums)` + `add(val)` that, after each insertion, returns the **kth largest** value in the whole stream so far (kth in sorted order, duplicates count — not kth distinct).
**Examples:** `k=3, [4,5,8,2]`; `add(3)→4, add(5)→5, add(10)→5, add(9)→8, add(4)→8`.
**Constraints:** `0 ≤ nums.length ≤ 10^4`, **`1 ≤ k ≤ nums.length + 1`** (so the initial pool can hold *fewer than k* elements — the load-bearing edge case), up to `10^4` `add` calls.

## Idea (1-liner)
This is [[1-215-kth-largest-element-in-an-array|LC 215]] made **online**: instead of one pass, keep the size-K min-heap **alive as a member** between calls. The heap always holds the k largest seen so far, so its top is the kth largest. Each `add` inserts and, **only if the heap now exceeds k**, evicts the smallest — that guard is the whole trick, because early on the heap may legitimately hold fewer than k elements.

## Approach
1. Keep a **min-heap** (`greater<int>`) and store `k` as a member.
2. Constructor: push each `nums[i]`; if `size() > k`, pop the min. Heap ends holding `min(len, k)` largest.
3. `add(val)`: `push(val)`; **if `size() > k`, pop**; return `top()`.

> **Load-bearing insight:** pop **conditionally** (`size() > k`), never unconditionally. Because `k ≤ len + 1`, the heap can start with `k-1` elements (even empty when `nums=[]`); a blind `push`+`pop` in `add` would shrink it below k — or call `top()` on an empty heap (UB) — and return the wrong value.

## Code (clean canonical)
```cpp
class KthLargest {
    priority_queue<int, vector<int>, greater<int>> minHeap;   // size-k min-heap
    int k;
public:
    KthLargest(int k, vector<int>& nums) : k(k) {
        for (int x : nums) {
            minHeap.push(x);
            if ((int)minHeap.size() > k) minHeap.pop();
        }
    }
    int add(int val) {
        minHeap.push(val);
        if ((int)minHeap.size() > k) minHeap.pop();           // guard — only pop when oversized
        return minHeap.top();                                 // kth largest so far
    }
};
```
**TC:** constructor O(n log k), `add` O(log k).  **SC:** O(k).

## My Solution (bug → fix)
```cpp
// BUGGY: add() pops unconditionally
int add(int val) {
    minHeap.push(val);
    minHeap.pop();          // ❌ shrinks heap below k when it holds < k elements
    return minHeap.top();   // ❌ top() on empty heap = UB
}
```
**Failed** on `k=1, nums=[]` → `add(-3),add(-2),add(-4),...` gave `-3,-2,-4,...` instead of `-3,-2,-2,...`. Root cause: the constructor correctly builds a heap of size `min(len,k)` — which can be **< k** since `k ≤ len+1` — but `add` always pops, so the heap never grows to `k` and keeps returning the just-inserted value (or reads an empty heap). **Fix:** store `k` and pop **only if `size() > k`** (the canonical block above). The constructor's `if (i >= k) pop` was fine — equivalent to the size guard — so the one-line fix is in `add`.

## Other Approaches
- **Re-sort / re-scan every `add`** *(brute)* — keep a list, sort on each call, index `n-k`. **O(add · n log n)** total → TLE at 10^4 calls. Never do this online.
- **Size-K min-heap kept across calls** *(optimal)* — the canonical block. `add` is **O(log k)**, space **O(k)**. The standard answer for any "kth largest **so far** / streaming cut-off" prompt. Quickselect (great for the *static* [[1-215-kth-largest-element-in-an-array|215]]) doesn't help here — it re-runs O(n) per `add` and doesn't maintain incremental state.
- **Balanced BST / multiset** — `multiset<int>` with an iterator parked at the kth-largest position; advance/retreat by one on each insert. O(log n) per add but O(n) space (keeps everything) and fiddlier — the size-k heap is strictly better when you only need *one* order statistic.

## Pitfalls
- **Unconditional pop in `add`** — the exact bug here. Pop **only when `size() > k`**; the heap may hold `< k` elements early because `k ≤ len + 1`.
- **`top()` on an empty heap is UB** — the conditional pop also prevents this on the first `add` when `nums=[]`.
- **Forgetting to store `k`** — `add` needs `k` at call time; make it a member (the array version [[1-215-kth-largest-element-in-an-array|215]] didn't need to keep state, this one does).
- **Min-heap, not max-heap** — kth *largest* → keep the k biggest in a **min**-heap; `top()` = kth largest.
- **`size()` is unsigned** — cast `k` to `int` (or make it `size_t`) in the comparison.
