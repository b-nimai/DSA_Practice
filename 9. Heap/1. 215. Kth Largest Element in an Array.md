# 1. 215. Kth Largest Element in an Array (LC 215) — Medium
🔗 [Open on LeetCode](https://leetcode.com/problems/kth-largest-element-in-an-array/)

**Pattern:** Heap — **Top-K with a size-K min-heap** (the seed problem of the whole Top-K family)
**Trigger:** "kth **largest** (not distinct), can you avoid sorting / do it on a stream" → keep a **min-heap of size K** of the largest-so-far; its top is the answer. Static array? **Quickselect** does it in average O(n).

## Problem (1-liner)
Return the **kth largest** element of `nums` in *sorted order* — the kth largest **value**, duplicates count (not the kth *distinct* value). Solve it without fully sorting if you can.
**Examples:** `[3,2,1,5,6,4], k=2 → 5` · `[3,2,3,1,2,4,5,5,6], k=4 → 4` (duplicates count).
**Constraints:** `1 ≤ k ≤ n ≤ 10^5`, `-10^4 ≤ nums[i] ≤ 10^4`. So `k` is always valid; `k == n` means the minimum.

## Idea (1-liner)
You don't need the whole array sorted — only the **k largest** values, and among those the *smallest* one is the kth largest. A **min-heap capped at size K** holds exactly the k biggest seen so far: whenever it overflows, popping the min evicts the current smallest, so what survives is always the top-k and `heap.top()` is the kth largest. O(n log k), and it works on a stream where you never see all of `nums` at once.

## Approach
1. Create a **min-heap** (`priority_queue` with `greater<int>`).
2. For each `x` in `nums`: `push(x)`; if `size() > k`, `pop()` the minimum.
3. After the pass the heap holds the **k largest** elements; return `heap.top()` (the smallest of them = kth largest).

> **Load-bearing insight:** for kth **largest** use a **min**-heap of size k (top = kth largest); for kth **smallest** flip to a **max**-heap of size k. The heap is capped at k, not n → space is O(k) and each op is O(log k), not O(log n).

## Code (clean canonical)
```cpp
class Solution {
public:
    int findKthLargest(vector<int>& nums, int k) {
        priority_queue<int, vector<int>, greater<int>> minHeap;   // size-k min-heap
        for (int x : nums) {
            minHeap.push(x);
            if ((int)minHeap.size() > k) minHeap.pop();           // drop the smallest
        }
        return minHeap.top();                                     // kth largest
    }
};
```
**TC:** O(n log k) — n pushes/pops, each O(log k).  **SC:** O(k) heap. Best answer for the **streaming / top-k** framing.

## My Solution
```cpp
class Solution {
public:
    int findKthLargest(vector<int>& nums, int k) {
        priority_queue<int, vector<int>, greater<int>> minHeap;
        for (int i = 0; i < nums.size(); i++) {
            if (i >= k && !minHeap.empty() && minHeap.top() < nums[i]) {
                minHeap.pop();
                minHeap.push(nums[i]);
            }
            if (i < k) minHeap.push(nums[i]);
        }
        return minHeap.top();
    }
};
```
**Correct**, same O(n log k) / O(k). Builds the heap from the first `k` (`i < k`), then for later elements replaces the min only if `nums[i]` beats it. The two `if`s are mutually exclusive (`i < k` vs `i >= k`) so there's no double-push; `!minHeap.empty()` is a harmless guard (heap is always size k once `i >= k`). Refinement: the canonical *push-then-pop-if-oversized* loop drops the `i >= k` / `i < k` bookkeeping and reads cleaner — same complexity. (Watch the `size()` vs `k` compare: `size()` is `size_t`, so cast `k` to avoid a signedness warning.)

## Other Approaches
- **Sort, index** *(brute)* — `sort(nums)`; return `nums[n-k]`. **O(n log n)** time, O(1) extra. Trivial and fine when n is small, but the problem explicitly nudges you past it.
- **Size-K min-heap** *(the canonical block)* — **O(n log k)** time, **O(k)** space. Reach for it on a **stream** or when `k ≪ n` (you never hold all of `nums`). This is the natural lead-in to [[2-703-kth-largest-element-in-a-stream|Kth Largest in a Stream]].
- **Quickselect (Hoare)** *(optimal on a static array)* — partition around a pivot; recurse only into the side holding rank `n-k`. **Average O(n)**, **worst O(n²)** (mitigate with a random pivot), O(1) extra. Fastest for a one-shot in-memory answer:
  ```cpp
  int quickselect(vector<int>& a, int lo, int hi, int target) { // target = index n-k
      int pivot = a[lo + rand() % (hi - lo + 1)], l = lo, r = hi;
      while (l <= r) {
          while (a[l] < pivot) l++;
          while (a[r] > pivot) r--;
          if (l <= r) swap(a[l++], a[r--]);
      }
      if (target <= r) return quickselect(a, lo, r, target);
      if (target >= l) return quickselect(a, l, hi, target);
      return a[target];
  }
  // call: quickselect(nums, 0, n-1, n-k);
  ```
- **`nth_element`** *(quickselect, batteries included)* — `nth_element(nums.begin(), nums.begin()+(n-k), nums.end()); return nums[n-k];`. Average **O(n)**, O(1) space. The one-liner to *state* in an interview after describing quickselect.

## Pitfalls
- **Min-heap for largest, max-heap for smallest** — the classic mix-up. Kth *largest* → keep the k biggest in a **min**-heap (top = answer).
- **Cap the heap at k, not n** — a plain max-heap of all n + k pops is O(n + k log n) and O(n) space; the size-k min-heap is O(n log k) / O(k). Don't let it grow to n.
- **`size()` is unsigned** — `minHeap.size() > k` mixes `size_t` and `int`; cast `k` (or make it `size_t`) to avoid the warning / edge surprises.
- **kth largest ≠ kth distinct** — duplicates count; don't dedupe (that's a different problem).
- **Quickselect worst case** — a sorted/adversarial array with a fixed pivot degrades to O(n²); **randomize the pivot**. Also remember the target index is **`n-k`**, not `k`.
