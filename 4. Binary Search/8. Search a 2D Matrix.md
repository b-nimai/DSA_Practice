# 8. Search a 2D Matrix (LC 74) — Medium
🔗 [Open on LeetCode](https://leetcode.com/problems/search-a-2d-matrix/)

**Pattern:** Binary Search — **flatten a fully-sorted matrix into one virtual 1D array** (rows sorted + each row's first > previous row's last ⇒ the row-major read is globally non-decreasing, so it's plain [[1-binary-search]] over an imaginary length-`m*n` array)
**Trigger:** "`m x n` matrix, **each row sorted** AND **first of each row > last of previous row** → is `target` present, in **O(log(m·n))**". The two properties together mean the whole grid reads as one sorted sequence in row-major order — the tell is *log(m·n)*, which forbids the staircase O(m+n) walk and demands a single binary search over `[0, m*n - 1]` with index `↔` `(row, col)` conversion via `/ cols` and `% cols`.

## Problem (1-liner)
Given an `m x n` matrix where each row is sorted and every row's first element exceeds the previous row's last → return `true` if `target` is in it, in **O(log(m·n))**.
**Example:** `matrix=[[1,3,5,7],[10,11,16,20],[23,30,34,60]], target=3 → true` · `target=13 → false`
**Constraints:** `1 ≤ m, n ≤ 100`, `−10⁴ ≤ matrix[i][j], target ≤ 10⁴`.

## Idea (1-liner)
Treat the matrix as a single sorted array of length `m*n` indexed `0 … m*n-1`; binary-search that, mapping a flat index `idx` back to a cell with `row = idx / cols`, `col = idx % cols`. **Divide and mod by the number of COLUMNS** (the row length), never the number of rows.

## Approach (binary search over the flattened index)
1. `rows = matrix.size()`, `cols = matrix[0].size()`. Closed window `low = 0, high = rows*cols - 1`, loop `while (low <= high)`, overflow-safe `mid = low + (high - low) / 2`.
2. Decode `mid` into a cell: `value = matrix[mid / cols][mid % cols]` — **`cols` is the divisor/modulus** because each row holds `cols` elements, so every block of `cols` flat indices is one row.
3. Standard 704 branching: `value == target` → `true`; `value > target` → `high = mid - 1`; `value < target` → `low = mid + 1`.
4. Fall out of the loop ⇒ `return false`.

## Code (clean canonical)
```cpp
class Solution {
public:
    bool searchMatrix(vector<vector<int>>& matrix, int target) {
        int rows = matrix.size(), cols = matrix[0].size();
        int low = 0, high = rows * cols - 1;          // flattened [0, m*n - 1]
        while (low <= high) {
            int mid = low + (high - low) / 2;
            int value = matrix[mid / cols][mid % cols]; // divmod by COLS
            if (value == target) return true;
            else if (value > target) high = mid - 1;
            else low = mid + 1;
        }
        return false;
    }
};
```
**TC:** O(log(m·n)).  **SC:** O(1).

## My Solution (original — buggy, the fix)
The submitted version named the dimensions **rows-as-`n`, cols-as-`m`**, then decoded the flat index using **`n` (the row count)** instead of the column count:
```cpp
class Solution {
public:
    bool searchMatrix(vector<vector<int>>& matrix, int target) {
      int n = matrix.size(), m = matrix[0].size();   // n = ROWS, m = COLS
      int low = 0, high = (m * n) - 1;
      while (low <= high) {
        int mid = low + (high - low) / 2;
        int value = matrix[mid / n][mid % n];          // ❌ divmod by n (rows)
        if (value == target) return true;
        else if (value > target) high = mid - 1;
        else low = mid + 1;
      }
      return false;
    }
};
```
**The bug:** the flat-index → cell conversion must divide/mod by the **row length = number of columns**, but the code used `n` = number of **rows**. The branching, window, and `high = m*n-1` are all correct — only the divisor is wrong.

**Why `[[1,1]], target=2` crashes:** `rows(n)=1, cols(m)=2`, so `high = 2*1-1 = 1`. First `mid=0` → `matrix[0/1][0%1] = matrix[0][0] = 1 < 2` → `low=1`. Next `mid=1` → `matrix[1/1][1%1] = matrix[1][0]` — **row index 1 doesn't exist** (only 1 row) → out-of-bounds read. It only surfaces when `cols > rows` (so `mid/n` overshoots the real row count); square or tall matrices can pass by luck, which is what makes it sneaky.

**The one-line fix:** decode with the column count — `matrix[mid / m][mid % m]` (with this file's naming where `m` = cols). Cleaner still: rename to `rows`/`cols` so the divisor reads as obviously the row length (see canonical).

74 is **[[1-binary-search]] in disguise**: the only new idea is the index `↔ (row, col)` arithmetic that lets one binary search range over a 2D grid. The compare/branch/window logic is byte-for-byte the 704 exact-match loop.

## Other Approaches
- **Staircase / search-space elimination.** Start top-right (`r=0, c=cols-1`): `> target` → `c--`, `< target` → `r++`, `==` → found. **O(m + n)** and dead simple, but it **ignores the "row.first > prev.last" property** and misses the required O(log(m·n)) — it's the right tool for this problem's harder cousin **LC 240** (rows and columns each sorted, but *not* globally sorted), where flattening is impossible.
- **Two binary searches.** First binary-search the column of row-lasts (`matrix[i][cols-1]`) to find the candidate row, then binary-search within that row. O(log m + log n) = same O(log(m·n)); two loops instead of one. The flattened single-search is tighter.

## Pitfalls
- **Divmod by COLUMNS, not rows.** `mid / cols`, `mid % cols`. Using the row count works only when `cols ≤ rows` and silently goes out-of-bounds otherwise (the `[[1,1]]` crash). Name the variables `rows`/`cols` so the divisor is self-evidently the row length.
- **`high = rows*cols - 1`, closed loop.** Pair `while (low <= high)` with `high = mid - 1` / `low = mid + 1` (the 704 exact-match window) — it must *find* a value, not a boundary, so don't borrow the half-open `high = mid` form.
- **`matrix[0].size()` assumes a non-empty first row.** Safe here (`m, n ≥ 1`); guard `matrix.empty() || matrix[0].empty()` if constraints ever allow empty.
- **Don't confuse with LC 240.** This problem's rows chain into one sorted sequence (binary-searchable); LC 240 only sorts each row and each column independently — there you *must* use the staircase walk, not flattening.
- **Overflow-safe midpoint.** `low + (high - low) / 2`; `m*n ≤ 10⁴` here so `(low+high)` wouldn't overflow, but keep the habit.
