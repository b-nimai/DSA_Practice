# 2. Search Insert Position (LC 35) — Easy
🔗 [Open on LeetCode](https://leetcode.com/problems/search-insert-position/)

**Pattern:** Binary Search — **lower bound / first-true** (the "where does it belong?" twin of exact-match [[1-binary-search]]: same halving loop, but it returns an *insertion point* instead of `-1` when the target is absent)
**Trigger:** "sorted distinct array → index of `target`, **or where it would be inserted** to stay sorted" → you want the **smallest index `i` with `nums[i] >= target`** (a.k.a. `lower_bound`). The "found" answer and the "insert here" answer are the *same* index, which is why no separate not-found branch is needed.

## Problem (1-liner)
Given a **sorted distinct** `nums` and a `target`, return the index of `target`, or the index where it would be inserted in order. Must be **O(log n)**.
**Example:** `nums=[1,3,5,6]` → `target=5 → 2` · `target=2 → 1` · `target=7 → 4`
**Constraints:** `1 ≤ n ≤ 10⁴`, `−10⁴ ≤ nums[i], target ≤ 10⁴`, distinct + sorted ascending.

## Idea (1-liner)
Binary-search for the **first position whose value is `>= target`**. If `target` exists, that position holds it; if not, it's exactly the slot it should slide into (and `n` if `target` exceeds everything). One monotonic boundary, found by the standard lower-bound loop ⇒ **O(log n) time, O(1) space.**

## Approach (two equivalent framings)
This is the place to internalize that **the same problem has two binary-search shapes**:

**A — reuse the exact-match loop, return `low`** (what your solution does):
1. Closed interval `low = 0`, `high = n - 1`, loop `while (low <= high)`.
2. On `nums[mid] == target` return `mid`; on `>` go left (`high = mid - 1`); on `<` go right (`low = mid + 1`).
3. **Fall out of the loop returning `low`.** The key insight: when the target is absent, `low` ends up at the first index `> target` — exactly the insertion point. (`low` only ever advances past elements strictly `< target`, so it lands on the boundary.)

**B — pure lower-bound (half-open, no equality branch):**
1. `low = 0`, `high = n` (note `n`, a *past-the-end* insertion slot, not `n - 1`).
2. Loop `while (low < high)`; `mid = low + (high - low) / 2`.
3. If `nums[mid] < target` → `low = mid + 1`; else (`>=`) → `high = mid` (**no `-1`** — `mid` might be the answer).
4. Return `low`. Collapses the `==`/`<`/`>` cases into one comparison; this is the template that generalizes to "minimum X such that `check(X)`".

Both are O(log n); **B** is the more reusable mental model, **A** is the smaller diff from [[1-binary-search]].

## Code (clean canonical — lower-bound form)
```cpp
class Solution {
public:
    int searchInsert(vector<int>& nums, int target) {
        int low = 0, high = nums.size();           // half-open [low, high), high = n
        while (low < high) {
            int mid = low + (high - low) / 2;
            if (nums[mid] < target) low = mid + 1;  // mid too small, exclude it
            else high = mid;                        // nums[mid] >= target, keep mid as candidate
        }
        return low;                                 // first index with nums[i] >= target
    }
};
```
**TC:** O(log n).  **SC:** O(1).

## My Solution (also correct — framing A)
```cpp
class Solution {
public:
    int searchInsert(vector<int>& nums, int target) {
      int low = 0, high = nums.size() - 1;
      while (low <= high) {
        int mid = low + (high - low) / 2;
        int num = nums[mid];
        if (num == target) return mid;
        else if (num > target) high = mid - 1;
        else low = mid + 1;
      }
      return low;
    }
};
```
This is **literally the [[1-binary-search]] loop with `return -1` swapped for `return low`** — a great way to see that "insert position" is just "exact match, but report where the search *settled* instead of giving up." Fully correct. The canonical version above is offered only as the more transferable lower-bound template; yours is equally valid and arguably clearer if you've just done 704.

## Family map — where this sits
| Problem | Loop shape | Returns | Note |
|---|---|---|---|
| [[1-binary-search]] (LC 704) | `low <= high`, `mid ± 1` | `mid` / `-1` | exact match; target may not exist → `-1` |
| **35 (this one)** | `low <= high` **or** `low < high` | `low` (insertion point) | lower-bound; "not found" becomes "insert here" |
| 34 First & Last Position | two lower-bound calls | `[first, last]` | lower-bound of `target` and of `target+1` |
| 875 Koko / 410 Split Array | `low < high`, `high = mid` | `low` | same lower-bound loop, `nums[mid] < target` → monotonic `check(mid)` |

The lesson 35 cements: **`return low` after the loop is the insertion point**, and that exact loop generalizes to every "binary search on the answer" problem later in this pattern.

## Other Approaches
- **Linear scan O(n)** — return the first index with `nums[i] >= target`, else `n`. Correct, but violates the required O(log n).
- **`std::lower_bound`** — `lower_bound(nums.begin(), nums.end(), target) - nums.begin()` is exactly framing B in one line; good to know it maps onto an STL primitive.

## Pitfalls
- **Returning `-1` out of habit.** This is the one-character trap coming from 704: the post-loop return must be **`low`** (the insertion slot), not `-1`. After the loop `low` is the count of elements strictly `< target`, which *is* the answer.
- **Wrong high bound for the lower-bound form.** Framing B uses `high = nums.size()` (= `n`), because the answer can legitimately be `n` (target bigger than every element). Using `n - 1` makes that case unreachable.
- **`high = mid - 1` in the lower-bound loop.** In framing B you must write `high = mid` (not `mid - 1`) — `mid` itself can be the first `>= target` element, and excluding it skips the answer.
- **`low < high` vs `low <= high` mismatch.** Half-open `[low, high)` pairs with `while (low < high)` + `high = mid`; closed `[low, high]` pairs with `while (low <= high)` + `high = mid - 1`. Mixing the two (e.g. `<=` with `high = mid`) infinite-loops. Pick one template and keep its pieces consistent.
- **Overflow midpoint.** Same as [[1-binary-search]]: `low + (high - low) / 2`, never `(low + high) / 2`.
