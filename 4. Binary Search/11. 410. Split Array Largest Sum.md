# 11. Split Array Largest Sum (LC 410) — Hard
🔗 [Open on LeetCode](https://leetcode.com/problems/split-array-largest-sum/)

**Pattern:** Binary Search — **on the answer** (no array to search; the answer is *the largest subarray sum*, and `possible(maxSum)` is a monotonic predicate — false when the cap is too tight, true once it's loose enough — so lower-bound the first `true`). This is the **"minimize the maximum"** archetype.
**Trigger:** "**split into `k` contiguous subarrays** so the **largest subarray sum is minimized**". The phrase *minimize the maximum* is the dead giveaway for binary-search-on-answer. The cap (largest allowed sum) and the number-of-pieces-needed move in opposite directions — bigger cap ⇒ fewer pieces — so feasibility (`pieces ≤ k`) is monotonic. Binary-search the **answer range** `[max(nums), sum(nums)]` with a sequential greedy fill, exactly like [[10-1011-capacity-to-ship-packages]] with `days → k`.

## Problem (1-liner)
Split `nums` into `k` non-empty **contiguous** subarrays so that the **largest** subarray sum is as **small as possible** → return that minimized largest sum.
**Example:** `[7,2,5,10,8], k=2 → 18` (`[7,2,5] | [10,8]`) · `[1,2,3,4,5], k=2 → 9` (`[1,2,3] | [4,5]`)
**Constraints:** `1 ≤ n ≤ 1000`, `0 ≤ nums[i] ≤ 10⁶`, `1 ≤ k ≤ min(50, n)`.

## Idea (1-liner)
Flip the question: instead of "what's the min largest sum?", ask "**given a cap `S`, can we split into ≤ k pieces** with every piece ≤ `S`?". Pieces-needed is **monotonically non-increasing in `S`**, so feasibility flips false→true exactly once. Binary-search the smallest feasible `S` in `[max(nums), sum(nums)]` — `lo = max(nums)` because any cap below the largest single element can't even hold that one element; `hi = sum(nums)` is the whole array as one piece (`k = 1`).

## Approach (binary search on the answer)
1. **Search space = candidate "largest sums", not indices.** `lo = max(nums)` (a piece must hold the biggest element alone), `hi = sum(nums)` (everything in one subarray).
2. **Feasibility predicate** `possible(maxSum)`: walk left→right with a running `sum`; start the piece count at **1** (you're already filling the first subarray). When `sum + num > maxSum`, **cut** — start a new subarray (`count++`, `sum = 0`) before adding `num`. Early-return false once `count > k`. Monotonic: larger cap ⇒ fewer cuts ⇒ once true, stays true.
3. **Why "≤ k" is the same as "exactly k".** If a cap is feasible with *fewer* than `k` pieces, you can always cut some piece further (every element is non-empty and `k ≤ n`) without raising any piece's sum. So testing "needs ≤ k pieces" correctly characterizes a feasible cap.
4. **Lower-bound loop** over the value range: on `possible(mid)` shrink right, else go right. Closed `while (lo <= hi)` + `hi = mid - 1` / `lo = mid + 1` lands `lo` on the minimum feasible cap.

## Code (clean canonical)
```cpp
class Solution {
    bool possible(const vector<int>& nums, int maxSum, int k) {
        int count = 1, sum = 0;                    // already filling piece 1
        for (int num : nums) {
            if (sum + num > maxSum) {              // doesn't fit current piece
                count++;                           // open a new subarray
                sum = 0;
                if (count > k) return false;       // too many pieces → infeasible
            }
            sum += num;                            // += not = (accumulate the piece)
        }
        return true;
    }
public:
    int splitArray(vector<int>& nums, int k) {
        int lo = *max_element(nums.begin(), nums.end());
        int hi = accumulate(nums.begin(), nums.end(), 0);
        while (lo <= hi) {                          // closed-interval lower bound
            int mid = lo + (hi - lo) / 2;
            if (possible(nums, mid, k)) hi = mid - 1;  // feasible → try a tighter cap
            else lo = mid + 1;                         // infeasible → loosen the cap
        }
        return lo;                                 // smallest feasible largest-sum
    }
};
```
**TC:** O(n · log(sum − max)).  **SC:** O(1).

## My Solution (original — correct & optimal)
The submitted version is **already correct and optimal** — same binary-search-on-answer with a closed-interval lower bound. The only differences from the canonical form are cosmetic. Original:
```cpp
bool possible(vector<int>& nums, int target, int k) {
    int count = 1, sum = 0;
    for (int num: nums) {
        if (num + sum > target) { count++, sum = 0; }
        sum += num;
        if (count > k) return false;
    }
    return true;
}
// splitArray: lo = max, hi = sum, closed loop, possible ? hi=mid-1 : lo=mid+1, return lo
```
What it got right (the things that bite people):
- **`sum += num`, never `sum = num`.** Accumulates the running piece — the exact bug that sank the first draft of [[10-1011-capacity-to-ship-packages]].
- **`count = 1`, not 0.** The first subarray exists from the first element; the in-loop `count++` counts only the **cuts**, so `total = 1 + #cuts`.
- **Predicate direction matches the branch.** `possible` is true when the cap is *loose enough*; on feasible go left (`hi = mid - 1`), and `return lo` lands on the first feasible cap.

Tiny refinements (style/robustness, not bugs):
- **`const vector<int>&`** on `possible` — signals it's read-only.
- **Overflow-safe habit:** `sum + num > target` is safe *here* (max total `1000·10⁶ = 10⁹` fits `int`), but `sum > target - num` (or a `long` accumulator) is the bug-proof form if limits ever grow. No `long` needed for these constraints.
- **Check placement:** moving `if (count > k) return false;` inside the cut-branch (only changes when a piece opens) is marginally tighter; the original's once-per-iteration check is equally correct.

## Other Approaches
- **DP (no binary search).** `dp[i][j]` = min largest-sum splitting `nums[0..i]` into `j` pieces, transition over the last cut: O(n²·k) time, O(n·k) space. Correct and the "classic" Hard-tier solution, but slower and heavier than binary search; reach for it only if asked for the partition itself or if monotonicity weren't available.
- **Linear scan of caps.** Try `S = max, max+1, …` until feasible — O(n·(sum−max)), blows the limit.
- **Looser low bound (`lo = 0` or `1`).** Still converges (predicate is false below `max(nums)`), but wastes `log` iterations and invites off-by-one. `lo = max(nums)` is the tight, self-documenting bound.

## Pitfalls
- **`sum += num`, never `sum = num`.** Overwriting the running piece silently undercounts cuts and returns a too-small (infeasible) answer.
- **`count = 1`, not 0.** The first subarray is already in progress; the in-loop `++` counts only new-piece cuts. Seeding at 0 undercounts by one piece.
- **`lo = max(nums)`, not 0/1.** A cap below the largest element can never hold it alone — start the range there (this is also why every `num ≤ mid` always holds, so the greedy never gets stuck).
- **"≤ k" not "== k".** Test pieces-needed `≤ k`; a partition using fewer pieces can always be cut up to exactly `k` (since `k ≤ n`) without raising any sum.
- **Window/branch must agree.** Closed `while (lo <= hi)` ⇒ `hi = mid - 1`; half-open `while (lo < hi)` ⇒ `hi = mid`. Mixing them (closed + `hi = mid` infinite-loops, half-open + `hi = mid - 1` skips the answer).
- **Overflow only at larger scales.** Here `sum ≤ 10⁹` fits `int`; if `nums[i]` or `n` grew, use `long` for `hi`/`sum` or write `sum > target - num`.
