# 6. Find First and Last Position of Element in Sorted Array (LC 34) — Medium
🔗 [Open on LeetCode](https://leetcode.com/problems/find-first-and-last-position-of-element-in-sorted-array/)

**Pattern:** Binary Search — **two boundary searches** (the [[2-search-insert-position]] lower-bound idea run twice: once biased left for the first occurrence, once biased right for the last)
**Trigger:** "sorted array **with duplicates** → the **first and last index** of `target`, or `[-1, -1]`, in **O(log n)**". The tell is *a range of equal values + log n*: a plain hit doesn't tell you where the block starts/ends, so on `nums[mid] == target` you **don't stop** — you keep shrinking toward one side to pin the boundary.

## Problem (1-liner)
A non-decreasing array (may be empty, may repeat) + `target` → return `[firstIndex, lastIndex]` of `target`, or `[-1, -1]` if absent, in **O(log n)**.
**Example:** `nums=[5,7,7,8,8,10], target=8 → [3,4]` · `target=6 → [-1,-1]` · `nums=[], target=0 → [-1,-1]`
**Constraints:** `0 ≤ n ≤ 10⁵`, `−10⁹ ≤ nums[i], target ≤ 10⁹`, non-decreasing.

## Idea (1-liner)
Run binary search twice. For the **first** index, on a hit record `mid` and keep searching **left** (`high = mid - 1`); for the **last**, on a hit record `mid` and keep searching **right** (`low = mid + 1`). Each pass is O(log n) ⇒ **O(log n) total, O(1) space.** If the first pass returns `-1`, the value is absent.

## Approach (two boundary passes)
1. **`lowerBound`** — closed interval, `index = -1`. On `nums[mid] == target`: save `index = mid`, then **`high = mid - 1`** (don't stop — a smaller index may exist). `nums[mid] > target` → `high = mid - 1`; `nums[mid] < target` → `low = mid + 1`. Return `index`.
2. **`upperBound`** — identical, but on a hit **`low = mid + 1`** (push right to find a larger index).
3. `searchRange` returns `{lowerBound(...), upperBound(...)}`. If `target` is absent both return `-1` ⇒ `[-1, -1]` naturally.

The single change between the two passes is the **tie-break direction** on equality (`high = mid - 1` vs `low = mid + 1`); everything else is the plain [[1-binary-search]] loop. Because the empty-array case makes `n = nums.size() - 1 == -1`, the loop guard `low <= high` is `0 <= -1` ⇒ false, so it safely returns `-1` without touching `nums`.

## Code (clean canonical)
```cpp
class Solution {
    int lowerBound(vector<int>& nums, int target, int low, int high) {
        int index = -1;
        while (low <= high) {
            int mid = low + (high - low) / 2;
            if (nums[mid] == target) { index = mid; high = mid - 1; }  // keep going left
            else if (nums[mid] > target) high = mid - 1;
            else low = mid + 1;
        }
        return index;
    }
    int upperBound(vector<int>& nums, int target, int low, int high) {
        int index = -1;
        while (low <= high) {
            int mid = low + (high - low) / 2;
            if (nums[mid] == target) { index = mid; low = mid + 1; }   // keep going right
            else if (nums[mid] > target) high = mid - 1;
            else low = mid + 1;
        }
        return index;
    }
public:
    vector<int> searchRange(vector<int>& nums, int target) {
        int n = nums.size() - 1;
        return { lowerBound(nums, target, 0, n), upperBound(nums, target, 0, n) };
    }
};
```
**TC:** O(log n) (two passes).  **SC:** O(1).

| **34 (this one)** | sorted, **duplicates** | `[first, last]` of `target` | left for first, right for last |

34 is the natural "blocks of duplicates" extension of the lower-bound template: instead of *where would it go*, you ask *where does the equal block start and end*.

## Other Approaches
- **Lower-bound primitives (`first >= target` and `first > target`).** Find `lo = firstIndex(nums[i] >= target)`; if `lo == n || nums[lo] != target` ⇒ absent. Else `hi = firstIndex(nums[i] > target) - 1`. Same O(log n), expressed with the pure half-open lower-bound template from 35 — slightly more reusable, but the "record-and-keep-shrinking" version here is just as clear.
- **One search to find any hit, then linear expand.** Binary-search to *any* occurrence, then walk left/right to the ends. Simple but **O(n)** worst case (e.g. all elements equal) — violates the required O(log n).
- **`std::lower_bound` / `std::upper_bound`.** The STL twins do exactly this; `upper_bound - 1` gives the last index. Great in real code, but the manual version is the point of the drill.

## Pitfalls
- **Stopping on the first `==`.** The whole trick is to *not* return on a hit — record it and keep shrinking toward the boundary. Returning early gives an arbitrary middle index, not the first/last.
- **Wrong tie-break direction.** First index needs `high = mid - 1` on equality; last needs `low = mid + 1`. Swapping them returns the opposite boundary.
- **Empty array / absent target.** `nums.size() - 1` is `-1` when empty; the `low <= high` guard must make the loop a no-op (it does) so you don't index `nums[-1]`. Absent target falls out as `-1` from both passes.
- **Overflow midpoint.** `low + (high - low) / 2`, never `(low + high) / 2` — `n` up to `1e5` is fine for `int`, but keep the habit.
- **Signed/unsigned mix.** `nums.size()` is unsigned; compute the high bound as `(int)nums.size() - 1` to avoid surprises if it ever feeds an unsigned context.
