# 12. Median of Two Sorted Arrays (LC 4) — Hard
🔗 [Open on LeetCode](https://leetcode.com/problems/median-of-two-sorted-arrays/)

**Pattern:** Binary Search — **on the partition** (not on a value or an answer-range). Binary-search *how many elements the smaller array contributes to the combined left half*; the partition of the larger array is then forced by arithmetic. A partition is correct when **every left element ≤ every right element** — a monotonic predicate, so lower-bound it.
**Trigger:** "**median of two sorted arrays in O(log(m+n))**". The `O(log)` requirement kills the merge (`O(m+n)`); the tell is *split both arrays into a left half and a right half of the right sizes such that `max(left) ≤ min(right)`* without merging. Binary-search the cut in the **smaller** array only.

## Problem (1-liner)
Given two sorted arrays, return the median of their union **in O(log(m+n))** (no full merge).
**Example:** `[1,3] & [2] → 2.0` · `[1,2] & [3,4] → 2.5` · `[2,3,5,8] & [10,12,14,16,18,20] → 11.0`
**Constraints:** `0 ≤ m, n ≤ 1000`, `1 ≤ m + n ≤ 2000`, `-10⁶ ≤ vals ≤ 10⁶`.

## Idea (1-liner)
The median splits the union into a left half and a right half, where **left has `(m+n+1)/2` elements** and `max(left) ≤ min(right)`. Pick `cut1` elements from `nums1` for the left; then `cut2 = (m+n+1)/2 − cut1` from `nums2` is **forced**. Binary-search `cut1` over `[0, m]` (search the **smaller** array so `cut2` never goes negative), checking the cross-condition `l1 ≤ r2 && l2 ≤ r1`.

## Approach (binary search on the partition)
1. **Always search the smaller array.** If `n1 > n2`, swap (recurse). Guarantees `cut2 = leftCount − cut1 ∈ [0, n2]`, so no negative/overflow partition.
2. **Left-half size** `leftCount = (n1 + n2 + 1) / 2`. The `+1` makes the **left side hold the extra element on odd totals**, so the median sits at `max(left)`.
3. **Search `cut1 ∈ [0, n1]`** (closed loop). For a candidate, `cut2 = leftCount − cut1`. Define the four boundary values with **sentinels** for the edges:
   - `l1 = cut1>0 ? nums1[cut1-1] : INT_MIN` (last of nums1's left), `r1 = cut1<n1 ? nums1[cut1] : INT_MAX` (first of nums1's right)
   - `l2 = cut2>0 ? nums2[cut2-1] : INT_MIN`, `r2 = cut2<n2 ? nums2[cut2] : INT_MAX`
4. **Cross-check** `l1 ≤ r2 && l2 ≤ r1` (each side's max ≤ the other side's min):
   - **valid** → even total: `(max(l1,l2) + min(r1,r2)) / 2.0`; odd total: `max(l1,l2)`.
   - `l1 > r2` → took **too many** from nums1 → `high = cut1 - 1`.
   - else (`l2 > r1`) → too few from nums1 → `low = cut1 + 1`.

## Code (clean canonical)
```cpp
class Solution {
public:
    double findMedianSortedArrays(vector<int>& nums1, vector<int>& nums2) {
        int n1 = nums1.size(), n2 = nums2.size();
        if (n1 > n2) return findMedianSortedArrays(nums2, nums1);  // search the smaller

        int leftCount = (n1 + n2 + 1) / 2;        // left half size (+1 → odd extra on left)
        int low = 0, high = n1;
        while (low <= high) {
            int cut1 = low + (high - low) / 2;
            int cut2 = leftCount - cut1;

            int l1 = cut1 > 0  ? nums1[cut1 - 1] : INT_MIN;   // index/guard both cut1
            int r1 = cut1 < n1 ? nums1[cut1]     : INT_MAX;
            int l2 = cut2 > 0  ? nums2[cut2 - 1] : INT_MIN;   // index/guard both cut2
            int r2 = cut2 < n2 ? nums2[cut2]     : INT_MAX;

            if (l1 <= r2 && l2 <= r1) {           // valid partition
                if ((n1 + n2) % 2 == 0)
                    return (max(l1, l2) + min(r1, r2)) / 2.0;
                return max(l1, l2);
            }
            if (l1 > r2) high = cut1 - 1;          // too many from nums1
            else         low  = cut1 + 1;          // too few from nums1
        }
        return 0.0;                                // unreachable on valid input
    }
};
```
**TC:** O(log(min(m, n))).  **SC:** O(1).

## My Solution (correct after fixing 2 `mid1`/`mid2` mix-ups)
The structure was right from the start; the only failures were two copy-paste slips where a `nums2` access used `mid1` instead of `mid2`. Both are now fixed. The bugs were:
```cpp
int l2 = (mid2 - 1) >= 0 ? nums2[mid1 - 1] : INT_MIN;   // ❌ was nums2[mid1-1] → nums2[mid2-1]
int r2 = mid1 < n2       ? nums2[mid2]     : INT_MAX;    // ❌ guard was mid1<n2 → mid2<n2
```
**The rule that prevents both: `mid1`/`n1` touch only `nums1`, `mid2`/`n2` touch only `nums2` — index *and* its bounds-guard must use the same variable.**
- **`l2` bug.** `l2` is nums2's last left element → `nums2[mid2 - 1]`. Worse, the guard (`mid2-1>=0`) and the index (`mid1-1`) **disagreed**, so when `mid2 ≥ 1` but `mid1 == 0` it read `nums2[-1]` → UB. Failed immediately on `[1,3] & [2]`: after swap `nums1=[2], nums2=[1,3]`, first step `mid1=0, mid2=2` → `nums2[-1]`.
- **`r2` bug.** `r2` exists only when `mid2 < n2`. Guarding with `mid1 < n2` lets `mid2 == n2` slip through (all of nums2 on the left) → reads `nums2[n2]` out of bounds.

Everything else — the `n1>n2` swap, `noLeftEle`, sentinels, branch direction, even/odd return — was already correct.

## Other Approaches
- **Merge / two-pointer to the middle.** Walk both arrays counting to position `(m+n)/2`. O(m+n) time, O(1) space — simplest and fine if `O(log)` isn't required, but violates the problem's complexity bound.
- **Full merge then index.** O(m+n) time **and** space — strictly worse; only a brute-force sanity check.
- **k-th element via recursive halving.** Generalizes to "find k-th of two sorted arrays" and also hits O(log(m+n)); more moving parts than the partition method for just the median.

## Pitfalls
- **`mid1`/`mid2` must not cross.** Every `nums1` access uses `mid1` (index + guard); every `nums2` access uses `mid2`. The classic bug is `nums2[mid1-1]` or a `mid1 < n2` guard → wrong value or out-of-bounds (`nums2[-1]` / `nums2[n2]`).
- **Search the smaller array.** Without the `n1 > n2` swap, `cut2 = leftCount - cut1` can go negative / exceed `n2`.
- **`leftCount = (n1+n2+1)/2`, with the `+1`.** Puts the extra element on the left for odd totals so the median is `max(left)`. Dropping the `+1` shifts odd-case indexing off by one.
- **Sentinels at the edges.** `INT_MIN` for an empty left side, `INT_MAX` for an empty right side — they make "take none / take all" of an array a legal, comparison-safe partition (exactly what the `[2,3,5,8] & [10,…]` case exercises, where all of nums1 lands left).
- **Even vs odd return.** Even: average the two middles `(max(l1,l2)+min(r1,r2))/2.0` — **use `2.0`** (float) not `2`. Odd: `max(l1,l2)`.
- **Branch direction.** `l1 > r2` ⇒ nums1's left reaches too high ⇒ move `high` left; otherwise move `low` right. Closed `while (low <= high)` pairs with `high = cut1-1` / `low = cut1+1`.
