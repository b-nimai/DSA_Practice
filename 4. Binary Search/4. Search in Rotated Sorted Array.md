# 4. Search in Rotated Sorted Array (LC 33) — Medium
🔗 [Open on LeetCode](https://leetcode.com/problems/search-in-rotated-sorted-array/)

**Pattern:** Binary Search — **modified exact-match on a rotated array** (the [[1-binary-search]] closed-interval loop, but since the array isn't globally sorted you first decide *which half is sorted*, then test whether `target` falls inside that sorted half)
**Trigger:** "ascending **distinct** array, **rotated** at an unknown pivot → index of `target` in **O(log n)**". The tell is *rotated + still wants log n*: a plain `nums[mid]` vs `target` compare is no longer enough, but **at every `mid` at least one side `[low..mid]` or `[mid..high]` is fully sorted**, and a sorted side lets you range-check `target` and discard half.

## Problem (1-liner)
A sorted-ascending distinct array, possibly rotated at an unknown index, plus `target` → return its index or `-1`, in **O(log n)**.
**Example:** `nums=[4,5,6,7,0,1,2]` → `target=0 → 4` · `target=3 → -1` · `nums=[1], target=0 → -1`
**Constraints:** `1 ≤ n ≤ 5000`, values unique, `−10⁴ ≤ nums[i], target ≤ 10⁴`.

## Idea (1-liner)
One binary search: compute `mid`, figure out **which half is sorted** (compare `nums[low]` with `nums[mid]`), then check if `target` lies within that sorted half's value range — if yes, search it; if no, search the other half. ⇒ **O(log n) time, O(1) space.**

## Approach (which-half-is-sorted)
1. Closed interval `low = 0`, `high = n - 1`, loop `while (low <= high)`; `mid = low + (high - low) / 2`.
2. **Hit:** `nums[mid] == target` → `return mid`.
3. **Left half `[low..mid]` is sorted** when `nums[low] <= nums[mid]`:
   - if `nums[low] <= target < nums[mid]` → target is inside it ⇒ `high = mid - 1`.
   - else ⇒ `low = mid + 1`.
4. **Otherwise the right half `[mid..high]` is sorted:**
   - if `nums[mid] < target <= nums[high]` → target is inside it ⇒ `low = mid + 1`.
   - else ⇒ `high = mid - 1`.
5. Fall out of the loop ⇒ `return -1`.

The crux is step 3's `nums[low] <= nums[mid]` test (note `<=`, not `<`): with `low == mid` the left "half" is a single sorted element, and `<=` keeps that case in the sorted branch. The endpoint comparisons are inclusive (`nums[low] <= target`, `nums[high] >= target`) so a target sitting *on* a boundary is caught; the `mid` side is exclusive because `nums[mid] == target` was already handled.

## Code (clean canonical)
```cpp
class Solution {
public:
    int search(vector<int>& nums, int target) {
        int low = 0, high = nums.size() - 1;
        while (low <= high) {
            int mid = low + (high - low) / 2;
            if (nums[mid] == target) return mid;

            if (nums[low] <= nums[mid]) {                 // left half [low..mid] sorted
                if (nums[low] <= target && target < nums[mid]) high = mid - 1;
                else low = mid + 1;
            } else {                                      // right half [mid..high] sorted
                if (nums[mid] < target && target <= nums[high]) low = mid + 1;
                else high = mid - 1;
            }
        }
        return -1;
    }
};
```
**TC:** O(log n).  **SC:** O(1).

## My Solution (also correct — identical logic)
```cpp
class Solution {
public:
    int search(vector<int>& nums, int target) {
      int low = 0, high = nums.size() - 1;
      while (low <= high) {
        int mid = low + (high - low) / 2;

        if(nums[mid] == target) return mid;
        else if (nums[low] <= nums[mid]) {
          if (nums[low] <= target && nums[mid] > target) high = mid - 1;
          else low = mid + 1;
        } else {
          if (nums[mid] < target && nums[high] >= target) low = mid + 1;
          else high = mid - 1;
        }
      }
      return -1;
    }
};
```
**Algorithmically flawless** — correct sorted-half detection, correct inclusive endpoint checks, O(log n)/O(1). Two purely cosmetic notes: (a) since `nums[mid] == target` is handled first, the `nums[mid] > target` / `nums[mid] < target` strictness isn't load-bearing — `>=`/`<=` would behave identically; (b) the snippet's class needs a trailing **`;`** after the closing brace to compile standalone (LeetCode's template usually supplies it). Nothing to change in the logic.

## Other Approaches
- **Find pivot, then plain binary search (two passes).** First binary-search the rotation index (smallest element, à la 153), then do an ordinary 704 search in the correct sub-range. Also O(log n) but two passes and more code; the one-pass "which half is sorted" version above is tighter.
- **Linear scan O(n).** Trivially correct, but violates the required O(log n).

## Pitfalls
- **`nums[low] < nums[mid]` (strict) breaks the `low == mid` case.** Use `nums[low] <= nums[mid]` so a single-element left side stays in the sorted branch; strict `<` can misroute size-1/2 windows.
- **Forgetting an inclusive endpoint.** The range test must include the sorted side's endpoint (`nums[low] <= target`, `target <= nums[high]`); using strict `<`/`>` there can miss a target that equals a boundary value.
- **Range-checking against the *unsorted* half.** You may only do the `low <= target < mid` style test on the half you've proven sorted. Testing the rotated half's raw endpoints is meaningless.
- **Overflow midpoint.** Same habit as the rest of the pattern: `low + (high - low) / 2`, never `(low + high) / 2`.
- **Assuming distinct.** This relies on uniqueness; with duplicates (LC 81) the `nums[low] == nums[mid]` tie is ambiguous and you must shrink `low`/`high` by one, degrading to O(n) worst case.
