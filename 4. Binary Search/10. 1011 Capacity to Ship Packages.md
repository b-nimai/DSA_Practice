# 10. Capacity To Ship Packages Within D Days (LC 1011) — Medium
🔗 [Open on LeetCode](https://leetcode.com/problems/capacity-to-ship-packages-within-d-days/)

**Pattern:** Binary Search — **on the answer** (no array to search; the answer is a *ship capacity*, and `possible(cap)` is a monotonic predicate — false for small capacities, true once the ship is big enough — so lower-bound the first `true`)
**Trigger:** "**minimum capacity / threshold** such that a **sequential greedy split** fits **within a budget** (`days ≤ D`), and feasibility **monotonically improves as capacity grows**". The tell is *minimize a value subject to a feasibility check that you compute by a left-to-right partition* — items shipped **in order**, no reordering or splitting. Binary-search the **answer range** `[max(weights), sum(weights)]`.

## Problem (1-liner)
Packages with `weights[i]` must ship **in the given order**; each day load consecutive packages without exceeding the ship's capacity → return the **least** capacity that ships everything within `days` days.
**Example:** `[1..10], days=5 → 15` · `[3,2,2,4,1,4], days=3 → 6` · `[1,2,3,1,1], days=4 → 3`
**Constraints:** `1 ≤ days ≤ n ≤ 5·10⁴`, `1 ≤ weights[i] ≤ 500`.

## Idea (1-liner)
Days-needed is a **monotonically non-increasing** function of capacity, so feasibility (`days ≤ D`) flips false→true exactly once. Binary-search the smallest feasible capacity in `[max(weights), sum(weights)]` — the **low bound must be `max(weights)`** (any capacity smaller can't even carry the heaviest single package), the high bound is `sum` (one day for everything).

## Approach (binary search on the answer)
1. **Search space = capacity values, not indices.** `lo = max(weights)` (must fit the biggest package alone, else it never ships), `hi = sum(weights)` (whole belt in one day).
2. **Feasibility predicate** `possible(cap)`: walk packages left→right keeping a running `load`; when `load + w` would exceed `cap`, **close the day** (`days++`, reset `load = 0`) before adding `w`. Start the count at **1** — you're already filling day 1 the moment you pick up the first package. Early-return false once `days > D`. Monotonic: larger `cap` ⇒ fewer days ⇒ once true, stays true.
3. **Lower-bound loop** over the value range: on `possible(mid)` shrink right, else go right. Closed `while (lo <= hi)` + `hi = mid - 1` / `lo = mid + 1` lands `lo` on the minimum feasible capacity.

## Code (clean canonical)
```cpp
class Solution {
    bool possible(const vector<int>& weights, int cap, int days) {
        int reqDay = 1, load = 0;                  // already on day 1
        for (int w : weights) {
            if (load + w > cap) {                  // doesn't fit today
                reqDay++;                          // open a new day
                load = 0;
            }
            load += w;                             // += not = (accumulate the day)
            if (reqDay > days) return false;       // too many days → infeasible
        }
        return true;
    }
public:
    int shipWithinDays(vector<int>& weights, int days) {
        int lo = *max_element(weights.begin(), weights.end());
        int hi = accumulate(weights.begin(), weights.end(), 0);
        while (lo <= hi) {                          // closed-interval lower bound
            int mid = lo + (hi - lo) / 2;
            if (possible(weights, mid, days)) hi = mid - 1;  // feasible → try smaller
            else lo = mid + 1;                               // infeasible → bigger ship
        }
        return lo;                                 // smallest feasible capacity
    }
};
```
**TC:** O(n · log(sum − max)).  **SC:** O(1).

## My Solution (original — correct after fixes)
The submitted version was right after **three fixes**, each instructive:
- **`sum = weight` → `sum += weight`.** The original *overwrote* the day's load with each package's weight instead of accumulating it, so `load` was always just the last package. `load + w > cap` almost never fired ⇒ days were massively undercounted ⇒ `possible()` returned true for capacities that were actually too small ⇒ final answer too low.
- **Per-package `ceil` predicate → sequential greedy.** An even earlier version did `reqDay += ceil(weight/mid)`, which is the **split-allowed** model (each package shipped independently). The problem ships packages **in order, accumulated into a day** — must be a left-to-right fill, not a per-item ceil.
- **Template consistency.** Either pair `while (lo < hi)` with `hi = mid` (half-open), **or** `while (lo <= hi)` with `hi = mid - 1` (closed). The original mixed `lo < hi` with `hi = mid - 1`, which can skip the answer. The accepted form is the closed version above.
- *Cosmetic:* a leftover `cout << maxi;` polluted stdout → "wrong answer" on the judge. Removed.

**Why `reqDay = 1`, not `0`:** `reqDay` is the day you're *currently filling*, and that day exists from the first package. The in-loop `reqDay++` counts only the **transitions** to a new day (overflow events), so `total = 1 + #overflows`. Seeding at 0 undercounts by exactly one day.

## Other Approaches
- **Linear scan of capacities.** Try `cap = max, max+1, …` until one fits in `days`. O(n · (sum − max)) — correct, but blows the limit.
- **Looser low bound (`lo = 1` or `0`).** Still converges since `possible` is false below `max(weights)`, but wastes `log` iterations and risks an off-by-one if the predicate isn't airtight. `lo = max(weights)` is the tight, self-documenting bound.

## Pitfalls
- **`load += w`, never `load = w`.** Overwriting the running load is the classic bug — it silently undercounts days and returns a too-small capacity.
- **Sequential greedy, not per-item `ceil`.** Packages ship in order and accumulate; `reqDay += ceil(w/cap)` is the wrong (split-allowed) model.
- **`reqDay = 1`, not 0.** The first day is already in progress; the in-loop `++` counts only new-day transitions.
- **`lo = max(weights)`, not 1.** Capacity below the heaviest package can never ship it — start the range there.
- **Window/branch must agree.** Closed `while (lo <= hi)` ⇒ `hi = mid - 1`; half-open `while (lo < hi)` ⇒ `hi = mid`. Mixing them (closed + `hi = mid` infinite-loops, half-open + `hi = mid - 1` skips the answer).
- **No debug prints.** A stray `cout` corrupts stdout and fails the judge.
