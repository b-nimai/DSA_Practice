# 3. First Bad Version (LC 278) — Easy
🔗 [Open on LeetCode](https://leetcode.com/problems/first-bad-version/)

**Pattern:** Binary Search — **lower bound / first-true** (the exact same boundary loop as [[2-search-insert-position]], but the monotonic predicate is an **API call** `isBadVersion(mid)` instead of an array comparison — your first taste of *binary-search-on-a-predicate*, not on values)
**Trigger:** "versions `1..n`, once one is bad **all later ones are bad** → find the **first** bad one with the **fewest API calls**" → the `false…false true…true` step is monotonic, so binary-search the **smallest index where `check(mid)` flips true**. "Minimize the number of calls" is the explicit O(log n) tell.

## Problem (1-liner)
Given `n` versions where badness is monotonic (`good…good bad…bad`) and an API `bool isBadVersion(v)`, return the **first bad version**, minimizing API calls.
**Example:** `n=5, bad=4` → `isBad(3)=false`, `isBad(5)=true`, `isBad(4)=true` ⇒ `4`. · `n=1, bad=1` → `1`
**Constraints:** `1 ≤ bad ≤ n ≤ 2³¹ − 1` (so the midpoint **must** be overflow-safe).

## Idea (1-liner)
The bad-flag sequence is `F F … F T T … T`; find the **leftmost `T`**. That's a textbook lower-bound: half-open window `[low, high]`, shrink toward the boundary, never discard a candidate `true`. ⇒ **O(log n) calls, O(1) space.**

## Approach (find-leftmost-true)
1. `low = 1`, `high = n` — the answer lives somewhere in `[low, high]` (1-indexed versions).
2. Loop `while (low < high)`; `mid = low + (high - low) / 2` (**never `(low + high)/2`** — `n` can be `INT_MAX`).
3. Call the API **once** per iteration:
   - `isBadVersion(mid) == false` → first bad is strictly to the right ⇒ `low = mid + 1`.
   - `isBadVersion(mid) == true` → `mid` *might be* the first bad one, so keep it ⇒ `high = mid` (**no `-1`**).
4. Loop exits when `low == high`, collapsed onto the boundary; return `low` (≡ `high`).

The shape is identical to lower-bound framing B in [[2-search-insert-position]] — only the comparison changed from `nums[mid] < target` to `!isBadVersion(mid)`. That substitution *is* the whole "binary search on the answer" idea the rest of this pattern leans on.

## Code (clean canonical)
```cpp
// bool isBadVersion(int version);  // API given to you

class Solution {
public:
    int firstBadVersion(int n) {
        int low = 1, high = n;                     // answer in [low, high]
        while (low < high) {
            int mid = low + (high - low) / 2;      // overflow-safe; n may be INT_MAX
            if (isBadVersion(mid)) high = mid;     // mid could be the first bad — keep it
            else low = mid + 1;                    // mid still good — first bad is later
        }
        return low;                                // low == high == first bad version
    }
};
```
**TC:** O(log n) API calls.  **SC:** O(1).

## My Solution (also correct — only a stored flag differs)
```cpp
class Solution {
public:
    int firstBadVersion(int n) {
      int low = 1, high = n;
      while(low < high) {
        int mid = low + (high - low) / 2;
        bool isBad = isBadVersion(mid);
        if (!isBad) low = mid + 1;
        else high = mid;
      }
      return low;
    }
};
```
Already **optimal** — minimal calls, correct boundary, overflow-safe midpoint. The only refinement is cosmetic: the `isBad` temp can be inlined (`if (isBadVersion(mid)) high = mid; else low = mid + 1;`) and leading with the *bad* case reads a touch more naturally. Keeping the named flag is perfectly fine. **Nothing to change for correctness or efficiency.**

| 875 Koko / 410 Split Array | `!check(mid)` | `low < high`, `high = mid` | `low` | same loop; `check` = a feasibility test over the answer range |

The lesson 278 cements: **lower-bound doesn't need an array** — any monotonic `false→true` predicate plugs straight into `high = mid` / `low = mid + 1` and `return low`.

## Other Approaches
- **Linear scan O(n)** — call `isBadVersion(1), (2), …` until the first `true`. Correct but worst-case `n` calls, violating "minimize the number of calls."
- **Closed-interval form** — `low=1, high=n, while(low<=high)`, on bad `high=mid-1` (track answer), on good `low=mid+1`. Works but needs an extra `ans` variable; the half-open `high = mid` form returns the boundary for free.

## Pitfalls
- **`(low + high) / 2` overflows.** With `n = 2³¹ − 1` the sum wraps to negative → UB / wrong mid. Always `low + (high - low) / 2`. This problem is the canonical reason that habit exists.
- **`high = mid - 1` skips the answer.** On a `true`, `mid` itself may be the first bad version; you must write `high = mid` (no `-1`), exactly as in [[2-search-insert-position]] framing B.
- **`low <= high` with `high = mid` infinite-loops.** Half-open `[low, high)` logic pairs `while (low < high)` with `high = mid`. Mixing `<=` and `high = mid` never shrinks a size-1 window. Keep the template pieces consistent.
- **Calling the API twice per loop.** Storing the result once (or inlining a single call) honors "minimize calls" — don't call `isBadVersion(mid)` in both the `if` and a later branch.
- **Starting `low = 0`.** Versions are **1-indexed**; `low = 1`. (Harmless here since version 0 is never bad, but it's a wasted call and wrong framing.)
