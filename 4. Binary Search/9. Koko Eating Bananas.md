# 9. Koko Eating Bananas (LC 875) — Medium
🔗 [Open on LeetCode](https://leetcode.com/problems/koko-eating-bananas/)

**Pattern:** Binary Search — **on the answer** (no array to search; the answer is a *speed* `k`, and `canFinish(k)` is a monotonic predicate — false for slow speeds, true once `k` is fast enough — so lower-bound the first `true`)
**Trigger:** "**minimum integer `k`** (rate / capacity / threshold) such that some cost stays **within a budget** (`hours ≤ h`), cost **monotonically decreases as `k` grows**". The tell is *minimize a value subject to a feasibility check* with no sorted input — you binary-search the **answer range** `[1, max(piles)]` and `feasible()` is the comparator. Slow → too many hours (false); past the threshold → fits (true); find the leftmost true.

## Problem (1-liner)
`n` piles, `piles[i]` bananas; guards return in `h` hours. At speed `k`/hr Koko clears `ceil(piles[i]/k)` hours per pile → return the **minimum** `k` so total hours `≤ h`.
**Example:** `piles=[3,6,7,11], h=8 → 4` · `[30,11,23,4,20], h=5 → 30` · `[30,11,23,4,20], h=6 → 23`
**Constraints:** `1 ≤ n ≤ 10⁴`, `n ≤ h ≤ 10⁹`, `1 ≤ piles[i] ≤ 10⁹`.

## Idea (1-liner)
Hours-needed is a **monotonically non-increasing** function of `k`, so feasibility (`hours ≤ h`) flips false→true exactly once. Binary-search the smallest feasible `k` in `[1, max(piles)]` — at `k = max(piles)` every pile takes 1 hour (`n ≤ h` guarantees this fits), so a feasible answer always exists in range.

## Approach (binary search on the answer)
1. **Search space = answer values, not indices.** `low = 1` (speed must be ≥ 1 — `k = 0` is meaningless *and* divides by zero), `high = max(piles)` (no point eating faster than the biggest pile — it's already 1 hr each).
2. **Feasibility predicate** `canFinish(k)`: sum `ceil(piles[i]/k)` = `(piles[i] + k - 1) / k` over all piles; **early-return false the moment the running total exceeds `h`**. Monotonic: larger `k` ⇒ fewer hours ⇒ once true, stays true.
3. **Lower-bound loop** over the value range: on `canFinish(mid)` keep `mid` as a live candidate and shrink right (`high = mid`), else `low = mid + 1`. `low == high` lands on the minimum feasible speed → return it.

## Code (clean canonical)
```cpp
class Solution {
    bool canFinish(const vector<int>& piles, int k, int h) {
        long hours = 0;                            // long: n·ceil can exceed int
        for (int p : piles) {
            hours += (p + k - 1) / k;              // ceil(p / k)
            if (hours > h) return false;           // early-exit (also caps the sum)
        }
        return true;
    }
public:
    int minEatingSpeed(vector<int>& piles, int h) {
        int low = 1, high = *max_element(piles.begin(), piles.end());
        while (low < high) {                       // half-open lower bound
            int mid = low + (high - low) / 2;
            if (canFinish(piles, mid, h)) high = mid;  // mid may be the answer
            else low = mid + 1;                        // too slow → faster
        }
        return low;                                // smallest feasible speed
    }
};
```
**TC:** O(n · log(max pile)).  **SC:** O(1).

## My Solution (original — correct, with refinements)
The submitted version was right after two fixes (`low = 1` to kill the division-by-zero, and aligning the predicate + branch direction). The accepted form used a closed loop with `high = mid - 1` and `return low`:
```cpp
class Solution {
  bool possible(vector<int>& piles, int b, int h) {
    int count = 0;
    for (int banana : piles) {
      count += (banana + b - 1) / b;
      if (count > h) return false;          // early-exit feasibility
    }
    return true;
  }
public:
    int minEatingSpeed(vector<int>& piles, int h) {
      int high = *max_element(piles.begin(), piles.end());
      int low = 1;
      while (low <= high) {
        int mid = low + (high - low) / 2;
        if (possible(piles, mid, h)) high = mid - 1;
        else low = mid + 1;
      }
      return low;
    }
};
```
**Correct and optimal** — closed-interval lower bound: on feasible shrink with `high = mid - 1`, and the post-loop `low` is the first feasible speed. Refinements, all cosmetic:
- **The `count > h` early-exit is load-bearing, not just a speedup.** Without it `count` could hit `n · max = 10⁴ · 10⁹ = 10¹³`, far past `int`. With it, `count` stays `≤ h + one increment ≈ 2×10⁹` — fits `int`, but *barely*; promoting to `long` removes the cliff entirely.
- **Half-open form drops the `±1` bookkeeping.** `while (low < high)` + `high = mid` (no `-1`) + `return low` keeps `mid` as a live candidate and is the canonical lower-bound shape — one fewer off-by-one to track. The closed version is equally correct.
- **Naming.** `possible`/`b` → `canFinish`/`speed` (or `k`) makes the predicate's direction self-documenting; `const vector<int>&` signals it's read-only.

Earlier wrong versions failed for instructive reasons: `low = 0` → `k = 0` → **division by zero**; and a predicate of `count >= h` (true when *too slow*) ran the search in the **wrong direction**, returning the max speed instead of the min.

## Other Approaches
- **Linear scan of speeds.** Try `k = 1, 2, 3, …` until one finishes in time. O(n · max) — correct but blows the time limit (`max` up to 10⁹). Only useful as a brute-force sanity check.
- **Looser upper bound.** `high = max(piles)` is the tight bound (at that speed every pile is exactly 1 hour, total `= n ≤ h`, always feasible). `high = sum(piles)` also works but wastes `log` factor. Prefer `max(piles)`.

## Pitfalls
- **`low = 1`, never `0`.** Speed 0 is meaningless and triggers `division by zero` in `(p + k - 1) / k`. The answer range starts at 1.
- **Predicate direction must match the branch.** `feasible = (hours ≤ h)` is true for *fast* speeds. On feasible go **left** (toward smaller `k`); on infeasible go right. Encoding the predicate as "too slow" (`hours >= h`) inverts the search and returns the wrong end.
- **Overflow in the hour sum.** `n · ceil` can reach ~10¹³ → use `long` for the accumulator, or rely on a strict `> h` early-exit (which caps it just under `int` range — fragile; prefer `long`).
- **Window/branch must agree.** Closed `while (low <= high)` ⇒ `high = mid - 1`; half-open `while (low < high)` ⇒ `high = mid` (no `-1`). Don't mix — closed-loop + `high = mid` infinite-loops, half-open + `high = mid - 1` can skip the answer.
- **`ceil` without floats.** `(p + k - 1) / k`, not `ceil((double)p / k)` — integer form avoids precision bugs at the 10⁹ scale.
- **`high = max(piles)`, not `sum`.** Faster than the biggest pile is wasted; the tight bound keeps `log` small (though both are correct).
