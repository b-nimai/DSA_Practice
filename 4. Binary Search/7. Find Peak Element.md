# 7. Find Peak Element (LC 162) — Medium
🔗 [Open on LeetCode](https://leetcode.com/problems/find-peak-element/)

**Pattern:** Binary Search — **on an unsorted array via a local slope** (no sorted order to exploit; the `nums[-1] = nums[n] = -∞` boundary guarantees an "uphill" direction that always leads to a peak)
**Trigger:** "find **any** peak (element strictly greater than both neighbors) in **O(log n)**, array **not sorted**, `nums[i] != nums[i+1]`". The tell is *log n on an unsorted array* — you can't binary-search a value, so binary-search the **slope**: compare `nums[mid]` to `nums[mid+1]` and always walk toward the higher neighbor. With `-∞` walls on both ends, walking uphill can never fall off without first hitting a peak.

## Problem (1-liner)
0-indexed array with `nums[i] != nums[i+1]` and imaginary `-∞` at both ends → return the index of **any** peak (strictly greater than both neighbors) in **O(log n)**.
**Example:** `nums=[1,2,3,1] → 2` · `nums=[1,2,1,3,5,6,4] → 5` (or `1`) — any valid peak index is accepted.
**Constraints:** `1 ≤ n ≤ 1000`, `−2³¹ ≤ nums[i] ≤ 2³¹−1`, `nums[i] != nums[i+1]`.

## Idea (1-liner)
At any `mid`, if `nums[mid] < nums[mid+1]` a peak is guaranteed somewhere to the **right** (you're on a rising slope walking into a region that must eventually fall — `nums[n] = -∞`); otherwise `mid` itself or something to its **left** is a peak. Half-open binary search on this slope test converges to a peak in **O(log n), O(1) space**.

## Approach (slope-directed binary search)
1. Half-open window `low = 0, high = n - 1`, loop `while (low < high)`, overflow-safe `mid = low + (high - low) / 2`.
2. **Compare only with the right neighbor** — `nums[mid+1]` is always in range because `mid < high` ⇒ `mid + 1 <= high`.
   - `nums[mid] > nums[mid+1]` → falling slope, peak is at `mid` or left → `high = mid` (keep `mid` — it's a live candidate, no `-1`).
   - else (`nums[mid] < nums[mid+1]`) → rising slope, peak strictly right → `low = mid + 1`.
3. The window shrinks every step and `low == high` lands on a peak → `return low`.

A peak is **mathematically guaranteed** (the `-∞` walls make at least one peak unavoidable), so there's no "not found" case — the loop always converges, no post-loop sentinel needed.

## Code (clean canonical)
```cpp
class Solution {
public:
    int findPeakElement(vector<int>& nums) {
        int low = 0, high = nums.size() - 1;
        while (low < high) {                       // half-open [low, high]
            int mid = low + (high - low) / 2;
            if (nums[mid] > nums[mid + 1]) high = mid;   // peak at mid or left
            else low = mid + 1;                          // peak strictly right
        }
        return low;                                // low == high == a peak
    }
};
```
**TC:** O(log n).  **SC:** O(1).

## My Solution (original — correct, with refinements)
The submitted version checks **both** neighbors at each `mid` (an explicit "is this a peak?" test) with `mid == 0` / `mid + 1 == n` boundary guards, plus `n == 1` and `n == 2` special cases, in a closed `low <= high` loop:
```cpp
class Solution {
public:
    int findPeakElement(vector<int>& nums) {
      int n = nums.size();
      if (n == 1) return 0;
      if (n == 2) return nums[0] > nums[1] ? 0 : 1;
      int low = 0, high = nums.size() - 1;
      while (low <= high) {
        int mid = low + (high - low) / 2;
        if ((mid == 0 || nums[mid] > nums[mid - 1]) && (mid + 1 == n || nums[mid] > nums[mid + 1])) return mid;
        else if ((mid == 0 || nums[mid] > nums[mid - 1])) low = mid + 1;
        else high = mid - 1;
      }
      return -1;
    }
};
```
**Algorithmically correct and O(log n)** — it walks toward the larger neighbor, which is the right invariant. Refinements, all cosmetic:
- **Drop the `n == 1` / `n == 2` special cases.** The boundary guards (`mid == 0`, `mid + 1 == n`) already handle size-1 and size-2 arrays inside the loop — those two lines are dead weight.
- **Compare only the right neighbor.** Since `nums[i] != nums[i+1]`, the single test `nums[mid] > nums[mid+1]` is enough to pick a direction; you don't need the left-neighbor check or the boundary guards at all (see canonical). Fewer comparisons, no edge-case clutter.
- **`return -1` is dead code.** A peak is guaranteed, so the loop always returns from inside; the canonical `low < high` form returns `low` and drops it.

Nothing to fix in the logic — the refinements just remove scaffolding.

## Other Approaches
- **Linear scan.** Return the first `i` with `nums[i] > nums[i+1]` (or `n-1`). O(n) — correct but ignores the required O(log n); good only as a sanity check.
- **Check-both-neighbors binary search (the original).** Explicitly test `mid` as a peak with boundary guards. Same O(log n), more comparisons and edge cases — the slope version is the clean form.
- **Compare to the left neighbor instead.** `nums[mid] > nums[mid-1] ? low = mid : high = mid - 1` (mirror image). Equivalent; the right-neighbor convention pairs naturally with the half-open `high = mid` window.

## Pitfalls
- **`<` vs `<=` must match the window.** Half-open here ⇒ `while (low < high)` + `high = mid` (no `-1`). Using `low <= high` with `high = mid` infinite-loops on a size-1 window; using `high = mid - 1` here can skip the peak.
- **Indexing `mid + 1` out of range.** Safe only because `low < high` ⇒ `mid < high` ⇒ `mid + 1 <= high`. If you loosen the loop to `<=`, `mid + 1` can read past the end.
- **Over-engineering with special cases.** `n == 1` / `n == 2` guards and dual-neighbor checks are unnecessary — the single slope test subsumes them.
- **Overflow midpoint.** `low + (high - low) / 2`, never `(low + high) / 2` — free correct habit even though `n ≤ 1000` here.
- **Expecting a "not found" return.** There's always a peak (`-∞` walls); a `return -1` after the loop is unreachable.
