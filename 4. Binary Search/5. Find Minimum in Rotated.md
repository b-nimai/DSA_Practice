# 5. Find Minimum in Rotated Sorted Array (LC 153) — Medium
🔗 [Open on LeetCode](https://leetcode.com/problems/find-minimum-in-rotated-sorted-array/)

**Pattern:** Binary Search — **walk toward the pivot on a rotated array** (a [[4-search-in-rotated-sorted-array]] cousin: no `target`, you binary-search the rotation point itself, which *is* the minimum)
**Trigger:** "ascending **distinct** array **rotated** at an unknown pivot → return the **minimum** in **O(log n)**". The tell is *rotated + wants the min in log n*: the smallest element is exactly the rotation pivot, and at each `mid` one half is sorted — compare `nums[mid]` against the **right end** `nums[high]` to decide which half still hides the pivot.

## Problem (1-liner)
A sorted-ascending array of **unique** values, rotated between `1` and `n` times → return the minimum element, in **O(log n)**.
**Example:** `nums=[3,4,5,1,2] → 1` · `nums=[4,5,6,7,0,1,2] → 0` · `nums=[11,13,15,17] → 11` (rotated `n` times = back to sorted).
**Constraints:** `1 ≤ n ≤ 5000`, `−5000 ≤ nums[i] ≤ 5000`, all values unique.

## Idea (1-liner)
The minimum is the rotation pivot. Binary-search with a half-open window converging on it: compare `nums[mid]` to `nums[high]` — `nums[mid] > nums[high]` means the dip is strictly right of `mid` (`low = mid + 1`), else the pivot is at `mid` or left (`high = mid`). When `low == high` it sits on the minimum. ⇒ **O(log n) time, O(1) space.**

## Approach (converge with `nums[high]`)
1. Half-open window `low = 0`, `high = n - 1`, loop `while (low < high)`; `mid = low + (high - low) / 2`.
2. **`nums[mid] > nums[high]`** → `mid` is in the "high" (pre-pivot) part, so the minimum is strictly to the right ⇒ `low = mid + 1`.
3. **else (`nums[mid] <= nums[high]`)** → `mid` could itself be the minimum (its segment to `high` is sorted) ⇒ keep it: `high = mid`.
4. Loop ends with `low == high` pointing at the pivot ⇒ `return nums[low]`.

**Why compare against `nums[high]`, not `nums[low]`?** For an already-sorted (or fully-rotated) array `nums[mid] >= nums[low]` is *always* true, so it tells you nothing. `nums[high]` is the unambiguous reference — that's why the textbook version uses it. And `high = mid` (no `-1`) is correct because `mid` itself is a live candidate; pairing it with `low < high` keeps the window shrinking without ever discarding the answer.

## Code (clean canonical)
```cpp
class Solution {
public:
    int findMin(vector<int>& nums) {
        int low = 0, high = nums.size() - 1;
        while (low < high) {
            int mid = low + (high - low) / 2;
            if (nums[mid] > nums[high]) low = mid + 1;  // min is strictly right
            else high = mid;                            // min is at mid or left
        }
        return nums[low];   // low == high points at the minimum
    }
};
```
**TC:** O(log n).  **SC:** O(1).

## My Solution (also correct — `nums[low]` compare + explicit `mini`)
```cpp
class Solution {
public:
    int findMin(vector<int>& nums) {
      int low = 0, high = nums.size() - 1, mini = INT_MAX;
      while (low <= high) {
        int mid = low + (high - low) / 2;

        if (nums[low] <= nums[mid]) mini = min(mini, nums[low]), low = mid + 1;
        else mini = min(mini, nums[mid]), high = mid - 1;
      }
      return mini;
    }
};
```
**Idea:** at each step test `nums[low] <= nums[mid]`:
- **true** → `[low..mid]` is sorted, so its smallest is `nums[low]`. Record it, discard the left half (`low = mid + 1`).
- **false** → the pivot lies in `[low..mid]` and `nums[mid]` is in the "small" part, so it's a min candidate. Record it, search left (`high = mid - 1`).

Tracking `mini` explicitly guarantees you never miss the answer — **algorithmically correct**, O(log n)/O(1). One refinement: the original used `mini = 5001`, which hardcodes the `nums[i] <= 5000` constraint; use **`INT_MAX`** so it doesn't silently break if the constraints change. The canonical version above drops `mini` entirely by letting the bounds converge.

## Other Approaches
- **Linear scan O(n).** Trivially `min(nums)` or "first element smaller than its predecessor", but violates the required O(log n).
- **`nums[mid]` vs `nums[low]` (my version).** Also O(log n) but needs the extra `mini` tracker and a closed `low <= high` loop, because comparing to `nums[low]` is uninformative on an already-sorted array — you can't converge by bounds alone, so you record candidates instead.

## Pitfalls
- **Comparing `nums[mid]` to `nums[low]` and trying to converge without a tracker.** On a sorted/fully-rotated array `nums[mid] >= nums[low]` always holds, so a pure bounds-converging loop misroutes. Either compare to `nums[high]` (canonical) or keep an explicit `mini`.
- **`high = mid - 1` in the canonical loop.** Here `high = mid` (no `-1`) because `mid` itself can be the minimum; pairing `high = mid` with `low < high` (half-open) is what keeps it correct and terminating. Using `mid - 1` can skip the answer.
- **Hardcoding the sentinel** (`mini = 5001`). Couples the code to the value constraint; use `INT_MAX`.
- **Overflow midpoint.** `low + (high - low) / 2`, never `(low + high) / 2` — same habit as the rest of the pattern.
- **Assuming distinct.** This relies on uniqueness; with duplicates (LC 154) `nums[mid] == nums[high]` is ambiguous and you must shrink `high--`, degrading to O(n) worst case.
