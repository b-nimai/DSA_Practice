# 1. Binary Search (LC 704) — Easy
🔗 [Open on LeetCode](https://leetcode.com/problems/binary-search/)

**Pattern:** Binary Search — **classic exact-match on a sorted array** (the family seed: every "binary search on answer" problem is this loop with `nums[mid]` replaced by a monotonic `check(mid)`)
**Trigger:** "**sorted** array + find the index of `target` in **O(log n)**" → the only data you need is "is `nums[mid]` too big, too small, or equal?", and sortedness guarantees that answer rules out half the array each step. Closed interval `[low, high]`, loop while `low <= high`.

## Problem (1-liner)
Given a **sorted ascending** `nums` and a `target`, return its index, or `-1` if absent. Values are unique.
**Example:** `nums=[-1,0,3,5,9,12], target=9 → 4` · `target=2 → -1`
**Constraints:** `1 ≤ n ≤ 10⁴`, `−10⁴ < nums[i], target < 10⁴`, all unique, sorted ascending. **Must be O(log n).**

## Idea (1-liner)
Keep a candidate window `[low, high]`. Look at the middle; if it equals `target` you're done, otherwise the sorted order tells you `target` lies strictly in the left or right half, so discard the other half. The window halves each iteration ⇒ **O(log n) time, O(1) space.**

## Approach
1. `low = 0`, `high = n - 1` — a **closed** interval where both ends are still live candidates.
2. Loop **while `low <= high`** (`<=`, not `<`, because a single-element window `low == high` still needs checking).
3. `mid = low + (high - low) / 2` — **overflow-safe midpoint**. `(low + high) / 2` can overflow `int` when both are near `INT_MAX`; the subtraction form can't. Harmless here (`n ≤ 10⁴`) but the habit matters once bounds are values, not indices (see [[2-search-insert-position]] and Koko 875).
4. Compare `nums[mid]` to `target`:
   - `== target` → return `mid`.
   - `> target` → answer is in the **left** half ⇒ `high = mid - 1`.
   - `< target` → answer is in the **right** half ⇒ `low = mid + 1`.
5. Loop exits with `low > high` ⇒ target never matched ⇒ return `-1`.

The `mid ± 1` (never `mid`) is what guarantees the window **strictly shrinks** every step — using `high = mid` here would infinite-loop when `low == high`.

## Code (clean canonical)
```cpp
class Solution {
public:
    int search(vector<int>& nums, int target) {
        int low = 0, high = nums.size() - 1;       // closed interval [low, high]
        while (low <= high) {
            int mid = low + (high - low) / 2;      // overflow-safe midpoint
            if (nums[mid] == target) return mid;
            else if (nums[mid] > target) high = mid - 1;   // discard right half
            else low = mid + 1;                            // discard left half
        }
        return -1;                                 // window emptied, not found
    }
};
```
**TC:** O(log n).  **SC:** O(1).

## My Solution (already canonical)
```cpp
class Solution {
public:
    int search(vector<int>& nums, int target) {
      int low = 0, high = nums.size() - 1;
      while (low <= high) {
        int mid = low + (high - low) / 2;
        int num = nums[mid];
        if (num == target) return mid;
        else if (num > target) high = mid - 1;
        else low = mid + 1;
      }
      return -1;
    }
};
```
Correct and idiomatic — overflow-safe `mid`, closed interval, `mid ± 1` shrink. The only stylistic note: caching `int num = nums[mid]` is fine (one read), the canonical version just inlines it.

## Family map — the two binary-search templates
| Template | Loop | Returns | Use when |
|---|---|---|---|
| **Exact match** (this one, LC 704) | `low <= high`, `mid ± 1` both sides | `mid` on hit, `-1` after loop | target must *exist*; you want its exact index |
| **Lower-bound / first-true** ([[2-search-insert-position]], 35) | `low < high`, `high = mid` (no `-1`) | `low` after loop (insertion point) | "smallest index where `nums[i] >= target`" / "minimum X such that `check(X)`" |

704 is the exact-match seed; 35 is the lower-bound seed. Everything else in this pattern (33 rotated, 153 min-in-rotated, 875 Koko, 410 split array) is one of these two loops with the comparison swapped for a **monotonic predicate**.

## Other Approaches
- **Linear scan O(n)** — trivially correct but violates the required O(log n) and ignores the sorted property.
- **`std::lower_bound` / `binary_search`** — STL gives `bool binary_search(...)` (presence only) or `lower_bound` (iterator to first `>= target`); fine in production, but interviews want the hand-rolled loop to prove you own the invariant.

## Pitfalls
- **`<` vs `<=` in the loop.** With a **closed** interval `[low, high]`, the terminating window is `low == high` (one live element) — it must still be checked, so the condition is `low <= high`. Using `<` skips that last element and can miss the target.
- **Forgetting `mid ± 1`.** The discarded half must *exclude* `mid` (you already tested it). Writing `high = mid` / `low = mid` keeps `mid` in range and **infinite-loops** when the window is size 1 or 2.
- **`(low + high) / 2` overflow.** Sum can exceed `INT_MAX`; always `low + (high - low) / 2`. Not triggered at `n ≤ 10⁴`, but it's a free correct habit and mandatory once you binary-search over a value range.
- **`nums.size()` is unsigned.** `high = nums.size() - 1` is fine here (`n ≥ 1`), but on an empty vector `size() - 1` wraps to a huge `size_t`; storing into `int high` is what saves you. Guard empty input if the constraints ever allow `n == 0`.
