# 8. Delete the Middle Node of a Linked List (LC 2095) — Medium
🔗 [Open on LeetCode](https://leetcode.com/problems/delete-the-middle-node-of-a-linked-list/)

**Pattern:** Fast & Slow Pointers — find-middle, but stop on the **predecessor** so you can unlink
**Trigger:** "delete the middle node (`⌊n/2⌋`, 0-indexed), return the head" → slow/fast to reach the middle, but keep a handle on the node *before* it so you can splice it out

## Problem (1-liner)
Given the head of a singly linked list, delete the middle node (the `⌊n/2⌋`-th node, 0-based) and return the modified head.
**Examples:** `[1,3,4,7,1,2,6]` → `[1,3,4,1,2,6]` (delete index 3) · `[1,2,3,4]` → `[1,2,4]` (delete index 2) · `[2,1]` → `[2]` (delete index 1)

## Idea (1-liner)
Standard middle-finding (`slow+=1`, `fast+=2`) lands `slow` on the middle — but to *delete* a singly-linked node you need its **predecessor**, so either track `prev` alongside `slow`, or pre-offset `fast` by two so `slow` halts one node early.

## Approach
1. **Single-node edge case:** `if (!head->next) return nullptr` — a 1-node list has its only node as the middle, so deletion leaves an empty list. (Constraints guarantee `head != nullptr`.)
2. **Walk** slow/fast from `head` keeping `prev` one step behind `slow`. When `fast` falls off (`!fast || !fast->next`), `slow` is on the middle and `prev` is its predecessor.
3. **Unlink:** `prev->next = slow->next`, then `delete slow` to free it.
4. **Return** `head`.

**Why the offset trick works:** starting `fast = head->next->next` advances it two nodes ahead, so the loop ends one iteration sooner — `slow` stops on the predecessor of the middle directly, removing the need for a separate `prev`.

## Code (clean canonical)
```cpp
class Solution {
public:
    ListNode* deleteMiddle(ListNode* head) {
        if (!head->next) return nullptr;          // single node → empty
        ListNode *slow = head, *fast = head->next->next;
        while (fast && fast->next) {              // slow stops on predecessor
            slow = slow->next;
            fast = fast->next->next;
        }
        ListNode* mid = slow->next;
        slow->next = mid->next;
        delete mid;
        return head;
    }
};
```
**TC:** O(n) (one pass)  **SC:** O(1)

## My Solution
```cpp
class Solution {
public:
    ListNode* deleteMiddle(ListNode* head) {
      if (!head->next) return nullptr;
      ListNode *slow = head, *fast = head, *prev = nullptr;
      while(fast && fast->next) {
        prev = slow;
        slow = slow->next;
        fast = fast->next->next;
      }
      prev->next = slow->next;
      delete(slow);
      return head;
    }
};
```

> **Verdict:** correct and optimal — O(n) time, O(1) space. The `prev`-tracking variant lands `slow` on the middle and `prev` on its predecessor; the single-node guard handles the only degenerate case (constraints guarantee `head != nullptr`, so `head->next` is safe). Traced clean on n = 1..5 and all three examples.
>
> **Refinements (style only):**
> - **Drop `prev` with an offset:** start `fast = head->next->next` and `slow` naturally stops on the predecessor — one fewer pointer to carry (see canonical above). Purely stylistic; the `prev` version is arguably more obvious.
> - **`delete slow;` not `delete(slow);`** — `delete` is an operator, not a function; the parens read like a call.

## Other Approaches
- **Two-pass (count then re-walk):** first pass counts `n`, second pass walks `⌊n/2⌋ - 1` steps to the predecessor and unlinks. Simple and foolproof, but 2× the traversal — the one-pass slow/fast is preferred.
- **Dummy node:** `dummy->next = head` lets the offset version handle even the single-node case without the early `return nullptr`, at the cost of one allocation/setup line.

## Pitfalls
- **Deleting needs the predecessor, not the node itself** — the core twist vs. plain "find middle" (LC 876). Forgetting `prev` (or the `fast` offset) leaves you unable to splice.
- **Single-node list:** the loop body never runs, so without the `if (!head->next)` guard `prev` stays `nullptr` and `prev->next` crashes. Handle it up front (or use a dummy).
- **Parity / which middle:** for `⌊n/2⌋` 0-indexed you want the *second* middle on even length — the standard `fast = head` start (or `head->next->next` offset) gives exactly that; don't start `fast = head->next` here (that targets the first middle).
- **Memory:** `delete` the unlinked node — LeetCode won't fail you for skipping it, but it's the correct habit and what an interviewer looks for.
- **Same backbone as [Middle of the Linked List](https://leetcode.com/problems/middle-of-the-linked-list/) (LC 876)** — identical traversal, just stop one node earlier to enable the unlink.
