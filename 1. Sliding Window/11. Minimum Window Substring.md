# 11. Minimum Window Substring (LC 76) — Hard
🔗 [Open on LeetCode](https://leetcode.com/problems/minimum-window-substring/)

**Pattern:** Sliding Window — **shortest** valid window, multiset cover via `need`/`window` + `formed` counter
**Trigger:** "**smallest/shortest substring of `s` that contains all of `t` (with duplicates)**" → variable window, grow `right` until valid, then **shrink `left` while still valid**, recording the min. Validity must be O(1) — track it with a `formed == required` counter, never by re-scanning a map.

## Problem (1-liner)
Given `s` and `t`, return the **minimum-length substring of `s`** that contains every character of `t` **including duplicates**, or `""` if none exists.
**Example:** `s = "ADOBECODEBANC", t = "ABC"` → `"BANC"`. `s="a", t="aa"` → `""`.

## Idea (1-liner)
This is the **shortest-window** variant (like [2. Minimum Size Subarray Sum](2.%20Minimum%20Size%20Subarray%20Sum.md)) but the validity test is "window contains `t` as a **multiset**" instead of a sum. The brute force re-checks coverage from scratch for every start index → O(m²). The fix: maintain the window's coverage **incrementally** and answer "is it valid?" in O(1) via a single `formed` counter — bump it only when a char's count *exactly hits* its required count.

## Approach
1. `need` = freq map of `t` (fixed); `required` = `need.size()` (distinct chars to satisfy).
2. Grow `right`: add `s[right]` to `window`; if its count just reached `need[c]`, `formed++`.
3. While `formed == required` (valid): record `[left, right]` if shorter, then shrink `left` — if removing `s[left]` would drop its count *below* `need`, `formed--` first.
4. Answer = best window seen, or `""` if none.

## Code (clean canonical — `need`/`window` + `formed`, O(m + n))
```cpp
class Solution {
public:
    string minWindow(string s, string t) {
        if (t.size() > s.size()) return "";

        unordered_map<char, int> need, window;
        for (char c : t) need[c]++;            // what the window must cover (with dups)

        int required = need.size();            // # of DISTINCT chars to satisfy
        int formed   = 0;                      // # of distinct chars currently satisfied
        int left = 0, bestLen = INT_MAX, bestStart = 0;

        for (int right = 0; right < s.size(); right++) {
            char c = s[right];
            window[c]++;                       // extend window to the right
            // only when count EXACTLY reaches need do we satisfy one more char
            if (need.count(c) && window[c] == need[c]) formed++;

            // window is valid → shrink from the left to minimise it
            while (formed == required) {
                if (right - left + 1 < bestLen) {     // record the smaller window
                    bestLen   = right - left + 1;
                    bestStart = left;
                }
                char lc = s[left];
                // about to remove lc: if it was exactly enough, we break validity
                if (need.count(lc) && window[lc] == need[lc]) formed--;
                window[lc]--;
                left++;
            }
        }
        return bestLen == INT_MAX ? "" : s.substr(bestStart, bestLen);
    }
};
```
**TC:** O(m + n) — each of `left`/`right` advances ≤ m; `need` built in O(n). **SC:** O(n + Σ) — the two maps (Σ = alphabet ≤ 52).

## My Solution

**Brute force (accepted, but O(m²) — for understanding only):**
```cpp
class Solution {
public:
    string minWindow(string s, string t) {
        if (t.size() > s.size()) return "";
        unordered_map<char, int> freq;
        for (int i = 0; i < t.size(); i++) freq[t[i]]++;
        int mini = INT_MAX / 2;
        string res = "";
        for (int i = 0; i < s.size(); i++) {
            unordered_map<char, int> freq1 = freq;   // re-checks coverage from scratch
            for (int j = i; j < s.size(); j++) {
                if (freq1.count(s[j])) {
                    freq1[s[j]]--;
                    if (freq1[s[j]] == 0) freq1.erase(s[j]);
                    if (freq1.empty() && mini > j - i + 1) {
                        res = s.substr(i, j - i + 1);
                        mini = j - i + 1;
                    }
                }
            }
        }
        return res;
    }
};
```
✅ Correct but **O(m²)** worst case (`m, n ≤ 10⁵` → up to 10¹⁰ ops; TLE-risk on a stricter judge). The wasted work is `freq1 = freq` + restarting the inner scan for *every* `i`.

**Sliding window (accepted, optimal — your canonical answer):**
```cpp
class Solution {
public:
    string minWindow(string s, string t) {
      if(t.size() > s.size()) return "";
      unordered_map<char, int>need, window;
      for(char c : t) need[c]++;

      int required = need.size(); // distinct chars to satisfy
      int left = 0, formed = 0, bestLen = INT_MAX, bestStart = 0;

      for(int right = 0; right < s.size(); right++) {
        char c = s[right];
        window[c]++;
        if(need.count(c) && need[c] == window[c]) formed++;
        while(formed == required) {
          if(right - left + 1 < bestLen) {
            bestLen = right - left + 1;
            bestStart = left;
          }
          char lc = s[left];
          if(need.count(lc) && need[lc] == window[lc]) formed--;
          window[lc]--;
          left++;
        }
      }
      return bestLen == INT_MAX ? "" : s.substr(bestStart, bestLen);
    }
};
```
✅ Correct, accepted, **O(m + n)** — this is the expected interview answer. Clean structure: grow-right / shrink-while-valid, with `formed` making the validity check O(1).

## Key Insight — why a `formed` counter, not `map.empty()`
The brute force's bottleneck is the validity test: copying `freq` and scanning until empty is O(distinct) **and repeated per start index**. The sliding window keeps **one** `window` map alive across the whole scan and collapses "is it valid?" into a single integer compare `formed == required`:

| | Brute force | Sliding window |
|---|---|---|
| `freq` handling | `freq1 = freq` reset every `i` | `need` fixed; `window` maintained, never reset |
| Validity check | `freq1.empty()` (O(distinct)) | `formed == required` (O(1)) |
| Start index | inner loop restarts at each `i` | `left` only moves forward, never restarts |
| Total | O(m²) | O(m + n) |

The subtle part is the `==` test, **not** `>=`: `formed` ticks up only at the *exact moment* a char's window count reaches its `need` count (extra copies beyond that don't re-bump it), and ticks down only when removal would take it *below* `need`. That keeps `formed` an exact count of "fully satisfied distinct chars".

## Other Approaches
- **Array `int[128]` instead of `unordered_map`** — chars are ASCII letters; two `int[128]` arrays replace both maps for a large constant-factor speedup (no hashing), same O(m + n).
- **`missing` integer instead of `need`/`window`** — track total *remaining* chars to cover: on add `if (need[c]-- > 0) missing--`, on remove `if (need[c]++ >= 0) missing++`; window valid when `missing == 0`. One map, fewer branches; functionally identical.
- **Brute O(m²)** — only to reason about correctness; not acceptable at `m = 10⁵`.

## Refinements still applicable
- **`unordered_map` → `int[128]`/`int[58]`** — drops hash overhead; `s`, `t` are `[A-Za-z]`.
- **Fold the lookup** — `need.count(c)` then `need[c]`/`window[c]` touches the same key 2–3×; with arrays it's a single index.
- **Signed/unsigned** — `right < s.size()` compares `int` with `size_t`; cache `int m = s.size()`.
- **`need.count(lc)` is redundant on the shrink** if you only ever incremented `window` for chars (cheap, but the `int[128]` form removes the question entirely).

## Pitfalls
- **`==` not `>=` when updating `formed`.** Using `>=` would bump `formed` on every extra copy of a char and overcount → window declared valid too early. Increment only when `window[c] == need[c]`.
- **Decrement `formed` *before* shrinking past the requirement.** Check `window[lc] == need[lc]` *before* `window[lc]--`; doing it after is off-by-one and keeps a stale-valid window.
- **Shrink with `while`, not `if`** — keep contracting as long as the window stays valid to find the true minimum (this is the "shortest" variant — record *while still valid*, then shrink; mirror of [2. Minimum Size Subarray Sum](2.%20Minimum%20Size%20Subarray%20Sum.md)).
- **Impossible-answer sentinel** — `bestLen = INT_MAX`; map back to `""`. Don't initialise to `0` (that's the *longest*-variant sentinel — different problem family).
- **Duplicates in `t` matter** — `t = "aa"` needs *two* `a`s; that's why `need`/`window` store **counts**, not a set. (`required` counts distinct chars; `need[c]` enforces the multiplicity.)
- **`t` longer than `s`** — early-return `""`; the loop would otherwise never reach `formed == required`.
