# 10. Subarrays with K Different Integers (LC 992) — Hard
🔗 [Open on LeetCode](https://leetcode.com/problems/subarrays-with-k-different-integers/)

**Pattern:** Sliding Window — **count subarrays**, "**exactly K**" via `atMost(K) − atMost(K−1)`
**Trigger:** "**count** subarrays with **exactly K** distinct" → there's no clean single window for *exactly* K, but there is for *at most* K. **`exactly K = atMost(K) − atMost(K−1)`**. Whenever you see "exactly K" + "count", reach for this decomposition.

## Problem (1-liner)
Given `nums` and `k`, return the **number of subarrays** whose count of distinct integers is **exactly `k`**.
**Example:** `nums=[1,2,1,2,3], k=2` → `7`. `nums=[1,2,1,3,4], k=3` → `3`.

## Idea (1-liner)
"Exactly K" has no monotone window (a window can have <K, =K, or >K distinct and growing/shrinking doesn't move that monotonically toward a single target). But **"at most K"** does: every subarray ending at `right` with ≤ K distinct is valid, so each step contributes `right − left + 1` subarrays. Then `exactly K = atMost(K) − atMost(K−1)` — every "≤ K" subarray that isn't "≤ K−1" has exactly K. Same count-window engine as the budget problems [4](4.%20Longest%20Repeating%20Character%20Replacement.md)/[7](7.%20Max%20Consecutive%20Ones%20III.md)/[8](8.%20Fruit%20Into%20Baskets.md), wrapped in a subtraction.

## Approach
1. `atMostK(nums, k)`: variable window, `freq` of values in window, running distinct budget.
2. For each `right`: add `nums[right]`; if it was new, spend a unit of budget.
3. While budget overspent (`> k` distinct): shrink `left`, refunding budget when a value's count hits 0.
4. Add `right − left + 1` to the count — all subarrays ending at `right` are now ≤ K distinct.
5. Answer = `atMostK(k) − atMostK(k−1)`.

## Code (clean canonical — array freq via the value bound, O(n)/O(n))
```cpp
class Solution {
    int atMostK(vector<int>& nums, int k) {
        int n = nums.size(), left = 0, count = 0;
        vector<int> freq(n + 1, 0);                 // 1 <= nums[i] <= n → array, no hashing
        for (int right = 0; right < n; right++) {
            if (freq[nums[right]]++ == 0) k--;      // new distinct → spend budget
            while (k < 0) {                         // overspent → shrink
                if (--freq[nums[left]] == 0) k++;   // value fully gone → refund
                left++;
            }
            count += right - left + 1;              // all subarrays ending at right are valid
        }
        return count;
    }
public:
    int subarraysWithKDistinct(vector<int>& nums, int k) {
        return atMostK(nums, k) - atMostK(nums, k - 1);
    }
};
```
**TC:** O(n) — two passes, each `left`/`right` moves ≤ n. **SC:** O(n) — the `freq` array.

## My Solution
```cpp
class Solution {
  int atMostK(vector<int>& nums, int k) {
    int left = 0, count = 0;
    unordered_map<int, int>freq;
    for(int right = 0; right < nums.size(); right++) {
      // If new element then decrease k
      if(freq[nums[right]] == 0) k--;
      // increase frequency
      freq[nums[right]]++;
      // if distrinct element > k
      while(k < 0) {
        freq[nums[left]]--;
        if(freq[nums[left]] == 0) k++;
        left++;
      }
      count += (right - left + 1);
    }
    return count;
  }
public:
    int subarraysWithKDistinct(vector<int>& nums, int k) {
      return atMostK(nums, k) - atMostK(nums, k - 1);
    }
};
```
✅ Correct and accepted. This is the **canonical interview answer** — the `atMost(K) − atMost(K−1)` decomposition is exactly what's expected here. O(n) / O(n), optimal complexity.

**Notable strength:** you use `k` itself as the running distinct budget and **never `erase`** from the map — stale keys sit at count 0, validity tracked via `k` rather than `freq.size()`. That deliberately sidesteps the classic "forgot to erase on zero → `size()` overcounts" bug seen in [8. Fruit Into Baskets](8.%20Fruit%20Into%20Baskets.md). Good design choice, worth keeping.

## Key Insight — why `exactly K = atMost(K) − atMost(K−1)`
For a fixed right endpoint, the set of valid left starts for "≤ K distinct" is a contiguous suffix — so it's *countable* in O(1) per step (`right − left + 1`). "Exactly K" is **not** a monotone window predicate, so it has no such direct count. But the subarrays with "exactly K distinct" are precisely those that are "≤ K" **and not** "≤ K−1". Subtracting the two at-most counts isolates that band. This is a **general counting trick**: any "count things with exactly K of a property" where "at most K" is a monotone window → `f(K) − f(K−1)`.

| | "at most K" | "exactly K" |
|---|---|---|
| Monotone window? | yes (suffix of valid lefts) | no |
| Direct count per `right`? | `right − left + 1` | not directly |
| This problem | `atMostK(k)` | `atMostK(k) − atMostK(k−1)` |

## Other Approaches
- **`unordered_map` freq (your version):** general — works for any value range. Use this if `nums[i]` weren't bounded. Here `1 ≤ nums[i] ≤ n` lets a plain `vector<int>` replace it for a large constant-factor win (no hashing).
- **Single-pass two-window merge:** maintain *two* left pointers (`≤K` and `≤K−1`) in one loop; answer accumulates `leftBig − leftSmall` per step. Same O(n), ~half the constant — but trickier to write bug-free. State the subtraction version first in an interview (it shows you understand *why*), mention the fuse only as an optimization.
- **Prefix-of-distinct / brute O(n²):** track distinct per start — too slow for `n = 2·10⁴` in the worst case; only a fallback to reason about correctness.

## Refinements still applicable
- **Array instead of `unordered_map`** — `1 ≤ nums[i] ≤ nums.length` ⇒ `vector<int> freq(n + 1, 0)`; drops hash overhead, same asymptotics, big practical speedup.
- **Fold lookups into the condition** — `if (freq[nums[right]]++ == 0) k--;` and `if (--freq[nums[left]] == 0) k++;` do one map/array touch instead of two on the same key.
- **Signed/unsigned** — `right < nums.size()` compares `int` with `size_t`; cache `int n = nums.size()`.
- **Overflow check** — max count ≈ `n²/2 ≈ 2·10⁸` for `n = 2·10⁴`, fits signed 32-bit `int` and LeetCode guarantees the answer fits, so `int` is fine *here*; on a larger-`n` variant use `long long`.

## Pitfalls
- **"exactly K" + "count" ⇒ subtraction trick.** Don't try to build a single window that holds "exactly K" — it isn't a monotone predicate and the standard window breaks.
- **`atMost(k−1)` with `k = 1` calls `atMostK(…, 0)`** — must correctly return 0 (no subarray has ≤ 0 distinct). The budget logic handles this: first element spends to `k = −1`, the `while` immediately evicts it, `count += 0`. Don't special-case it incorrectly.
- **Erase-vs-budget:** if you switch to `freq.size()` as the distinct count you are *forced* to `erase` keys on zero or it overcounts. The `k`-budget form avoids this — don't mix the two.
- **Count window, not length window** — accumulate `right − left + 1` (number of subarrays), not `max`/`min` of widths. Different goal from the longest/shortest variants (#2–#9).
- Same count-window engine as #4 / #7 / #8 (budget shrink), but the **answer is a sum of counts**, wrapped in `f(K) − f(K−1)`. Recognise the wrapper, not just the inner window.
