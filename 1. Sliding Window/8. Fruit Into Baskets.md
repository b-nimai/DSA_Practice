# 8. Fruit Into Baskets (LC 904) — Medium
🔗 [Open on LeetCode](https://leetcode.com/problems/fruit-into-baskets/)

**Pattern:** Sliding Window — variable size, longest valid window (at most **2 distinct** values)
**Trigger:** "pick fruits left→right, only **2 baskets**, one fruit type each, stop at a 3rd type" → strip the story: **longest subarray containing at most 2 distinct values**. Grow `right`, shrink/slide `left` while `#distinct > 2`.

## Problem (1-liner)
Array `fruits` (`fruits[i]` = fruit type at tree `i`); collect from a contiguous run of trees using **2 baskets**, each holding **one type only** → return the **max number of trees** (= longest subarray with ≤ 2 distinct values).
**Example:** `fruits=[1,2,1]` → `3`. `fruits=[0,1,2,2]` → `3` (trees `[1,2,2]`). `fruits=[1,2,3,2,2]` → `4` (trees `[2,3,2,2]`).

## Idea (1-liner)
"2 baskets, one type each" is a disguised **"≤ 2 distinct"** constraint — the generic *"longest substring with at most K distinct"* with `K = 2`. Track type→count in a map; grow `right`; when the map holds > 2 keys, shrink `left` (decrement, erase on zero) until back to ≤ 2; record the widest window. Same family as [3. Longest Substring Without Repeating Characters](3.%20Longest%20Substring.md) — the budget here is **distinct keys**, not duplicate count.

## Approach
1. `left = 0`, `best = 0`, `unordered_map<int,int> cnt`.
2. For each `right`: `cnt[fruits[right]]++`.
3. While `cnt.size() > 2`: `if (--cnt[fruits[left]] == 0) cnt.erase(fruits[left]); left++;`.
4. `best = max(best, right - left + 1)`.
5. Return `best`.

## Code (clean canonical — non-shrinking window, O(n)/O(1))
```cpp
int totalFruit(vector<int>& fruits) {
    int left = 0;
    unordered_map<int, int> cnt;
    for (int right = 0; right < (int)fruits.size(); right++) {
        cnt[fruits[right]]++;
        if (cnt.size() > 2) {                       // window invalid: slide forward by one
            if (--cnt[fruits[left]] == 0) cnt.erase(fruits[left]);
            left++;
        }
    }
    return (int)fruits.size() - left;               // width is monotone → final width is the answer
}
```
**TC:** O(n)  **SC:** O(1) — map never exceeds 3 entries.

## My Solution
**Version 1 — `while`-shrink + `maxi` (the standard variable-window form):**
```cpp
class Solution {
public:
    int totalFruit(vector<int>& fruits) {
        int maxi = 0, size = fruits.size();
        int left = 0;
        unordered_map<int, int> mpp;

        for (int right = 0; right < size; right++) {
            mpp[fruits[right]]++;
            while (mpp.size() > 2) {
                mpp[fruits[left]]--;
                if (mpp[fruits[left]] == 0) {
                    mpp.erase(fruits[left]);
                }
                left++;
            }
            maxi = max(maxi, right - left + 1);
        }
        return maxi;
    }
};
```
✅ Correct and accepted. O(n) / O(1) — already optimal in complexity. `mpp.size()` (distinct types in window) is exactly the right state; `while`-shrink restores validity, `maxi` tracks the widest valid window. Clearest version to *state first* in an interview — the `unordered_map` makes it read as the general "≤ K distinct" template (here `K = 2`).

**Version 2 — `if`-slide, non-shrinking window:**
```cpp
class Solution {
public:
    int totalFruit(vector<int>& fruits) {
        int left = 0;
        unordered_map<int, int> mpp;
        for (int right = 0; right < fruits.size(); right++) {
            mpp[fruits[right]]++;
            if (mpp.size() > 2) {
                if (--mpp[fruits[left]] == 0)
                    mpp.erase(fruits[left]);
                left++;
            }
        }
        return fruits.size() - left;
    }
};
```
✅ Correct and accepted. The non-shrinking refinement: `while`→`if`, drop `maxi`, return `fruits.size() - left`. Same O(n) / O(1), tighter form — **same trick as #4 and #7**.

## Key Insight — why `while`→`if` (non-shrinking window) works
Once the window is the widest valid one seen, it never needs to **shrink** — only to keep from growing past that size. So when a 3rd type appears (`cnt.size() > 2`), advance **both** ends by one (`left++` now, `right` next iteration): the window slides forward without losing width.

Because the window width is then **monotonically non-decreasing**, `fruits.size() - left` at the end *is* the answer — no `maxi` needed. **Exact same trick as [4. Longest Repeating Character Replacement](4.%20Longest%20Repeating%20Character%20Replacement.md) and [7. Max Consecutive Ones III](7.%20Max%20Consecutive%20Ones%20III.md)** — `if`, not `while`. Recognising the shared template across #4 / #7 / #8 is the point — solve one, you have all three.

| | V1 `while`-shrink | V2 `if`-slide |
|---|---|---|
| On invalid (`> 2` types) | contract `left` until ≤ 2 again | move `left` once; window never shrinks |
| Answer | track `maxi` each step | `n - left` at the end |
| Complexity | O(n) / O(1) | O(n) / O(1) |

## Dry run — `fruits=[1,2,3,2,2]`, V2 `if`-slide
| right | fruits[right] | map after add | size > 2? | left | width = right-left+1 |
|------|------|------|------|------|------|
| 0 | 1 | {1:1} | no | 0 | 1 |
| 1 | 2 | {1:1, 2:1} | no | 0 | 2 |
| 2 | 3 | {1:1, 2:1, 3:1} | **yes** → drop fruits[0]=1, erase 1 | 1 | 2 |
| 3 | 2 | {2:2, 3:1} | no | 1 | 3 |
| 4 | 2 | {2:3, 3:1} | no | 1 | **4** |

`n - left = 5 - 1 = 4` ✅. `left` only advances when a 3rd type forces it, so width never drops below the best seen.

## Other Approaches
- **V1 `while`-shrink + `maxi`:** the standard variable-window form; clearest to explain, present first.
- **Generalize to "≤ K distinct":** replace the literal `2` with a parameter `k` — identical code solves LC 340 *Longest Substring with At Most K Distinct Characters*. Worth stating aloud as the general form.
- **Last-seen-index (2 vars, no map):** since `K = 2` is tiny, track the two active types and the last index of the second type — O(1) space without a map. Faster constant factor but **fragile and 2-only**; mention only as a micro-optimization, don't lead with it.

## Refinements still applicable
- **Signed/unsigned:** `right < fruits.size()` compares `int` with `size_t`. V1 already caches `int size = fruits.size();` — good; in V2 cast `(int)fruits.size()` or cache it too.
- **Combine decrement + check:** prefer `if (--mpp[fruits[left]] == 0) mpp.erase(fruits[left]);` (V2 style) over V1's decrement-then-reread — avoids a second map lookup on the same key.

## Pitfalls
- **Decode the story first** — "2 baskets, one type each, stop at a 3rd" *is* "≤ 2 distinct". Don't model baskets/stopping literally.
- **Erase on zero** — if you decrement but don't `erase` the key, `cnt.size()` overcounts and the window never reopens. This is the #1 bug here.
- **`size()` is the validity signal**, not a fixed count of *anything* — it's the number of *distinct keys currently in the window*.
- Variable, not fixed window — width is whatever ≤ 2 distinct allows; don't slide a constant-width window.
- "At most 2" → `size > 2` is invalid (strictly greater); a window of exactly 2 types is fine.
- In V2 the window **never shrinks** — that's why `n - left` works; don't also add `maxi` and shrink, you'd conflate the two templates.
- Same engine as #3 / #4 / #7 (count/budget windows) — different from the fixed-window anagram problems (5, 6); don't mix templates.
