# 6. Find All Anagrams in a String (LC 438) — Medium
🔗 [Open on LeetCode](https://leetcode.com/problems/find-all-anagrams-in-a-string/)

**Pattern:** Sliding Window — fixed size (`width = p.length`) + frequency match
**Trigger:** "return **all** start indices of `p`'s anagrams in `s`" → fixed-width window, compare letter frequencies, collect every match

## Problem (1-liner)
Given `s` and `p` (lowercase), return **all start indices** of substrings of `s` that are anagrams of `p`.
**Example:** `s = "cbaebabacd", p = "abc"` → `[0,6]`. `s = "abab", p = "ab"` → `[0,1,2]`.

## Idea (1-liner)
Anagram = same multiset of letters. Slide a window of **exactly `p.length`** over `s`; every window whose letter-frequency matches `p`'s contributes its left index to the answer. Same engine as [5. Permutation in String](5.%20Permutation%20in%20String.md) — only difference: **collect all matches** instead of returning on the first.

## Approach
1. If `p.size() > s.size()` → `{}` (no window can fit).
2. Build `need[26]` from `p`, and `window[26]` from the **first `p.size()` chars** of `s`.
3. If they match → push index `0`.
4. Slide: for `right = p.size() .. s.size()-1`, add `s[right]`, drop `s[left]`; if match, push `left+1`.
5. Return `result`.

## Code (clean canonical — `found` counter, O(1)/step)
```cpp
vector<int> findAnagrams(const string& s, const string& p) {
    if (p.size() > s.size()) return {};
    vector<int> result;

    int need[26] = {0};
    for (char c : p) need[c - 'a']++;

    int required = p.size(), left = 0, found = 0;
    for (int right = 0; right < (int)s.size(); right++) {
        if (need[s[right] - 'a']-- > 0) found++;          // consumed a still-needed char
        if (right - left + 1 > required) {
            if (need[s[left] - 'a']++ >= 0) found--;       // released a char that was counted
            left++;
        }
        if (found == required) result.push_back(left);
    }
    return result;
}
```
**TC:** O(n)  **SC:** O(1) — single fixed 26-slot array, output not counted

## My Solution
**Version 1 — two frequency arrays + `==`, build-then-slide (O(26·n)):**
```cpp
class Solution {
public:
    vector<int> findAnagrams(string s, string p) {
      if(p.size() > s.size()) return {};
      vector<int>result;
      vector<int>freqP(26, 0), freqS(26, 0);

      for(int i = 0; i < p.size(); i++) {
        freqP[p[i] - 'a']++;
        freqS[s[i] - 'a']++;
      }

      int left = 0;
      if(freqP == freqS) result.push_back(left);

      for(int right = p.size(); right < s.size(); right++) {
        freqS[s[right] - 'a']++;
        freqS[s[left] - 'a']--;
        left++;
        if(freqS == freqP) result.push_back(left);
      }
      return result;
    }
};
```
✅ Correct and accepted. Clearest framing to state first in an interview: build the first window, compare, slide one in / one out. Cost is the `freqS == freqP` 26-element rescan every step → O(26·n) ≈ O(n).

**Version 2 — matched-bucket counter, O(1) per slide:**
```cpp
class Solution {
public:
    vector<int> findAnagrams(string s, string p) {
      if(p.size() > s.size()) return {};
      vector<int>result;
      vector<int>need(26, 0), window(26, 0);

      for(int i = 0; i < p.size(); i++) {
        need[p[i] - 'a']++;
        window[s[i] - 'a']++;
      }

      int match = 0;
      for(int i = 0; i < 26; i++) {
        if(need[i] == window[i]) match++;
      }

      int left = 0;
      if(match == 26) result.push_back(left);

      for(int right = p.size(); right < s.size(); right++) {
        int in = s[right] - 'a';
        int out = s[left] - 'a';

        // Add incoming char
        if(need[in] == window[in]) match--;
        if(need[in] == ++window[in]) match++;

        // Remove outgoing char
        if(need[out] == window[out]) match--;
        if(need[out] == --window[out]) match++;

        left++;
        if(match == 26) result.push_back(left);
      }
      return result;
    }
};
```
✅ Correct and accepted. Removes the O(26) rescan: `match` = how many of the 26 buckets currently equal their target. A bucket can only change equality on the **one slot** that the in/out char touches, so each step is O(1) → true O(n).

## Key Insight — the `match`-counter update is "check-before, check-after"
For every bucket you mutate, equality with `need` can flip at most once. So around each `++`/`--`:
- **Before** the change, if `need == window` for that slot, it *was* matched → on the verge of breaking, so `match--`.
- **After** the change, if `need == window` now, it *became* matched → `match++`.

`++window[in]` / `--window[in]` are **pre**-increment so the post-state is compared. Net effect per char: `match` moves by −1, 0, or +1. Window is a valid anagram iff `match == 26`.

> Two equivalent O(1) refinements exist: this **matched-bucket counter** (count of 26 slots equal — Version 2) and the **single `found` progress counter** (canonical code; see [5. Permutation in String](5.%20Permutation%20in%20String.md) for why its add uses `> 0` but remove uses `>= 0`). Either is fine in an interview; know one cold.

## Key Insight — same engine as Permutation in String
LC 438 and LC 567 are the **same fixed-window-anagram algorithm**. The only delta:

| | LC 567 (Permutation in String) | LC 438 (Find All Anagrams) |
|---|---|---|
| On a match | `return true` immediately | `result.push_back(left)` and **keep going** |
| No match ever | `return false` | `return result` (possibly empty) |

Solve one → solve the other by changing the match action.

## Dry run — `s="cbaebabacd"`, `p="abc"`, Version 2, `match` of 26
Initial window = `"cba"` (indices 0..2) → `window` matches `need` on all 26 → `match=26`, push `0`.
Then each step swaps one char out (`left`) and one in (`right`):

| right | in | out | after add | after remove | match | left | push? |
|------|----|-----|-----------|--------------|-------|------|-------|
| 3 | e | c | c-slot/e-slot break | — | 24 | 1 | |
| 4 | b | b | … | … | 24 | 2 | |
| 5 | a | a | … | … | 24 | 3 | |
| 6 | b | e | e-slot heals | b rebalances | 26 → window `"bac"` | 4 | **4? → 6** |
| … | | | | | | | |

Window `"bac"` occupies indices 6..8 and is the second anagram; `match` returns to 26 there and `left` (after `left++`) equals the start index `6` → push `6`. Final `result = [0, 6]` ✅. (`match == 26` is evaluated **after** `left++`, so the pushed value is the window's true start.)

## Other Approaches
- **Sort every window:** sort `p` once; for each window sort and compare → O(n · k log k). Simple, slow.
- **Version 1 (`==` on 26-array):** clearest to explain; present first, then offer a Version 2 / canonical O(1)-per-step refinement.

## Refinements still applicable
- **Signed/unsigned warning:** `int i = 0; i < p.size()` and `right < s.size()` compare `int` with `size_t`. Cast `(int)p.size()` / `(int)s.size()` or use `size_t`.
- **Avoid input copies:** `string s, string p` copies both; `const string& s, const string& p` does not. LeetCode accepts the const-ref signature.
- **`left` is redundant:** it is always `right - p.size()` (fixed window) — can index `s[right - p.size()]` directly and drop the variable.

## Pitfalls
- **Fixed, not variable, window:** width is exactly `p.length`. Don't grow/shrink by validity — slide one in, one out.
- Check `p.size() > s.size()` up front, else the initial-window build reads out of bounds.
- `s.size()` is unsigned — cast to `int` before comparing in the `for`.
- **Keep going after a match** — vs. LC 567 the point is collecting *all* indices, not the first.
- Push the **left index** (`left` after `left++`, i.e. `right - p.size() + 1`), not `right`.
- In Version 2 use **pre**-`++`/`--` so the after-check sees the post-state; mixing in post-increment silently miscounts `match`.
- Contrast with the *variable*-window longest problems (3, 4): different template, don't mix them.
