# 5. Permutation in String (LC 567) — Medium
🔗 [Open on LeetCode](https://leetcode.com/problems/permutation-in-string/)

**Pattern:** Sliding Window — fixed size (`width = s1.length`) + frequency match
**Trigger:** "does s2 contain a permutation / anagram of s1" → fixed-width window, compare letter frequencies

## Problem (1-liner)
Given `s1` and `s2` (lowercase), return `true` if **some substring of `s2` of length `s1.length` is a permutation of `s1`**.
**Example:** `s1 = "ab", s2 = "eidbaooo"` → `true` (window `"ba"`). `s1 = "ab", s2 = "eidboaoo"` → `false`.

## Idea (1-liner)
A permutation = same multiset of letters. Slide a window of **exactly `s1.length`** over `s2`; the answer is `true` iff some window's letter-frequency matches `s1`'s.

## Approach
1. If `s1.size() > s2.size()` → `false` (no window can fit).
2. Build `freqS1[26]` from `s1`, and `freqS2[26]` from the **first `s1.size()` chars** of `s2`.
3. If `freqS1 == freqS2` → `true`.
4. Slide: for `right = s1.size() .. s2.size()-1`, add `s2[right]`, drop `s2[left++]`, compare again.
5. Return `false` if no window matched.

## Code (clean canonical)
```cpp
bool checkInclusion(const string& s1, const string& s2) {
    if (s1.size() > s2.size()) return false;

    int freq[26] = {0};
    for (char c : s1) freq[c - 'a']++;

    int need = s1.size(), left = 0, found = 0;
    for (int right = 0; right < (int)s2.size(); right++) {
        if (freq[s2[right] - 'a']-- > 0) found++;     // consumed a still-needed char
        if (right - left + 1 > need) {
            if (freq[s2[left] - 'a']++ >= 0) found--;  // released a char that was counted
            left++;
        }
        if (found == need) return true;
    }
    return false;
}
```
**TC:** O(n)  **SC:** O(1) — single fixed 26-slot array

## My Solution
**Version 1 — two frequency arrays, build-then-slide (O(26·n)):**
```cpp
class Solution {
public:
    bool checkInclusion(string s1, string s2) {
      if(s1.size() > s2.size()) return false;
      vector<int>freqS1(26, 0), freqS2(26, 0);
      // Build s1 & s2 intial frequency
      for(int i = 0; i < s1.size(); i++) {
        freqS1[s1[i] - 'a']++;
        freqS2[s2[i] - 'a']++;
      }
      if(freqS1 == freqS2) return true;
      // Check for remaining part
      int left = 0;
      for(int right = s1.size(); right < s2.size(); right++) {
        freqS2[s2[right] - 'a']++;
        freqS2[s2[left] - 'a']--;
        left++;
        if(freqS1 == freqS2) return true;
      }
      return false;
    }
};
```

**Version 2 — single array + `found` counter, O(1) per slide:**
```cpp
class Solution {
public:
    bool checkInclusion(string s1, string s2) {
      if(s1.size() > s2.size()) return false;
      int freq[26] = {0};
      // Building frequency with s1
      for(int i = 0; i < s1.size(); i++) {
        freq[s1[i] - 'a']++;
      }
      int need = s1.size(), left = 0, found = 0;
      for(int right = 0; right < s2.size(); right++) {
        // if before decrement > 0 then found one required char
        if(freq[s2[right] - 'a']-- > 0) found++;
        // if window out of s1.size then
        if(right - left + 1 > need) {
          // if before increment it >= 0 then we leave one reqire char
          if(freq[s2[left] - 'a']++ >= 0) found--;
          left++;
        }
        if(need == found) return true;
      }
      return false;
    }
};
```

## Key Insight — what `freq[c]` means in Version 2
After building from `s1`, `freq[c]` = **how many more `c` the window still needs**:
- `freq[c] > 0` → still need `c`
- `freq[c] == 0` → window has exactly enough `c`
- `freq[c] < 0` → window has a **surplus** of `c` (more than needed, or `c` not in `s1`)

`found` = number of "needed slots" currently filled; answer found when `found == need`.

## Key Insight — why `> 0` on add but `>= 0` on remove (the operators are NOT symmetric)
- **Add** (`freq[s2[right]-'a']-- > 0`): post-decrement tests the value **before** the decrement. Adding `c` is progress only if `c` was *still needed* → `freq > 0`. Using `>= 0` here would wrongly count every char **not in `s1`** (their `freq` starts at `0`) — the algorithm would return `true` almost immediately. So strict `> 0` is required.
- **Remove** (`freq[s2[left]-'a']++ >= 0`): we lose a filled slot only if the char being released **was actually counted**, i.e. the window was at-or-below the requirement → `freq >= 0` before the increment. If `freq < 0` we're only shedding surplus, so `found` stays.
- **Symmetry that explains it:** add un-fills on crossing `freq: 1 → 0`; remove un-fills on crossing `freq: 0 → 1`. Both are the *same boundary at 0*, approached from opposite directions — hence one `>` and one `>=`. Swapping them is a silent bug even though some test sets may not catch it.

## Dry run — `s1="ab"`, `s2="eidboaoo"`, `need=2`
Start `freq[a]=1, freq[b]=1` (rest 0), `left=0, found=0`.

| right | char | freq before-- | `>0`? | found | shrink? (left char, freq before++, `>=0`?) | found | left |
|------|------|------|------|------|------|------|------|
| 0 | e | 0 | no | 0 | size 1, no | 0 | 0 |
| 1 | i | 0 | no | 0 | size 2, no | 0 | 0 |
| 2 | d | 0 | no | 0 | remove `e`, −1, no | 0 | 1 |
| 3 | b | 1 | **yes** | 1 | remove `i`, −1, no | 1 | 2 |
| 4 | o | 0 | no | 1 | remove `d`, −1, no | 1 | 3 |
| 5 | a | 1 | **yes** | 2 | remove `b`, 0, **yes** → −1 | 1 | 4 |
| 6 | o | −1 | no | 1 | remove `o`, −2, no | 1 | 5 |
| 7 | o | −1 | no | 1 | remove `a`, 0, **yes** → −1 | 0 | 6 |

`found` hits 2 momentarily at `right=5` but the window is `"boa"` (size 3) at that moment; the shrink immediately removes `b` and `found` drops to 1. Result: **`false`** ✅. The `found == need` check is meaningful only *after* the window is trimmed back to exactly `need`, which is why it sits at the end of the loop body.

## Other Approaches
- **Sort every window:** sort `s1` once; for each window sort and compare → O(n · k log k). Simple but slow.
- **Two frequency arrays + `==`** (Version 1): rebuild/slide a 26-array and compare with `==` each step → O(26·n) ≈ O(n). Clearest to explain in an interview; state this first, then offer Version 2 as the O(1)-per-step refinement.

## Pitfalls
- **Fixed, not variable, window:** width is exactly `s1.length`. Don't grow/shrink by validity — slide one in, one out.
- Check `s1.size() > s2.size()` up front, else the initial-window build reads out of bounds.
- `s2.size()` is unsigned — cast to `int` before comparing in the `for` (`right < (int)s2.size()`).
- Operator direction in the `found` trick: add is `> 0`, remove is `>= 0` — see insight above; do not "tidy" them to match.
- Evaluate `found == need` **after** the shrink, not before — a too-wide window can momentarily reach `need`.
- Contrast with [4. Longest Repeating Character Replacement](4.%20Longest%20Substring%20without%20Repeating%20char.md): there the window is *variable* and shrinks with a single `if`; here it is *fixed* — different template, don't mix them.
