# 12. Sliding Window Maximum (LC 239) — Hard
🔗 [Open on LeetCode](https://leetcode.com/problems/sliding-window-maximum/)

**Pattern:** Sliding Window — **fixed window + monotonic deque** (deque holds *indices*, values strictly/weakly decreasing front→back; front = current window max)
**Trigger:** "**max (or min) of every fixed-size-`k` window** as it slides" → can't re-scan each window (O(n·k)); maintain a **decreasing deque of indices** so the answer is always `dq.front()` in O(1) amortized.

## Problem (1-liner)
Given `nums` and a window size `k` sliding left→right one step at a time, return the **maximum of each window**.
**Example:** `nums = [1,3,-1,-3,5,3,6,7], k = 3` → `[3,3,5,5,6,7]`. `nums=[1], k=1` → `[1]`.

## Idea (1-liner)
Brute force takes the max of every length-`k` window → O(n·k). The key observation: if `nums[i] <= nums[j]` and `i < j`, then `i` is **useless forever** — any window containing `i` also contains the larger/later `j`. So keep a deque of indices whose values are **monotonically decreasing**; the front is always the window's max. Each index is pushed and popped at most once → **O(n)**.

## Approach
1. Iterate `i` over `nums`, deque stores **indices** (not values).
2. **Expire front:** if `dq.front()` has fallen out of the window (`<= i - k`), `pop_front()`.
3. **Maintain monotonicity:** while `nums[dq.back()] <= nums[i]`, `pop_back()` — those indices can never be a future max.
4. `push_back(i)`.
5. Once the first full window is formed (`i >= k - 1`), append `nums[dq.front()]` to the answer.

## Code (clean canonical — monotonic deque, O(n) / O(k))
```cpp
class Solution {
public:
    vector<int> maxSlidingWindow(vector<int>& nums, int k) {
        vector<int> ans;
        deque<int> dq;                              // indices, nums[] decreasing front→back
        for (int i = 0; i < (int)nums.size(); i++) {
            // 1. drop the front if it slid out of the window
            if (!dq.empty() && dq.front() <= i - k) dq.pop_front();
            // 2. drop smaller/equal tail values — they can never be a future max
            while (!dq.empty() && nums[dq.back()] <= nums[i]) dq.pop_back();
            // 3. add current index
            dq.push_back(i);
            // 4. first window complete → front is its max
            if (i >= k - 1) ans.push_back(nums[dq.front()]);
        }
        return ans;
    }
};
```
**TC:** O(n) — every index is pushed once and popped at most once (amortized O(1)/step). **SC:** O(k) — the deque never holds more than one full window's worth of indices.

## My Solution
```cpp
class Solution {
public:
    vector<int> maxSlidingWindow(vector<int>& nums, int k) {
      vector<int>ans;
      deque<int>dq;
      for(int i = 0; i < nums.size(); i++) {
        // Remove the element index which are out of window
        if(!dq.empty() && dq.front() <= (i - k)) dq.pop_front();
        // Pop back the index which are smaller than current element index
        while(!dq.empty() && nums[dq.back()] <= nums[i]) dq.pop_back();
        // Store current element to the deque
        dq.push_back(i);
        // Store window maximum Element
        if(i >= k - 1) ans.push_back(nums[dq.front()]);
      }
      return ans;
    }
};
```
✅ Correct, accepted, **O(n) / O(k)** — this *is* the canonical answer. One nit: `i < nums.size()` compares `int` with `size_t`; with `i - k` the subtraction is fine here because `dq.front()` is `int`, but caching `int n = nums.size()` removes the unsigned-comparison smell.

## Key Insight — store indices, keep values decreasing
The deque is **not** a window of values — it's a shortlist of *candidate maxima*, by index, in decreasing value order:

| | Brute force | Monotonic deque |
|---|---|---|
| Per-window max | re-scan `k` elements | read `dq.front()` in O(1) |
| Why valid | — | a smaller earlier element is dominated by a larger later one *for every window they share* |
| Index vs value | — | store **indices** so step 2 can detect "out of window" by position |
| Total | O(n·k) | O(n) |

Two reasons the deque holds **indices, not values**: (a) the front-expiry test `dq.front() <= i - k` needs position, and (b) duplicates with `<=` in step 3 still get evicted, so the front is always the *most recent* maximal index — correct even with repeated values.

## Other Approaches
- **Max-heap of `(value, index)`** — push every element, pop the top while its index is out of window. Simple, but **O(n log n)** and O(n) space (lazy deletion).
- **Block / sparse-table (DP) — left & right max scans** — precompute prefix-max within each `k`-block and suffix-max; each window's max = `max(suffix[l], prefix[r])`. O(n) time, O(n) space, no deque.
- **`multiset`** — insert `nums[i]`, erase `nums[i-k]`, read `*rbegin()`. O(n log k); concise but slower than the deque.
- **Brute force** — max of every window, O(n·k); only viable for tiny `k`.

## Refinements still applicable
- **`int n = nums.size()`** — avoids the signed/unsigned comparison and the `(i - k)` mixed-type subtraction.
- **`ans.reserve(nums.size() - k + 1)`** — exact output size is known; skip reallocations.
- **`< ` vs `<=` in step 3** — both are correct; `<=` keeps the deque shorter (evicts equal values). Don't switch to `<` thinking it's "safer" — it just wastes space.

## Pitfalls
- **Deque stores indices, not values** — you need the index for the window-expiry check (step 2). Storing values loses position and breaks correctness.
- **Expire the front by *index*, not value** — `dq.front() <= i - k` (the index left the window), not "front value is small".
- **Emit only after the first full window** — guard with `if (i >= k - 1)`; emitting earlier produces maxes of partial windows.
- **`while` (not `if`) on the back-pop** — multiple stale tail indices may need eviction in one step.
- **`<=` (not `<`) when popping the back** keeps a single representative for duplicate values and ensures the front index is the freshest max; with `<` the deque grows with equal values (still correct, just wasteful).
- **This is the *fixed-window* family** (like [1. Maximum Average Subarray I](1.%20Maximum%20Average%20Subarray%20I.md) / [9. Maximum Number of Vowels](9.%20Maximum%20Number%20of%20Vowels.md)) but the per-window aggregate (max) isn't decomposable like a sum — hence the deque instead of an `+in / -out` running value.
