# 4. Longest Repeating Character Replacement (LC 424) — Medium
🔗 [Open on LeetCode](https://leetcode.com/problems/longest-repeating-character-replacement/)

**Pattern:** Sliding Window — variable size (grow `right`, shrink `left` while *invalid*) + frequency map
**Trigger:** "longest substring of one letter with ≤ k replacements" → window is valid while `windowLen - maxFreq <= k`

## Problem (1-liner)
Given a string `s` (uppercase letters) and an integer `k`, you may replace at most `k` characters. Return the **length of the longest substring** that becomes one repeated letter.
**Example:** `s = "AABABBA", k = 1` → `4` (replace middle `A`/`B` to get `"AABBBB"` or `"BBBB"` window).

## Idea (1-liner)
For any window, the cheapest way to make it uniform is to keep the **most frequent character** and replace the rest. So the window is valid iff `windowLen - maxFreq <= k`. Grow `right`; if invalid, slide `left` forward; track the largest window seen.

## Approach
1. `left = 0`, `maxLen = 0`, `maxFreq = 0`, `hash[26] = {0}`.
2. For `right = 0 .. n-1`:
   - `hash[s[right] - 'A']++`.
   - `maxFreq = max(maxFreq, hash[s[right] - 'A'])`.
   - If `(right - left + 1) - maxFreq > k` (invalid): `hash[s[left++] - 'A']--`.
   - `maxLen = max(maxLen, right - left + 1)`.
3. Return `maxLen`.

## Code (clean canonical)
```cpp
int characterReplacement(string s, int k) {
    int n = s.size(), left = 0, maxLen = 0, maxFreq = 0;
    int hash[26] = {0};
    for (int right = 0; right < n; right++) {
        hash[s[right] - 'A']++;
        maxFreq = max(maxFreq, hash[s[right] - 'A']);
        if ((right - left + 1) - maxFreq > k) {
            hash[s[left] - 'A']--;
            left++;
        }
        maxLen = max(maxLen, right - left + 1);
    }
    return maxLen;
}
```
**TC:** O(n)  **SC:** O(1) — fixed 26-slot array

## My Solution
```cpp
class Solution {
public:
    int characterReplacement(string s, int k) {
        int left = 0, right = 0, size = s.size(), maxFreq = 0, maxLen = 0;
        int hash[26] = {0};

        while(right < size) {
            hash[s[right] - 'A']++;
            maxFreq = max(maxFreq, hash[s[right] - 'A']);

            if((right - left + 1) - maxFreq > k) {
                hash[s[left] - 'A']--;
                left++;
            }
            maxLen = max(maxLen, right - left + 1);
            right++;
        }
        return maxLen;
    }
};
```

## Key Insight — why `if` is enough (not `while`)
Most sliding-window templates use `while` to keep shrinking until the window is valid again. Here we use a single `if` and that is intentional:
- The window grows by exactly **one** character per iteration of `right`. So when it becomes invalid, it is invalid by at most 1 — sliding `left` forward by 1 is enough to restore the size balance.
- Even if we *did* shrink more than once, the window would only get **smaller**, and a smaller window can never improve `maxLen` (we want the **maximum** length). So extra shrinking would do no useful work — it would just slide the window forward without ever beating the current best.
- In effect, the window never actually shrinks below its largest valid size: it only **slides forward** by one when invalid, preserving the best length seen so far so that future expansions can try to beat it.

## Key Insight — why we don't decrement `maxFreq` on shrink
- `maxFreq` may become **stale** (higher than the true max in the current window) after `left` moves forward. That's fine.
- A stale `maxFreq` only makes the validity check `windowLen - maxFreq > k` **more lenient**, never stricter.
- `maxLen` only updates when we find a window **strictly larger** than the current best — and beating the best requires a genuinely larger `maxFreq`, which can only happen when `right` legitimately bumps it up.
- So skipping the recompute saves an O(26) scan per shrink without ever producing a wrong answer.

## Other Approaches
- **Brute force:** for every `(i, j)`, count the most frequent char in `s[i..j]` and check `(j-i+1) - maxFreq <= k` → O(n² · 26).
- **`while`-shrink variant:** keeps the window strictly valid by shrinking until `windowLen - maxFreq <= k`. Also correct, but does extra work — the `if` version produces the same `maxLen` cheaper because of the insight above.

## Pitfalls
- **`maxFreq` update direction**: `maxFreq = max(maxFreq, hash[...])`, not `if (maxFreq > hash[...]) maxFreq = hash[...]` (that sets it to the *smaller* value — silent bug).
- Don't try to "fix" `maxFreq` on shrink by scanning the map — unnecessary, see insight above.
- `if` here works because window grows by 1 per step; in problems where the window can break by > 1 in a single step (e.g. `567 Permutation in String` distinct-count condition), use `while`. Don't carry the `if` habit blindly.
- Constraint says uppercase letters only → `hash[26]`; for general ASCII use `hash[128]`.
- Contrast with [3. Longest Substring Without Repeating Characters](3.%20Longest%20Substring.md): there shrinking *must* be `while` because adding one char can require popping several to clear the duplicate.
